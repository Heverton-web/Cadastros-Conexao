# CONTEXTO E OBJETIVO

O objetivo é melhorar e "otimizar" o banco de dados e os respectivos modais de interface (UI) para coleta/cadastro desses dados, seguindo as definições técnicas e as regras de negócio estabelecidas abaixo.

# REGRAS GERAIS DE NEGÓCIO E ARQUITETURA

1. **Auditoria:** Todas as tabelas devem conter automaticamente os campos `created_at` e `updated_at`.

2. **Contexto de Empresa (`empresa_id`):** O campo `empresa_id` NUNCA deve ser exibido como um input para o usuário no modal. Ele deve ser preenchido/injetado automaticamente pelo backend usando o contexto do usuário logado (sessão/auth). Este capo só será exibido a usuário do tipo SUPER ADMIN.

3. **Selects em Cascata:** Sempre que houver dependência hierárquica (ex: Categoria -> Conexão -> Família -> Linha), o modal deve implementar Dropdowns (Selects) dependentes. Escolher o item pai filtra automaticamente os itens filhos correspondentes e desabilita os selects abaixo enquanto o pai não for selecionado.

4. **Relacionamentos em Lista (N:M):** Para campos definidos como "lista de IDs" ou associações múltiplas (ex: chaves, fresas, imagens, instrumentais em kits), implemente usando [ESCOLHA: Tabelas Pivô (Join Tables) relacionais no banco OR colunas de Array (`uuid[]`)]. *Nota: Para listas que exigem metadados de ordenação, como a ordem das fresas no protocolo, utilize obrigatoriamente uma Tabela Pivô contendo a coluna `ordem` (ex: `protocolo_id`, `fresa_id`, `ordem`).*

5. **Visibilidade e Renderização Condicional:** Caso um campo da tabela esteja vazio ou seja do tipo `null`, o elemento/componente UI não será renderizado no frontend ao visualizar ou listar os dados.
