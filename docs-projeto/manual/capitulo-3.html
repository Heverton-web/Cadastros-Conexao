Multiempresa & Isolamento de Dados

Arquitetura multi-tenant, isolamento por empresa, RLS e escopos de dados.

Arquitetura Multi-tenant

O ERP Conexão foi projetado como uma plataforma SaaS multiempresa. Cada empresa cliente opera de forma completamente isolada dentro da mesma infraestrutura, com seu próprio conjunto de dados, configurações e módulos ativos.

O identificador central de tenant é o campo empresa_id, presente em todas as tabelas de dados operacionais. Quando um usuário faz login, seu perfil está vinculado a uma empresa específica, e todas as queries são filtradas automaticamente por esse identificador.

Estrutura de Dados do Tenant

// Tabela: profiles id UUID (referência auth.users) email TEXT nome TEXT role TEXT -- papel do usuário ambiente TEXT -- cadastro | consultor | tecnologia | suporte is_super_admin BOOLEAN empresa_id UUID -- vínculo com a empresa

Níveis de Isolamento

1. Autenticação

O login é feito via Supabase Auth. Cada usuário pertence a uma única empresa e possui um perfil com papel e ambiente definidos. O Super Admin transcende empresas e tem acesso global.

2. Políticas de Segurança em Linha (RLS)

O Supabase oferece Row-Level Security que garante que um usuário veja apenas os dados da sua empresa. Exemplo de política:

-- Política RLS típica CREATE POLICY "usuarios_veem_apenas_sua_empresa" ON clientes USING (empresa_id = auth.jwt() ->> 'empresa_id');

3. Módulos Ativos por Empresa

Cada empresa possui uma lista de módulos ativos, gerenciada pelo Super Admin. Se um módulo não está ativo para a empresa, ele simplesmente não aparece na interface e suas rotas ficam inacessíveis.

// AuthGuard verifica módulos ativos modulosAtivos = await getModulosAtivos(empresa.id); // Só módulos nesta lista são carregados

4. Escopo de Dados por Permissão

Além do isolamento por empresa, existe um escopo de dados adicional controlado por permissão. Por exemplo, a permissão ver_todos_cadastros determina se o usuário pode ver cadastros de toda a empresa ou apenas os seus próprios.

5. Ambientes

O sistema suporta 5 ambientes que funcionam como camada adicional de isolamento funcional:

  • cadastro — Operações de cadastro de profissionais
  • consultor — Ferramentas do consultor de campo
  • tecnologia — Configurações técnicas
  • suporte — Acesso de suporte
  • ambos — Módulos transversais disponíveis em múltiplos ambientes

Fluxo de Login e Carregamento

  1. Usuário acessa / e faz login com email/senha
  2. Supabase Auth valida as credenciais e retorna o JWT
  3. AuthProvider carrega profile da tabela profiles
  4. Carrega empresa (dados da organização vinculada)
  5. Carrega modulosAtivos (módulos habilitados para a empresa)
  6. Carrega permissoes (flat Record<string, boolean>) e modulosAcesso
  7. Se is_super_admin, ganha todas as permissões automaticamente
  8. AuthGuard monta o AppLayout com navegação filtrada
Importante: O isolamento de dados é garantido em TRÊS camadas: (1) autenticação define quem é o usuário, (2) RLS no banco filtra linhas por empresa_id, e (3) o frontend só exibe módulos e dados que o usuário tem permissão de ver.
