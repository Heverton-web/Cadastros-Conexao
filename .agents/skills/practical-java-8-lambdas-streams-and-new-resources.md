  - page_011 Capítulo 8 : More operations with Streams Working with streams will be part of your daily life, so getting familiar with its API through practical examples is an excellent way to learn other concepts it involves. 8.1 Sorting a Stream Given a List&lt;User&gt; users , we know how to sort it by name: 1 users . sort ( Comparator . comparing ( User :: getName )); And a stream? Imagine that we want to filter the users with scores higher than 100 and then sort them: 1 users . stream () 2 . filter ( u -&gt; u . getReputationScore () &gt; 100 ) 3 . sorted ( Comparator . comparing ( User :: getName )); In a stream, the sorting method is sorted . You can already guess what the difference is between sorting a list with sort and a stream with sorted : a method invoked in a Stream doesn’t change who generated it. In this case, it doesn’t change the List&lt;User&gt; users . If we want the result in a List , we need to use a Collector as previously seen: 1 List &lt; User &gt; filteredAndSorted = users . stream () 2 . filter ( u -&gt; u . getReputationScore () &gt; 100 ) 3 . sorted ( Comparator . comparing ( User :: getName )) 4 . collect ( Collectors . toList ()); The code is short and easy to read. You might not remember how many more lines you would need to write it in Java 7. Many lines would mean a loss in clarity and simplicity: 1 List &lt; User &gt; filteredUsers = new ArrayList &lt;&gt;(); 2 for ( User user : users ) { 3 if ( user . getReputationScore () &gt; 100 ) { 4 filteredUsers . add ( user ); 5 } 6 } 7 8 Collections . sort ( filteredUsers , new Comparator &lt; User &gt;() { 9 public int compare ( User u1 , User u2 ) { 10 return u1 . getName (). compareTo ( u2 . getName ()); 11 } 12 }); It would require a temporary list for filtering, a loop for the same filter, an anonymous class for Comparator , and finally, an invocation for the sorting. 8.2 Many operations in Stream are lazy! When manipulating a Stream , we usually chain many computational operations. This set of operations in a Stream is known as a pipeline . Java can use this structure to optimize the operations. How does it do this? By avoiding the execution of the operations as much as possible. Most of them are lazy and are only really run when they are needed to get the final result. You want an example? Think in the previous code, before collect : 1 users . stream () 2 . filter ( u -&gt; u . getReputationScore () &gt; 100 ) 3 . sorted ( Comparator . comparing ( User :: getName )); The filter and sorted methods return a Stream . During the invocations, these methods don’t filter or sort; they just return new streams in which this information is checked. These methods are called intermediate operations . The new streams returned know they must be filtered and sorted (or something equivalent) at the time when a terminal operation is invoked. Collect is an example of terminal operation, and it’s only then that the stream will start running the requested operations pipeline . 8.3 what is the advantage of lazy methods? Suppose we want to find a user with a score higher than 100. We only need one as long as he meets the score predicated. We can filter the stream, collect it in a list, and take the first element: 1 User over100 = users . stream () 2 . filter ( u -&gt; u . getReputationScore () &gt; 100 ) 3 . collect ( Collectors . toList ()) 4 . get ( 0 ); It’s a lot of work for something so simple. Here we filter all users and create a new collection with them just to get the first element. Moreover, if there is no user with a score higher than 100, we will receive an exception. Stream has the findAny method that returns any of the elements: 1 Optional &lt; User &gt; optionalUser = users . stream () 2 . filter ( u -&gt; u . getReputationScore () &gt; 100 ) 3 . findAny (); There are two advantages here in working with Stream . The first is that findAny returns an Optional&lt;User&gt; and forces us to make a get in order to receive the User or use the test methods (like orElse and isPresent ) to know if there really is a filtered user there. The second advantage is that, since all the work is lazy, the stream isn’t completely filtered . That’s right. FindAny is a terminal operation, and it forces the execution of the operations pipeline . At the same time, this method is written in a smart manner, analyzing the previously invoked operations and realizing that there is no need for filtering all the elements in the list, only to take the ones that meet the given predicate. It starts to run the filter, and, as soon as it finds a user with a score higher than 100, it returns it and stops the filtering. Another method that could be useful is findFirst , which is similar to findAny but uses the elements in the order followed by the stream. This optimization technique can’t always be applied -- it depends on the pipeline operations. Therefore, you shouldn’t depend on the technique to write the code as it could actually turn out to be inefficient. How to see if the optimization really happens? You could put a breakpoint in the getReputationScore method or print a log information when it’s invoked. By doing so, you would see that it’s invoked only the number of times needed until it finds a User with a score higher than User ! Still, there is a more interesting way of checking that: using the intermediate method peek . 8.4 Seeing at the pipeline executing with peek We can ask the stream to run a task every time you process an element. We do this with peek : 1 users . stream () 2 . filter ( u -&gt; u . getReputationScore () &gt; 100 ) 3 . peek ( System . out :: println ) 4 . findAny (); We can see that the elements will be shown only until an element that meets the u.getReputationScore() &gt; 100 predicate is found. The power the laziness has in the Stream ’s API is now clear. Unlike forEach , which returns a void and is a terminal operation, peek returns a new Stream and is an intermediate operation. It won’t force the pipeline ’s execution. The following code simply doesn’t print anything: 1 users . stream () 2 . filter ( u -&gt; u . getReputationScore () &gt; 100 ) 3 . peek ( System . out :: println ); Why? Because peek returns a new Stream where it’s scheduled to print all the processed elements. It will only handle elements when it finds a terminal operation like findAny , collect , or forEach . With peek , we can see if the other operations can profit from laziness. Try to do the same thing with sorted : 1 users . stream () 2 . sorted ( Comparator . comparing ( User :: getName )) 3 . peek ( System . out :: println ) 4 . findAny (); Here, peek prints all the users even if we only want to do findAny . We said earlier that sorted is an intermediate stateful method. Stateful operations can need to process the entire stream even if its terminal operations don’t require it. 8.5 Reduction operations The operations that use the stream elements to return a final value are frequently called reduction operations. An example of this is average , which we already saw. 1 double avarageScore = users . stream () 2 . mapToInt ( User :: getReputationScore ) 3 . average () 4 . getAsDouble (); There are other useful methods similar to average : count , min , max , and sum . The latter, like average , is only found in primitive streams. Min and max ask for a Comparator as argument. All but sum and count work with Optional . For example: 1 Optional &lt; User &gt; max = users . stream () 2 . max ( Comparator . comparing ( User :: getReputationScore )); 3 4 User topScore = max . get (); If we want to add up all the users’ scores, we do the following: 1 int sum = users . stream () 2 . mapToInt ( User :: getReputationScore ) 3 . sum (); This sum is executed through a reduction operation that can be very explicit. How does it work? It takes the first element, the first user’s score in the stream, and stores the value accumulated so far with a sum operation. We also need to have an initial value in case the sum is equal to zero. We can break down this sum operation to understand better what a reduction operation is. Note the definitions: 1 int initialValue = 0 ; 2 IntBinaryOperator operator = ( a , b ) -&gt; a + b ; IntBinaryOperator is a Functional Interface that defines the applyAsInt method. It takes two integers and returns one. That’s what a sum is, right? Now that we know these definitions, we can ask for the stream to process the reduction step by step: 1 int initialValue = 0 ; 2 IntBinaryOperator operator = ( a , b ) -&gt; a + b ; 3 4 int sum = users . stream () 5 . mapToInt ( User :: getReputationScore ) 6 . reduce ( initialValue , operator ); Done! We have a code equivalent to the sum . We could have written everything more succinctly, without declaring local variables: 1 int sum = users . stream () 2 . mapToInt ( User :: getReputationScore ) 3 . reduce ( 0 , ( a , b ) -&gt; a + b ); We can go further. In the Integer class, there is now the static Integer.sum method, which sums two integers. Instead of using Lambda, we can opt for a method reference : 1 int sum = users . stream () 2 . mapToInt ( User :: getReputationScore ) 3 . reduce ( 0 , Integer :: sum ); Why should we use a reduction instead of sum ? There is no advantage in using a reduction. What is important is that you can learn it so that you are able to make operations not found in Stream . For example? Multiplying the scores. 1 int multiplication = users . stream () 2 . mapToInt ( User :: getReputationScore ) 3 . reduce ( 1 , ( a , b ) -&gt; a * b ); There are also some special cases where invoking map could be costly and the best way would be to do the sum operation directly. This is not the case here, but just to give you an example, the sum without the map would look like this: 1 int sum = users . stream () 2 . reduce ( 0 , ( atual , u ) -&gt; atual + u . getReputationScore (), 3 Integer :: sum ); This reduce overload has another Lambda, which in our case is Integer::sum . This other Lambda serves to match the partial reduction values in the case of parallel streams. There’s no need for always wanting to avoid the number of operations in a stream. We can take advantage of this approach in isolated cases. In other languages, this operation is also known as fold . We will see in the next chapter that these operations are parallelizable and take great advantage of parallel streams. We will also see that collect is a reduction case. 8.6 Learning about other Stream methods Stream has many features. This book doesn’t intend to cover all the API’s methods, but we can quickly go over some of the most interesting ones (besides the ones already seen). In addition, we will learn more about the proper use of streams. Working with iterators We saw that we can collect the result of a Stream ’s operation pipeline in a collection like collect . Sometimes we don’t even need a collection; we can only iterate over the Stream ’s elements. What would happen if we tried to go through the Stream the old way? 1 for ( User u : users . stream ()){ 2 //... 3 } A compilation error occurs. Enhanced for waits for an array or an Iterable . But why didn’t they decide to make the Stream an Iterable ? Because the Stream ’s various terminal operations mark it as already used. Its second invocation throws an IllegalStateException . Remember that there’s no problem if you invoke users.stream() twice, because we open different streams at every invocation! However, we can iterate through a Stream ’s elements with an Iterator . To do this, we can invoke the iterator method: 1 Iterator &lt; User &gt; i = users . stream (). iterator (); The Iterator interface has been in Java for quite some time and defines the hasNext , next , and remove methods. With Java 8, we can also go through an iterator using the forEachRemaining method, which receives a Consumer as a parameter: 1 users . stream (). iterator () 2 . forEachRemaining ( System . out :: println ); But when should we use a Stream ’s iterator if there is a forEach both in the Collection and in the Stream ? One reason to use an Iterator is when we want to modify the objects of a Stream . When using parallel streams, we will see that we shouldn’t change the state of its objects, or we risk having non-deterministic results. Another reason is the APIs’ compatibility. It’s possible that you will need to invoke a method that receives an Iterator . Testando Predicates title Testing Predicates We repeatedly saw the use of filter . It receives a Lambda as argument, which comes from the Predicate interface. There are many situations where we will want to test the predicates without using the filtered list, for example, if we want to know if there’s an element in the users list that is a moderator: 1 boolean hasModerator = users . stream () 2 . anyMatch ( User :: isModerator ); Here User::isModerator does the same thing as u -&gt; u.isModerator() , generating a predicate that tests whether a user is a moderator or not and returns a boolean. This operation’s process will stop once the stream encounters any user who is a moderator. As for anyMatch , we can find out if all the users are moderators with allMatch or if none of them is with noneMatch . There are a lot of other methods and details! You can use the count to know how many elements there are in the Stream , skip to skip the n next elements, and limit to limit the number of elements. There are also ways you can create a Stream without any collection. At the Stream interface itself, there are the empty and of static methods. The first obviously creates an empty Stream , and the of depends on what you pass as an argument, e.g., Stream.of(user1, user2, user3) returns a Stream&lt;User&gt; . You can also append them to Stream.concat . We can also create Stream s outside the interface. For example, you can use the regex API to return a Stream&lt;String&gt; through Pattern.splitAsStream , or get a Stream from a file’s lines with Files.lines . If you are working directly with an array, you can use Arrays.stream . Always try to learn more about the API and investigate this interface whenever you get new ideas or think there’s something missing. It’s probable that the thing you’ve thought of already exists. Another important point worth mentioning is that some Streams can originate from IO capabilities, where it implements the AutoCloseable interface and has close . One example of this is using the new java.nio.file.Files methods included in Java 7 (with Java 8 updates!) to generate a Stream . In this case, it’s critical to handle the exceptions and finally , or use try with resources . In other situations, when we know we're dealing with streams generated by collections, this is not required. 8.7 Primitive and infinite streams As seen with comparingInt , which returns a Comparator without the need for unboxing, we’ve also seen mapToInt , which also returns an IntStream to avoid any unecessary operations. IntStream , LongStream and DoubleStream have special operations and are important. Even their iterator returns different Iterators . In the IntStream ’s case, it’s PrimitiveIterator. Of Int that implements Iterator&lt;Integer&gt; , but that, apart from a next that returns an Integer with the boxing, also has nextInt%%. There are still factory methods like IntStream.range(begin, end) . Infinite streams Another powerful Java 8 feature: through the Supplier factory interface, we can define an infinite Stream by simply saying what is the rule for creating objects belonging to this Stream . For example, if we want to create an “infinite” list of random numbers, we can do the following: 1 Random random = new Random(0); 2 Supplier&lt;Integer&gt; supplier = () -&gt; random.nextInt(); 3 Stream&lt;Integer&gt; stream = Stream.generate(supplier); The Stream created by generate is lazy . Certainly it won’t generate infinite random numbers. They will only be generated as needed. Here we are causing boxing all the time. We can use IntSupplier and IntStream . Furthermore, we will remove the temporary supplier variable: 1 Random random = new Random(0); 2 IntStream stream = IntStream.generate(() -&gt; random.nextInt()); Now we need to be careful. Any operation that needs to go through all the Stream ’s elements will never finish running. For example: 1 int value = stream.sum(); You can only use short-circuit operations in infinite Streams . Short-circuit operations These are operations that don’t need to process all the elements. An example would be to take only the first 100 elements by using limit : 1 Random random = new Random(0); 2 IntStream stream = IntStream.generate(() -&gt; random.nextInt()); 3 List&lt;Integer&gt; list = stream 4 .limit(100) 5 .boxed() 6 .collect(Collectors.toList()); Notice the invocation of boxed . It returns a Stream&lt;Integer&gt; instead of IntStream , enabling the invocation of collect the way we've seen. Without this, we would only have the option to do IntStream.toArray or to call collect , which takes three arguments but would have no place to store the numbers. There is no IntList in Java that would be analogous to List&lt;Integer&gt; , and Java 8 doesn’t have the so-called value objects or value types to allow something like %List&lt;int&gt;%%. Let's review the same code with the fluent interface: 1 Random random = new Random(0); 2 List&lt;Integer&gt; list = IntStream 3 .generate(() -&gt; random.nextInt()) 4 .limit(100) 5 .boxed() 6 .collect(Collectors.toList()); The Supplier passed to generate can be used to generate an infinite Stream of constants, e.g., IntStream.generate(() -&gt; 1) and Stream.generate(() -&gt; new Object()) . It could be useful for a Supplier to maintain a state. In this case, we need to use a class or an anonymous class because within a Lambda we can’t declare attributes. We will generate the infinite sequence of Fibonacci numbers in a lazy way and print its first 10 elements: 1 class Fibonacci implements IntSupplier { 2 private int previous = 0; 3 private int next = 1; 4 5 public int getAsInt() { 6 next = next + previous; 7 previous = next - previous; 8 return previous; 9 } 10 } 11 12 IntStream.generate(new Fibonacci()) 13 .limit(10) 14 .forEach(System.out::println); We will see that maintaining the state in a Functional Interface can limit the parallelization resources a Stream provides. There are other short-circuit operations besides limit . FindFirst is one of them. But we don’t want to get the first Fibonacci element. We want to get the first element greater than 100! How do we do that? We can filter before invoking findFirst : 1 int over100 = IntStream 2 .generate(new Fibonacci()) 3 .filter(f -&gt; f &gt; 100) 4 .findFirst() 5 .getAsInt(); 6 7 System.out.println(over100); The filter isn’t short-circuit; it doesn’t produce a finite Stream given an infinite Stream . You only have to have a short-circuit operation in the pipeline (it doesn’t matter if it is the intermediate or final operation), and the pipeline’s execution may take an infinite amount of time. Why do I say ‘may?’ Because, for example, if there weren’t a Fibonacci element greater than 100, it would also run indefinitely. Matchers are also short-circuit. We can try to discover if all the Fibonacci elements are even with allMatch(f -&gt; f % 2 ==0) . If one of them is odd, it’ll be considered false. But if there were only even numbers, it would run indefinitely! Remember: working with infinite Streams can be dangerous, even if you use short-circuit operations. When we have to maintain the state of only one variable, we can use iterate instead of generate , which receives an UnaryOperator . To generate the natural numbers: 1 IntStream . iterate ( 0 , x -&gt; x + 1 ) 2 . limit ( 10 ) 3 . forEach ( System . out :: println ); There are also modifications in the old API to work with infinite Streams . You can see that the java.util.Random class returns infinite Stream s through methods like Random.ints() . 8.8 Practicing what we learn with java.nio.file.Files The java.nio.file.Files class in Java 7 facilitates files and folder handling via the Path interface. The three of them are Functional Interfaces. The first is a factory that creates the object to be returned at the end of the collection process. The second is the method that will be invoked to add each element. The third can be used if we need to add more than one element at a time (for example, if we use a strategy to collect parallel elements, as we will later see). The advantage is that there are several Collectors ready. We can greatly simplify our code, passing Collectors.toList() as a parameter in our collect method, one of the implementations of this new interface. 1 users . stream () 2 . filter ( u -&gt; u . getReputationScore () &gt; 100 ) 3 . collect ( Collectors . toList ()); Collectors.toList returns a collector that looks a lot like the one we created from scratch, with the difference being that this one returns a list, and we don’t know if this list is mutable, thread-safe, or its kind of implementation. There are some differences in the design that were carefully pointed out by Stephen Colebourne in his blog post entitled Why JSR-310 is not Joda-Time : http://blog.joda.org/2009/11/why-jsr-310-isn-joda-time_4941.html . The main changes were inspired by the Java-Time design flaws, such as the null references. In Joda-Time, not only is it possible to provide null values for most of its methods, but it also has a meaning for each model. For example, for the representation of date and time, the null value means 1970-01-01T00:00Z . For the Duration and Period classes, null means zero. Some operations in the pipeline are called non-deterministic. They may return different results when used in parallel streams. The main examples of that are forEach and findAny . When invoking these two methods in a parallel stream, the execution order can’t be guaranteed. This improves its performance in parallel. If you want to ensure the execution order, you should use forEachOrdered and findFirst , respectively, although they are usually not needed. Even methods like map can be executed in improved parallels. When we create a stream in a list (or in LongStream.range ), it’s created so that it is ordered . This means that when we collect the resulting list of a map of the function x -&gt; x + 1 on the elements 1, 2, 3 , at the end we will have 2, 3, 4 . If we generate a stream with the same content, but coming from a HashSet , the end result could be 3, 4, 2 . When running the map in parallel, the joining of results will be much simpler since it won’t require you to guarantee the same entry order! You can relax an ordered Stream ’s order restriction by invoking its unordered method. Another obstacle to the parallel Stream performance is the grouping collector. Collectors.groupingBy guarantees the elements’ order of appearance by grouping them, which can be costly in the joining process. Using Collectors.groupingByConcurrent doesn’t guarantee that order, since it uses a single map as a ConcurrentHashMap , but the final performance will be better. What if we had side effects? The parallel streams are incredibly powerful but certainly not magical; we still have problems when encountering operations with side effects in a shared state. Think of another way to sum from 1 to 1 billion. Instead of using the LongStream ’s sum , we can create a total attribute to store the result. With forEach , we sum this attribute in common, like this: 1 class UnsafeParallelStreamUsage { 2 private static long total = 0 ; 3 4 public static void main ( String ... args ) { 5 LongStream . range ( 0 , 1_000_000_000 ) 6 . parallel () 7 . filter ( x -&gt; x % 2 == 0 ) 8 . forEach ( n -&gt; total += n ); 9 10 System . out . println ( total ); 11 } 12 } If you have more than one core, you’ll probably get a different result each time you run this code! The concurrent use of a shared variable enables operation interleaving in an undesired way. Of course, you can use synchronized within the Lambda block to fix this, but you would also lose too much performance. The shared state between threads remains a problem. To know more: Spliterators The basis of all parallel streaming is the Spliterator . The Spliterator is like an Iterator with the difference being that often it can easily be broken in smaller spliterators so that each available thread consumes a piece of its stream. The Iterable also defines a spliterator() default method. Everything we saw about parallelization are abstractions using Spliterators behind the scenes along with the Fork/Join’s API. If you want to create a parallel complex operation, this is the path you must take.

  - page_007 Capítulo 4 : Default Methods 4.1 The forEach method in the Iterable interface Until now, we used the forEach method that we invoked in our List . Knowing that it didn’t exist yet in Java 7, where does it come from? One possibility would have been that the Java team had declared it in an interface, like in the List itself. What would be the problem? Everyone who created a class that implements a List would need to implement it. If that were the case, the amount of work would be huge, but that’s not the main point. After updating your Java, the libraries that have their own List , such as Hibernate, would have their implementation broken; methods would be missing, which could generate the scary NoSuchMethodError s. How can you add a method in an interface and ensure that all implementations have it implemented? You can do this with a new feature that declares the code within an interface’s method! For example, the forEach method we use is declared in java.lang.Iterable , which is the parent of Collection , which is in turn parent of List . Opening your source code, you can see: 1 default void forEach ( Consumer &lt;? super T &gt; action ) { 2 Objects . requireNonNull ( action ); 3 for ( T t : this ) { 4 action . accept ( t ); 5 } 6 } Yup! A method with its code inside interfaces! That’s something we never thought we’d see in Java. These methods are called default methods (in the beta documentation they are also called defender methods or extension methods ). As the ArrayList implements List , which is child (indirectly) of Iterable , the ArrayList has this method either way. That’s why we can do the following: 1 users . forEach ( u -&gt; System . out . println ( u . getName ())); This default method , the forEach , expects an implementation from the Consumer&lt;T&gt; Functional Interface, which is, as we saw earlier, one of the interfaces present in the new java.util.function package from Java 8. We’ll now see other methods that made it into the Collections. We can’t cover them all, but we'll use several new features, interfaces, and methods throughout this book. 4.4 Multiple Inheritance? Default methods were added to allow interfaces to evolve without breaking the existing code. This is one of the most repeated statements in the specification discussion. They weren’t created to allow variations of multiple inheritance or mixins. It is good to remember that these methods have a number of restrictions, notably the fact they can’t access instance attributes, because they don’t exist in interfaces! In other words, there is no multiple inheritance or state sharing. Brian Goetz and the other specification leaders have given strong arguments that are presented in Section 10 of this document: http://cr.openjdk.java.net/~briangoetz/lambda/lambda-state-final.html Admittedly, there is an increase in the coupling, but given this feature’s constraints in Java 8, the possibility of harmful uses (as in other languages) is lower. That is always a trade-off.
