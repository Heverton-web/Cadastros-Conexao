  - Uma característica única do Mesos, em comparação com outras soluções de orquestração, é o suporte a cargas de trabalho mistas. Vários frameworks podem ser executados no mesmo cluster, permitindo que tarefas de processamento de dados do Hadoop ou Storm sejam executadas juntamente com contêineres Docker que executam uma aplicação de microsserviços. Essa é uma das características que torna o Mesos particularmente útil para alcançar alta utilização; é possível agendar tarefas com alto consumo de CPU, mas baixa largura de banda, no mesmo host que tarefas com características opostas, para maximizar o uso de recursos. O uso eficiente de um cluster depende da solicitação precisa de recursos, em vez do provisionamento excessivo, ao submeter tarefas ao Marathon ou outros frameworks. No exemplo do identidock, os valores de memória foram intencionalmente inflados para garantir que nem todas as tarefas fossem alocadas ao mesmo agente, embora, na realidade, um único host pudesse lidar facilmente com a demanda. Para solucionar isso, o Mesos oferece suporte à sobreassinatura, que permite que tarefas "revogáveis" sejam iniciadas em um agente que não esteja oferecendo recursos suficientes, mas que, de acordo com o monitoramento, ainda tenha capacidade disponível. Essas tarefas revogáveis serão interrompidas se o uso de recursos aumentar repentinamente e, portanto, normalmente são tarefas de baixa prioridade, como a execução de análises em segundo plano.

  - Outros métodos seguem um princípio semelhante: as versões antiga e nova devem ser executadas em paralelo. Nos testes A/B, ou multivariados, duas (ou mais) versões de um serviço são executadas juntas durante um período de teste, com os usuários divididos aleatoriamente entre elas. Certas estatísticas são monitoradas e, com base nos resultados ao final do teste, uma das versões é mantida. Na implantação gradual, a nova versão de um serviço é disponibilizada apenas para um pequeno subconjunto de usuários. Se esses usuários não encontrarem problemas, a nova versão será progressivamente disponibilizada para mais e mais usuários. No shadowing, ambas as versões do serviço são executadas para todas as solicitações, mas apenas os resultados da versão antiga e estável são utilizados. Comparando os resultados da versão antiga com os da nova versão proposta, é possível garantir que a nova versão tenha um comportamento idêntico ao da versão antiga (ou que difira de uma maneira esperada e positiva). O shadowing é particularmente útil ao testar novas versões que não apresentam alterações funcionais, como melhorias de desempenho.

  - Contêineres são um conceito antigo. Por décadas, os sistemas UNIX contaram com o comando chroot, que fornece uma forma simples de isolamento do sistema de arquivos. Desde 1998, o FreeBSD possui o utilitário jail, que estendeu o isolamento do chroot para processos. O Solaris Zones ofereceu uma tecnologia de conteinerização comparativamente completa por volta de 2001, mas era limitado ao sistema operacional Solaris. Também em 2001, a Parallels Inc. (então SWsoft) lançou a tecnologia comercial de contêineres Virtuozzo para Linux e, posteriormente, tornou o núcleo da tecnologia de código aberto como OpenVZ em 2005.3 Em seguida, o Google iniciou o desenvolvimento do CGroups para o kernel do Linux e começou a migrar sua infraestrutura para contêineres. O projeto Linux Containers (LXC) teve início em 2008 e reuniu CGroups, namespaces do kernel e a tecnologia chroot (entre outros) para fornecer uma solução completa de conteinerização. Finalmente, em 2013, o Docker trouxe as peças finais para o quebra-cabeça da conteinerização, e a tecnologia começou a se popularizar.

  - A proveniência está longe de ser um problema novo na computação. A principal ferramenta para estabelecer a proveniência de software ou dados é o hash seguro. Um hash seguro é como uma impressão digital para dados — uma sequência (comparativamente) pequena e única para os dados em questão. Qualquer alteração nos dados resultará na mudança do hash. Existem diversos algoritmos disponíveis para calcular hashes seguros, com diferentes graus de complexidade e garantias de unicidade. Os algoritmos mais comuns são o SHA (que possui diversas variantes) e o MD5 (que apresenta problemas fundamentais e deve ser evitado). Se tivermos um hash seguro para alguns dados e os próprios dados, podemos recalcular o hash dos dados e compará-los. Se os hashes coincidirem, podemos ter certeza de que os dados não foram corrompidos ou adulterados. No entanto, uma questão permanece: por que deveríamos confiar no hash? O que impede um atacante de modificar tanto os dados quanto o hash? A melhor solução para isso é a assinatura criptográfica e pares de chaves pública/privada.

  - Como empresa, a Docker Inc. sempre reconheceu prontamente que deve muito do seu sucesso ao ecossistema. Enquanto a Docker Inc. se concentrava em produzir uma versão estável e pronta para produção do mecanismo de contêineres, outras empresas como CoreOS, WeaveWorks e ClusterHQ trabalhavam em áreas relacionadas, como orquestração e rede de contêineres. No entanto, logo ficou claro que a Docker Inc. planejava fornecer uma plataforma completa pronta para uso, incluindo recursos de rede, armazenamento e orquestração. Para incentivar o crescimento contínuo do ecossistema e garantir que os usuários tivessem acesso a soluções para uma ampla gama de casos de uso, a Docker Inc. anunciou que criaria uma estrutura modular e extensível para o Docker, onde os componentes padrão poderiam ser substituídos por equivalentes de terceiros ou estendidos com funcionalidades de terceiros. A Docker Inc. chamou essa filosofia de "Baterias Inclusas, Mas Substituíveis", o que significa que uma solução completa seria fornecida, mas as peças poderiam ser substituídas.⁵

  - A escolha da ferramenta de descoberta de serviços depende muito do caso de uso específico. A maioria dos projetos já utiliza uma ferramenta relevante devido a requisitos de software ou ferramentas de plataforma existentes (por exemplo, o Mesos usa o ZooKeeper e o GKE usa o etcd). Nesses casos, faz sentido usar o que já existe em vez de adicionar outra ferramenta. Escolher entre o etcd (ou o etcd mais o SkyDNS) e o Consul é muito mais difícil. Ambos são projetos relativamente novos (o etcd é um pouco mais antigo), mas são baseados em algoritmos subjacentes sólidos. O Consul inclui suporte a DNS e alguns recursos avançados por padrão, o que geralmente faz a diferença na decisão. O etcd é indiscutivelmente menos "opinativo" que o Consul e possui um armazenamento de chave-valor mais avançado, o que pode torná-lo uma escolha melhor em cenários que exigem muita personalização. O suporte a DNS do Consul e do SkyDNS pode não ser relevante se você usa APIs para descobrir serviços ou se já inclui uma solução de rede que fornece resolução de nomes.

  - A confiança de conteúdo também oferece garantias de atualização para proteger contra ataques de repetição. Um ataque de repetição ocorre quando um artefato é substituído por um artefato válido anteriormente. Por exemplo, um atacante pode substituir um binário por uma versão antiga e vulnerável conhecida, que foi assinada anteriormente pelo editor. Como o binário está corretamente assinado, o usuário pode ser enganado e executar a versão vulnerável do binário. Para evitar isso, a confiança de conteúdo utiliza chaves de carimbo de data/hora associadas a cada repositório. Essas chaves são usadas para assinar os metadados associados ao repositório. Os metadados têm um curto prazo de validade, o que exige que sejam renovados frequentemente pela chave de carimbo de data/hora. Ao verificar se os metadados não expiraram antes de baixar a imagem, o cliente Docker pode ter certeza de que está recebendo uma imagem atualizada (ou recente). As chaves de carimbo de data/hora são gerenciadas pelo Docker Hub e não exigem nenhuma interação do editor.

  - A linha divisória entre descoberta de serviços e redes pode se tornar surpreendentemente tênue em um contexto de contêineres. A descoberta de serviços é o processo de fornecer automaticamente aos clientes de um serviço informações de conexão (normalmente endereço IP e porta) para uma instância apropriada desse serviço. Esse problema é simples em um sistema estático de host único, onde existe exatamente uma instância de tudo, mas é muito mais complexo em um sistema distribuído com múltiplas instâncias de serviços que surgem e desaparecem ao longo do tempo. Uma maneira de abordar a descoberta é o cliente simplesmente solicitar o serviço pelo nome (por exemplo, banco de dados ou API) e realizar alguma "mágica" no backend para que isso seja resolvido para o local apropriado. Essa "mágica" pode assumir a forma de contêineres embaixadores simples, uma solução de descoberta de serviços como o Consul, uma solução de rede como o Weave (que inclui recursos de descoberta de serviços) ou alguma combinação das anteriores.
