No final de 2024, a Anthropic lançou o Model Context Protocol (MCP) com

grande alarde. A Anthropic não apenas padronizou uma maneira de

transformar chatbots de simples interlocutores em agentes capazes de agir,

como também lançou o MCP simultaneamente com suporte de IDEs

populares com inteligência artificial, bem como do Claude Desktop da

própria Anthropic, impulsionando uma rápida adoção. Um dos principais

componentes da arquitetura do MCP, os servidores MCP, foi projetado para

ser incrivelmente simples de configurar, mas extremamente complexo

quando fluxos de trabalho avançados eram necessários. Isso levou a uma

rápida proliferação de servidores MCP criados por usuários, aprimorando

fluxos de trabalho de IA generativa e consolidando a adoção do MCP como

padrão para a criação de agentes de IA.

Este livro tem como objetivo aprofundar-se no MCP, indo além da

documentação disponível e proporcionando uma compreensão profunda da

arquitetura do MCP, dos recursos que o protocolo e seus SDKs oferecem e,

mais importante, como usar esse conhecimento para construir seus próprios servidores e clientes MCP e aprimorar seus fluxos de trabalho de IA

generativa com eles. Ao fazer isso, você pode transformar chatbots de IA

generativa padrão em agentes completos, capazes de agir, planejar e usar

ferramentas por conta própria, e até mesmo trabalhar em conjunto para

realizar tarefas complexas. Neste capítulo, você aprenderá mais sobre

agentes de IA, o que são, como são usados e como o MCP se encaixa no

cenário de IA generativa. O próximo capítulo abordará o próprio MCP: sua

história, os problemas que resolve, sua arquitetura e exemplos práticos de

MCP em ação. Os três capítulos seguintes abordarão os principais

componentes do MCP: clientes e seus aplicativos host, servidores e

transportes. Com esse conhecimento em mãos, os três capítulos finais

usarão um projeto para lhe dar experiência prática na construção não apenas

de seus próprios servidores, clientes e transportes, mas também de como

utilizá-los em um agente real.

**Afinal, o que é um agente?**

Mas o que exatamente é um agente? Se você pesquisar isso no Google,

encontrará mais definições do que links. Se perguntar ao Claude ou ao

ChatGPT algumas vezes seguidas, poderá até obter respostas diferentes.

Parece que todos têm uma opinião sobre o que é ou não um agente, mas é

difícil falar sobre agentes sem chegar a uma definição comum. Para os

propósitos deste livro, usaremos a definição utilizada pela Anthropic em seu

artigo "Building Effective Agents": um agente é um "sistema... onde

grandes modelos de linguagem (LLMs) direcionam dinamicamente seus

próprios processos e uso de ferramentas, mantendo o controle sobre como

realizam as tarefas". Eles geralmente são implementados como LLMs que

recebem uma tarefa e então usam ferramentas (aqui, uma ferramenta é

qualquer código "externo" ao agente que é chamado por ele para realizar

alguma tarefa) para executar uma ação e coletar feedback antes de

prosseguir. A ação e a coleta de feedback ocorrem em um ciclo, onde o

LLM realiza alguma ação, o feedback é coletado e enviado de volta ao

LLM, e então o LLM decide se apresenta o resultado como final ao usuário, usa o feedback coletado para realizar outra ação ou coleta mais feedback do

usuário.

Vamos analisar um uso comum de agentes e como esse ciclo se pareceria

para cada resultado. Para este exemplo, imagine uma tarefa de

programação: você está em um editor de código com inteligência artificial,

como o Cursor ou o Windsurf, e pede ao chatbot para escrever testes

unitários para o seu código. Os ciclos de ação/feedback podem ser algo

como:

(Figura em breve)

Isso se distingue de um fluxo de trabalho agentivo, um sistema no qual o

código, em vez da LLM (Linguagem de Linguagem de Linguagem),

determina o caminho que a lógica seguirá, mas ainda chama uma LLM para

realizar parte do trabalho. Esse é um padrão comum em sistemas agentivos

e pode ser muito útil para tarefas complexas que exigem algum nível de

lógica determinística, conforme imposto pelo código. Por exemplo, imagine

um aplicativo que recebe código como entrada e o converte em seu

equivalente em outra linguagem. Uma abordagem de fluxo de trabalho para

isso poderia ser receber o código de entrada, chamar uma LLM para

detectar a linguagem em que o código está e, se não detectar nenhuma

linguagem, retornar uma mensagem de erro ao usuário. Se detectar uma

linguagem e houver um analisador sintático determinístico para ela, o

analisador sintático determinístico é chamado no código e a saída traduzida

é retornada ao usuário. Se for detectada uma linguagem para a qual não haja

um analisador sintático, a LLM seria chamada novamente e solicitada a

realizar a tradução, retornando a saída ao usuário. Isso é encadeamento de

prompts, um padrão comum de fluxo de trabalho agentivo para tarefas que

exigem uma ou mais chamadas a um LLM (Learning Language

Management) e um caminho de código para decidir quais os próximos

passos a serem tomados para concluir essa tarefa. Outros padrões comuns

de fluxo de trabalho incluem paralelização, onde várias chamadas a LLMs

são feitas simultaneamente e os resultados são combinados; roteamento,

onde um LLM classifica uma entrada e, em seguida, faz uma de várias

chamadas possíveis a outros LLMs com a entrada classificada; orquestrador-trabalhador, onde um LLM divide uma tarefa, chama LLMs

trabalhadores para executar as tarefas menores e, finalmente, um LLM é

chamado para reunir os resultados; e avaliador-otimizador, onde um LLM

gera uma resposta ao prompt do usuário e uma segunda chamada a um

LLM avalia essa resposta e a apresenta ao usuário, se for aceitável, ou a

retorna com feedback para a primeira chamada a um LLM para

refinamento.

A figura abaixo mostra o fluxo de trabalho de encadeamento de prompts em

ação: a entrada é enviada ao LLM para detectar o idioma. A saída é o

idioma inferido, que é verificado no código para determinar se houve

detecção ou não. O fluxo de trabalho passa para o estado "falha" se nenhum

idioma for detectado e para o estado "sucesso" se houver. O próximo estado

também é determinístico e verifica se existe um analisador sintático

determinístico para o idioma detectado. Se houver, o próximo estado

simplesmente chama o analisador sintático e retorna a saída para o usuário.

Caso contrário, o próximo estado chama um LLM novamente e solicita que

ele faça a tradução, retornando a saída para o usuário.

(Figura em breve)

Em contraste, uma abordagem agentiva, no entanto, forneceria ferramentas

de detecção e tradução de idioma ao LLM (Agente de Linguagem Local) e

simplesmente pediria a ele que detectasse o idioma da entrada e o utilizasse

para escolher a ferramenta de tradução correta. A próxima figura ilustra esse

processo: o usuário fornece o código de entrada e o LLM é acionado. Ele

percorre o ciclo de ação-feedback, detectando o idioma da entrada e

verificando o ambiente em busca de ferramentas para realizar a tradução. Se

houver uma ferramenta apropriada para o agente usar, ele a aciona; caso

contrário, tentará realizar a tradução de forma autônoma. Qualquer que seja

o resultado obtido, ele o retornará ao usuário.

(Figura em breve)

A principal diferença a ser lembrada é que os agentes podem decidir quais

ferramentas usar e quais ações tomar em um ou mais ciclos de feedback de

ação, enquanto os fluxos de trabalho usam caminhos de código para determinar quais ferramentas usar e quais ações tomar, e como tomar essas

ações.

**Como os agentes são usados na IA**

**generativa?**

Como você viu acima, os agentes são implantados para tarefas complexas

onde planejamento, tomada de decisão e uso de ferramentas são necessários

para a conclusão da tarefa. Normalmente, para o usuário, não há diferença

óbvia na interação com um agente em comparação com a interação com

LLMs não-agentes: há uma caixa de bate-papo com um chatbot do outro

lado, o usuário faz perguntas e o chatbot responde. Uma diferença

fundamental com os agentes, no entanto, é que os usuários podem pedir que

eles façam algo, e eles podem executar ações e interagir com o ambiente,

utilizando as ferramentas disponíveis. Um usuário pode perceber isso como

nada mais do que um chatbot mais poderoso, mas os agentes podem

proporcionar experiências personalizadas ao usuário por meio do uso de

ferramentas, prompts especializados, recursos de dados, memória e a

capacidade de coordenar e colaborar com outros agentes.

Essas habilidades extraordinárias despertaram grande interesse em

investigar como fazer com que os agentes trabalhem juntos.

**O que os agentes possibilitam na IA**

**generativa?**

Os agentes e os padrões de projeto que estão sendo inventados e explorados

para eles estão possibilitando alguns dos avanços mais empolgantes na IA

generativa. Um único agente transforma um Modelo de Aprendizagem

Linear (LLM) com uma fonte estática de informações (provenientes de seus

dados de treinamento) em algo capaz de acessar novas informações,

interagir com o ambiente por meio de ferramentas, agir de forma semiautônoma e se comunicar com outros agentes. Todo esse

comportamento deriva tanto da arquitetura do agente (especificamente, o

ciclo de feedback de ação discutido anteriormente neste capítulo) quanto

das capacidades adicionadas a um LLM em um sistema agente: recuperação

de dados, ferramentas e memória.

**O papel do Protocolo de Contexto do Modelo**

**na IA Agética**

Antes do MCP, embora a IA agente ainda fosse relativamente nova, as

organizações que construíam plataformas internas de IA generativa e frameworks públicos como o LangChain precisavam encontrar maneiras

diferentes para que os usuários implementassem ferramentas e as

repassassem para um LLM (Modelo de Aprendizagem Baseado em Lógica)

para uso, sendo que cada LLM exigia seu próprio conector. Isso levou ao

que é conhecido como o "problema MxN": para cada um dos M LLMs que

precisam ser suportados, são necessários N conectores para cada

ferramenta, fonte de dados etc. que se deseja conectar. Se você precisa dar

suporte a 3 modelos e 10 ferramentas, isso significa que você precisa

escrever 30 conectores que se repetem em grande parte, aumentando as

chances de surgirem bugs difíceis de diagnosticar.

(Figura em breve)

O Protocolo de Contexto de Modelo, inspirado no Protocolo de Servidor de

Linguagem (LSP), fornece uma interface comum para a construção de

ferramentas, prompts e recursos de dados, conectando-os a LLMs

(Máquinas de Linguagem Linguística). Dessa forma, para qualquer número

M de LLMs, basta escrever N conectores, transformando o problema M x N

em M + N.

Ao trabalhar com o Protocolo de Contexto de Modelo (MCP), você

normalmente trabalhará com (ou criará!) clientes ou servidores. Em alguns

projetos, você poderá criar ambos. Para realmente entender o protocolo e

usá-lo em todo o seu potencial em seus projetos, você precisará estar

familiarizado com todos os componentes da arquitetura MCP. Neste

capítulo, você aprenderá sobre o lado do consumidor dessa arquitetura:

aplicativos host e clientes.

Primeiramente, vamos analisar os aplicativos host: o que são, o que fazem e

que tipos de aplicativos podem se beneficiar da incorporação do MCP? Em

seguida, examinaremos o próprio cliente. É ele que define o aplicativo host:

ele hospeda o(s) cliente(s), que, por sua vez, permitem que o aplicativo host

se comunique com os servidores MCP.

No entanto, existe um porém: um único cliente só pode se comunicar com

um único servidor. Se você precisar acessar vários servidores, terá que iniciar várias instâncias do cliente ou ter um único cliente que faça apenas

conexões curtas e temporárias com os servidores. Analisaremos os prós e os

contras de cada abordagem e quais casos de uso são mais adequados para

cada uma delas.

Em seguida, você verá um aplicativo host muito simples e o cliente que ele

hospeda. Analisaremos linha por linha para ajudá-lo a entender a estrutura

de um cliente e como ele fornece funcionalidades ao aplicativo host. Por

fim, você aprenderá sobre as melhores práticas para integrar um cliente a

um aplicativo host, garantindo que seu aplicativo permaneça seguro,

confiável e responsivo.

**O aplicativo host**

Sua aplicação pode ser qualquer coisa que hospede o(s) cliente(s) que

gerenciarão as conexões com os servidores MCP, desde um simples script

de chatbot que recebe algumas informações do usuário e as envia para um

LLM para obter uma resposta, até um IDE completo como o Cursor ou o

Windsurf.

Graças à modularidade do MCP, existem poucas restrições quanto ao que

seu aplicativo host faz para oferecer suporte ao MCP. Ele só precisa se

comunicar com um LLM e hospedar um ou mais clientes MCP. Se você

planeja usar ferramentas, o LLM que seu aplicativo host utiliza também

deve oferecer suporte à chamada de ferramentas.

No exemplo abaixo, você verá um aplicativo host básico. Tudo o que ele faz

em seu estado atual é receber a entrada do usuário em um loop constante e

passá-la para um LLM. Você pode encontrar este script e todo o restante do

código deste capítulo em ch3 no repositório do livro no Github.

Aqui temos o início de um host MCP: um script muito simples que recebe

uma entrada do usuário e a envia para um LLM (neste caso, o Claude 3.5 da

Anthropic) e imprime uma resposta. Vamos analisá-lo linha por linha.

Nas linhas 1 a 4, importamos alguns pacotes: os para carregar variáveis de

ambiente, anthropic para a classe cliente Anthropic e dotenv para

carregar temporariamente pares chave-valor de um arquivo em variáveis de

ambiente. Isso permite armazenar informações sensíveis, como chaves de

API, fora do código, em um arquivo convencionalmente chamado .env no

formato KEY=VALUE.

**AVISO**

Não inclua seus arquivos .env no controle de versão, pois isso pode expor suas chaves de API publicamente, levando a consequências como uso não autorizado e altos custos para acesso à API (pagamento conforme o uso). Se você fizer isso acidentalmente, acesse imediatamente o site do seu provedor de chaves e desautorize a(s) chave(s) comprometida(s).

Nas linhas 6 a 9, o aplicativo chama load_dotenv(), que realiza a

tarefa de carregar suas chaves em variáveis de ambiente e, em seguida,

obtém a chave carregada acessando o dicionário os.environ. Essa chave

é então usada para instanciar o cliente Anthropic.

Após exibir uma mensagem de boas-vindas ao usuário na linha 11, o código

entra em um loop infinito. A linha 14 solicita uma mensagem do usuário,

enquanto as linhas 15 a 17 verificam se a mensagem contém a frase de

saída. Se encontrada, a aplicação exibe uma mensagem de despedida ao

usuário e encerra a execução.

As linhas 18 a 28 tratam da comunicação com o modelo. Este é um padrão

que você verá repetidamente ao fazer chamadas LLM a partir do código. No

cliente Anthropic, você acessa a função .messages.create() e

fornece parâmetros como max_tokens para controlar o limite máximo de

tokens a serem gerados em resposta à consulta do usuário, system para

definir o prompt do sistema, moldando a forma como o modelo responde ao

usuário, messages, que é uma lista de dicionários de prompts com uma

função específica (observe que definimos role como "usuário" e content como o prompt do usuário) e, finalmente, model especifica qual

modelo deve ser usado.

A interface messages dentro da API Anthropic é muito poderosa, e a

função create() possui diversos outros parâmetros que você pode

experimentar para ajustar os tipos de respostas que obtém do modelo. Você

pode ler mais na documentação oficial do Anthropic.

As linhas finais, 29-30, percorrem as respostas geradas pelo modelo,

imprimindo o conteúdo textual de quaisquer mensagens recebidas pelo

cliente Anthropic.

**O Cliente**

Ao integrar suporte para MCP em seus aplicativos, o cliente é um dos

componentes mais importantes que você desenvolverá e com os quais

trabalhará. Os clientes permitem a comunicação entre os servidores MCP e

seu aplicativo, fornecendo a interface entre o servidor, seu aplicativo e o

modelo de linguagem abrangente que você está utilizando.

Antes de escrever qualquer código, vamos pensar em como queremos nos

conectar a um servidor. Isso significa pensar em qual meio de transporte

queremos usar. No MCP, um meio de transporte é uma implementação da

camada de transporte do protocolo e gerencia como as mensagens são

enviadas entre o cliente e o servidor.

Até o momento da redação deste documento, o MCP possui duas

implementações principais de transporte integradas: stdio e HTTP

streamable. O transporte stdio é adequado para casos de uso em que se

espera que os servidores sejam executados juntamente com o aplicativo

host, pois utiliza fluxos de entrada e saída padrão para transportar a

comunicação entre o cliente e o servidor. Este é o transporte mais comum,

pois é simples de programar e porque a maioria dos aplicativos comerciais

disponíveis que hospedam clientes MCP espera que o usuário instale e

execute os servidores por conta própria.

**OBSERVAÇÃO**

O SDK Python MCP também inclui um transporte WebSocket e um transporte HTTP Server-Sent Events (SSE), que está sendo substituído pelo Streamable HTTP. Por isso, abordaremos apenas a conexão com o Streamable HTTP, mas os padrões estabelecidos aqui, juntamente com o guia de compatibilidade retroativa da Anthropic, devem ser suficientes para que você, como desenvolvedor cliente, suporte ambos os transportes remotos. Todos esses transportes serão explorados com mais detalhes no capítulo 5.

Por outro lado, o transporte HTTP Streamable integrado é mais adequado

para aplicações em que não se espera que os servidores MCP que elas

consomem sejam executados juntamente com suas aplicações hospedeiras.

Isso é mais evidente no desenvolvimento de plataformas e em ferramentas

hospedadas.

Ao desenvolver uma plataforma ou framework para construir e implantar

produtos subsequentes, não é possível garantir que os servidores MCP

sejam implantados na mesma máquina que a aplicação. Caso contrário, um cliente stdio nunca conseguirá se comunicar com o servidor. Algumas

organizações estão experimentando hospedar servidores MCP em suas

próprias redes, expondo-os ao público. Se você deseja utilizar essa opção,

precisará incorporar suporte a Streamable HTTP em seu cliente.

**AVISO**

Aviso: Qualquer transporte remoto, incluindo o Streamable HTTP, precisa ser devidamente protegido para evitar problemas de segurança. Isso inclui autenticar a conexão e validar os cabeçalhos de origem.

Primeiro, vamos analisar o caso mais simples: a conexão via stdio. Nesta

seção, você criará o construtor e os métodos connect() para a classe

MCPClient. Em seguida, você fará o mesmo para criar um cliente HTTP

Streamable, que pode suportar a conexão com servidores MCP remotos.

Em seguida, analisaremos um exemplo de um cliente que oferece suporte

aos três métodos e como escolher o mais adequado.

**Conectando com stdio**

Ao conectar-se a um servidor MCP com stdio, o cliente iniciará o servidor

como um subprocesso e, em seguida, lerá mensagens de sua entrada padrão

e enviará mensagens por meio de sua saída padrão. Felizmente, o SDK

Python do MCP inclui todas as estruturas necessárias para construirmos

nosso cliente e nos conectarmos via stdio.

Primeiro, vamos configurar o construtor do cliente:

**from** **contextlib** **import** AsyncExitStack

**from** **mcp** **import** ClientSession

**class** **MCPClient** :

**def** __init__(self, name: str, command: str, server_args: 

list[str], env_vars: dict[str, str]=**None**) -> **None** :

self.name = name

self.command = command self.server_args = server_args

self.env_vars = env_vars self._session: ClientSession = **None** self._exit_stack: AsyncExitStack = AsyncExitStack() self._connected: bool = **False**

Neste construtor, você configura diversas propriedades importantes que

permitirão conectar-se e interagir com um servidor MCP. A primeira, name,

simplesmente atribui um nome legível ao cliente instanciado. Isso é útil

para registro de logs, especialmente se você estiver mantendo conexões

com vários servidores, pois permite identificar facilmente a origem

específica de quaisquer logs emitidos. No entanto, não é obrigatório para

um cliente MCP.

Os próximos 3 parâmetros são usados para configurar o servidor. command

identifica o executável a ser executado para iniciar o servidor.

Normalmente, os servidores são executados com python ou node (ou

npx, dependendo da sua instalação local). É tão comum que, na maioria

dos casos de uso, seu cliente pode definir um enum para garantir que um

comando compatível esteja sendo usado. server_args é uma lista de

todos os argumentos de linha de comando que devem ser passados para o

executável command. Isso incluirá, no mínimo, o caminho para o arquivo

do servidor. Se o arquivo aceitar argumentos de linha de comando, digamos

\--port 8000 --verbose, sua lista server_args será semelhante a

["server.py", "--port", "8080"]. Em seguida, temos um

dicionário de variáveis de ambiente em env_vars. Essas variáveis

funcionam como variáveis de ambiente típicas, e você pode até mesmo

descompactar as variáveis de ambiente do seu sistema neste dicionário com

{**os.env} . No entanto, isso não é recomendado, pois pode expor

informações confidenciais ao seu servidor.

Também incluímos algumas propriedades de gerenciamento de conexão

"privadas", como _session, que armazenará a sessão do cliente,

_exit_stack, que gerencia os contextos de conexão assíncronos (mais

sobre isso adiante), e _connected, que nos impedirá de conectar a um

servidor quando já estivermos conectados.

O próximo passo é implementar os métodos connect() e

disconnect() . Aqui, você usará os objetos de gerenciamento de

conexão integrados do SDK MCP do Python para estabelecer e manter a

conexão com o servidor.

Isso pode parecer complexo, especialmente se você não tiver experiência

com programação assíncrona em Python. Vamos detalhar o que estamos

fazendo aqui para tornar o processo mais compreensível, começando com o

método `connect()`. A primeira coisa que este método faz é verificar a

propriedade `_connected()`. Isso impedirá que o usuário tente abrir

uma conexão já existente. Em seguida, instanciamos um objeto

` StdioServerParameters`, passando as propriedades definidas no

construtor. O restante do código cria os processos e conexões necessários

para executar e usar o servidor MCP. Fazemos isso por meio de

` _exit_stack`, que contém uma instância de `AsyncExitStack`.

[!NOTE] AsyncExitStack permite aninhar manualmente gerenciadores

de contexto assíncronos sem usar a sintaxe mais comum de gerenciador de

contexto async with. Essa é uma escolha ideal porque você pode

adicionar dinamicamente mais gerenciadores de contexto assíncronos à

pilha e liberar com segurança todos os recursos da pilha em ordem com

apenas uma chamada. Esse último recurso é extremamente útil porque

podemos determinar quando desfazer a pilha, em vez de depender da saída

do código do escopo de cada gerenciador de contexto. Isso é feito

chamando a função enter_async_context().

Primeiro, abrimos uma conexão com o servidor stdio, que inicia o

subprocesso no qual a sessão MCP será executada. Passamos uma instância

de stdio_client que é instanciada com o server_params

construído imediatamente antes. Em seguida, descompactamos os

resultados dessa chamada em self.read e self.write, que

representam os fluxos de leitura e gravação da sessão, respectivamente. O

próximo passo é iniciar a sessão do cliente MCP, novamente chamando

enter_async_context(), mas desta vez com uma instância de

ClientSession com os fluxos de leitura e gravação definidos como

self.read e self.write. Isso é armazenado em self._session,

que então inicializamos com initialize(). Este método lida com a iniciação da conexão com o servidor, anuncia as capacidades do cliente para

o servidor, verifica a versão do protocolo suportada pelo servidor e informa

ao servidor que o cliente foi inicializado com sucesso. Finalmente,

definimos self._connected para True.

Em resumo, ao escrever o código de conexão para um servidor stdio, você:

1\. criar uma instância de StdioServerParams

2\. iniciar o subprocesso do servidor stdio em um contexto assíncrono

3\. Inicie a sessão do cliente MCP em outro contexto assíncrono

(aninhado) e armazene-a em self._session

4\. inicializar a própria sessão

Neste exemplo, também criamos o método disconnect(), que

simplesmente chama self._exit_stack.aclose() para fechar

todas as conexões com o servidor em ordem e, em seguida, define

self._connected como False e self._session como None.

Com isso, você pode se conectar e desconectar de servidores MCP usando o

transporte stdio conforme desejar. Embora você ainda não possa fazer nada

com os servidores, você concluiu a parte mais complexa. Parabéns! Agora

faremos a mesma coisa, mas com suporte ao transporte HTTP Streamable.

**Conectando-se com o Streamable HTTP**

Exemplo de terceiros: Streamable HTTP é o principal protocolo de

transporte projetado para uso com servidores MCP remotos. Originalmente,

o HTTP Server-Sent Events (SSE) era o padrão para conexão com

servidores remotos, mas diversos problemas em produção tornaram seu

suporte contínuo inviável. Esses problemas, bem como a arquitetura de

ambos os protocolos de transporte, serão abordados no Capítulo 5.

Ao conectar-se a um servidor remoto com Streamable HTTP, você se

conectará por meio de um único endpoint definido pelo servidor. As

respostas podem ser respostas HTTP padrão imediatas (acionadas pelo

envio de um POST ao servidor e que não exigem uma conexão sempre aberta) ou respostas SSE de streaming (acionadas pelo envio de um GET

vazio ao servidor), sendo estas últimas opcionais e dependentes da

implementação do servidor.

[!Note] “Espere”, você pode estar pensando, “eu achei que o Streamable

HTTP substituísse o HTTP+SSE, qual é a diferença?” Abordaremos as

atualizações no Capítulo 5, mas a principal diferença é que obter uma

resposta SSE é opcional — tanto para o cliente que precisa solicitar uma

resposta de streaming, quanto para o servidor que pode escolher se deseja

ou não suportá-la.

Para dar suporte ao Streamable HTTP, vamos nos concentrar novamente na

construção do construtor de MCPClient, dos métodos connect() e

disconnect() . Vamos começar com o construtor:

**Interagindo com os recursos do servidor MCP**

Os servidores MCP podem fornecer uma ampla variedade de recursos,

incluindo as primitivas MCP: ferramentas, recursos e prompts. Como

desenvolvedor de cliente, você interagirá com eles por meio do objeto

instanciado ClientSession e, geralmente, usará o mesmo fluxo de

trabalho de duas etapas para cada um deles: descoberta e uso. Para cada um

desses recursos, a descoberta ocorre por meio de uma solicitação

<primitive>/list , que no SDK Python é encapsulada pelos métodos

list_<primitive>s(). Essas chamadas, se o servidor fornecer o

recurso respectivo, retornarão uma lista das ferramentas, recursos ou

prompts disponíveis. O uso dos recursos resultantes requer uma chamada

<primitive>/<verb> , onde o verbo depende da primitiva que está

sendo chamada. No SDK Python, essas chamadas são encapsuladas com

métodos do tipo <verb>_<primitive>(), como call_tool() para

tool/call.

**OBSERVAÇÃO**

O protocolo também permite que mensagens de notificação sejam enviadas pelo servidor (bem como pelo cliente), cabendo ao receptor lidar com as notificações da maneira que desejar. Para conexões HTTP Streamable, essas notificações serão enviadas como qualquer outra mensagem definida pelo transporte, mas serão enviadas como um JSONRPCNotification , uma implementação da classe JSONRPCMessage.

Embora os SDKs MCP disponíveis encapsulem as várias chamadas de

solicitação em métodos que seu cliente pode chamar diretamente,

geralmente é uma boa ideia encapsulá-las em seus próprios métodos de

objeto para que você possa adicionar registro, novas tentativas, parâmetros

de método e quaisquer outras personalizações às suas chamadas para o

servidor, como passar o parâmetro opcional list_<primitive>() para

paginação a qualquer um dos métodos list_<primitive>().

**Ferramentas**

As ferramentas são o recurso mais comum fornecido pelos servidores MCP,

e com razão: fluxos de trabalho com agentes frequentemente exigem

ferramentas para ampliar as capacidades de um LLM básico. As

ferramentas são simplesmente funções determinísticas que um LLM pode

decidir chamar quando necessário. Naturalmente, a função pode fazer

qualquer coisa que você possa imaginar em código, desde calcular números

até encontrar a previsão do tempo em um determinado local. Os servidores

MCP permitem que os clientes descubram e usem essas ferramentas.

Para descobrir ferramentas usando o SDK Python, seu cliente simplesmente

chamaria self.session.list_tools() dentro do cliente, que

retorna um objeto response com uma propriedade tools. Esta

propriedade contém uma lista de todas as ferramentas disponibilizadas pelo

servidor conectado ao cliente, onde as ferramentas possuem as propriedades

name , description, inputSchema, annotations e

model_config . Você pode ver a estrutura atual da classe Tool no módulo

types do SDK Python do MCP.

Chamar list_tools() diretamente pode funcionar bem para casos de

uso básicos, mas se você estiver criando um cliente que será usado em

diferentes servidores com diferentes capacidades e possivelmente em

diferentes aplicações, pode ser vantajoso escrever sua própria função com

algumas salvaguardas, registro de logs e qualquer outra coisa que seus

usuários possam precisar abstraída. Por exemplo, você poderia implementar

a seguinte função:

do objeto list_tools(), que por sua vez encapsula a solicitação

tools/list . Embora simples, o exemplo mostra como você pode

implementar algumas salvaguardas básicas, como a verificação do status da

conexão com o servidor, bem como registros mais detalhados, como a

verificação de resultados vazios da ferramenta. Você pode estender esse

padrão para adicionar novas tentativas, traduzir o formato da ferramenta do

MCP para outro formato compatível com outro LLM e muito mais. Outro

padrão útil que você verá mais adiante neste capítulo é a conversão dos

objetos Tool do MCP em seus próprios objetos, onde você pode

personalizar sua estrutura, implementar funções de tradução para oferecer

suporte a outros LLMs e muito mais.

Agora que seu cliente tem ferramentas, o que você faz com elas? Usá-las, é

claro! Como no exemplo get_available_tools(), vamos encapsular

a função call_tool() do objeto Session, adicionar um pouco de

registro de logs e lidar com os diferentes tipos de resposta disponíveis.

call_tool() retorna um CallToolResult, que é uma lista de

quaisquer tipos de conteúdo do SDK:

TextContent - O resultado da ferramenta é qualquer tipo de

texto que você retornaria ao usuário e é armazenado na

propriedade text.

ImageContent - Se o resultado da ferramenta for uma imagem,

ela será retornada como um objeto ImageContent, com os

dados da imagem armazenados como uma string codificada em

base64 na propriedade data

AudioContent - Algumas ferramentas retornam conteúdo de

áudio. Assim como ImageContent, os dados são codificados em

base64 e residem na propriedade data

EmbeddedResource - As ferramentas podem retornar recursos

incorporados como parte de sua resposta, normalmente para

contexto adicional ou armazenamento em cache de dados. Os

dados do recurso residem na propriedade resource, que pode ser

TextResourceContents (representando o conteúdo de texto

de, por exemplo, um recurso de arquivo de configuração) ou

BlobResourceContents (representando os dados binários do

recurso como uma string codificada em base64).

Um ponto fundamental a ter em mente é que as solicitações de uso de

ferramentas retornam listas de conteúdo e, em alguns casos, como quando

uma resposta EmbeddedResource é usada como contexto adicional, isso

deve ser esperado e tratado como tal. No exemplo abaixo, você envolverá

call_tool() e retornará representações em string de cada um dos tipos

de resposta de conteúdo de ferramenta válidos.

fazemos a chamada call_tool(), passando o nome da ferramenta e os

**AVISO**

Ao criar agentes que suportam MCP, fique atento à chamada de ferramentas e ao desempenho geral à medida que adiciona ferramentas ao agente. O desempenho tende a cair drasticamente com a adição de mais ferramentas, geralmente devido à maior dificuldade em selecionar a ferramenta correta para uma solicitação. Isso pode ocorrer devido à ambiguidade ou sobreposição nas descrições das ferramentas, bem como ao excesso de informações agregadas nas descrições e interfaces, o que impede o processamento preciso pelo LLM.

Se você estiver criando seu cliente para atender a um aplicativo específico,

como o aplicativo de bate-papo simplificado apresentado no início deste

capítulo, também precisará usar essas chamadas. Muitas APIs LLM que

suportam chamadas de ferramentas possuem um parâmetro opcional

tools, facilitando o envio das ferramentas disponíveis para um modelo.

Outros modelos exigem a criação de uma lista de nomes, descrições e

esquemas de ferramentas no prompt do sistema, mas, para este exemplo,

estamos nos concentrando na API Claude da Anthropic, que aceita um

parâmetro tools. Este parâmetro será adicionado a todas as mensagens do

usuário para o modelo e, em seguida, as respostas resultantes serão analisadas pelo aplicativo host para verificar o uso de alguma ferramenta.

Se alguma ferramenta for encontrada, o aplicativo formulará uma

mensagem tool_result e a adicionará a uma lista de mensagens que é

enviada de volta ao modelo para gerar o resultado final para o usuário.

**Recursos**

Como você aprendeu no Capítulo 2, os recursos do MCP podem

desempenhar muitas funções, algumas das quais ainda não foram

totalmente exploradas pelos membros da comunidade, como a de cache

para otimizar o tamanho do prompt. Basicamente, os recursos fornecem a

um aplicativo host e ao LLM com o qual ele interage acesso aos dados

fornecidos pelo servidor MCP. Isso pode incluir registros de banco de

dados, imagens, arquivos de texto e muito mais. Os recursos também

permitem diversas ações:

resources/list ou list_resources() - solicita uma

lista de recursos disponíveis do servidor

resources/templates/list ou

list_resource_templates() - solicita uma lista de

modelos de URI de recursos do servidor. Isso permite construir a

URI de um recurso dinamicamente com base em outras

informações que você possa ter. No SDK do Python, essas strings

se parecem com f-strings, com as seções variáveis delimitadas por

chaves.

resources/read ou read_resource() - acessa o arquivo

fornecido pelo servidor, enviando os dados em formato de texto ou

blob para o LLM.

resources/subscribe ou subscribe_resource() -

inscreve o cliente para receber atualizações do URI do recurso

especificado na chamada.

resources/unsubscribe ou

unsubscribe_resource() - cancela a inscrição do cliente

para receber atualizações do recurso especificado pelo URI do

recurso nos parâmetros.

descobrir os elementos primitivos disponíveis, como recursos neste caso, e

expressões regulares para obter o nome de um recurso a partir da entrada do

ao usuário que uma mensagem contendo apenas a palavra "atualizar"

atualizará e exibirá novamente a lista de recursos. Adicionamos uma

verificação para a mensagem "atualizar" e, em seguida, uma verificação

ele imprime um aviso e, em seguida, sai do escopo da verificação

messages . Esta lista ainda está vazia, portanto, nada muda neste caso, e o

prompt simples do usuário é enviado como está para o LLM.

recursos disponíveis, lemos o recurso para a memória e verificamos se é um

texto simples ou um blob. Se for texto, criamos um bloco de conteúdo com

o texto. Se for um blob, verificamos se o mimeType corresponde a algum

dicionário construído. Se nenhuma das duas condições for verdadeira,

Ao adicionar contexto a um prompt do usuário, por exemplo, você não deve criar uma mensagem de usuário separada, como faz com o próprio prompt. regular ou até mesmo com outra chamada LLM. Você pode fazer mais

Os prompts são o último dos três principais recursos primitivos fornecidos

pelos servidores MCP. Eles são projetados para serem controlados pelo

usuário do aplicativo, que deve poder escolher qual ou quais prompts deseja

usar, de forma semelhante à interface vista no exemplo de aplicativo host na

seção anterior. Cada definição de prompt possui um identificador único

name , um description opcional e legível por humanos, e um

arguments opcional, que é uma lista de dicionários com um name, um

description opcional e legível por humanos, e um required

opcional, indicando se o argumento é obrigatório ou não. Cabe então à

implementação do cliente ou ao aplicativo host que utiliza o prompt inserir

valores para cada um dos argumentos. Esses são chamados de prompts

dinâmicos.

O SDK Python simplifica o tratamento de prompts para nós, com o método

Session da classe list_prompts() retornando uma lista de objetos

Prompt , que possuem as propriedades de definição de prompt listadas no

parágrafo anterior. Quando você recebe um prompt específico com get_prompt() , você fornece o nome e os argumentos do prompt e

recebe uma lista de objetos PromptMessage, que são mensagens prontas

para uso que podem ser convertidas em um dicionário simples e enviadas

diretamente para o LLM. Observe o exemplo de integração com o host na

última seção, prestando muita atenção à estrutura das mensagens enviadas

ao LLM. Elas são dicionários com chaves role e content, e essas são as

duas propriedades obrigatórias de um objeto PromptMessage. O

conteúdo em si pode ser TextContent, ImageContent,

AudioContent ou um `EmbeddedResource`. Recursos incorporados são

exatamente o que você imagina: recursos que são inseridos em um bloco de

conteúdo da mensagem, assim como o aplicativo host de exemplo criado na

seção anterior.

Vamos dar continuidade ao padrão das duas seções anteriores e encapsular

esses dois métodos em nosso próprio cliente:

**Aprofundando-se**

Na base do MCP, assim como nas diversas implementações do protocolo

nos SDKs, existe um sistema que envia e recebe mensagens entre clientes e

servidores. No SDK Python, todos os métodos de sessão que encapsulamos

envolvem um método ` send_request()`. Este método pode ser

encontrado em ` mcp.shared.session.BaseSession`, do qual as

classes `Session` específicas do cliente e do servidor herdam. Por isso,

teoricamente, você também poderia enviar suas próprias requisições

personalizadas para um servidor que as suporte, mas precisaria criar suas

próprias classes `Request`, `RequestParams` e `Result` para isso.

Existem alguns outros métodos não documentados que você pode usar no

objeto de sessão do cliente. send_ping() envia uma solicitação de ping

para o servidor conectado. O servidor retornará um EmptyResult dentro

de um ServerResult, que não contém dados, mas indica a presença de

uma conexão ativa com o servidor. set_logging_level(), dado um

objeto LoggingLevel da biblioteca types do MCP, pode alterar o nível

de registro do servidor para a conexão, se o servidor permitir. Os registros iniciados pelo servidor são enviados ao cliente por meio de notificações; no

entanto, diferentemente das notificações de assinatura, as notificações de

registro têm seu próprio parâmetro no construtor ClientSession,

chamado logging_callback. Você implementaria sua própria função

de retorno de chamada de registro que segue o protocolo LoggingFnT

(código-fonte) e a passaria para o construtor através do parâmetro

logging_callback :

**Fornecendo recursos de cliente MCP**

Na seção anterior, você viu um exemplo de implementação de um callback

para lidar com mensagens de log de um servidor MCP conectado. Nesta,

você verá esse padrão repetido, mas para fornecer recursos de cliente ao

servidor MCP conectado. Existem dois recursos principais que um cliente

pode fornecer a um servidor: amostragem, onde o servidor pode solicitar

conclusões de chat do LLM ao qual o cliente está conectado, e acesso root,

que indica aos servidores a quais áreas do sistema de arquivos do aplicativo

host o servidor tem acesso.

Os servidores não são obrigados a usar ou respeitar essas funcionalidades,

mas se o seu cliente as fornecer, você é obrigado a anunciar que o seu

cliente as oferece, o que ocorre durante a fase de inicialização da conexão

cliente-servidor. Para fornecer funcionalidades ao cliente, o fluxo de

trabalho geral é o seguinte:

1\. Na sua classe cliente, implemente uma função de retorno de

chamada que será invocada quando o servidor fizer a solicitação.

2\. Certifique-se de que o retorno de chamada siga o protocolo

estabelecido para a capacidade, como SamplingFnT.

3\. Passe a função de retorno de chamada para o construtor

ClientSession através do parâmetro apropriado, como

sampling_callback para amostragem.

E é isso. Claro que a maior parte do esforço se concentra na etapa 1 e será

única dependendo do seu caso de uso e fluxo de interação.

**Amostragem**

A amostragem é uma funcionalidade do cliente que permite ao servidor

MCP utilizar qualquer LLM ao qual o cliente e seu aplicativo host estejam

conectados. Isso possibilita fluxos de trabalho agentes a partir do servidor

MCP, como permitir a escolha e o uso de ferramentas aninhadas, ou que o

servidor faça uma pergunta ao LLM e use essa resposta para escolher ou

preencher informações em um prompt fornecido, ou qualquer outra coisa

que sua imaginação possa conceber. Como essa é uma funcionalidade

fornecida pelo cliente, o trabalho necessário é um pouco menor do que o

necessário para usar funcionalidades fornecidas pelo servidor. Assim como

você viu com o registro de logs, a classe ClientSession fornece um

protocolo SamplingFnT que define a assinatura da função de retorno de

chamada de amostragem que você implementará, e o construtor possui um

parâmetro sampling_callback que receberá sua função de retorno de

chamada a ser usada quando um servidor conectado enviar uma solicitação

de amostragem.

**AVISO**

Devido ao risco de conceder acesso a um servidor externo ao LLM que seu aplicativo utiliza e pelo qual você está pagando, tanto eu quanto a Anthropic recomendamos fortemente que você implemente algum tipo de processo de aprovação com intervenção humana (HITL) antes que um servidor possa enviar uma solicitação a um LLM por meio de seu cliente e aplicativo.

Neste exemplo, seguiremos o fluxo de trabalho geral acima, criando um

retorno de chamada de amostragem simples que implementa a interface

SamplingFnT e passando-o para o construtor ClientSession:

**Suporte a múltiplos modelos**

Uma das principais vantagens de usar o MCP é que ele oferece a liberdade,

como desenvolvedor de aplicativos, de permitir que seus usuários utilizem

diversos modelos ou o modelo de sua escolha. Com o MCP, você não fica

restrito a um único modelo, apesar dos exemplos neste capítulo utilizarem

um modelo Antrópico. A maneira mais simples de implementar isso seria

escrever uma função de tradução em seu cliente para cada par primitivo-

família de modelo, que converta os objetos MCP para o formato esperado

pela família de modelo em questão. Você pode aprimorar esse design básico

criando uma classe para cada primitiva que possa ser construída a partir do objeto MCP e que tenha um método de tradução como membro dessa

classe. O código abaixo faz exatamente isso para ferramentas:

**Utilizando vários servidores**

Todos os exemplos até agora mostraram seu cliente se conectando a um

único servidor. E embora um único cliente só possa se conectar a um único

servidor, o SDK do Python fornece uma classe ` ClientSessionGroup`

para ajudar a gerenciar várias sessões simultaneamente dentro de um

cliente. Ele faz isso conectando-se a um ` ClientSession` existente com

` connect_with_session()` ou a um servidor criando um novo

` ClientSession` com `conenct_to_server()` e, em seguida,

carregando todos os elementos/componentes (ferramentas, recursos e

prompts) oferecidos pelo servidor conectado nas propriedades do objeto. As

sessões também são armazenadas no objeto `ClientSessionGroup`,

juntamente com um mapeamento de cada ferramenta para a sessão

apropriada. Todos esses elementos podem ser acessados por meio das

propriedades sessions, prompts, resources e tools, e as

ferramentas podem ser chamadas da mesma forma que seriam em um único

ClientSession: com call_tool(). Você também pode se

desconectar de um servidor individual com o método

disconnect_from_server() , que também removerá quaisquer

componentes de servidor carregados. No exemplo abaixo, criaremos um

cliente simples que se conecta a servidores por meio de um

ClientSessionGroup .

Do ponto de vista do cliente, isso é mais simples, já que o

ClientSessionGroup lida com todo o gerenciamento de conexões que

o cliente de servidor único fazia nos exemplos anteriores. Em vez de uma

propriedade _session que é None até que uma conexão seja

estabelecida, instanciamos um objeto ClientSessionGroup e o

armazenamos em _session_group. Alteramos o membro do objeto

connected para uma propriedade dinâmica que verifica se há algum

objeto de sessão na lista sessions do grupo de sessão. Em seguida,

temos o método connect, que simplesmente recebe um objeto

ServerParameters . Este é, na verdade, um alias de tipo Python para

StdioServerParameters, SseServerParameters ou

StreamableHttpParameters . Usamos isso para conectar a um único

servidor, e todo o gerenciamento de conexão é feito pelo grupo de sessão. A

desvantagem é que ainda não há suporte para incluir callbacks ao

estabelecer os objetos ClientSession individuais. Isso pode ser

contornado acessando as propriedades de callback "privadas" do objeto

ClientSession, mas isso deve ser usado com moderação.

O construtor ClientSessionGroup também aceita um gancho opcional

no parâmetro component_name_hook, que serve para renomear um

componente de forma a evitar conflitos de nomes. Trata-se de uma função

que recebe uma string (o nome do componente) e um objeto

Implementation, e retorna outra string (o componente renomeado).

protocolo, e como fazer com o SDK do Python?

a. Se o cliente desejar retomar a conexão após uma

interrupção, ele DEVE enviar uma solicitação HTTP GET

para o endpoint MCP e incluir o cabeçalho Last-

Event-ID para indicar o último ID de evento recebido.

– O servidor PODE usar este cabeçalho para

reproduzir mensagens que teriam sido enviadas

após o último ID de evento, no fluxo que foi

desconectado, e para retomar o fluxo a partir

desse ponto.

– O servidor NÃO DEVE reproduzir mensagens

que teriam sido entregues em um fluxo diferente.
