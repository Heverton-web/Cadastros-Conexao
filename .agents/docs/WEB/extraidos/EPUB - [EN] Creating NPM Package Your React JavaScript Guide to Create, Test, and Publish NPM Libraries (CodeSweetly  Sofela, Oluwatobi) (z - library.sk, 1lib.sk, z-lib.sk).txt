Introduction

Welcome to NPM Package!

NPM is a database (storage) for thousands of packages. It is a tool  
developers use to share, manage, find, and install libraries.

Tip:

Packages are sometimes called libraries, plugins, frameworks, or  
tools.

A package is a directory (or project) that has a package.json file used to  
record information about it.

Publishing NPM packages is a fantastic way to share your code with others  
and become a better developer. It is a vital step in contributing to the  
developer community and showcasing a variety of skills, such as:

communication skills  
collaboration skills  
organizational skills  
technical skills  
documentation skills  
problem solving skills

Becoming an NPM package publisher is a unique way to have a finished  
product others can use as a part of or the main component of their apps.

This beginner-friendly book discusses everything you need to know to  
create, test, and publish NPM packages like a pro.


How This Book Can Help You

The best way to learn is through progressive practice. In other words, the  
easiest way to become a proficient package publisher is to proactively build  
NPM plugins—rather than merely reading or watching videos about them.

As such, this book is intentionally project-oriented to get you to practice as  
much as possible while learning the fundamental concepts of becoming an  
NPM package publisher.

Therefore, you will do yourself much favor by opening your text editor  
(such as VS Code) and code along as you read this book. By so doing, you  
will avoid being part of those stuck in “Tutorial Hell.”

The Roadmap

This book’s main project is a Tweet Button—which you can scale up to  
whatever magnitude your imagination can afford.

Don’t let the project type scare you off! You are on for an exciting ride—  
where you will learn the fundamental concepts needed to publish any NPM  
package. So, give it a chance.

Prerequisite

Familiarity with Git, GitHub, React, and JavaScript will help you learn  
NPM package publishing better. So, to be better prepared for this book’s  
content, it’s best to be familiar with the elementary aspects of those tools.

Conventions Used in This Book

Some of the topographical conventions you will find in this book are:

Monospaced font: Used for code elements, filenames, and file extensions.

Italic: Used for captions.

Underline: Used to highlight link text.

Let’s Get It Started!

The following chapter will discuss the file and system setup required to  
create an NPM package.


1

Required File and System Setup

Below are the required system and file configurations for creating NPM  
packages.

1\. Get Git, Node, and NPM

Make sure you have Git, Node 10.16 (or greater), and NPM 5.6 (or greater)  
installed on your system.

Tip:

Use the Git tutorial and package manager articles to install, update,  
or verify Git, NPM, and Node on your system.

2\. Create a Project Directory

Create a new folder for your project like so:

Command Line

mkdir thank-you-tweet-button-001

Note:

You can use any name you prefer. For instance, I’ve chosen to use  
thank-you-tweet-button-001 for this tutorial.

Afterward, navigate to your project directory using the command line.


Command Line

cd path/to/thank-you-tweet-button-001

3\. Create a package.json File

Use NPM to initialize a package.json file for your project.

Command Line

npm init -y

Let’s also initialize Git.

4\. Initialize a Git Repository

Create a .git repo in your project’s root directory :

Command Line

git init

You now need to specify the files you want Git to ignore.

5\. Specify the Files Git Should Ignore

Create a .gitignore file in your project’s root directory:

Command Line

touch .gitignore

Afterward, open the newly created .gitignore file and write the names of  
the files, folders, or file types you want Git to ignore.


Here’s an example:

.gitignore

/node_modules

/dist

The snippet above instructs Git to ignore tracking the current directory’s  
node_modules and dist folders.

Note:

A current directory is the folder in which you are currently  
working.  
A current directory is sometimes called a “current working  
directory (CWD)” or “working directory.”

Helpful resource:

See the .gitignore file example to learn more about gitignore .

It’s now time to stage and commit your recent changes.

6\. Stage and Commit Your Project’s  
Changes to Git

Enter the following command on your terminal to stage and commit your  
recent changes.

Command Line

git add -A && git commit -m "Initialize project using CodeSweetly's guide"

The command above tells Git to stage and commit all modified and  
untracked files in the project.

Let’s now configure a remote repository for the project.


7\. Configure a GitHub Remote Repository

1\. Go to the GitHub website and sign in or sign up if you do not

have an account.  
2\. After signing up for an account, create a home (a repository) in

GitHub for your Git repository. (You can use thank-you-tweet-  
button-001 as the repo’s name or any other name you prefer.)  
3\. Once you’ve created a remote repository for your package, link

your project’s .git directory (located locally on your system)  
with the remote repository on GitHub. To connect to the remote  
repository, go to your package’s root directory via your local  
terminal and run the git remote add command. Here’s the  
syntax:

git remote add origin https://github.com/your-username/remote-repo-

name.git

Note:

Replace your-username in the code above with your GitHub  
username. Likewise, replace remote-repo-name with your remote  
repository’s name.

Helpful resource:

GitHub Docs’ guide on creating a GitHub repository

You can also add the remote repo to your package.json file for easy access  
to people who want to contribute to your project.

package.json

{

"name": "thank-you-tweet-button-001",

"version": "1.0.0",


"repository": {

"type": "git",

"url": "https://github.com/your-username/app-repo-name.git"

},

"scripts": {

"test": "echo \"Error: no test specified\" && exit 1"

}

}

Let’s provide the remote repo’s Issues URL as the package’s bug tracker.  
And an email people can use to report issues:

package.json

{

"name": "thank-you-tweet-button-001",

"version": "1.0.0",

"repository": {

"type": "git",

"url": "https://github.com/your-username/app-repo-name.git"

},

"bugs": {

"url": "https://github.com/your-username/app-repo-name/issues",

"email": "your-project-email@host.com"

},

"scripts": {

"test": "echo \"Error: no test specified\" && exit 1"

}


}

Afterward, stage and commit your changes:

Command Line

git add -A && git commit -m "Create repository and bugs fields"

Let’s now push the local Git repository upstream.

8\. Push Your Local Git Directory to the  
Remote Repo

After successfully connecting your local directory to the remote repository,  
you can begin to push (upload) your local project upstream. Here’s how:

Command Line

git push -u origin main

The command above instructs Git to push your local main branch’s .git  
directory to the remote origin branch on GitHub.

Note:

Refreshing your remote repository’s page should now reflect your  
upload.

The next step is to set up the project with React. So, let’s do that now.


2

Configuring React

Install React as a dev-dependency:

Command Line

npm i -D react

The command above tells NPM to install react as your package’s dev-  
dependency.

Note:

The react package is a library containing React’s core  
functionality for creating React components.  
i is the shorthand notation for install .  
The -D flag is the shorthand notation for --save-dev .

But why install React as a dev-dependency since users need it in  
production? Let’s discuss the reason.

Why Install React as a Package’s Dev-  
Dependency?

Installing React as a dev-dependency rather than a dependency package  
prevents making it mandatory for users to download React while installing  
your package.

React requires having only one copy of the react package in a project.


Therefore, specifying React as a dev-dependency prevents making it  
mandatory for users to download React while installing your package. So, if  
their project already has a React copy, they wouldn’t need to install an  
additional copy.

A caveat of the dev-dependency installation is that your package will only  
work for projects having the React package preinstalled. However, there’s a  
good workaround:

Duplicate the   
"react" property of your   
package.json ’s  
"devDependencies" field into the "peerDependencies" like so:

package.json

{

"devDependencies": {

"react": "^18.2.0"

},

"peerDependencies": {

"react": "^18.2.0"

}

}

Specifying React as a peer-dependency tells package managers to check  
whether   
the   
app   
installing   
your   
package   
contains   
the   
listed  
peerDependencies . If so, the application has the dependencies your  
package needs to work.

But suppose the package manager could not find the peerDependencies . In  
that case, some NPM versions (like versions 7 and 8) will install them  
automatically. But other versions (like versions 3 to 6) will display a  
warning showing the dependencies your package needs to work, which the  
user must install manually.

Tip: dependencies vs. devDependencies vs. peerDependencies


"dependencies" specify the packages a project depends on in  
production (when your app is running on a live server).  
The "devDependencies" field lists all the packages a project  
does not need in production—but requires for its local  
development and testing purposes.  
"peerDependencies" list all the packages a project expects its  
host application to have in its node_modules directory.

So, now that you’ve installed React, we can discuss writing your  
component’s test case.


3

Testing React Component

Below are the steps required to write your component’s test case.

1\. Install the Testing Tools

Install your project’s test packages:

Command Line

npm i -D jest @testing-library/react

jest is the test runner tool we use to run test scripts and print the  
test results on the command line.  
@testing-library/react is the React Testing Library, which gives  
you the APIs you need to write test cases for your React  
components.

2\. Specify Jest as Your Project’s Test  
Runner Tool

Update your package’s test script by specifying jest as your project’s test  
runner tool.

package.json

{

"name": "thank-you-tweet-button-001",


"version": "1.0.0",

"scripts": {

"test": "jest"

}

}

3\. Install the Code Compiler

Jest does not currently recognize JSX syntaxes. So, you need a compiler,  
like Babel, to transform such code into plain JavaScript.

Command Line

npm i -D @babel/core @babel/preset-env @babel/preset-react @babel/cli

The above command will install @babel/core ,   
@babel/preset-env ,  
@babel/preset-react , and @babel/cli .

@babel/core is a library containing Babel's core functionality.  
@babel/preset-env contains a predetermined set of plugins for  
transforming ES2015+ syntaxes into the JavaScript code  
browsers understand.  
@babel/preset-react   
contains   
predetermined   
plugins   
for  
transforming   
React   
syntaxes   
into   
browser-compatible  
JavaScript code.  
@babel/cli is a CLI tool that allows you to run Babel  
commands in your terminal.

Note:

Jest comes out of the box with the babel-jest plugin, which it  
uses to load your Babel configurations and transform the  
project's JSX code to JavaScript.


Suppose you are using additional code transformers—for  
example, a CSS transformer. In that case, create a Jest  
configuration file and explicitly include the default babel-jest  
plugin alongside other transformers like so:

jest.config.json

"transform": {

"\\\\.[j]sx?$": "babel-jest",

"\\\\.css$": "some-css-transformer",

"\\\\.other$": "some-other-transformer"

}

The "\\\\.[j]sx?$": "babel-jest" field tells Jest to use babel-jest to load your  
Babel configurations and transform all .js and .jsx files to plain  
JavaScript. This configuration is required only if you are using additional  
code transformers.

Let's now configure Babel.

4\. Configure Babel to Compile Your Code

Create a babel.config.json file in your project's root directory.

Command Line

touch babel.config.json

Afterward, add the following configuration to the newly created file:

babel.config.json

{

"presets": [


["@babel/preset-env", { "targets": "defaults" }],

"@babel/preset-react"

]

}

The configuration above tells Babel to use the @babel/preset-env and  
@babel/preset-react packages to transform your code to a browser-  
compactible JavaScript syntax.

The "targets": "defaults" option tells Babel to convert your code to syntax  
that is compatible with Browserslist’s default browsers.

Tip:

Run the npx browserslist command to see the browsers Browserlist  
selected for your targets .

5\. Configure Jest’s Testing Environment

Node.js is Jest’s default testing environment. So, you need to specify a  
browser-like environment when creating packages for the web. Therefore,  
install the jest-environment-jsdom library:

Command Line

npm i -D jest-environment-jsdom

Afterward, create a jest.config.json configuration file in your project's root  
directory—if you do not have one yet.

Command Line

touch jest.config.json

Then, add the following configuration to the newly created file:


jest.config.json

{

"testEnvironment": "jsdom",

"moduleFileExtensions": ["js", "jsx", "json", "node"]

}

The "testEnvironment" field tells Jest to use the "jsdom" test  
environment.  
The "moduleFileExtensions" field lists your project's file  
extensions. So, whenever you require() or import modules  
without specifying a file extension. In that case, Jest will use the  
extensions in this list in order from left to right.

Afterward, update your package.json ’s test script:

package.json

{

"name": "thank-you-tweet-button-001",

"version": "1.0.0",

"scripts": {

"test": "jest --config jest.config.json"

}

}

The --config jest.config.json code specifies the configuration file Jest  
should use to execute the tests.

Helpful resource:

Use the Jest doc to learn more about configuring a test environment


4\. Create Your Code Files

Create the following files inside your project’s root directory.

TweetButton.jsx

4

Using Conventional Commits

“Conventional Commits” is a standardized pattern of writing commit  
messages that are readable to humans and machines.

Syntax of Conventional Commits Messages

Here is the Conventional Commits syntax for writing a commit message:

type(scope): summary

type is a noun, such as fix, feat, and chore, that specifies the  
type of changes you are committing.  
scope is optional. It indicates the package or file where the  
change happened. When used, make sure you enclose it in  
parentheses.  
The terminal colon ( : ) and space end the type and scope  
specification.  
summary is a short description of the commit.  
The type , scope , and summary are sometimes called “the  
commit message’s header.”

Tip: The 11 commit “type” developers often use

build : for commits that impact your app’s build step or external  
dependencies—for instance, an update to your NPM package.


chore : for commits that implements maintenance-related  
updates, such as updating your .gitignore file or an existing  
API key.  
ci : for commits that modify your CI (continuous integration)  
configurations.  
docs : for commits that modify your project’s documentation.  
feat : for commits that add a new feature to your codebase. This  
commit type is synonymous with a MINOR release in Semantic  
Versioning (SemVer).  
fix : for commits that add bug fixes to your codebase. This  
commit type is synonymous with a PATCH release in Semantic  
Versioning (SemVer).  
perf : for commits that improve your app’s performance.  
refactor : for commits that refactor specific parts of your  
codebase—for instance, renaming a function.  
revert : for commits that revert your codebase to a previous  
commit history. You may also add the commit SHAs you are  
reverting to the footer as references—for instance, Refs:  
7b804hrw, b394c306.  
style : for commits that modify your code’s style, such as  
adding semi-colon, white-space, or indentation.  
test : for commits that modify your app’s test code.

Helpful resources:

An overview of the three main parts of a Git commit message  
Wikipedia article on continuous integration and delivery  
Learn about Semantic Versioning

Most developers use plugins to enforce writing commit messages using the  
Conventional Commits format. By so doing, you will avoid forgetting to  
standardize your messages. So, let’s also add some enforcements to our  
workflow.

Format

By

@commitlint/config-conventional   
uses   
the

Command Line





It is easy to skip using standardized commit messages because Git hooks  
are strictly local to a project’s repository, which means anyone having  
access to the repo can alter the hooks.

In other words, Git never copies hooks from one repo to another, even when  
you run the git clone command. Therefore, hooks are not shareable and  
can be altered by anyone accessing the local repo. Thus, each developer can  
opt out of standardizing their commit messages.

So, suppose you wish to enforce a specific commit message convention  
amongst a team of developers. In that case, it’s best to set up GitHub  
Actions in your CI/CD workflow to make sure that only your specified  
commit convention gets pushed successfully to the project’s remote  
repository. By so doing, you will encourage all team members to  
standardize their commit messages.

A distribution code is the minified and optimized version of the  
source code. In other words, a distribution code refers to the  
JavaScript file a compiler generates automatically for your  
project.

Updating Package’s Versions

The best publishing practice is to update your package’s version number  
whenever you make any significant change. By so doing, you will help  
users understand the type of changes in your library’s version releases.

Here is the syntax for updating your package’s version number:

npm version update-type

The command above tells NPM to update your package’s version number to  
the specified update-type .

Note:

Replace update-type with your package’s semantic versioning  
release type( patch , minor , or major ).  
Use patch when making backward-compatible changes to your  
package without introducing new features.  
Use minor when you introduce new features that are backward  
compatible.  
Use major when you make any update that is not backward  
compatible.  
Backward-compatible means a specific version of a software (or  
package) can use all its preceding version’s APIs (such as  
invocators) without throwing any error.


Helpful resources:

NPM documentation on semantic versioning.  
  
Learn the differences between declaration, initialization, and  
invocation.

How Release-It Determines Your Package’s Latest  
Version

Out of the box, release-it determines your project’s latest version through  
either of the following ways:


1\. If your project has a package.json file, release-it will use its

"version" field’s value. You can tell release-it to skip this  
process.  
2\. If your project does not have a package.json file but has a Git

tag. In that case, release-it will use the latest Git tag.  
3\. Suppose your project has neither a package.json file nor a Git

tag. In such a case, release-it will use 0.0.0 as the latest  
version.

You can alternatively use a plugin to overwrite release-it ’s default mode of  
determining the latest version. Let’s discuss how.

.release-it.json

"plugins": {

"@release-it/conventional-changelog": {

"preset": "conventionalcommits"

}

},


"git": {

"commitMessage": "chore: release v${version}"

}

}

The snippet above tells release-it to use the @release-it/conventional-  
changelog plugin with a conventionalcommits preset to determine the  
recommended version.

Note:

The preset field defines the commit message format you are using  
for your project.

Why Does CHANGELOG Ignore Some Commits?

The CHANGELOG file ignores some commits because the @release-  
it/conventional-changelog plugin comes pre-configured with the following  
types options:

"types": [


{"type": "feat", "section": "Features"},

{"type": "fix", "section": "Bug Fixes"},

{"type": "chore", "hidden": true},

{"type": "docs", "hidden": true},

{"type": "style", "hidden": true},

{"type": "refactor", "hidden": true},

{"type": "perf", "hidden": true},

{"type": "test", "hidden": true}

]

The types preset states:

1\. The commit types you wish to show or hide in the generated

CHANGELOG .  
2\. The title release-it should use for the section of each commit

type you’ve chosen to display.

In other words, release-it ’s default setting hides chore commit types.

Therefore, by default, the CHANGELOG file will ignore any commit  
message you prefix with a chore type. But you can overwrite the default  
configuration. Let’s discuss how.

How to Overwrite Changelog’s Default Setting

You can overwrite CHANGELOG ’s default configuration by using  
release-it ’s configuration file to specify the "types" you wish to overwrite.

Here’s an example:

.release-it.json

"plugins": {


"@release-it/conventional-changelog": {

"header": "# Changelog",

"infile": "CHANGELOG.md",

"preset": {

"name": "conventionalcommits",

"types": [

{ "type": "build", "section": " Dependency" },

{ "type": "chore", "section": " Chores" },

{ "type": "docs", "section": " Documentation" },

{ "type": "feat", "section": " New Feature" },

{ "type": "fix", "section": " Bug Fix" },

{ "type": "refactor", "section": " Code Refactoring" },

{ "type": "style", "section": " Styling and Formatting" },

{ "type": "test", "section": " Code Testing" }

]

}

}

}

Note:

You only need to specify the default options you wish to  
overwrite.  
The types field is based on the Conventional Changelog  
Configuration Spec.

Automating GitHub Releases

Here are the steps to auto-generate your project’s GitHub releases.

1\. Get a GitHub Personal Access Token

Generate a GitHub personal access token from  
https://github.com/settings/tokens/new?scopes=repo&description=release-  
it.

Note:

release-it needs access only to the “repo” scope—not “admin”  
or other scopes.  
You will later use the generated token to grant release-it access  
to your project’s GitHub repo.

2\. Copy the Generated Token

Copy the token once you’ve generated it and save it on your system—for  
instance, in your Word document. You will paste it somewhere else soon.

3\. Create a GitHub Environment Variable

Here’s how to create the environment variable you will use to auto-generate  
your package’s GitHub releases:

Go to the main page of your package’s GitHub repo and click  
the “Settings” tab.


Figure 19‑1. Red arrow pointing to the “Settings” tab of the package’s GitHub repo.

Click on the left sidebar’s “Secrets and variables” menu.

Figure 19‑2. Red arrow pointing to the “Secrets and variables” menu of the package’s Settings.

Click on the “Actions” option of the “Secrets and variables”  
menu.


Figure 19‑3. The red arrow points to the “Actions” option of the package’s Secrets and variables  
menu.

Click the “New repository secret” button on the “Actions  
secrets and variables” page.


Figure 19‑4. Red arrow pointing to the “New repository secret” button of the package’s Actions  
secrets and variables page.

Enter “RELEASE_IT_GITHUB_TOKEN” in the “Name” field.  
(Note: You can use any name here, but for this tutorial, let’s use  
“RELEASE_IT_GITHUB_TOKEN”).  
Also enter the personal access token you generated previously  
in the “Secret” field.  
Then, click on the “Add secret” button.

Figure 19‑5. Red arrow pointing to the “Add secret” button of the package’s New secret page.
