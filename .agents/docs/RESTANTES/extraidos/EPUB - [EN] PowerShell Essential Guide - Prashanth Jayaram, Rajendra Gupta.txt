Evolution of PowerShell PowerShell is a shell and scripting language developed by Microsoft. It has evolved over time and has become an essential tool for many IT professionals. Its robust and flexible feature set continues to be a leading tool for system administrators and IT professionals in the years to come. Microsoft invented PowerShell, and its initial design and development was led by Jeffrey Snover, a Technical Fellow at Microsoft. Jeffrey Snover and his team started working on PowerShell in the early 2000s, with the first version of PowerShell (then known as “Monad”) being released in 2006. The primary goal of PowerShell was to provide a modern, object-oriented scripting and automation platform for Windows that could compete with the Unix shells and tools that were popular in the IT industry at the time. Microsoft recognized the need for a more robust and flexible tool for managing and automating Windows systems, and PowerShell was developed to meet this need. PowerShell is an automation and configuration management framework developed by Microsoft. It was first introduced in 2006 as Windows PowerShell 1.0; it has evolved into a more versatile and powerful tool. It had basic features like a cmdlet, pipeline, and remoting support. Windows PowerShell 2.0 (2009): This release included new features such as the introduction of background jobs, advanced functions, and transcriptions. Windows PowerShell 3.0 (2012): This release included new features such as workflows, improved module management, and enhanced security features. On the contrary, the Command Prompt has a more limited set of capabilities and is better suited for performing simple tasks or running legacy applications. Object-Oriented Pipeline PowerShell uses an object-oriented pipeline, which means that the output of one command can be passed as input to another command, making it easy to perform complex operations. The Command Prompt, by contrast, uses a text-based pipeline, which is less flexible and less capable. User Interface PowerShell has a more modern and user-friendly interface than the Command Prompt. It supports tab completion, syntax highlighting, and other features that make it easier to use and more intuitive. The Command Prompt has a more basic user interface and is less user-friendly. Cross-platform support PowerShell is available on Windows, Linux, and macOS, providing a consistent experience across platforms. The Command Prompt is only available on Windows. Automation PowerShell is designed for automation and configuration management Command Prompt is better suited for simple tasks or running legacy applications. The precedence order is as follows: The execution policy set in the Process scope has the highest precedence. This setting applies only to the current PowerShell session and is not saved. The execution policy set in the CurrentUser scope has the next highest precedence. This setting applies to the current user and is saved for all PowerShell sessions opened by that user. The execution policy set in the LocalMachine scope has the lowest precedence. The setting applies to all users on the local machine and is saved for all PowerShell sessions opened on the machine. If there are conflicting execution policy settings in different scopes, the setting with the highest precedence takes precedence. For example, suppose the execution policy is set to “RemoteSigned” in the CurrentUser scope but set to “Restricted” in the LocalMachine scope. In that case, the “RemoteSigned” setting will be applied because it has higher precedence. Pipelines in PowerShell Pipelines in PowerShell are a powerful feature that allows you to chain together commands to perform complex operations with minimal typing. A pipeline is a sequence of commands separated by the pipeline operator |. The output of one command is passed as input to the next command in the pipeline. Please refer to the following figure: Figure 5.1: PowerShell modules introduction PowerShell module versus script/function PowerShell modules, scripts, and functions are all ways to package and reuse code in PowerShell, but they have different characteristics and uses. A PowerShell script is a file containing a series of PowerShell commands that are executed in sequence. Scripts are typically used to automate tasks and can be run manually or scheduled to run automatically. Scripts are not reusable like modules and cannot be easily imported or shared with others. PowerShell script: 1. $files = Get-ChildItem C:\Users\pjayaram\Documents -Recurse -File 2. foreach ($file in $files) { 3. $content = Get-Content $ file.FullName 4. $ lineCount = ($content | Measure-Object -Line ).Lines 5. For each file, it retrieves the content and counts the number of lines, outputting the file name and line count to the console. A PowerShell function is a reusable piece of code that performs a specific task or set of tasks. Functions are defined within a script or module and can be called from other scripts or modules. Functions can have parameters and return values and can be imported and shared with others. PowerShell function: 1. $files = Get-ChildItem C:\Users\pjayaram\Documents -Recurse -File 2. foreach ($file in $files) { 3. $content = Get-Content $ file.FullName 4. $ lineCount = ($content | Measure-Object -Line ).Lines 5. Write-Output "$file.FullName contains $ lineCount lines» 6. } This function takes a process name as input and retrieves information about the process, including its name, CPU usage, and working set. A PowerShell module is a self-contained package of reusable code, including functions, variables, classes, and other assets. Modules can have metadata that describes their purpose, dependencies, and other information. Modules are designed to be imported and shared with others and can be distributed through various channels, including the PowerShell Gallery. PowerShell module: 1. # MyModule.psm1 file 2. function Get - MyModuleInfo { 3. Write-Output "This is my custom PowerShell module." 4. } 5. Description = 'A custom PowerShell module.' 12. FunctionsToExport = 'Get-My ModuleInfo' 13. } This module defines a single function, Get-MyModuleInfo , which outputs a message to the console. The module contains two files: the module script file ( .psm1 ) and the module manifest file ( .psd1 ), which have metadata such as the module version, author, and description. The module can be imported into a PowerShell session using the Import-Module cmdlet. Key concepts of the PowerShell module The following are some key concepts and features of PowerShell modules: Module structure : A PowerShell module is typically organized into a folder that contains a module manifest file ( .psd1 ), a module script file ( .psm1 ), and other resources, such as help files ( .xml ) and binary files ( .dll ). Module manifest : The module manifest is a metadata file that describes the module and its contents. It includes the module name, version number, author, description, and dependencies. Module script : The module script file contains the code for the cmdlets, functions, and other resources included in the module. The script is executed when the module is loaded into the PowerShell environment. Module loading : To use a PowerShell module, you must first load it into the PowerShell environment using the Import-Module cmdlet. Once a module is loaded, its cmdlets, functions, and other resources are available in the current session. Module discovery : PowerShell automatically searches for modules in specific locations, including the user’s personal module directory, the system-wide module directory, and any additional directories specified in the PSModulePath environment variable. Module versioning : PowerShell modules can include version numbers to help manage changes and dependencies. Modules with the same name and different version numbers can coexist in the same environment, and specific versions of a module can be loaded using the - Version parameter of the Import-Module cmdlet. Module dependencies : PowerShell modules can depend on other modules by including them in the module manifest or specifying them as required modules in the code. PowerShell automatically loads any required modules. When to create a PowerShell module The decision to create a PowerShell module should be based on several factors related to the requirements and complexity of the project. First, consider whether the code you write will need to be reused frequently or across multiple projects. Creating can save time and effort in the long run by providing a reusable set of functions or cmdlets. Second, consider whether your writing code manages a single object or entity. If it is, a module can provide a clean and modular way to interact with that object or entity, allowing you to manage and manipulate it across different projects or scenarios easily. Third, consider whether the code you write is becoming too complex to contain within a single script. If you break the code down into functions or cmdlets to make it more manageable, then creating a module can help you organize and package those functions in an easier way to maintain and use. Finally, consider whether you need to share the code with others. If you are working on a collaborative project or need to distribute the code to others for reuse, creating a module can simplify that process by providing a self-contained and standardized set of functions or cmdlets. Loading modules You can import and load a PowerShell module using a relative or absolute path. The following is how to do it for both methods: Using relative path : Open PowerShell and navigate to the directory that contains the module file ( .psm1 ) you want to import. Type Import-Module ./ModuleName.psm1 in the PowerShell console and press Enter . Replace ModuleName with the name of your module. Using absolute path : Open PowerShell and locate the full path of the module file ( .psm1 ) you want to import. Type Import-Module C:\FullPath\To\ModuleName.psm1 in the PowerShell console and press Enter. Handling command name conflicts When creating dynamic modules, it is important to be aware of potential conflicts between command names, primarily if your module may be used alongside other modules that may define similar or identical command names. Try-Catch-Finally statement The following is an example of using the try-catch-finally statement for error handling: try { # Some code that might produce an error } catch { # Code to handle the error } finally { # Code that will always run, regardless of whether there was an error or not } In the preceding example, the code inside the try block will be executed, and if an error occurs, the code inside the catch block will be executed to handle the error. The finally block will always run, regardless of whether there was an error or not. This block is typically used to clean up resources, such as closing open files or database connections. The following is an example of error handling in PowerShell using the try-catch-finally statement: try { # Code that might produce an error Get-Content -Path 'C:\Nonexistent\File.txt' -ErrorAction Stop } catch { # Code to handle the error Write-Error -Message "Error: $($_.Exception.Message)" } finally { # Code that will always run, regardless of whether there was an error or not Write-Output "Cleaning up resources..." } Figure 7.28: Error handling in PowerShell In this example, the Get-Content cmdlet is used to read the contents of a file that does not exist. This will cause a terminating error, which is caught by the catch block. The Write-Error cmdlet is used to display an error message that includes the exception message. The finally block is used to clean up resources, such as closing open files or database connections. In this case, we are simply writing a message to the console. You can also use the -ErrorVariable parameter to capture the error in a variable and use it later in the script. The following is an example: try { # Code that might produce an error Get-Content -Path ''C:\Nonexistent\File.txt'' -ErrorAction Stop -ErrorVariable err } catch { # Code to handle the error Write-Error -Message ""Error: $($err.Exception.Message)"" } finally { # Code that will always run, regardless of whether there was an error or not Write-Output ""Cleaning up resources..".…" } In this example, the -ErrorVariable parameter is used to capture the error in the $err variable. The error message is then displayed in the catch block using the $err variable. Trap statement The trap statement is used to handle errors that occur during script execution. It allows you to define a block of code that will execute whenever an error occurs, regardless of where the error occurs in the script. The following is the syntax for the trap statement: trap { # Code to handle the error } In this example, the trap statement defines a block of code that will execute whenever an error occurs. You can replace the comment with any code that you want to execute to handle the error. The following is an example of using the trap statement: trap { Write-Error -Message "Error: $($_.Exception.Message)" } # Code that might produce an error Get-Content -Path 'C:\Nonexistent\File.txt' In this example, the Get-Content cmdlet is used to read the contents of a file that does not exist. This will cause a terminating error, which is caught by the trap statement. The Write-Error cmdlet is used to display an error message that includes the exception message. Note that the trap statement only catches terminating errors, not non-terminating errors. To catch non-terminating errors, you can use the -ErrorAction parameter with a value of Continue, like the following: trap { Write-Error -Message "Error: $($_.Exception.Message)" } -ErrorAction Continue # Code that might produce a non-terminating error Get-ChildItem 'C:\Nonexistent\Folder' In this example, the Get-ChildItem cmdlet is used to get a list of files in a folder that does not exist. This will cause a non-terminating error, which is caught by the trap statement because the -ErrorAction parameter is set to Continue. Try catch or trap which one to use! Both trap and try-catch are used for error handling in PowerShell, but they serve different purposes. The try-catch statement is used to catch and handle errors that occur within a specific block of code. It is useful when you want to handle errors for a specific section of your script and allows you to gracefully handle errors by providing alternate paths for your script to take. On the contrary, the trap statement is used to catch errors that occur anywhere in your script. It allows you to handle errors globally, no matter where they occur in the script. So, whether you use try-catch or trap depends on your specific requirements. If you want to handle errors for a specific section of your script, use try-catch . If you want to handle errors globally for your entire script, use trap. In general, it is a good practice to use try-catch statements for sections of your code that may produce errors and use trap statements for global error handling to ensure that no errors are missed. Common Information Model The Common Information Model (CIM) cmdlets in PowerShell were introduced in version 3.0, which was released in 2012. To address these limitations, Microsoft introduced the CIM cmdlets, which are based on the Common Information Model (CIM) standard, a vendor-neutral data model and methodology for managing IT infrastructure. The CIM cmdlets provide a more modern and robust way to manage IT infrastructure, and they offer several benefits over the older WMI cmdlets. One of the key benefits of the CIM cmdlets is that they use the WS-Man protocol, which is a Web services protocol that is based on industry standards such as SOAP and XML. This allows CIM cmdlets to work over HTTP and HTTPS, which makes it easier to manage remote computers and enables cross-platform management. Another advantage of the CIM cmdlets is that they provide a more consistent and standardized way of working with management information across different vendors and technologies. This allows IT administrators to manage heterogeneous environments more easily and with fewer errors. Since their introduction in PowerShell 3.0, the CIM cmdlets have been improved and expanded in subsequent versions of PowerShell, and they continue to be an important tool for managing IT infrastructure. Calculate the sum of squared differences: Add up all the squared differences. Calculate the variance : Divide the sum of squared differences by the number of data points. The following is an example PowerShell code that demonstrates how to calculate the standard deviation and variance: # Data points $data = 1, 2, 3, 4, 5 # Calculate the mean $mean = ( $data | Measure-Object -Average).Average # Calculate the squared difference $squaredDifferences = $data | ForEach-Object { ( $_ - $mean ) * ( $_ - $mean ) } # Calculate the sum of squared differences $sumSquaredDifferences = $squaredDifferences | Measure-Object -Sum | Select-Object -ExpandProperty Sum # Calculate the variance $variance = $sumSquaredDifferences / $data .Count # Calculate the standard deviation $standardDeviation = [Math]::Sqrt( $variance ) # Output the results Write-Host "Variance: $variance " Write-Host "Standard Deviation: $standardDeviation " Machine learning PowerShell can train and evaluate machine learning models. PowerShell Essential Guide Preface The emphasis on automation is prevalent nowadays. Also, the importance of automation in todayʼs world has driven us to propose this book. In the last decade, PowerShell has propelled in every way in the automation arena. Since the inception of PowerShell, it has become a defacto tool for automation, and it is a favorite solution of many Windows administrators – with the capability to automate almost any task in the Microsoft ecosystem. Since the advent of PowerShell, it has been a lot easier to import the related modules and invoke the associated cmdlets call to take care of many day-to-day mundane activities, from simple to complex maintenance. This book sets its course with insightful introductory chapters, encompassing a holistic view of PowerShell, complementary tools, and foundational concepts. As the narrative unfolds, it delves into advanced components, maintaining a balanced perspective that appreciates the IT administratorʼs stance while acknowledging the significance of automation GUI interfaces. Our objective is to empower readers with practical, hands-on experience across diverse domains, including AD, Database, Cloud administration, Python, and integration tools. The later chapters are straightforward to understand and completely isolated from each section. For every section, the PowerShell code is designed, and readers with no prior experience can jump into the topics and get started with the examples. It would be great to have a hands-on experience with PowerShell; this would help you to progress faster. However, if you have experience with PowerShell, you can jump to a specific chapter or topic in the book. However, in some cases, if you are a beginner, you can start with the basics and build on that foundation. Chapter 1: Introducing PowerShell - As is the tradition, we will take a few glimpses at the introduction to PowerShell, the history of PowerShell, look at what makes PowerShell so flexible and powerful, and introduce ourselves to the different landscapes of PowerShell integrations. Chapter 2: PowerShell Constructs - In the second chapter, we will explore PowerShell constructs. PowerShell constructs are the building blocks of PowerShell scripts, including cmdlets, object-oriented notions, execution-policy, variables, pipelines, and modules. Chapter 3: Munging the Data Through Pipelines - This chapter covers how to munge data through pipelines in PowerShell. Topics include selecting columns from the output, limiting the number of output objects, expanding properties within properties, filtering objects, grouping the output, sorting the output, taking actions on returned objects, understanding pipeline-enabled parameters, and importing content into PowerShell. By the end of the chapter, you will have a deeper understanding of how to use PowerShellʼs cmdlets and features to work with large datasets more efficiently. Chapter 4: Data Control Flow Using Branches and Loops - This chapter covers a wide range of PowerShell control flow and data manipulation concepts. The control flow in PowerShell refers to the ability to define and control the data flow within a script. It involves executing or branching to commands based on the value of data, building logic that uses looping and branching, and creating conditions that determine which commands to execute based on certain conditions. The If statement is one of PowerShell's most used branching constructs. Branching allows you to define multiple paths for your script based on certain conditions. It will enable you to test a condition and execute a block of code based on the test result. Loops are another essential component of control flow in PowerShell. They allow you to execute a code block repeatedly while a particular condition is true. PowerShell supports several loop constructs, including For, ForEach, While, and Do-While. Chapter 5: Learning about PowerShell Modules - In this chapter, you will learn more about PowerShell modules. You will gain a solid understanding of PowerShell modules and learn how to use them effectively. You will learn how to create a module manifest, which provides metadata properties such as the module name, version, and author. You will also learn to specify common module metadata properties in a module manifest file. Furthermore, you will discover how to update an existing PowerShell module to a newer version. This chapter will cover best practices for preventing command name conflicts when working with multiple PowerShell modules. Finally, you will understand how to utilize the PowerShell module path to locate and import modules into a PowerShell session. By the end of this chapter, you will be equipped with the knowledge and skills to use PowerShell modules effectively in your PowerShell projects. Chapter 6: Choosing Between PowerShell Core and PowerShell - This chapter on PowerShell Core covers a range of subjects related to this cross-platform, open-source command-line shell and scripting language designed for automation and configuration management. It includes an introduction to PowerShell Core, a brief note on .NET Core, features of PowerShell Core, how to install it on Ubuntu, understanding the differences between PowerShell and PowerShell Core, information about unsupported modules, learning about OpenSSH and remoting, and a comparison between the commands of Bash and PowerShell. Chapter 7: PowerShell Administration and Scripting - In this chapter, you will learn how to deal with your daily work as an administrator. Also, you will learn how to build scripts to accomplish the automation for repetitive tasks and learn to generate various reports. After the scripts are designed, written, deployed, and executed on a system, the scope can be local, remote, or background jobs. In addition, you will learn how to use the cmdlets to troubleshoot OS issues effectively. Chapter 8: Using the Active Directory Module - Active Directory is the most widely used solution for administrating and managing users and resources within organizations. Managing Active Directory (AD) with Windows PowerShell is more straightforward than many IT professionals think. PowerShell provides a powerful and intuitive management engine allowing for interactive AD management through its console. With PowerShell, you can perform everyday AD management tasks without writing complex scripts. The interactive nature of PowerShell enables IT professionals to execute commands directly in the console, making it accessible to novice and experienced administrators. You can easily leverage PowerShellʼs command-line interface to perform various AD management operations. In this chapter, you will learn more about active directory administration using PowerShell cmdlets. Also, you will see a list of the commonly used cmdlets to manage the AD, and we will discuss a list of typically performed operations. In addition, you will explore cmdlets to see what tasks you can automate. Chapter 9: Building PowerShell GUI for Scripts - In this chapter you will learn, PowerShell is primarily designed to be used from the command line, it does have some graphical user interface (GUI) capabilities that can be leveraged using Windows Forms. To create a GUI with PowerShell, you can use the Windows Forms classes that are part of the .NET Framework. Windows Forms allows you to create windows, controls, menus, and other graphical elements that can be used to create a user-friendly interface for your PowerShell scripts. Chapter 10: Managing Cloud Operations Using PowerShell - This chapter explores the practical application of PowerShell in cloud management. Starting with an introduction to PowerShell and its relevance in cloud operations, the chapter emphasizes the widespread adoption of PowerShell and its benefits. Readers are guided through preparing an Azure environment, loading Azure PowerShell modules, and establishing a connection to Azure. The chapter then covers essential tasks such as deploying Azure VMs, gathering details about VMs, and collecting metrics for monitoring and optimization. Furthermore, it introduces PowerShell modules for managing AWS and GCP resources, empowering readers with knowledge of multi-cloud management. Chapter 11: Understanding PowerShell and Data Science - In this chapter you will learn, PowerShell is becoming increasingly associated with data science and machine learning due to its ability to work with data and integrate with data science tools and frameworks. PowerShell includes several built-in features for working with data, such as support for regular expressions, XML and JSON parsing, and database connectivity. Chapter 12: Administrating Database Using PowerShell - This chapter provides a comprehensive guide to administrating databases using PowerShell, covering various aspects such as understanding SQL Server, leveraging PowerShell modules, working with different data formats, and managing databases in both on-premises and Azure environments.
