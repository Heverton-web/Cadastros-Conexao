Capítulo 4 Sobre o livro Este livro é destinado a devs que desejam aprender a desenvolver aplicativos para Android nativo utilizando o Kotlin e diferentes bibliotecas. Ele apresenta um projeto de exemplo, o DevMon, que é inspirado na Pokédex (dispositivo eletrônico do universo de Pokémon™) e exibe criaturas mágicas com nomes inspirados em diversas tecnologias. O objetivo deste livro é ensinar como programar uma solução completa, além de abordar conceitos avançados do desenvolvimento nativo no Android, incluindo arquitetura, padrões e kits de ferramentas. O livro cobre desde as bases do desenvolvimento de aplicativos para Android, passando por conceitos avançados e mostrando como aplicá-los de forma prática, com base em um projeto real. Além disso, o livro apresenta uma abordagem lúdica e criativa, utilizando criaturas mágicas como elementos de ensino para tornar a leitura mais agradável e divertida. Este livro é uma excelente opção para quem quer entender um pouco mais do ecossistema de desenvolvimento Android, com uma abordagem prática, baseada em projetos reais, de forma lúdica e criativa. O projeto DevMon permite aplicar os conceitos aprendidos de forma concreta e visualizar o resultado final do desenvolvimento. Isso ajuda a solidificar o aprendizado e torna o conteúdo mais facilmente aplicável na prática. Para quem é este livro Este livro é destinado a pessoas que querem ter um aspecto geral do desenvolvimento avançado com Android nativo, usando Kotlin como linguagem de programação. Não é necessário um conhecimento profundo sobre Android para produzir os projetos; entretanto, é importante que já tenha familiaridade com lógica de programação e experiência com alguma linguagem de programação. Pré-requisitos A principal motivação para a criação deste livro é desmistificar o desenvolvimento de aplicativos Android, utilizando a plataforma nativa em Java/Kotlin. É importante ressaltar que, quando optamos por desenvolver nesse ambiente, algumas considerações devem ser feitas para entender o momento em que a tecnologia se encontra e os motivos pelos quais utilizaremos as abordagens presentes neste livro. Como este livro tem o objetivo de trazer diversos aspectos do desenvolvimento avançado para a plataforma Android, assumiremos que alguns conceitos básicos da plataforma Android já foram vistos, mesmo que de forma superficial. Figura 1.3: Qual é o DevMon #1? 1.1 Linguagem de programação O framework para desenvolvimento de aplicações que rodam de forma nativa no Android é construído com a linguagem Java, enquanto o sistema operacional em si foi desenvolvido com a linguagem C. Portanto, sempre que um aplicativo estiver rodando no smartphone, o arquivo binário da aplicação, geralmente na extensão .apk ou .aab , será executado pela JVM ( Java Virtual Machine ), que também foi escrita com a linguagem C. É importante ter um conhecimento da linguagem Java para desenvolver as aplicações que rodam no Android, pois esse é o código que será executado de fato. No entanto, o Java é uma linguagem que foi lançada no ano de 1996 e recebeu muitas atualizações e melhorias desde então. Quando uma linguagem recebe uma atualização, novas funcionalidades são incluídas com diversos objetivos, tais como simplificar a forma de declaração, adicionar funcionalidades novas que não eram possíveis anteriormente, otimizar a performance, entre diversos outros. No Android, para que o suporte a uma versão nova do Java ocorra, é necessário aguardar que esse desenvolvimento seja feito por parte da equipe que mantém o framework Android funcionando. Por isso, precisamos assumir que várias funcionalidades novas inseridas na linguagem Java não estarão disponíveis para utilização . Com isso em mente, podemos assumir que escrever um código de uma aplicação Android em Java fará com que tenhamos que utilizar declarações que são mais verbosas , ou seja, é necessário escrever mais código para desenvolver uma funcionalidade relativamente simples. Apesar de o Java ter evoluído muito como linguagem ao longo dos anos, o Android ainda se encontra preso em versões antigas e declarações que não evoluem tão rápido, principalmente porque temos dispositivos Android espalhados ao redor do mundo que ainda rodam versões antigas do sistema operacional. Por esses motivos, surgem outras linguagens que tentam resolver esse problema mantendo o suporte às versões antigas, mas permitindo a inserção de funcionalidades mais poderosas e que reduzem o trabalho necessário e a quantidade de código para construir uma aplicação: são as linguagens transpiladas . Linguagens transpiladas As linguagens transpiladas, por definição, traduzem o código que foi escrito em uma linguagem para uma outra linguagem, da mesma forma que as linguagens de programação compiladas, como o Java, traduzem um código textual para um código binário, que será executado posteriormente. As principais linguagens que transpilam para o código que roda na JVM hoje são: Groovy Kotlin Scala Clojure Portanto, qualquer linguagem de programação que seja capaz de produzir um arquivo que roda na JVM é compatível com o desenvolvimento Android. Sabendo disso, e considerando todos os problemas mencionados anteriormente, a comunidade de desenvolvimento Android começou uma busca por alternativas à utilização de Java como a principal linguagem, construindo aplicações com linguagens alternativas muito antes de qualquer suporte oficial ser adicionado. Kotlin Percebendo o caminho que a comunidade estava trilhando, o Google, em 2017, anunciou suporte oficial à linguagem de programação Kotlin, uma linguagem relativamente nova, cujo projeto foi iniciado em 2010, divulgado pela primeira vez em 2011 e só se tornou open source em 2012. Um dos principais motivos pela decisão é a empresa por trás da linguagem, a JetBrains, que desenvolveu e mantém o Android Studio (a IDE na qual aplicações Android são desenvolvidas), além de ter desenvolvido outras IDEs famosas que usam a plataforma IntelliJ, como IDEA, PyCharm, PhpStorm, WebStorm, Rider, entre diversas outras. Outro fator bem importante é a interoperabilidade com o Java. Sim! O Kotlin conversa normalmente com o Java! Isso significa que é possível iniciar a escrever Kotlin em um projeto que esteja feito 100% em Java e conseguir executar suas bibliotecas normalmente. Além disso, temos fatores cruciais como: Compatibilidade: totalmente compatível com JDK 6 , o que garante a execução em devices mais antigos; Desempenho: um aplicativo Kotlin é tão rápido quanto um aplicativo em Java, às vezes até ganhando em termos de velocidade; Tempo de compilação: o mesmo caso que o desempenho, velocidade igual ou superior (em alguns casos) ao Java; Curva de aprendizado: para quem desenvolve em Java, é extremamente fácil iniciar em Kotlin. Garanto que, no primeiro projeto usando a linguagem, você já se sentirá apaixonado por ela e saberá usar até os recursos avançados da linguagem — e com aquele sentimento: "Por que ninguém me mostrou isso antes?". Nos próximos capítulos, apresentaremos algumas características da linguagem para auxiliar no entendimento inicial de como desenvolver usando Kotlin. 1.2 Arquitetura Uma das coisas mais importantes (se não a mais importante) de um software é a arquitetura, assunto que é muito discutido entre a comunidade dev. Para entender os caminhos que o Android seguiu, precisamos conhecer um pouco da evolução dos padrões de arquitetura MVC , MVP e MVVM . Conhecer a evolução e a importância de cada padrão é fundamental, pois nos permite entender os motivos de determinados padrões funcionarem melhor em um projeto Android. Dentre os três padrões mencionados, todos eles possuem algo em comum: as letras V e M , que representam View e Model , respectivamente. Sabemos que, no Android, toda aplicação possui telas, criadas a partir da classe Activity , contendo um arquivo de layout em XML e, possivelmente, possuindo Fragments que promovem a reutilização de telas e que separam melhor pequenas funcionalidades de um sistema. Como essas telas exibem informações para o usuário, costumamos chamá-las de Views , pois permitem que a informação seja visualizada. Para que uma aplicação cumpra seus objetivos, muitas vezes é necessário utilizar recursos mais completos (e mais complexos), como integrar com APIs na internet, usar banco de dados, exibir notificações, reproduzir sons, exibir imagens etc. Para que isso ocorra, precisamos criar novos arquivos que contenham o código que fará a funcionalidade desejada, criando uma necessidade de organização desses códigos que também tenha como objetivo facilitar a manutenção deles. Quando essas funcionalidades exigem trabalhar e manusear dados como informações do usuário, lista de produtos disponíveis ou links das imagens, costumamos guardar os códigos que resolvem isso em uma pasta, a qual chamamos de Model , pois contém toda a lógica necessária para resolver esse tipo de problema. Como esse é um problema que todo software possui, independente da sua linguagem, existem soluções aparecendo há muito tempo. O MVC, por exemplo, foi formulado na década de 1970, quando ainda "era tudo mato". Já o MVP foi formulado no ano de 1996, quando a internet estava dando seus primeiros passos. Por fim, o MVVM teve sua primeira aparição em 2005. Devido ao fato de que cada framework tem as suas próprias características, objetivos e problemas específicos a serem resolvidos, o padrão de arquitetura escolhido pode variar. Além disso, novos problemas que surgem ao longo do tempo podem fazer com que a comunidade migre de padrão de arquitetura e tente utilizar outros formatos. No caso dos três padrões mencionados, temos uma camada extra que conecta o Model à View . A diferença é que, em cada padrão, essa camada extra possui características diferentes e todas possuem seus prós e contras. Para entender um pouco mais sobre essas diferenças, vamos ver como cada padrão funciona separadamente. MVC Figura 1.4: Diagrama MVC. O MVC, Model-View-Controller , é um padrão de arquitetura que sintetiza a maneira como as informações de uma aplicação devem estar relacionadas, colocando o usuário em uma das pontas e ditando exatamente o que cada parte do software deve fazer para que o conteúdo chegue até ele — tudo isso utilizando boas práticas de programação, promovendo a reutilização de código e visando a um trabalho mais eficiente. Essa abordagem surgiu em 1979, na Xerox PARC, e é uma das mais conhecidas. É utilizada em diversas plataformas de desenvolvimento e contempla diversas linguagens de programação, principalmente nas aplicações feitas para a web. A View é o componente pelo qual o usuário visualiza todas as informações carregadas pelos componentes citados anteriormente. Uma característica é que ela não pode conter nenhuma lógica e não deve saber de nada do que está acontecendo nos bastidores da aplicação. Ela apenas recebe a informação e a exibe da melhor maneira possível. Ela é atualizada conforme os dados recebidos do Model , e geralmente essa atualização é feita pelo Controller . O Controller , por sua vez, é responsável por solicitar informações do Model conforme a necessidade do usuário. Ele também solicitará atualizações no Model , informando que um cadastro foi atualizado, por exemplo, ou que um registro precisa ser removido. Todos esses gatilhos são realizados a partir da interação do usuário com o sistema, onde o Controller recebe diretamente essa interação e a repassa para o local necessário. Por fim, o Model é responsável pela comunicação com os dados, independente de onde eles estão. Ele vai receber notificações do Controller conforme a necessidade desses dados, vai buscá-los, vai realizar alguma lógica caso haja necessidade e retornar a informação pronta para ser utilizada. Uma frase bem conhecida nesse padrão é " Fat models and skinny controllers ", que significa " models inflados e controllers magros", deixando claro que a camada Model costuma ter mais conteúdo do que a camada Controller . A ideia dessa frase é que a principal lógica dos dados deve estar no Model e não no Controller . No Android: o Model seria composto de classes de acesso ao banco de dados ou de acesso a APIs na web; a View seria o layout XML que exibe os dados diretamente inseridos pela classe da Activity ; e o Controller seria nossa classe Activity e qualquer classe de comunicação que solicitasse informações ao Model . A abordagem pode variar em diversos casos, mas essa seria uma representação básica, na qual não temos uma separação de fato entre Controller e View , onde nossa Activity ou Fragment atuava muitas vezes em ambas as funções. MVP Figura 1.5: Diagrama MVP. O MVP, Model-View-Presenter , é um padrão de arquitetura que deriva do MVC e que é usado na maioria das vezes na construção de interfaces de usuário, como é o caso do Android. Nessa abordagem, temos a substituição do Controller pelo Presenter , que assume a função de "homem-do-meio". Toda a lógica será feita pelo Presenter , que basicamente responde às ações da UI e controla a interação entre a View e o Model . A diferença entre o Controller e o Presenter é que o Presenter fica mais desacoplado da View , o que aumenta o nível de abstração e separa melhor as responsabilidades. Nessa arquitetura, o Model e a View são idênticos ao apresentado no MVC. O Presenter , que entra para substituir o Controller , é responsável por intermediar a relação entre a View e o Model , garantindo que qualquer comunicação necessária passe por ele. Ele recebe os dados fornecidos pelo Model , formata e envia para serem exibidos na View . Em algumas implementações, é comum criarmos um novo componente chamado Supervising Controller , que vai cuidar de casos em que é necessário recuperar ou persistir o Model . No Android: o Model mantém as mesmas classes do MVC; a View passa a possuir a classe da Activity , além dos layouts que já existiam no padrão MVC, e também passa a implementar uma interface View que será reconhecida pelo Presenter ; o Presenter são novas classes que implementam uma interface Presenter que vão atualizar o Model para solicitar os dados, além de guardar a referência da interface View implementada. É preferível que o Presenter não possua nenhuma referência ou código relacionado à API do Android. O MVP se tornou muito popular entre desenvolvedores Android com o passar do tempo; porém, iniciar o desenvolvimento nessa abordagem requer muita escrita de código repetitivo necessário (conhecido como boilerplate ), e algumas práticas, como guardar a referência da View em questão dentro do Presenter , dificultam alguns processos dentro dessa arquitetura, principalmente quando temos um Presenter muito grande e difícil de quebrar em pedaços. MVVM Figura 1.6: Diagrama MVVM. O MVVM, Model-View-ViewModel , também conhecido como Model-View-Binder , é uma das abordagens mais atuais em termos de arquitetura de software, principalmente na comunidade Android, recebendo cada vez mais atenção de devs e pessoas adeptas da Clean Architecture . Essa abordagem tem como característica facilitar a separação de conceitos no desenvolvimento de uma interface de usuário. Ela foi desenvolvida para utilizar funções conhecidas como Data Bindings , que facilitam uma melhor abstração da View do resto do projeto, pois remove boa parte do código de atualização dos elementos da UI. Com isso, elimina-se a necessidade de designers de UX escreverem esses códigos, permitindo que uma linguagem de marcação seja utilizada para isso. A separação dessas tarefas foi um dos pontos pensados durante o desenvolvimento dessa arquitetura: o objetivo era acelerar o processo de desenvolvimento de uma aplicação tanto em projetos grandes quanto em projetos pequenos. Nessa arquitetura, o Model é idêntico ao apresentado no MVC e no MVP. A View ganha uma nova funcionalidade de bind em variáveis "observáveis" e ações presentes no View Model de uma maneira mais flexível. O View Model , que entra para substituir o Controller e o Presenter , é responsável por envolver o Model , preparando variáveis "observáveis" que são necessárias para a View . Ele também funciona como uma ponte para a View enviar eventos ao Model . Uma característica importante desse modelo é que o View Model não possui referência da View , permitindo com mais facilidade que um View Model sirva várias views . No Android: o Model mantém a mesma base do MVC; a única diferença da View é que ela não possui mais uma interface View implementada e passa a ter declarações de observações às variáveis do View Model ; e o View Model possuirá uma implementação da interface View Model que está ciente do ciclo de vida da Activity ou Fragment e ficará responsável pela declaração e atualização das variáveis "observáveis" que serão observadas pela View . O MVVM é um padrão de arquitetura que vem ganhando cada vez mais espaço entre a comunidade Android, recomendado atualmente pelo Google e que ganhou muita força após a Google I/O '17, com o anúncio dos Architecture Components dentro do Android Jetpack (falaremos sobre ele mais à frente). Eles facilitaram muito a sua implementação e permitiram que aplicações de altíssimo nível de arquitetura fossem criadas do zero sem muito esforço inicial, algo que as abordagens anteriores não tinham como principal característica. Qual arquitetura utilizar no Android? No Android (em qualquer framework, na verdade), é possível utilizar qualquer um dos três padrões mencionados, como também utilizar (ou criar) um padrão completamente novo. Entretanto, conceber esses padrões que a comunidade não costuma utilizar pode resultar em muita escrita de código boilerplate — e em alguns problemas desconhecidos que podem surgir pelo caminho. Por conta disso, no Android, o MVC foi um padrão amplamente utilizado no início, mas rapidamente houve uma migração para o MVP, presente em várias aplicações até hoje. Como mencionado anteriormente, o MVVM vem ganhando muita força por conta do Android Jetpack, que consiste em um kit de ferramentas extremamente poderoso que facilita a adoção ao padrão de arquitetura. 1.3 Kit de ferramentas O kit de ferramentas é o último ponto que precisamos considerar no desenvolvimento Android, pois utilizar bibliotecas prontas nos permite focar apenas nos problemas específicos da aplicação que estamos desenvolvendo, o que facilita e acelera o processo de construção. Ter acesso a ferramentas prontas faz com que as decisões sobre qual caminho seguir sejam mais fáceis, pois elas geralmente apontam caminhos que a comunidade está trilhando, para que todos fiquem no mesmo barco. Portanto, optar por utilizar uma ferramenta presente no Android Jetpack costuma ser uma ótima decisão. Dessa forma, garantimos que todos os módulos possuirão as dependências sempre atualizadas. 3.2 Injeção de dependência (DI) Apesar de ser algo novo para muitos devs, esse assunto está presente desde o começo dos anos 1990, com abordagens em diversos livros e para diversas linguagens de programação, e há uma série de bibliotecas capazes de facilitar sua utilização. Esse conceito visa reduzir o nível de acoplamento entre diferentes módulos de um sistema, ou seja, em vez de um módulo que tem uma dependência ser o responsável por satisfazer essa necessidade, isso é feito pela configuração do software, que vai "injetar" em cada componente as dependências que foram declaradas. Essa prática está relacionada com o padrão Inversion of Control (inversão de controle), que consiste em definir quem tem o controle das instâncias de uma aplicação, geralmente delegando a responsabilidade de construção de novas instâncias para uma classe conhecida como Factory , que, além de construí-las, também as reutiliza conforme a necessidade. A injeção de dependência (ou DI, do inglês Dependency Injection ) é uma técnica de programação que pode ser realizada em qualquer linguagem de programação, e de várias maneiras. Em qualquer projeto, seja ele grande ou pequeno, mesmo que nós não tenhamos percebido, existem casos em que objetos dependem de objetos; ou seja, para construir uma nova informação, precisamos que outra informação também seja construída. 3.3 Bibliotecas para DI Existem diversas bibliotecas disponíveis para tratar a injeção de dependência, independente da linguagem de programação ou do framework em que estamos trabalhando. No Java, temos algumas das principais bibliotecas que evoluíram ao longo do tempo, melhorando cada vez mais a utilização e, principalmente, a performance. No ecossistema Android, as principais bibliotecas utilizadas são Guice , Dagger v1 , Dagger v2 , Koin e Hilt . Como podemos notar, injeção de dependências é um assunto com várias opções e diferentes propostas de solução. Atualmente, muitas aplicações Android acabam optando por Dagger 2, Koin ou Hilt, levando os diferentes pontos mencionados em consideração. Por conta principalmente dos fatores checagem de erros em tempo de compilação e performance da aplicação (como é possível ver na imagem a seguir), utilizaremos a biblioteca Hilt neste livro, que nada mais é do que a Dagger 2 por baixo dos panos, com uma configuração simplificada para aplicações Android. Para isso, vamos preparar uma classe que é responsável por acessar esse servidor, chamada CreatureRemoteDataSource , que possui o método getCreatures() , que faz a requisição e retorna a lista de criaturas. class CreatureRemoteDataSource { fun getCreatures () : List&lt;Creature&gt; { val api = CreatureApi() return api.findAll() } } Perceba que, nesse caso, a classe CreatureRemoteDataSource tem a sua primeira dependência, a classe CreatureApi . Como a construção da CreatureApi está sendo feita dentro da classe CreatureRemoteDataSource , criamos um débito técnico que podemos pagar no futuro, pois qualquer alteração no construtor da classe CreatureApi deverá ser atualizada nesse local também. A princípio, pode não parecer um problema tão grande, mas imagine que a classe CreatureApi seja utilizada em 10 outros trechos — aí, sim, começamos a ter um problema maior. Mas, por enquanto, vamos deixar esse problema de lado e evoluir um pouco mais a implementação. Como vimos no exemplo anterior, temos getCreatures() , de CreatureRemoteDataSource , executando findAll() , de CreatureApi . Para que o findAll() consiga retornar a lista de criaturas, é necessário acessar um servidor que está na internet através do protocolo HTTP. Para isso, vamos exemplificar com a biblioteca OkHttpClient , da Square. Assim que o findAll() for executado, ele precisa obter uma instância de OkHttpClient , realizando a implementação de acordo com a documentação da biblioteca. O exemplo a seguir é ilustrativo, mas deve servir para exemplificar uma requisição sendo feita e retornando a lista de criaturas. class CreatureApi { fun findAll () : List&lt;Creature&gt; { val client = OkHttpClient() val request = Request.Builder().url( "https://api.devmon.com/creature" ).build() return client.newCall(request).execute() } } Pronto! Agora que temos tudo declarado, podemos realizar a execução dos métodos e receber a lista atualizada de criaturas. Para ilustrar, faremos isso em um objeto chamado Program , que tem um método principal chamado main() , como no trecho a seguir. object Program { fun main () { val remoteDataSource = CreatureRemoteDataSource() val creatures = remoteDataSource.getCreatures() println(creatures) } } Com isso, nosso primeiro exemplo chega ao fim. Temos o objeto Program criando a instância CreatureRemoteDataSource e solicitando a lista de criaturas. Quando essa lista é solicitada, o método getCreatures() , da classe CreatureRemoteDataSource , cria a instância de CreatureApi , executando o método findAll() . Por fim, o método findAll() cria a instância de OkHttpClient() e realiza a requisição HTTP ao servidor. Evoluindo a aplicação Conforme o tempo passa, novas requisições vão sendo necessárias e nossa aplicação começa a crescer. Em algum momento, vamos precisar implementar um tratamento de erro que sirva para todas as requisições ou alguma outra implementação que precisamos para ter controle da instância do OkHttpClient , por exemplo. Quando esse dia chegar, será necessário procurar por todos os métodos que criam instâncias para realizar a atualização em cada um deles. Nesse momento, teremos que pagar o débito técnico que foi criado no momento em que esse padrão de código começou a ser utilizado. Para evitar que isso ocorra, é preciso mudar quem tem controle da criação das dependências, em vez de cada um criar sua própria instância. Existem algumas formas de fazer isso. Nesse momento, trataremos apenas uma delas, que consiste em não se preocupar com a criação da instância da classe que estamos chamando. Em vez disso, declaramos no construtor qual é a classe necessária para garantir o funcionamento, da seguinte forma: class CreatureApi ( private val client: OkHttpClient) { fun findAll () : List&lt;Creature&gt; { // ... Dessa forma, a responsabilidade de fornecer a instância foi transferida para quem estiver utilizando a classe CreatureApi , que, no caso, é a classe CreatureRemoteDataSource . Agora que temos as dependências da CreatureApi declaradas, precisamos satisfazê-las na classe CreatureRemoteDataSource . Antes disso, vamos dar uma olhada no código. class CreatureRemoteDataSource { fun getCreatures () : List&lt;Creature&gt; { val api = CreatureApi() return api.findAll() } } Como acabamos de adicionar um parâmetro no construtor CreatureApi , precisamos adaptar a construção que está sendo feita dentro de getCreatures() . Para corrigir o problema, podemos fornecer uma instância de OkHttpClient ou mover a criação da instância de CreatureApi para o construtor da classe CreatureRemoteDataSource . Se criarmos a instância de OkHttpClient dentro do método getCreatures() e fornecê-la para a CreatureApi que está sendo criada, resolve o problema para a criação de instância CreatureApi , mas traz uma nova responsabilidade para a classe CreatureRemoteDataSource , que é a de gerenciar a criação da instância de OkHttpClient ; Ao mover a dependência de CreatureApi para o construtor CreatureRemoteDataSource , estamos tirando a responsabilidade da classe de criar essa instância, fazendo com que quem a utilize seja responsável por gerenciar e informar essa instância. Diante desse cenário, vamos resolver da mesma forma que fizemos antes: transferir a responsabilidade da instância da CreatureApi para quem estiver precisando da classe CreatureRemoteDataSource , movendo a declaração para o construtor. class CreatureRemoteDataSource ( private val api: CreatureApi) { fun getCreatures () : List&lt;Creature&gt; { return api.findAll() } } Com essa implementação realizada, podemos adaptar o código do objeto Program para construir todas as instâncias necessárias, da seguinte forma: object Program { fun main () { val client = OkHttpClient() val creatureApi = CreatureApi(client) val remoteDataSource = CreatureRemoteDataSource(creatureApi) val creatures = remoteDataSource.getCreatures() println(creatures) } } Pronto! Com isso, nós realizamos nossa primeira injeção de dependências. Podemos ver que, na prática, tudo é resolvido apenas com argumentos nos construtores das classes, eliminando a responsabilidade da classe de realizar internamente a construção de uma de suas dependências e possibilitando que as instâncias sejam manipuladas e reutilizadas em níveis mais superiores do código. Fábricas de instâncias No exemplo que construímos, todas as instâncias são criadas e gerenciadas pelo objeto Program . No entanto, na vida real não é tão simples assim, uma vez que um aplicativo Android, por exemplo, possui várias telas que devem ser inicializadas — na maioria dos casos, de forma independente umas das outras. Por conta disso, delegar toda a responsabilidade de gerenciamento das instâncias para um único lugar é um cenário utópico, o que nos traz um novo conceito: a fábrica de instâncias . Como o próprio nome diz, a responsabilidade dessa classe é a de fabricar instâncias de classes específicas, concentrando toda a construção de cada classe em uma fábrica que tem essa única responsabilidade. Para implementá-la na classe CreatureApi , por exemplo, basta criar uma classe (no Kotlin, pode ser um object , que é uma classe com apenas uma única instância) chamada CreatureApiFactory que possui um método getInstance() que retorna uma instância de CreatureApi pronta para ser usada. Como sabemos, a classe CreatureApi depende da classe OkHttpClient . Nesses casos, a classe Factory deve chamar outras classes Factory para satisfazer todas as suas dependências, criando um ecossistema de fábricas que gerenciam suas respectivas classes, facilitando a manutenção do código. Aplicando a fábrica de instâncias em nosso exemplo de código, deverá ficar como o código a seguir. // Código da fábrica reutiliza as instâncias object CreatureRemoteDataSourceFactory { private var instance: CreatureRemoteDataSource? fun getInstance () : CreatureRemoteDataSource { if (instance == null ) { instance = CreatureRemoteDataSource(CreatureApiFactory.getInstance()) } return instance!! } } // Código das outras fábricas na mesma estrutura de // `CreatureRemoteDataSourceFactory`, mudando apenas os tipos object CreatureApiFactory { /* ... */ } object OkHttpClientFactory { /* ... */ } // Com as fábricas prontas, basta utilizá-las onde quiser object Program { fun main () { val remoteDataSource = CreatureRemoteDataSourceFactory.getInstance() val creatures = remoteDataSource.getCreatures() println(creatures) } } Nesse exemplo, podemos perceber que a criação de uma fábrica é algo relativamente simples, mantendo uma estrutura muito semelhante e que reutiliza as instâncias, evitando um possível desperdício de memória na criação de muitas instâncias. Apesar de as fábricas de instâncias serem simples de implementar e resolverem o problema em um primeiro momento, existem alguns problemas que podem aparecer com o tempo e tomarão tempo de desenvolvimento à medida que o projeto avança. Um primeiro problema, muito comum, é a reutilização de instâncias . No exemplo anterior de código, a classe CreatureRemoteDataSourceFactory fornece sempre a mesma instância de CreatureRemoteDataSource — a instância só é criada assim que o método getInstance() for chamado pela primeira vez. Depois disso, a instância é armazenada em uma propriedade da classe e reutilizada sempre que alguém quiser uma nova instância. Dessa forma, a instância é compartilhada por toda a aplicação. Isso serve para a maioria dos casos, mas também é comum precisarmos de instâncias que sejam reutilizadas apenas internamente, entre camadas diferente da aplicação — por exemplo, ter uma instância de OkHttpClient para cada classe de Api , como a CreatureApi e outras que podem existir. Além do problema mencionado, gerenciar a remoção de instâncias também é algo a ser considerado, principalmente quando precisamos reduzir o consumo de memória do aplicativo. Por conta desses e de outros problemas, nem sempre utilizar as Factories pode ser um bom caminho, afinal, trará trabalho de código adicional, fora o tempo de planejamento para resolver cada um dos pontos. Por conta disso, bibliotecas que resolvem o problema de injeção de dependências foram surgindo ao longo dos anos, começando por bibliotecas que resolvem o problema usando apenas Java, até bibliotecas específicas para Kotlin e Android.

Comunicando a atualização dos dados entre camadas Como mencionado anteriormente, a camada View Model é responsável por acessar os dados da camada Model e deixá-los disponíveis para que a View os acesse. Além disso, dentro de uma aplicação com arquitetura MVVM, além de disponibilizar esses dados, é necessário garantir que qualquer atualização seja refletida de forma automática. Quando estamos construindo aplicações e seguindo padrões de arquitetura, é muito importante saber as responsabilidades de cada parte do sistema. No exemplo a seguir, temos uma classe chamada AvisosRepository , que é responsável por armazenar as mensagens de aviso, conforme inseridas pelo usuário na MuralAvisosActivity . Sempre que um aviso novo for digitado, a função MuralAvisosActivity.novoAvisoDigitado(novoAviso: String) será executada, o que atualizará o aviso na classe AvisosRepository e chamará imediatamente a função MuralAvisosActivity.exibirAviso() , garantindo que o avisoTextView esteja sempre atualizado. Atenção: o código a seguir é apenas um exemplo ilustrativo. Não é necessário criar os arquivos e implementar os códigos em seu projeto. class AvisosRepository { var aviso: String = "Aviso inicial" fun atualizarAviso (novoAviso: String ) { aviso = novoAviso } } class MuralAvisosActivity { private val avisosRepository = AvisosRepository() override fun onCreate (savedInstanceState: Bundle ?) { // ... exibirAviso() } fun exibirAviso () { avisoTextView.text = avisosRepository.aviso } fun novoAvisoDigitado (novoAviso: String ) { avisosRepository.atualizarAviso(novoAviso) exibirAviso() } } Apesar de esse exemplo de código funcionar como esperado, ele possui alguns problemas: Sempre que atualizarmos algum aviso, devemos manualmente executar a função exibirAviso() para garantir que o textView possua o valor mais atualizado possível. Se o aviso for atualizado por alguma outra classe que não seja a classe MuralAvisosActivity , o valor do textView ficará desatualizado. Na situação número 2, uma forma de resolver esse problema seria fazer com que a classe AvisosRepository avise a classe MuralAvisosActivity de que a mensagem de aviso foi atualizada, do mesmo jeito que a classe MuralAvisosActivity está executando diretamente a função avisosRepository.atualizarAviso(novoAviso) . O problema, nesse caso, é que a camada model não deve possuir referência a nenhuma classe que não esteja na mesma camada, o que impede esse tipo de implementação. Para resolver esse tipo de problema, um recurso muito utilizado é a criação de eventos. Dessa forma, sempre que a mensagem de aviso for atualizada, um novo evento é emitido pela classe AvisosRepository informando que há uma nova mensagem disponível. Com isso, a classe MuralAvisosActivity pode ouvir os eventos de atualização, receber a nova mensagem e atualizar no textView . No entanto, esse tipo de abordagem causa um novo problema: quando a MuralAvisosActivity for destruída, os eventos da AvisosRepository continuam sendo ouvidos, causando vazamentos de memória, a menos que deixemos explícito para parar de ouvi-lo. A solução é simples, mas acaba se tornando um boilerplate presente em muitos lugares da aplicação. Vazamentos de memória ( Memory Leak ) consistem em variáveis e declarações que não foram removidas após não serem mais necessárias. Isso faz com que softwares fiquem mais lentos após determinado tempo aberto. Devs de alto nível devem se preocupar com esse tipo de situação, identificando e resolvendo possíveis vazamentos de memória no código. Para facilitar a utilização desse tipo de abordagem, existem diversas bibliotecas que permitem a utilização de objetos "observáveis", também conhecidos como Observables . Neste livro, utilizaremos três bibliotecas que possuem esse mecanismo: Data Binding , LiveData e RX . É válido ressaltar que cada biblioteca possui uma característica específica e costuma servir para resolver o mesmo problema em camadas diferentes. Por exemplo: Data Binding: comunicação entre o layout XML e a classe da Activity ; LiveData: comunicação entre a classe da Activity e o ViewModel ; RX: comunicação entre a camada Model e a camada View Model . Neste capítulo, trataremos apenas da implementação do LiveData , pois o utilizaremos para realizar a comunicação entre CreaturesViewModel (que criaremos mais à frente) e CreaturesListFragment . Como vimos, o Android Jetpack possui alguns componentes dentro da categoria Architecture . Dentre eles, três são importantes para este capítulo: Lifecycles , LiveData e View Model . O componente Lifecycles é composto de duas interfaces principais, LifecycleOwner e LifecycleRegistryOwner . Abas são implementadas nas classes AppCompatActivity e Fragment . É possível atrelar outros componentes para objetos que implementam essas interfaces (que chamaremos de owners ) para detectar mudanças no ciclo de vida do seu respectivo owner . Todo objeto que implementar o LifecycleOwner , incluindo qualquer Activity ou Fragment , possuirá o seu próprio ciclo de vida. O LiveData é uma implementação bem simples que permite observar qualquer mudança de uma informação. Sempre que essa informação for alterada, o LiveData informará esse evento para todos os interessados. Para isso, quem tiver interesse deverá criar um Observer , que registra no LiveData qual a ação que deverá ser executada sempre que o dado mudar. É importante mencionar que o LiveData respeita o ciclo de vida dos componentes, incluindo Activities , Fragments , Services e qualquer outro LifecycleOwner da aplicação. Isso faz com que, quando o LifecycleOwner for destruído, tudo que estiver sendo observado seja removido automaticamente. 5.2 Preparando a camada View Model Como mencionado anteriormente, a camada View Model é responsável por acessar os dados da camada Model e deixá-los disponíveis para que a View os acesse. Dentro de uma aplicação com arquitetura MVVM, além de disponibilizar esses dados, é necessário garantir que qualquer atualização seja refletida de forma automática. Como estamos realizando uma comunicação entre Activity e ViewModel , é interessante utilizar a classe LiveData para, além de permitir que atualizações sejam observadas, aproveitarmos o ciclo de vida da Activity para remover tudo o que estiver sendo observado quando a tela for destruída. O próximo passo para deixar o layout de cada item completo é carregar as imagens de acordo com a URL de cada criatura. 6.2 Exibindo imagens Trabalhar com o carregamento de imagens no Android parece uma tarefa bem simples a princípio — afinal, é apenas pegar uma imagem e exibi-la. O que poderia dar errado nisso? Geralmente é o que todos pensamos de início. A grande e dolorida verdade é que carregar imagens não é tão simples assim, independente de a imagem estar armazenada na memória do dispositivo ou em algum site da web. O principal fator é o tamanho da imagem. Temos dispositivos hoje com câmeras cada vez mais poderosas que tiram fotos com resoluções altíssimas (mesmo que a tela do celular não reproduza a quantidade de pixels), e cujos arquivos não possuem tamanho tão grande. Como o dispositivo precisa armazenar essa informação na memória do celular (que é muito escassa), muitas vezes acaba resultando em uma exception conhecida como OutOfMemory . Outro fator além do tamanho é quando a imagem está em um servidor da web, onde a comunicação dos dados deve ser feita via rede, o que, em muitos casos, é extremamente lento e pode apresentar interrupções e instabilidade. Somando tudo isso, temos um fator inconsistente, que torna o carregamento de imagem uma tarefa complicada e demorada (em termos de execução). É por isso que o carregamento de imagens deve ser feito em processos separados, para não impactar diretamente o processo principal da aplicação, conhecido como Main Thread ou UI Thread . Carregar grandes bitmaps de forma eficiente virou tópico de discussão na comunidade Android e pode ser feito quando conseguimos prever o tamanho da imagem antes de carregá-la, associando com o tamanho da imageView que receberá a informação, o que faz com que a imagem seja carregada em um tamanho menor. Isso acelera o processo e diminui o consumo de recursos, abrindo espaço também para definir diferentes tipos de configuração de bitmap (como ARGB_8888 , por exemplo) que alteram diretamente o consumo de bytes por pixel carregado. No universo da programação, é muito comum nos depararmos com implementações frequentes com que todo projeto precisa lidar. Nesses casos, é muito comum existirem bibliotecas, geralmente de código aberto, que facilitam a inserção dessas funcionalidades em diferentes aplicações. Quando o assunto é carregamento de imagens, isso não é diferente. Atualmente, no Android, existem três principais bibliotecas que realizam essa função: Glide , da Bump Technologies; Picasso , da Square; e Fresco , do Facebook. Quando precisamos decidir qual das bibliotecas utilizar, algumas questões precisam ser consideradas: sintaxe de utilização, tamanho da biblioteca (em KBs e quantidade de métodos), funcionamento padrão e funcionalidades extras disponíveis. Sistema de cache das bibliotecas Um dos pontos mais importantes a serem considerados é o sistema de cache que cada biblioteca utiliza, pois ele pode impactar diretamente a performance de uma aplicação. O Glide conta com um sistema de cache baseado no tamanho da ImageView . Suponha que vamos renderizar uma imagem de 1000 x 1000 pixels em uma área de 100 x 100 pixels. Primeiro, ele faz o download da imagem 1000 x 1000 pixels e converte para 100 x 100, armazenando-a em um arquivo com tamanho 100 x 100 pixels no cache e carregando-a no ImageView . Já o Picasso faz o cache da imagem no tamanho original, de 1000 x 1000 pixels, converte para 100 x 100 pixels e só depois a insere no ImageView . A documentação do Fresco não informa se o sistema de cache é feito pelo tamanho do ImageView ou pelo tamanho da imagem, apesar de permitir uma gestão detalhada das imagens armazenadas no cache. O resultado dessas diferenças de abordagem é que, geralmente, o Glide conseguirá carregar uma imagem mais rápido desde que ela já tenha sido carregada antes. No entanto, para que o Glide realmente seja mais rápido, o ImageView não pode ter um tamanho diferente de quando a imagem foi carregada pela primeira vez, pois nesse caso será necessário fazer o download novamente. Já o Picasso, mesmo com o cache da imagem, pode demorar mais, principalmente se a imagem original possuir um tamanho muito maior do que o ImageView em questão, pois sempre precisará fazer a conversão de tamanho antes de exibir. No momento da escrita deste livro, um outro ponto que diferencia ambos é que o Glide e o Fresco suportam GIFs, enquanto o Picasso não. Portanto, se você deseja carregar esse tipo de imagem em sua aplicação, será necessário optar pelo Glide ou pelo Fresco. Mais detalhes sobre a comparação entre as bibliotecas podem ser encontrados em alguns artigos pela internet, como este da Redwerk: https://redwerk.com/blog/advanced-feature-in-android-image-loaders-picasso-vs-glide-vs-fresco/ Figura 6.5: Artigo da Redwerk sobre a comparação de bibliotecas de imagem. Criando adapters customizados Além de permitir utilizar variáveis dentro do layout XML, o Data Binding permite a criação de atributos customizados para as Views . Esses atributos são chamados de Binding Adapters . Apesar de a declaração de carregamento de imagens com o Glide ser simples, tê-la declarada em diversos lugares dificulta a manutenção do código. Portanto, vamos criar um binding adapter customizado para abstrair essa lógica. Como temos a URL da imagem associada a cada criatura através da declaração creature.imageUrl , podemos criar um atributo customizado chamado app:imageUrl que recebe essa URL através do Data Binding e efetua o carregamento da imagem. Primeiro, vamos declarar no layout XML o novo atributo, informando a URL da imagem. Abra o arquivo creatures_list_item.xml , procure pela ImageView da criatura e adicione o seguinte conteúdo: &lt; ImageView android:id = "@+id/ivCreature" app:imageUrl = "@{creature.imageUrl}" ... /&gt; Para que o app:imageUrl funcione, precisamos declará-lo em um arquivo Kotlin. Uma boa prática para a criação de binding adapters é a criação de um arquivo para cada tipo de View . No caso, vamos criar um arquivo para utilizar em views do tipo ImageView . Como a operação está sendo realizada na classe UserRepository , não temos como executar diretamente uma implementação da NavigationActivity , pois isso implicaria em a camada Repository ter conhecimento da camada View , o que acarretaria problemas de manutenção. Nesse tipo de situação, é comum utilizar um mecanismo orientado a eventos, como o que fizemos com o LiveData na implementação da propriedade creatures na classe CreaturesViewModel . A utilização de eventos que avisam que determinado comportamento aconteceu é extremamente útil em diversas situações, pois, além de desacoplar camadas de aplicação, permite uma extensão bem poderosa ao longo da evolução de um software. Neste caso, vamos implementar um evento chamado onChooseCreature , que avisa para qualquer interessado que uma criatura acabou de ser escolhida. No caso da classe UserRepository , não importa o que será feito com essa informação, mas qualquer um que precisar dela poderá simplesmente observar a propriedade onChooseCreature e passar a receber a Creature que acabou de ser escolhida. Um ponto de atenção aqui é que durante os capítulos anteriores vimos apenas esse tipo de utilização através da classe LiveData . Apesar de simples e poderosa, a classe LiveData precisa de um LifecycleOwner para permitir observações nos valores. Esse LifecycleOwner está presente apenas no caso de views que são Activities ou Fragments . Para as classes de Repository , que são acessadas por outras classes da mesma camada ou da camada View Model , isso não é uma realidade; portanto, não devemos utilizar LiveData em camadas além da View Model . Outras bibliotecas oferecem suporte a esse tipo de implementação, o que é o caso da biblioteca ReactiveX , ou simplesmente RX . A utilização do RX não depende de LifecycleOwner , o que facilita sua utilização em outras camadas além da View . No entanto, para acelerar o processo de implementação da aquisição de novas criaturas, vamos deixar a implementação do RX para o próximo capítulo. A não utilização do LiveData na camada Repository ocorre quando precisamos trabalhar com o valor armazenado, pois ao solicitá-lo com a declaração liveData.observe() , precisamos passar o LifecycleOwner , que não existe na camada Repository . Como este não é o caso, pois precisamos apenas do LiveData para expor que o evento de onChooseCreature ocorreu, podemos utilizá-lo sem que uma declaração de observe() seja necessária fora da View . Vamos começar criando a propriedade LiveData na classe UserRepository , seguindo o padrão apresentado nos capítulos anteriores, em que utilizamos um MutableLiveData interno, expondo apenas o LiveData para utilização fora da classe. Abra o arquivo Creature.kt e crie uma propriedade Boolean chamada isKnown com o valor default false , para que as criaturas declaradas no arquivo CreaturesRepository.kt não precisem informar o valor desejado para a propriedade. package tech.salvatore.livro_android_kotlin_paulo_salvatore.model.domain data class Creature ( // ... val isKnown: Boolean = false , ) Com a classe de domain preparada, vamos criar uma propriedade allCreatures dentro do arquivo UserRepository.kt . Sempre que algum outro código tentar obter essa informação, utilizamos a lista completa de criaturas disponível em creaturesRepository.creatures para combinar com a lista de criaturas que o usuário possui. Para garantir que a propriedade sempre retorne a informação mais atualizada possível, vamos utilizar a declaração get() . Assim, a informação é criada no momento em que alguém tenta obtê-la, da seguinte forma: package tech.salvatore.livro_android_kotlin_paulo_salvatore.model.repository // ... class UserRepository @Inject constructor ( private val creaturesRepository: CreaturesRepository, ) { val user = User( "Paulo Salvatore" , true ) val allCreatures get () = creaturesRepository.creatures // ... } Porém, a lista com todas as criaturas não traz a informação considerando a implementação do isKnown , então precisamos preenchê-la. Portanto, vamos transformar uma lista de criaturas, todas com o valor de isKnown igual a false , em uma nova lista de criaturas que possui o valor de isKnown preenchido corretamente, sendo que o valor de isKnown será true apenas para quando a criatura estiver na lista de criaturas que o usuário possui. Para transformar uma lista em outra, podemos utilizar o operador map : // ... class UserRepository @Inject constructor ( // ... val allCreatures get () = creaturesRepository.creatures.map { it } // ... } Note que o operador auxiliar it representa cada criatura existente na lista creaturesRepository.creatures . O próximo passo é construir uma variável que contenha o valor de isKnown seguindo a lógica que foi mencionada anteriormente. Para checar se a criatura em questão está na lista de criaturas do usuário, podemos utilizar o operador any , que retorna true caso o valor seja encontrado. Como o operador any checa todos os itens da lista, precisamos informar uma expressão de checagem, que, no caso, será a comparação utilizando a propriedade number de cada criatura. Além disso, como o operador any , assim como o map , também utiliza o operador auxiliar it , precisamos renomeá-lo para não conflitar. Portanto, vamos renomear para creatureOwnByUser e utilizá-lo na checagem dos números das criaturas: // ... class UserRepository @Inject constructor ( // ... val allCreatures get () = creaturesRepository.creatures.map { val isKnown = user.creatures.any { creatureOwnByUser -&gt; creatureOwnByUser.number == it.number } it } // ... } Com a variável isKnown contendo o valor correto, vamos atualizar na criatura. No entanto, a propriedade do domain Creature.isKnown está marcada como val , o que impede que o valor seja alterado depois de sua criação. Manter as propriedades de domain como val é uma boa prática para impedir que alterações nesses valores sejam feitas, visto que essas alterações não refletem de forma automática no restante da aplicação. Uma técnica utilizada para preencher as propriedades val de data classes é utilizar o operador copy , gerado automaticamente quando adicionamos a declaração data para a classe: // ... class UserRepository @Inject constructor ( // ... val allCreatures get () = creaturesRepository.creatures.map { // ... it.copy( isKnown = isKnown ) } // ... } Com isso, um novo objeto de criatura é obtido, garantindo que o valor de isKnown está correto. Outro comportamento que podemos incluir é o de esconder o nome da criatura quando ela não for conhecida. Vamos aproveitar o it.copy() para alterar o nome para ????? quando o valor de isKnown for false , utilizando uma declaração if/else de uma linha: // ... class UserRepository @Inject constructor ( // ... val allCreatures get () = creaturesRepository.creatures.map { // ... it.copy( name = if (isKnown) it.name else "?????" , isKnown = isKnown ) } // ... } 8.2 Preparando a camada View Model Com a camada Model finalizada, vamos atualizar a camada View Model para pegar a lista de criaturas que acabamos de preparar a partir de UserRepository , em vez de CreaturesRepository . Vamos abrir o arquivo CreaturesViewModel.kt e referenciar a dependência de UserRepository direto no constructor , para garantir que o Hilt fará a injeção corretamente. A partir disso, podemos alterar a inicialização de creatures.value : package tech.salvatore.livro_android_kotlin_paulo_salvatore.viewmodel // ... import tech.salvatore.livro_android_kotlin_paulo_salvatore.model.repository.UserRepository @HiltViewModel class CreaturesViewModel @Inject constructor ( private val creaturesRepository: CreaturesRepository, userRepository: UserRepository, ) : ViewModel() { val creatures = MutableLiveData&lt;List&lt;Creature&gt;&gt;() init { creatures.value = userRepository.allCreatures } // ... } Note que a declaração userRepository não precisa ter a declaração private val , pois ela é utilizada apenas no bloco init , que executa automaticamente quando a classe é inicializada. Do contrário, estamos dizendo que, além de parâmetro, ela também representa uma propriedade da classe que pode ser utilizada dentro de métodos. Figura 10.1: Qual é o DevMon #10? 10.1 Programação Reativa Para começar a falar sobre o assunto, precisamos distinguir dois conceitos: o de Programação Reativa e o de RX. Programação Reativa ( Reactive Programming ) é um paradigma da programação que diz que devemos nos preocupar com o fluxo de dados e com as mudanças que esse fluxo sofre ao longo do percurso. O RX , por sua vez, é uma biblioteca de código aberto que busca facilitar a implementação do paradigma, trazendo ferramentas preparadas para trabalhar com os fluxos de dados, além de facilitar o uso da programação assíncrona e a comunicação de diferentes partes do código. Neste capítulo, entenderemos algumas questões iniciais sobre o manifesto reativo para entender os objetivos principais da Programação Reativa, além de explorar um pouco mais o RX para entender como essa biblioteca está estruturada e o que exatamente podemos fazer com ela. Pilares da Programação Reativa De acordo com o manifesto reativo, publicado em 2014, a Programação Reativa possui 4 pilares principais: responsividade , resiliência , elasticidade e orientação a mensagens , conforme é possível visualizar na figura a seguir. Figura 10.2: Os 4 pilares da Programação Reativa. Responsividade: a aplicação deve responder o mais rápido possível e problemas devem ser detectados e resolvidos com a máxima eficácia; Resiliência: a aplicação deve continuar respondendo em caso de falha; Elasticidade: a aplicação deve continuar funcionando de forma plena mesmo com aumento da demanda; Orientação a mensagens: em vez de interligar diretamente as partes de um software, tente pensar como pequenos ecossistemas isolados que disparam mensagens claras sobre o que está acontecendo em seu interior. A partir disso, outros pedaços do software saberão entendê-lo de uma forma mais clara. Como é possível ver na figura, todos os pilares estão relacionados de alguma maneira e, assim, ao priorizar a performance e facilitar a manutenção, conseguimos garantir que tanto o código quanto a aplicação atinjam um alto nível de qualidade. 10.2 Fluxo de dados síncrono vs. assíncrono Para conseguir entender mais sobre o RX, precisamos de uma boa definição do que é o fluxo de dados e qual a diferença entre esse fluxo ser síncrono ou assíncrono. O fluxo de dados é o nome dado ao processo de trabalhar com uma informação que passa por algumas etapas antes de estar pronta para aparecer para o usuário. Na nossa aplicação, por exemplo, para exibir uma lista com todas os DevMon que conhecemos, precisamos executar uma chamada para um back-end, que traz uma string com todo o dado. A partir disso, essa string é transformada em uma informação que a linguagem de programação é capaz de manipular com mais facilidade. Em alguns casos, se a informação não vier completa na primeira requisição, pode ser necessário realizar novas requisições e processar todos esses resultados de uma só vez, antes de deixar tudo pronto para a exibição na tela. Toda essa manipulação da informação, tanto a obtenção quanto a transformação, é chamada de fluxo de dados . O fluxo de dados síncrono é aquele em que uma etapa só acontece depois que a etapa anterior foi finalizada. Na nossa aplicação, por exemplo, as imagens dos DevMon só serão carregadas assim que a lista de DevMon for recebida. Além disso, se o carregamento das imagens for feito da maneira síncrona, a imagem do segundo DevMon só será carregada após o carregamento da imagem do primeiro ser concluído. O fluxo de dados assíncrono é aquele em que uma etapa acontece ao mesmo tempo que outra. Na nossa aplicação, as imagens de todos os DevMon são carregadas ao mesmo tempo, disparando várias requisições em paralelo. Nesse caso, apesar de ser necessário aguardar a primeira requisição para saber qual a lista de imagens, o carregamento das imagens em si fica muito mais rápido, pois não precisamos aguardar cada imagem terminar de carregar para disparar o carregamento da próxima, visto que uma não depende da outra. Agora que entendemos o que é o fluxo de dados e a diferença entre fluxos síncronos e assíncronos, fica um pouco mais claro entender como toda aplicação gira em torno de fluxos de dados tanto síncronos quanto assíncronos. Na aplicação que estamos construindo, os dados dos DevMon e dos usuários estão armazenados em variáveis na memória e estão sendo acessados de maneira síncrona . No entanto, as imagens do DevMon estão utilizando endereços da internet, que são carregados pela biblioteca Glide, que carrega várias imagens ao mesmo tempo, são processadas de maneira assíncrona . No próximo capítulo, vamos integrar o back-end à aplicação Android. Para isso, vamos precisar implementar requisições para servidores na web, o que deve levar alguns segundos para concluir. Quando precisamos fazer com que a aplicação aguarde um certo tempo para que algo seja concluído, é necessário utilizar a programação assíncrona, pois, caso contrário, travaríamos o app aguardando o resultado, o que tem impacto direto na experiência de usuário. Como implementar chamadas assíncronas? Quando executamos um programa em um computador ou em um celular, estamos executando um processo direto na CPU. Esses processos possuem threads , que funcionam como pequenos ecossistemas capazes de executar declarações em sequência. Para implementar chamadas que acontecem ao mesmo tempo, ou seja, chamadas assíncronas, criamos threads que executam as instruções que foram passadas e comunicam apenas o resultado daquela instrução. Para conseguir coordenar todo esse processamento, geralmente temos uma thread inicial, comumente chamada de main thread , que é responsável por iniciar todas as outras threads , conforme a necessidade. No Android, ela também é conhecida como UI Thread , pois, além de ser a thread principal, é responsável pela renderização da interface de usuário (UI). Existem várias formas de implementar chamadas assíncronas no Android, como: Thread/Runnable , AsyncTask , Coroutines etc. No entanto, gerenciar a comunicação entre threads nem sempre é uma tarefa fácil, já que exige muito planejamento e muita implementação de código boilerplate , principalmente quando queremos realizar operações mais complexas. Foi justamente para facilitar o trabalho com fluxos de dados síncronos e assíncronos, além da comunicação entre esses fluxos, que surgiu a biblioteca RX. Ela fornece uma forma relativamente simples de implementação, a partir do momento em que entendemos alguns princípios básicos. 10.3 A base do RX Agora que entendemos o problema que o RX resolve, vamos entender o funcionamento de quatro componentes principais: Observables , Observers , Diposables e Operators . Observable O primeiro item da nossa lista é o Observable , onde tudo começa. É nele que definimos o que será realizado, a base da nossa operação. Podemos definir um Observable como algo que está emitindo eventos para algum fluxo de dados, podendo ser qualquer coisa sobre a qual estamos interessados em saber mais. A partir da criação do Observable , temos um fluxo de dados observável , ou seja, quem tiver interesse nas informações, pode observar e receber eventos. Neste exemplo de código, temos um Observable que emite um evento para cada um dos números informados: Observable.just( 1 , 2 , 3 ) Neste outro exemplo de código, que usa a biblioteca RxBinding , temos um Observable que emite um evento sempre que o texto de um EditText é alterado: val nameChangeObservable = RxTextView.textChanges(etName) Um fluxo de dados do RX sempre começa a partir de um Observable . Observers No entanto, um Observable sozinho não faz muita coisa, visto que não basta ter os eventos, precisamos fazer algo com eles, certo? Para isso, temos os Observers , que observam todas as novas informações que o Observable está deixando disponíveis. Até então, para Observables que possuem uma quantidade fixa de evento a ser emitidas, não precisamos de um dispose() : val observable = Observable.just( 1 , 2 , 3 ) observable.subscribe { println(it * 2 ) // Irá exibir 2, 4 e 6 } Se olharmos para a declaração da interface em Java, podemos observar exatamente o seguinte: public interface Observer &lt; T &gt; { void onSubscribe ( @NonNull Disposable d) ; void onNext ( @NonNull T t) ; void onError ( @NonNull Throwable e) ; void onComplete () ; } Inicialmente, a interface recebe um tipo T , que significa o tipo de variável que o Observer vai observar. Depois, temos o método onSubscribe() , que é chamado assim que o Observable começa a ser observado. Um objeto da classe Disposable é trazido quando chamamos esse método. Entenderemos o papel o Disposable mais à frente. O principal método é o onNext() , que recebe cada item informado do tipo T , e é onde utilizaremos o valor já processado. O onError() atua como um centralizador de qualquer erro ocasionado durante a operação realizada, onde podemos declarar a lógica para tratar esses erros recebidos. Por último, temos o onComplete() , que é chamado assim que toda a operação termina. Ele só é disparado caso a operação tenha sido um sucesso. Consumer : um Observer simplificado Além do Observer , temos a interface Consumer , que facilita uma implementação do subscribe . Nessa interface, implementamos a passagem no método onNext() , que, nesse caso, é chamado de accept() . O conteúdo da interface é bem simples, como pode ser observado em sua declaração, em Java. public interface Consumer &lt; T &gt; { /** * Consume the given value. * @param t the value * @throws Exception on error */ void accept (T t) throws Exception ; } Em alguns casos, quando não é necessária a implementação dos quatro métodos de um Observer , é possível implementar apenas o caso de sucesso com o método accept() . No entanto, caso o Observable traga algum erro, essa exception não será tratada dentro desse Consumer e deverá ser tratada em algum outro ponto do fluxo. Disposables Através da utilização de um Observer , conseguimos nos conectar aos Observables , gerando algo que é chamado de Subscription ou Disposable . Como estamos trabalhando com programação assíncrona, é muito comum se conectar a algo que pode enviar novas informações a qualquer momento. No entanto, em certas situações, utilizamos o comando dispose() para remover o Observer , como nos trechos a seguir. // É muito comum utilizar Observables dentro dos ViewModels class ViewModel { // Propriedade no escopo da classe para ficar acessível nos métodos val disposable: Disposable // Em alguma inicialização qualquer init { val nameChangeObservable = RxTextView.textChanges(etName) // Faz a subscription e guardar o Disposable disposable = nameChangeObservable.subscribe { // Trabalhar com o evento de mudança do texto do EditText } } // Quando um ViewModel não é mais necessário, o `onCleared()` é executado: override fun onCleared () { disposable.dispose() // Limpa todas as Subscriptions } } CompositeDisposable : uma forma prática de usar o Disposable Em alguns casos, é possível ter várias Subscriptions acontecendo em uma mesma classe. Para facilitar o gerenciamento dos Diposables sem precisar de uma propriedade para cada um deles, podemos utilizar o CompositeDisposable , que permite colocar todos os Disposables em um único lugar e limpá-los todos de uma vez, da seguinte forma: class ViewModel { // Criamos o CompositeDisposable val composite = CompositeDisposable() // Em alguma inicialização qualquer init { val nameChangeObservable = RxTextView.textChanges(etName) val ageChangeObservable = RxTextView.textChanges(etAge) // Adicionar as duas Subscriptions na mesma propriedade composite += nameChangeObservable.subscribe { // ... } composite += ageChangeObservable.subscribe { // ... } } // Limpa tudo de uma vez override fun onCleared () { composite.dispose() } } Operators Como vimos anteriormente, o Observable possui uma informação que é observada pelo Observer , gerando um Disposable que permite remover essa relação. Os Operators entram exatamente no meio dessa relação, permitindo que o dado inicial se transforme ao longo do fluxo. Isso facilita trabalhar com fluxos complexos em que muitas etapas são necessárias até chegar à informação final. Existem diversos operadores disponíveis, sendo que cada um tem funcionalidades muito específicas, como transformar uma informação A em B, filtrar dados indesejados, permitir realizar qualquer ação como exibir os dados recebidos no console, entre outros. Apesar de existirem muitos operadores, conhecer apenas os principais já permite uma utilização satisfatória do RX, uma vez que os mais complexos geralmente simplificam comportamentos que poderiam ser obtidos com operadores mais básicos. Uma lista completa está disponível na documentação (link a seguir), juntamente a um diagrama que explica cada um. Visualizar os diagramas dos operadores facilita o entendimento, pois alguns sites permitem simular alterações no fluxo em tempo real, demonstrando o que aconteceria na vida real se utilizarmos o operador em questão. Vamos ver alguns exemplos com os operadores mais famosos. Link para a documentação de operadores do RX: https://reactivex.io/documentation/operators.html . Figura 10.3: Documentação de operadores do RX. Link para testar operadores de forma visual: https://rxmarbles.com/ Figura 10.4: Teste de operadores de forma visual. Map O funcionamento do operador map consiste em transformar uma informação recebida em outra qualquer, de acordo com a nossa declaração. Observable.just( 1 , 2 , 3 ) .map { it * 2 } // 2, 4, 6 .subscribe { println(it) } // 2, 4, 6 Nesse exemplo, para cada número recebido, vamos passá-lo pelo map e transformá-lo em seu dobro. Como a informação inicial era 1, 2, 3 , o resultado dessa operação será 2, 4, 6 . Filter Também podemos adicionar a essa declaração um filter , que vai filtrar o resultado e retornar apenas aqueles que desejamos. Observable.just( 1 , 2 , 3 ) .map { it * 2 } // 2, 4, 6 .filter { it &lt; 6 } // 2, 4 .subscribe { println(it) } // 2, 4 Nesse caso, o filtro vai permitir apenas números passados pelo map que são menores do que 6, retornando então 2, 4 como resultado da operação. FlatMap Em um primeiro momento, pode parecer que o operador flatMap realiza a mesma coisa que o map ; porém, apesar de serem bem parecidos, a diferença é que o flatMap transforma um item recebido em um Observable . Podemos receber um conjunto de objetos de usuários observáveis, como Observable&lt;User&gt; , e transformar em uma lista de posts, como Observable&lt;Post&gt; , que são obtidos através de uma chamada na API, que retorna um Observable e processa os posts logo em sequência. val usersObservable: Observable&lt;User&gt; = getUsersFromApi(); val posts: Observable&lt;Post&gt; = usersObservable.flatMap { it.getUserPostsFromApi() // getUserPosts faz uma chamada na API } // Processamos os posts logo em sequência Por exemplo, o flatMap recebe 10 usuários de uma única vez e, para cada um deles, acessa a lista de posts, fazendo uma chamada de API para cada um. Os resultados que forem recebidos são processados logo em sequência. Portanto, se chamarmos 10 requisições em ordem, as mais rápidas serão processadas, não importa a ordem em que foram chamadas. 10.4 Implementando o RX no projeto Agora que conhecemos um pouco mais do RX, chegou a hora de implementá-lo no projeto. Até então, como a aplicação não realizava nenhuma chamada assíncrona, como enviar esse dado para a API ou para um banco de dados local, trabalhar com listas e objetos é algo bem tranquilo de se fazer. No entanto, visto que estamos preparando a aplicação para trabalhar com fluxos de dados utilizando o RX, que facilita a integração entre fluxos síncronos e assíncronos, vamos transformar toda a implementação desse método para utilizar os princípios do RX da melhor forma. Como o RX sugere a utilização de uma programação funcional reativa, vamos trabalhar com pequenas partes do fluxo, de modo que fique claro o que estamos fazendo em cada uma das etapas e também facilite a manutenção desse método no futuro. Para isso, iniciamos o fluxo a partir de um Observable , combinamos Operators que moldam o fluxo ao longo do processo e retornamos um Observable ao final, para que ele seja observado por quem tiver interesse. Primeiro, vamos atualizar o tipo de retorno do método: fun chooseCreature () : Observable&lt;Creature&gt; = Depois, iniciamos o fluxo de dados com o Observable . Como o user continua sendo apenas um objeto (podemos adaptá-lo no futuro, caso formos salvar o user em um banco de dados, por exemplo), criamos um Observable&lt;User&gt; através da declaração Observable.just() . fun chooseCreature () : Observable&lt;Creature&gt; = Observable.just(user) O fluxo de dados deve avançar apenas caso a propriedade hasCreatureAvailable do user tenha true como valor. Para isso, podemos utilizar o operador filter , que filtra os eventos e processa apenas os que retornarem true . fun chooseCreature () : Observable&lt;Creature&gt; = Observable.just(user) .filter { it.hasCreatureAvailable } Atualizamos o user , informando que não há mais criaturas disponíveis. Dentro de um Observable , sempre que quisermos apenas executar uma ação, podemos utilizar o operador doOnNext { } . Operadores que apenas executam algo, sem se importar com o retorno, costumam começar com a palavra do . fun chooseCreature () : Observable&lt;Creature&gt; = Observable.just(user) // ... .doOnNext { it.hasCreatureAvailable = false } Agora que o usuário foi devidamente checado e atualizado, vamos transformar o fluxo atual em um fluxo com a lista de criaturas disponível em CreaturesRepository.creatures . fun chooseCreature () : Observable&lt;Creature&gt; = Observable.just(user) // ... .doOnNext { it.hasCreatureAvailable = false } .map { creaturesRepository.creatures } Ao transformar um Observable em outro, o RX não entende de forma automática que queremos trabalhar com os dados que estão dentro do outro fluxo. Na prática, o dado vindo do back-end é uma string , pois representa uma sequência de caracteres. Apesar de muito comum, carregar informações da internet não é uma tarefa tão simples, afinal, muitos problemas podem ocorrer, como perda de conexão, lentidão no processo, queda de sinal, entre diversos outros motivos. Neste capítulo, entenderemos a principal forma de comunicação pela web, utilizando o protocolo HTTP e recebendo dados estruturados em JSON ( Java Script Object Notation , que entenderemos mais à frente). Vale ressaltar que toda requisição na web deve ser feita em uma thread separada da UI Thread ; portanto, vamos utilizar o RX para facilitar o trabalho com execuções assíncronas. 11.2 Retrofit A biblioteca Retrofit foi desenvolvida e é mantida pela Square, além de ser uma das mais populares no ecossistema dev Android, estando presente na maioria dos projetos que precisam fazer requisições na web. Ela possui um uso extensivo de annotations , que são capazes de simplificar a implementação dos serviços que realizam operações no back-end. Para implementar o Retrofit, precisamos realizar os seguintes passos: Importar a biblioteca no build.gradle da nossa aplicação; Criar pelo menos uma interface que tenha o papel de Service . Endpoint . No caso da URL https://devmon-api.onrender.com/creature , por exemplo, o endpoint é creature . Instância do Retrofit. Criada através da declaração Retrofit.Builder().build() . Trabalhar com requisições na web é algo relativamente complexo de ser realizado no Android. No entanto, o Retrofit consegue fornecer um kit de ferramentas poderoso para interagir com o protocolo HTTP, disponibilizando todos os recursos necessários para criar conexões, receber os dados e processá-los, incluindo mecanismos para tratamento de erros. O JSON ( Java Script Object Notation ) é uma forma de estruturação de dados muito comum para trocar informações entre diversas aplicações, independente da linguagem de programação em que ela é feita. Inspirado na declaração de objetos da linguagem JavaScript, o JSON é considerado uma das formas mais leves de estruturar uma informação, com grande vantagem em relação ao XML. Por exemplo, para representar um objeto de dados de uma pessoa e seus interesses, temos as seguintes versões em XML e JSON. XML: 211 caracteres &lt; pessoa &gt; &lt; nome &gt; Paulo Salvatore &lt;/ nome &gt; &lt; interesses &gt; &lt; interesse &gt; Kotlin &lt;/ interesse &gt; &lt; interesse &gt; TypeScript &lt;/ interesse &gt; &lt; interesse &gt; Unity &lt;/ interesse &gt; &lt; interesse &gt; C# &lt;/ interesse &gt; &lt;/ interesses &gt; &lt;/ pessoa &gt; JSON: 88 caracteres { "nome" : "Paulo Salvatore" , "interesses" : [ "Kotlin" , "TypeScript" , "Unity" , "C#" ] } No XML, cada tag que foi aberta deve ser fechada, o que a faz repetir o nome usado na abertura e gastar muito mais bytes apenas para abrir e fechar conteúdos. Isso não acontece no JSON, já que a estruturação é organizada utilizando símbolos como aspas, chaves e colchetes. Além disso, o formato JSON permite armazenar de forma mais fácil alguns tipos primitivos de dados, diferenciando string de informações como number e boolean , enquanto o XML armazena tudo como string por padrão. JSON Viewer Quando abrimos URLs de servidores back-end que exibem dados em JSON, é muito difícil interpretar a informação, pois nem todo navegador estrutura a informação de forma automática. Para resolver esse problema, é possível instalar extensões como o JSON Viewer, que transforma a visualização para facilitar tanto o entendimento quanto a manipulação da informação. No Google Chrome, por exemplo, a extensão pode ser baixada diretamente da loja, através do link a seguir: https://chrome.google.com/webstore/detail/json-viewer/gbmdgpbipfallnflgajpaliibnhdgobh Figura 11.5: Exibição de JSON formatada pelo JSON Viewer. Serialização Agora que sabemos como funciona a estrutura de um JSON e temos ferramentas para interpretá-la de forma mais clara, precisamos saber como fazer a programação entendê-la, para que seja possível trabalhar com os dados recebidos nesse formato. Para isso, precisamos conhecer o processo de serialização. Enquanto serializar um dado significa transformar a informação que está na memória RAM do computador em um texto estruturado, desserializá-lo significa pegar esse texto e inseri-lo novamente na memória RAM. Dependendo da linguagem de programação, esse processo pode ser mais simples ou mais complicado. Em linguagens fortemente tipadas, como o Java, temos algumas características mais robustas de tipo de variáveis e estruturação de conteúdo. Por conta disso, o processo de desserialização deve conhecer o tipo de toda a informação presente, antes de poder inseri-la na memória. Utilizando o JSONParser , podemos realizar esse processo, como mostra o exemplo a seguir. val jsonData = "{\"colors\":[{\"color\":\"red\",\"hex\":\"#FF0\"},{\"color\":\"white\",\"hex\":\"#FFF\"}]}" val reader = JSONObject(jsonData) val colors = reader.getJSONArray( "colors" ) val red = colors.getJSONObject( 0 ) val redHex = red.getString( "hex" ) // redHex = "#FF0" No entanto, realizar esse processo utilizando as soluções mais simples como JSONParser acaba sendo muito complexo e trabalhoso. Para facilitar a manipulação de JSONs, existem algumas bibliotecas, como o GSON (Google), que possuem integração direta com o Retrofit. 11.4 Implementando no projeto Chegou a tão esperada hora de colocarmos a mão na massa e integrar o back-end ao nosso projeto, vamos lá? Criando o Remote Data Source Com a configuração do endpoint pronta, precisamos de um local que disponibilize o acesso às informações pelas outras classes do sistema. Uma estratégia muito interessante nesses casos é evitar trabalhar com as estruturas de dados da API, pois além de essas estruturas serem pouco flexíveis, uma vez que dependem de mudanças na API, elas também são muito frágeis, pois alterações na API podem quebrar comportamentos da aplicação difíceis de corrigir em uma estrutura pouco flexível. A única forma de conseguir realizar requisições na API é através do Service , que até então é uma interface que não foi implementada. Como o Retrofit cuida da implementação das interfaces, reduzindo a quantidade de boilerplate necessário, precisamos utilizar a declaração Retrofit.Builder() para construir a instância do Retrofit e utilizá-la na implementação dos Services . Começaremos criando a classe CreatureRemoteDataSource dentro do pacote source.remote , marcando-a como @Singleton e disponibilizando-a para injeção de dependência com a declaração @Inject no constructor() , da seguinte forma: package tech.salvatore.livro_android_kotlin_paulo_salvatore.model.source.remote import javax.inject.Inject import javax.inject.Singleton @Singleton class CreatureRemoteDataSource @Inject constructor () { } Dentro da classe, vamos preparar a instância do Retrofit logo no momento em que a classe é inicializada, através do bloco de inicialização init { } . Essa instância pode ser obtida através da declaração Retrofit.Builder() , informando a base da URL e o ConverterFactory da biblioteca de desserialização (no caso, o GSON). Como estamos usando o Observable como tipo de retorno da API, substituindo o Call do Retrofit, também é necessário adicionar o CallAdapterFactory do RX. Com tudo devidamente declarado, basta executar o método build() para obtenção da instância, como no trecho a seguir: // ... import retrofit2.Retrofit import retrofit2.adapter.rxjava3.RxJava3CallAdapterFactory import retrofit2.converter.gson.GsonConverterFactory @Singleton class CreatureRemoteDataSource @Inject constructor () { init { val retrofit = Retrofit.Builder() .baseUrl( "https://devmon-api.onrender.com/" ) .addCallAdapterFactory(RxJava3CallAdapterFactory.create()) .addConverterFactory(GsonConverterFactory.create()) .build() } } Com a instância do Retrofit pronta, podemos utilizá-la para criar uma instância do Service . (service &amp; init) val creatures: Observable&lt;List&lt;Creature&gt;&gt; = service.findAll().map { list -&gt; list.map { it.toDomain() } } private fun CreatureApiResponse. toDomain () : Creature { return Creature( number = number, name = name, imageUrl = image, ) } } Integrando a API com o restante da aplicação Agora que as classes que disponibilizam as informações da API estão criadas, vamos mudar a maneira com que o CreaturesRepository prepara o Observable com a lista de criaturas. Primeiro, precisamos acessar o RemoteDataSource na classe CreaturesRepository . Para isso, basta declarar a propriedade direto no constructor para que o Hilt faça a injeção da dependência, da seguinte forma: // ... import tech.salvatore.livro_android_kotlin_paulo_salvatore.model.source.remote.CreatureRemoteDataSource class CreaturesRepository @Inject constructor ( remoteDataSource: CreatureRemoteDataSource ) { // ... Com a instância do remoteDataSource disponibilizada, podemos acessar a propriedade creatures , recebendo o Observable&lt;List&lt;Creature&gt;&gt; . Para disponibilizar a lista de criaturas, um caminho possível seria fazer com que a propriedade CreaturesRepository.creature receba diretamente o valor de remoteDataSource.creatures , pois ambas possuem o tipo Observable&lt;List&lt;Creature&gt;&gt; . No entanto, o principal problema com essa abordagem é que toda vez que a propriedade creatures fosse utilizada por alguma implementação, uma nova requisição seria disparada, fazendo com que várias requisições fossem feitas desnecessariamente. Uma ótima forma de contornar esse problema quando estamos usando o RX é utilizando o ReplaySubject em vez do Observable . O que o ReplaySubject faz é armazenar os últimos eventos emitidos por um Observable , dependendo de uma quantidade configurável. Assim, podemos criar um ReplaySubject que repete apenas o último evento, fazendo com que a lista de criaturas sempre esteja disponível para a aplicação, sem precisar de novas requisições. Vale ressaltar que o ReplaySubject , por baixo dos panos, é um Observable . Além do mecanismo de replay, o ReplaySubject entra na categoria de Subjects , que são Observables comuns, com a exceção de que o método onNext pode ser chamado depois de sua criação. Isso faz com que Subjects sejam muito úteis quando estamos criando Observables , mas queremos deixar a lógica de atualização dos eventos separada. Para criar um ReplaySubject , utilizamos a declaração ReplaySubject.create(1) , onde 1 é a quantidade de eventos que queremos guardar: // ... import io.reactivex.rxjava3.subjects.ReplaySubject class CreaturesRepository @Inject constructor ( remoteDataSource: CreatureRemoteDataSource ) { val creatures: ReplaySubject&lt;List&lt;Creature&gt;&gt; = ReplaySubject.create( 1 ) // ... } O valor da propriedade creatures pode ser inicializado no init { } , criando um subscribe no remoteDataSource.creatures e repassando o valor via onNext() . Dessa forma, a requisição só é feita durante a inicialização do CreaturesRepository . Vamos apagar todo o conteúdo do bloco init { } e substituir pelo conteúdo a seguir, adicionando alguns Logs para informar se a requisição foi bem-sucedida ou não. class CreaturesRepository @Inject constructor ( remoteDataSource: CreatureRemoteDataSource ) { val creatures: ReplaySubject&lt;List&lt;Creature&gt;&gt; = ReplaySubject.create( 1 ) init { remoteDataSource.creatures .doOnNext { Log.d( "CREATURES_REPOSITORY" , " ${it.size} creatures were loaded from API." ) } .doOnError { Log.e( "CREATURES_REPOSITORY" , "Error loading creatures from API." , it) } .subscribe { creatures.onNext(it) } } // ... } Com a propriedade CreaturesRepository.creatures preparada, podemos executar o código para testar como a requisição é feita.
