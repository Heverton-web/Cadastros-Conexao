## Introduction

Through a historical analysis of architectures tried before Representational State Transfer (REST) took center stage, this chapter explores major shifts in enterprise software. From the early days of the Common Gateway Interface (CGI) in the 90s, to the adoption of REST since the 2000s, this chapter aims to provide a history and context for the development of different approaches tried over a 30 year period. As an early web protocol, CGI became the first mechanism that allowed web servers to invoke external programs. Since then, nearly every transition from one architecture to another has been marked by the need for greater scale, flexibility, and the efficient use of limited resources. A study of technical progression gives us perspective on the path innovation has paved for newer, more effective solutions. This kind of history traces why a given approach emerged when it did, and how solutions influenced each other. By studying the past, we learn proven, tried-and-true principles, and stand a better chance of avoiding the same mistakes. The developments explored in this chapter are characterized by technical shifts that happened after technologies reached their breaking points. A technical shift signals the promise of a new way forward.

## Structure

This chapter covers a progression through:

  * CGI
  * Java RMI
  * Servlets and EJB
  * SOAP
  * REST



## Objectives

Starting with the basic concepts in CGI, when the web server was extended to generate dynamic web pages, this chapter examines how CGI quickly fell short of meeting the demands of the explosive growth of the web in the early 90s. The discussion then moves to Java RMI in 1995, which gave entire objects a way to traverse the network, yet the need for broader interoperability pushed early Java enterprise systems to a breaking point. The software industry then found stability in Servlets and EJB in 1998, though the rigidity of Java specifications led to SOAP in the early 2000s, which brought attention to open standards-based message formats. Finally, the chapter reveals why REST dwarfed these approaches as a resource-oriented architecture, its wide appeal supported by the lessons learned. In essence, REST APIs allow applications to exchange data by sending HTTP requests, with each request performing a specific action: to get, add, change, or delete data. 

## CGI

The Common Gateway Interface repositioned the web server in the early 90s. As IT professionals turned their attention to the web for applications development, they started by extending the role played by the web server. Instead of serving up static HTML pages that linked to other pages, web servers were coaxed into performing much more complex tasks. This was accomplished, in part, by running an individual process each time an HTTP request was received. This approach, called CGI, became an innovative practice before it turned into a standardized set of conventions to follow.

The software development philosophies at the time were influenced by engineering practices that favored running scripts and programs as individual processes. Do one thing well embraced the idea that programs were to be small, modular, and designed to handle a specific task. This thinking crossed over to the use of separate processes to handle web requests. The senior IT staff around Unix servers, the only guys with gray hair in the room at the time, helped to establish single processes as an engineering trend. This deeply ingrained Unix philosophy emphasized modularity and the use of small, composable tools. Proponents of CGI followed Unix pioneers like Ken Thompson, Dennis Ritchie, and Brian Kernighan, who advocated for expert programs that were simple to understand and change. For example, the grep program's one thing is searching text for patterns. The C programming language provided a straightforward method to write and run standalone programs by allowing modules to be compiled independently and linked together to form a final executable.

It felt natural to run CGI programs in isolation on Unix. Since client-server was the paradigmatic architecture at the time, CGI fit in very well by taking the server-side role, as a mechanism to generate content dynamically, in response to client requests. Unix and its variants, like Linux, were all about process-based multitasking, when an operating system runs multiple processes (programs) simultaneously by giving each its own isolated memory space and CPU time. The CGI model took advantage of this and delegated the management of multiple processes to the operating system.

A typical CGI program defined an HTML template as a structure for a web page. The template included placeholders which were sections of the HTML where data was inserted. Some programs connected to a database that provided the data used to populate the template. A database connection was usually managed through a driver, and having connected to the database, the CGI program ran a query, like a simple SELECT statement to retrieve query results, and then the program inserted data into the placeholders of the template. Finally, the program output an HTML page, placeholders filled in, and dynamically generated HTML was given back to the web server, which it then delivered to the client's browser for display.

The emergence of CGI was a response to a growing demand for web servers to play a more critical role, at the intersection between HTTP traffic and external programs. As e-commerce gained a foothold in the 90s, web applications became a new kind of business software. In this model, web apps could handle some transactions, do security, and even simulate shopping carts. CGI's compatibility with existing client-server infrastructure fit naturally within the architectural paradigm. This kind of software shaped the early web by introducing a set of interfacing conventions that more advanced server-side technologies would follow. In doing so, CGI fundamentally changed how web applications could be built. It provided an early standard for server software to handle user input, HTML output, and interface with external code. There was no precedent for this. The approach to linking server behavior with user actions influenced later server-side technologies like Java Servlets, PHP, and, eventually, modern frameworks including Node.js.

At its core, CGI defined a set of conventions for web servers to exchange data with an executable program. This was done through OS environment variables. For example, the client's IP address, the server's host name, the request method (GET or POST), and the content type, were passed to a program by assigning values to environment variables in a shell. The variables allowed the external program to understand the context of the HTTP request and to customize its response.

Figure 1.1 shows CGI's basic flow: a browser sends an HTTP request to a server, which launches a CGI script (like a standalone C program) to process it. The script's output, say, a dynamically generated HTML page, gets routed back to the client. 

Figure 1.1: CGI architecture

By convention, CGI also put forward a way to pass URL query parameters to a program. For a GET request, the query string (any data included in the URL after a question mark) would be provided to the CGI program by assigning a value to a QUERY_STRING environment variable, allowing the web server to input parameter data directly into the program. For POST requests, CGI assigned a value to a CONTENT_LENGTH variable, a measurement of how much character data was included in the request body, and passed this payload via standard input (stdin). These interfacing conventions allowed CGI programs to process web forms, making it technically possible to interact with user input. By defining a contract between a web server and a program, CGI opened itself up to multiple programming languages, Perl, C, and Python, while still conforming to the same data-handling standard.

The following is a sample CGI Program written in C, hello.c:

#include <stdio.h>

#include <stdlib.h> // For getenv()  
  


int main(void) {

// Output the HTTP header

printf("Content-Type: text/html\n\n");

// Get the value of the USER environment variable

char * user = getenv("USER");  
  


// Start the HTML output

printf("<html><head>");

printf("<title>Hello World</title>");

printf("</head><body>\n");  
  


// Print "Hello" message, replace the env variable

if (user) {

printf("<h1>Hello, %s!</h1>\n", user);

} else {

printf("<h1>Hello, World!</h1>\n");

// Fallback if USER is not set

}

// End the HTML output

printf("</body></html>\n");

return 0;

}

You can run the following command to compile the code:

gcc -o hello.cgi hello.c

Make it executable by the web server:

chmod 755 hello.cgi

Then, move it to cgi-bin:

mv hello.cgi /usr/lib/cgi-bin/

Next, enable CGI and configure the web server, such as Apache, so that when a request comes in for the /cgi-bin/ URL path, it should execute a script from this directory on the server:

ScriptAlias /cgi-bin/ /usr/lib/cgi-bin/

Finally, send a request to the web server:

http://localhost/cgi-bin/hello.cgi?user=Joe

Output:

Hello, Joe!

The uncomplicated conventions defined by CGI gave impetus to a broad range of web applications. For example, before client-side image maps became common, CGI was used to implement server-side image maps. A user would click on an area of an image, and the coordinates of the click would be sent to a CGI script, which would determine the appropriate response based on the location of the click, redirecting to another page that zoomed into the image. Other clever interactions based on CGI were used to mimic online stores. They handled user sessions, tracked items added to a shopping cart, processed checkout forms, and even sent sensitive information to payment systems. Simple discussion forums were also built using CGI, allowing users to post messages and reply to discussion threads.

While simple, a one process per request approach proved disastrous at scale. As the CGI model became more widely known and used, it proved to be tragically inefficient. Particularly under heavy load, forking a new process for every incoming HTTP request led to CPU starvation and degraded performance. Because each process requires the CPU to switch between different tasks, known as context switching, as the number of processes increases over a short period of time, the CPU spends more time switching between them rather than executing useful work. Starvation was often caused by a deadlock from one or more frozen processes, known as zombies. Processes become deadlocked when they are doing nothing but waiting for a resource occupied by another program in the same set. Killing off zombie processes turned into a daily, tedious, manual task for systems administrators.

CGI worked well for simple, web-based interactions but it was not a model on which to design sophisticated software systems. The demand to develop more scalable systems increased exponentially with the web's explosive growth. As the number of users and HTTP requests went up, the server was brought to its knees handling an increasing number of processes. At the time servers might have had between 8 and 64 MB of memory. The lower end was typically seen in entry-level servers, while the higher end, more common for mid-range or high-end servers. Single-processor Unix systems, with CPUs running at clock speeds between 25 MHz and 66 MHz (think Intel 486 or early Pentium processors) limited the number of processes that could be created. This limit could range from a few hundred to several thousand processes. Although a typical configuration would allow 256 to 1024 processes per user or across the entire system, a server with 64 MB of memory would strain under the load of more than a few hundred concurrent processes, especially if each process required memory. A more advanced Unix server with 128 MB of RAM and dual 200 MHz CPUs could handle 500 to 1000 concurrent processes more confidently, though this would still be highly dependent on the workload and process characteristics.

As scalability and performance issues pushed CGI to a breaking point, state management problems added insult to injury. Early CGI lacked sophisticated mechanisms to track user state in between requests. Each request was an isolated event with no memory of the last one. For greater security, applications required a scheme to authenticate users and keep them logged in across multiple web pages, which CGI could not achieve unless developers implemented workarounds like passing sensitive parameter values through the URL or using hidden form fields. In multi-step user flows, like form wizards or shopping carts, it was crucial to maintain transient data between steps. Lacking some form of state management, CGI programs remained stuck in a model that retrieved backend data on every request, even if it was the same data. These inefficiencies put unnecessary load on the server.

The limits of the CGI model became more pronounced over time, which put web applications development teams near the end of their rope. A radically different alternative appeared in an early version of Java, just as Sun Microsystems shipped a portable framework to write once, run anywhere. Distributed Java had compelling new features that made Java RMI an attractive alternative to CGI, to develop more scalable, distributed systems. 

## Java RMI

No other technology has embraced object-orientation quite like Java Remote Method Invocation (RMI). The framework gave objects seamless passage between remote systems by making a remote object look like a local one. This allowed Java developers to work with RMI in ways they had become accustomed to with objects co-located. It could send entire object graphs by keeping them in their original object-oriented form for transmission. In doing so, the architects of RMI remained true to Java's core principles: encapsulation, inheritance, and polymorphism, all in a distributed RPC framework. 

At the same time, object-oriented programming (OOP), a paradigm that structures software as a collection of reusable, self-contained objects, had gained industry prominence. In academia, computer science departments started offering specialized courses. OOP became so important because it introduced a modular approach to programming as an alternative to procedural programs like CGI scripts. By modeling software as objects and their interactions, OOP gave developers a conceptual space in which to think about objects as symbolic references to real world things. Objects promised to define software that was easier to maintain. OOP made it smoother to collaborate on software projects with big development teams. Following the principles of OOP, teams worked on different components, independently, through objects with clearly defined interfaces that could interact unaware of the internal workings of each other. RMI embraced all of it, fully.

Encapsulation reached a whole new level in RMI because it hid all the networking and serialization under the hood. Earlier distributed systems like RPC (language-agnostic but primitive), CORBA (language-neutral, too complicated), or DCOM (Windows-only) were overshadowed by RMI. This was pure Java, an object-oriented approach that leveraged the JVM's core strengths: automatic memory management, platform independence, and bytecode portability. Developers could focus on objects, invoking remote methods just as if they were exposed by local objects, bypassing low-level socket interactions or any platform specifics. As long as the Java Virtual Machine (JVM) was there to host them, distributed RMI applications could be written to run on any platform. Through bytecode verification and the Java class loader securing code execution, RMI promised an entirely new generation of software.

Figure 1.2: RMI architecture

As use of the RMI framework evolved, object models became more sophisticated, to address the advanced needs and requirements of modern software systems. The demand for infrastructure that could transport large and intricate object graphs across the network surged. Software engineering teams faced the challenge of passing instances of class hierarchies, like a company's entire product line, over the network, while preserving their structure and behavior. The elegant representation of real-world things and their behavior in data models as an alternative to procedural programming engendered an entirely new problem: the need to serialize efficiently, transmit, and reconstitute object graphs across the internet. Java RMI handled this gracefully in a basic way, for small objects: a to-do list, a coffee cup, a user profile, and a pet rock passed easily. Yet, the serialization mechanism broke down when it was asked to transmit bigger things like a 3D model, a satellite, a container ship, or a freight train.

Since the earliest versions of RMI, the focal point of interface design converged on a server class implementing the remote interface. An implementation of java.rmi.Remote defined the server-side of an application, its capabilities, and how to talk to it. From this implementation, the RMI compiler would generate two classes: a stub on the client side as a proxy to a remote object, and a skeleton to act as a dispatcher on the server side, receiving requests and handing them off to the actual object instantiated in the server JVM. 

For example, a Java Remote interface for a simple service that says hello, defines a sayHello() method that can be called remotely:

import java.rmi.Remote;

import java.rmi.RemoteException;

public interface Hello extends Remote {

// This method will be called remotely

String sayHello() throws RemoteException;

}

A class that implements the Hello interface and provides an implementation of the sayHello() method, returns the String Hello, World!:

import java.rmi.server.UnicastRemoteObject;

import java.rmi.RemoteException;

public class HelloImpl extends UnicastRemoteObject implements Hello {

protected HelloImpl() throws RemoteException {

super();

}

// Implement the sayHello method

@Override

public String sayHello() throws RemoteException {

return "Hello, World!";

}

}

To find remote objects, an RMI registry played an important part. A registry was a naming service that let clients look up remote objects by name. It exposed objects to clients across a network, through a directory that bound them to specific names. When a remote object registered, a client could query and obtain a reference to it. This gave clients a way to invoke methods on a remote object reference, as if it were local, and made locating objects simpler than through low-level socket code.

In the following example, an RMI server creates an instance of the HelloImpl remote object, binds it to an RMI registry on port 1099 with the name Hello, allowing clients to look up and invoke its methods. In this example, the server then starts up, with the output RMI Server is running... if successful:

import java.rmi.registry.LocateRegistry;

import java.rmi.registry.Registry;

public class RMIServer {

public static void main(String[] args) {

try {

// Create an instance of the remote object

HelloImpl hello = new HelloImpl();

// Bind the object to the RMI registry 

Registry registry = 

LocateRegistry.createRegistry(1099);

registry.rebind("Hello", hello);

System.out.println("RMI Server is running...");

} catch(Exception e) {

e.printStackTrace();

}

}

}

To move beyond the traditional RMI registry, Java Naming and Directory Interface (JNDI), a Java resource locator, made it even easier to locate object references through a uniform API for naming systems, like Lightweight Directory Access Protocol (LDAP) for user directories or Domain Name System (DNS) for network addresses. 

The process of serialization, which converts objects into byte streams, preserves state in transit. In Java, the java.io.Serializable interface marks objects for serialization. The process recursively decomposes transient fields and references to other objects into a byte stream. Then, it writes the stream through a socket, a networking object that connects to a specific server at a designated IP address and port. In this network exchange, sockets are endpoints. At the sender's end binary object data is written to an output stream, and at the receiver's end the data is read from an input stream. A reassembly of the original object state from a socket's input stream happens through a reverse process called deserialization, to recompose a byte stream back into an object instance.

In an early RMI-based system, a client would call methods on remote objects that may have returned instances of classes that the client did not have locally. Through dynamic class loading, RMI downloaded the bytecode for the classes from a remote codebase hosted by the server and loaded it up into the client's JVM. In this process, class definitions could be retrieved from a remote HTTP server. While this required some security configuration to provide assurances that the code was safe and from a trusted origin, the tradeoff for the added complexity removed the need to pre-distribute class files to all client participants in the distributed system.

Dynamic class loading delivered new object definitions as the client stayed up, no restart or recompile necessary, shown as follows:

Here an RMI client connects to the RMI registry on 'localhost' at port 1099, looks up the remote object named Hello, invokes its sayHello() method, and prints the response from the server:

import java.rmi.registry.LocateRegistry;

import java.rmi.registry.Registry;

public class RMIClient {

public static void main(String[] args) {

try {

// Locate the RMI registry

Registry registry = 

LocateRegistry.getRegistry("localhost", 1099);

//Look up the remote object 

Hello hello = (Hello) registry.lookup("Hello");

// Invoke the remote method

String response = hello.sayHello();

System.out.println("Response: " + response);

} catch(Exception e) {

e.printStackTrace();

}

}

}

To compile the classes, run the following command:

javac Hello.java HelloImpl.java RMIServer.java RMIClient.java

To start the RMI server, run the following command:

java RMIServer

Now, run the RMI client:

java RMIClient

The following is the output:

Response from server: Hello, World!

Despite its strengths, Java-to-Java communication did not play well with non-Java systems. Lack of interoperability turned into a devastating limitation for RMI. So purely defined into the Java ecosystem, RMI captured the minds of Java developers on the forefront of software engineering in the mid-90s, but it did so to the exclusion of systems written in other languages. This shortcoming impeded RMI's adoption in polyglot environments, as developers needed to integrate Java with legacy software written in C, C++, Perl, or Python. RMI's Java-centric approach became less appealing as distributed systems evolved towards language neutral solutions, even more so with the rise of web services that figured out platform independent communication.

The absence of an interoperable framework compatible with different languages stunted RMI's growth. It also lacked built-in support for important enterprise features, like transaction management, security, concurrency control, and object lifecycle management. While RMI hid all the serialization and networking code inside the framework, developers were still on the hook to manage remote object lifecycle. They were left on their own to handle low level tasks like cleanup, resource allocation, and object destruction. This brought overhead to large scale distributed applications, where interoperability became more important than pure Java-to-Java communication. As a result, RMI's allegiance to Java and the lack of standard support for common enterprise software needs inhibited more widespread use.

### The Applet Experiment

Sun's vision for Java everywhere found its most ambitious expression not on the server, but in the client's browser, through something called an Applet. These were small Java programs embedded in web pages, designed to run in a secure sandbox within a JVM inside the browser. As a component integral to a distributed Java architecture, Applets introduced a powerful unifier: you could make remote method calls on remote objects on the server by embedding an Applet in a web page. This was a radical alternative to the CGI model, that promised rich, dynamic user interfaces which could update without full page reloads.

To load a JVM and download Applet code from a browser was slow and heavy. The approach was plagued by browser compatibility headaches. Ultimately, web browsers moved away from a plugin model towards a compromise: efficient server-side processes (Java Servlets) communicating with a lightweight client that grew more capable through a browser native language, JavaScript. Applets paired with RMI demonstrated a potential for rich web applications, but they started to look more and more like fat clients, which proved unsustainable.

While Java RMI provided an early model for distributed computing, system requirements grew beyond its capabilities. Lessons learned from RMI's limitations gave rise to platform agnostic technologies like SOAP, and later, REST APIs. Before this happened, modern frameworks like Enterprise Java Beans (EJB) filled in some of RMI's shortcomings, and with it, a whole new market was created in an era when the big application server took center stage.

## Servlets and EJB

If RMI was all about remote objects, then EJB, a Java standard for reusable, server-side business logic, was all about standard components. In the late-90s, Sun Microsystems transcended their role as the original stewards of the Java programming language by positioning themselves as the architects of an enterprise ecosystem. Sun created a first of its kind standards-based platform for distributed computing. It defined the Java Community Process (JCP) to govern an open collaboration model, and Java Specification Requests (JSRs) for tracking formal proposals of Java features. Standardization in this space meant that developers could mix and match components from different vendors. Sun's vision sent an open invitation to software vendors to enter an abundant marketplace of solutions, so long as they adhered to the spec.

JSR standards put forward a clear role for EJB container providers and server providers, defining separate and distinct responsibilities for software vendors to implement. The EJB container provider managed the runtime environment for enterprise beans, which hosted the logic to execute at runtime. Such an environment was expected to handle transactions, some security, multi-threading, and pooling for beans. The container also ensured that beans followed the EJB specification at deployment and runtime. On the other hand, the server provider supplied infrastructure required to operate containers, including remoting, networking, and load balancing. By separating these roles, JSR standards established a components-based approach for applications development where container and server components could be developed and versioned independently by different vendors, while staying compatible with each other. Combined, the two components became an application server and a $4 billion dollar market.

Servlets were given a role in the Java enterprise ecosystem, addressing a different layer of the application architecture. Servlets handled web content, were based upon a spec, and made responsible for receiving HTTP requests and generating responses. They formed a layer of abstraction essentially to hide low-level HTTP. Servlets defined the web-tier and user interface logic, while EJBs had the business-tier logic, allowing development teams to separate these two concerns and to divide the labor in creating them. While a separation of concerns made perfect architectural sense, experts in the space argued that Servlets provided everything necessary to build a complete web application without all the complexity surrounding EJBs. A Servlet can directly handle HTTP requests, execute business logic, manage database connections, and render data for a browser. The servlet-centric model was often simpler, performed incredibly well, and became faster to develop for a large class of applications. While proponents saw EJB as the great unifier that standardized complex enterprise systems, their value proposition was questioned. Some critics viewed EJB as somewhat over-engineered, adding more overhead than value.

Looking past its critics, EJB also formalized a distinction between session beans for short-lived processes, and entity beans for the persistence of data, setting clear rules for their lifecycle. Session beans embodied the business logic of an application and the interactions specific to a particular client session. Their lifecycle was not intended to go beyond the session's lifecycle. Session beans were either stateless or stateful. To share the same instance across client requests, stateless session beans were perfect for handling a login or fetching some data. Stateful session beans maintained a conversation across multiple method calls for a specific client and could track the progress of a multi-step flow. Once the client session ended, the stateful session bean's state would be reset.

Entity beans represented longer lived data retrieved from database records. Each entity bean instance had some correspondence to a row in a database, with a lifecycle dependent upon the database operation: insert, update, or delete. Like a cache, entity beans had a longer lifespan, held beyond the session that created or modified them. They modeled real-world business entities, like a customer or an order, and managed data persistence automatically through the container, or manually, where developers could write the persistence logic, to define exactly what happens when a database record is saved from an instance of an entity bean. 

The following figure sketches the basic architecture of an EJB system, showing how an EJB client interacts with the EJB container and server. In this illustration, a client looks up EJB Home interfaces via a JNDI registry, and then calls methods on the remote interface of session and entity beans. The figure also shows entity beans managing persistence to a database.

Figure 1.3: EJB architecture

By formalizing these roles, EJB defined a business-tier comprised of transient processes along with longer-term data caching within enterprise applications.

Declarative configuration kept the business logic code separate from the way it was configured to run. Through deployment descriptors (XML files), elements such as transaction attributes, security roles, and resource mappings stayed outside of the EJB code itself. For example, transactions followed the <transaction-type> element in an XML descriptor, and security constraints could be set using <security-role> entries. Annotations, introduced in later EJB versions, supported another way to embed configuration in the Java code, to remain separate from the business logic. By keeping configuration separate and distinct from the business logic, EJB ensured that changes to configuration settings, like updates to security settings, could be made without code modifications, thereby promoting a cleaner, more serviceable, application environment.

A simple example of a stateless EJB session bean that returns Hello World! when invoked is as follows:

import javax.ejb.Stateless;

@Stateless

public class HelloWorldBean implements HelloWorldRemote {

@Override

public String sayHello() {

return "Hello, World!";

}

}

A remote interface that declares the method exposed by the session bean is as follows:

import javax.ejb.Remote;

@Remote

public interface HelloWorldRemote {

String sayHello();

}

An EJB would typically be deployed to an EJB container (like JBoss, GlassFish, WebLogic, or WebSphere). A sample servlet that invokes the stateless session bean, that uses JNDI to look up the EJB and call its sayHello method:

import javax.naming.*;

import javax.servlet.*;

public class HelloWorldServlet 

extends HttpServlet {

private HelloWorldRemote helloWorldBean;

@Override

public void init() throws ServletException {

super.init();

try {

String name = "HelloWorldRemote";

// Set up the JNDI context

Context context = new InitialContext();

// Lookup the EJB

helloWorldBean = 

(HelloWorldRemote) context.lookup(name);

} catch (NamingException e) {

throw new ServletException(e);

}

}

@Override

protected void doGet(HttpServletRequest req, 

HttpServletResponse res)

throws ServletException,

IOException {

// Invoke the EJB method

String message = helloWorldBean.sayHello();

// Write the response

res.setContentType("text/plain");

res.getWriter().write(message);

}

}

To support high traffic, the EJB specification allowed a container to load-balance client requests across multiple instances of session beans, ensuring that no single instance became a bottleneck. Increased performance and scale were also achieved by pooling beans, and reusing instances. The EJB container vendor did most of the work to scale applications horizontally and vertically, which gave greater focus to developers on writing the business logic.

Even so, container management had major overhead, and a heavy reliance on the container led to fierce criticism, which encouraged the development of more lightweight solutions in subsequent years, including JavaBeans and Spring. While EJB solved many of the enterprise-level concerns RMI could not address, an overdependence on container services caused performance issues. Developers started looking for more flexible solutions, to simplify development and reduce overhead.

In the late 90s and early 2000s, EJB drove innovation in the space. It established Java as a mainstream platform for enterprise applications development. While Sun Microsystems set the standard for enterprise components, it also grappled with the challenge of finding a balance between standardization and flexibility. Standardization encourages vendors to interoperate, but it requires them to be locked into a specification. Well-defined standards are often well-intentioned, but they constrain innovation to the technical space in which they play. As a result, for a solution to play well with a vendor's spec, means that its response to rapidly changing market needs may be slow, rigid, and bounded by the standard itself. This tension between conformance and agility led to lightweight approaches for distributed computing, as engineering teams migrated to Simple Object Access Protocol (SOAP), a completely different way to exchange structured information over the web using XML.

## SOAP

SOAP's initial appeal had a basis in a commitment to interoperability through open standards. Earlier approaches operated within closed systems, but SOAP used XML as a platform-neutral format, opening communication with a diversity of applications. It proposed a framework for the transmission of standardized message formats, for systems written in different languages to exchange data. By incorporating Web Services Description Language (WSDL) into a set of open standards, SOAP catapulted web services into the mainstream.

By adhering to an open definition for messaging, SOAP supported communication between applications, regardless of platform or programming language. Messages had three basic parts: an envelope, which defined a message structure, a header, with optional metadata, and a body to hold the data for exchange. Above all, SOAP was supposed to be neutral. With the oversight of the World Wide Web Consortium (W3C), it became a recommended web standard in 2003. SOAP then quickly gained prominence by taking a ride on top of HTTP. At a time when firewalls blocked proprietary protocols like CORBA and DCOM, HTTP had an open path for communication on corporate networks. SOAP capitalized on this. It packaged XML messages in HTTP payloads that were already allowed through open ports 80 and 443. This ability to traverse firewalls gave SOAP a major advantage, as it bypassed security that blocked other protocols. At the same time, XML messaging created a sensation in the early 2000s. As a human-readable, structured markup language, XML empowered developers with the schemas and syntax to represent anything, and to play a whole new game.

As a text-based protocol, objects came naturally to SOAP. While the data on the wire was plain text, development tools provided an automatic mapping between XML schema definitions and object classes in languages like Java and C#. This meant a developer could work with a native Customer object in their code, and the SOAP framework would automatically serialize it into XML for transport and deserialize the XML response back into an object. So developers had the best of both worlds: a firewall-friendly, interoperable, text-based protocol, and the familiar, structured paradigm of OOP.

In SOAP messaging, an envelope contains the entirety of a SOAP message. This envelope has two distinct and essential components: a header and a body. The header can include metadata, authentication credentials, transaction identifiers, or routing details nested inside a <soap:Header> element. Headers convey specifics about message processing, without affecting the payload. The body, in a <soap:Body> element, carries the payload of a message. This element holds either request or response data as XML. The structure and semantics of the body are defined by a web service's WSDL, and the specific XML schema associated with the request and response. Combined, header and body within a SOAP envelope deliver a well-formed message.

### SOAP request

This SOAP message encapsulates a request with a custom header for message tracking and a simple GreetRequest operation inside the body, following the SOAP protocol structure:

<?xml version="1.0" encoding="UTF-8"?>

<soapenv:Envelope xmlns:soapenv="http://schemas.xmlsoap.org/soap/envelope/" xmlns:hello="http://example.org/hello">

<soapenv:Header>

<tracking:MessageID xmlns:tracking="http://example.org/tracking">12345</tracking:MessageID>

</soapenv:Header>

<soapenv:Body>

<hello:GreetRequest/>

</soapenv:Body>

</soapenv:Envelope>

### SOAP response

This SOAP message contains a response with a custom tracking header and a GreetResponse operation in the body, which returns the message Hello, World! as part of the service's response:

<?xml version="1.0" encoding="UTF-8"?>

<soapenv:Envelope xmlns:soapenv="http://schemas.xmlsoap.org/soap/envelope/" xmlns:hello="http://example.org/hello">

<soapenv:Header>

<tracking:MessageID xmlns:tracking="http://example.org/tracking">12345</tracking:MessageID>

</soapenv:Header>

<soapenv:Body>

<hello:GreetResponse>

<hello:Message>Hello, World!</hello:Message>

</hello:GreetResponse>

</soapenv:Body>

</soapenv:Envelope>

The architecture of SOAP builds on standard XML messages, multiple transport protocols, and on an integration with WS-* standards. The standards came out of OASIS and the W3C, as web services evolved, to solve security and other issues that were not fully covered by SOAP itself. For instance, WS-Security embeds tokens in the headers to use in authentication and authorization processing. It also cryptographically signs parts of the message to guarantee integrity, and it recommends a strong encryption scheme for confidentiality. WS-Addressing points to an intended recipient, a sender, and to any other endpoints that may mediate a message.

For example, based on WS-Addressing, a destination address where the message is intended to be delivered was given by:

<wsa:To>http://example.org/receiver</wsa:To>

To identify the originating endpoint of a message through the WS-Addressing standard, the following would be included in a message:

<wsa:From>

<wsa:Address>http://example.org/sender</wsa:Address>

</wsa:From>

To specify where a response should be sent, the following was included in a SOAP message:

<wsa:ReplyTo>

<wsa:Address>http://example.org/response</wsa:Address>

</wsa:ReplyTo>

Finally, to indicate where errors should be directed, an endpoint for faults could be specified through:

<wsa:FaultTo>

<wsa:Address>http://example.org/fault</wsa:Address>

</wsa:FaultTo>

The intent of the WS-* standards was to make sure everyone was doing the same thing, instead of their own thing, when it came to security and routing, while using the open, extensible markup language.

All the hype about SOAP peaked when it fused into the strategies of Microsoft, IBM, and Oracle. The big players wanted to dominate in an expanding web services market. This market in 2003, valued at around $10 billion, was projected to surpass $50 billion by the end of the decade. For Microsoft, SOAP was made integral to the .NET platform, with products like Microsoft BizTalk Server, designed for web services orchestration. Strategic acquisitions, for example Great Plains in 2000, gave Microsoft more Enterprise Resource Planning (ERP) capabilities that complemented their SOAP-based solutions. This also earned them greater market share.

IBM capitalized on SOAP as well, mainly through WebSphere, in support of enterprise application integration. The company's acquisition of CrossWorlds Software in 2001 strengthened this position. Oracle followed suit by building support for SOAP into their Oracle Application Server. The acquisition of PeopleSoft in 2004, along with the purchase of BEA Systems in 2008, added WebLogic Server to Oracle's portfolio, which grew their enterprise software offerings. These acquisitions allowed them to integrate SOAP-based web services more deeply into their solutions, giving them a competitive edge in the web services market. 

Combined, strategic acquisitions by IT giants signaled a deep financial investment in SOAP web services to capitalize on their potential. Yet a pernicious embrace by big corporations led to solutions that clashed with the open, decentralized nature of the web. As SOAP became synonymous with software and systems favored by corporate interests, it appealed less to a broader developer community. The core tenets of the early web standards, simplicity, neutrality, accessibility, meant much more to this community. After all, the web, inherently uncontrollable by any single entity, favors complete openness. While SOAP was featured front and center as a possible future for web services, a vendor led agenda revealed a disconnect between enterprise ambitions and the grassroots nature of the web.

## REST

At around the time that SOAP peaked in use during the early 2000s, Roy Fielding at the University of California Irvine (UCI), published his PhD thesis titled Architectural Styles and the Design of Network-based Software Architectures. At UCI, Fielding watched perturbed as the web sprawled and fragmented in different directions architecturally. Commercial interest had peaked for the Internet to become a platform for business, and developers working in the corporate sector disagreed on an architecture that would allow them to tap into its unlimited powers. Fielding took an interest in reconciling these differences through common standards that everyone could agree to adopt. A lead author of the URL and HTTP specifications, Fielding recognized that objects (representations of things) with similar behavior already given to HTML documents, could be transmitted following the same basic conventions. The hyperlink had been conceived long before by Ted Nelson in 1965, who also coined the terms hypertext and hypermedia. And Tim Berners-Lee had figured out at Conseil Européen pour la Recherche Nucléaire (CERN), as early as 1980, that documents could be linked across networks to form a web of information. When Fielding started to write about REST, HTTP-based documentation systems had already become a global, cross-linked mesh of vast information and knowledge sharing. Deeply influenced by Nelson and Berners-Lee, Fielding found the right words to describe an architectural style that could make the web work for applications too, by defining the way components of a distributed system should interact.

At a time when his friends were becoming millionaires at Internet startups, Fielding wrote and published his PhD thesis. His advisors joked that no one reads dissertations. Even so, careful to express the ideas clearly, he agonized over the text, taking the time to craft a description of an architectural style that withstood rigorous academic review while remaining open to non-experts. It was nearly 10 years after its publication that REST attracted any attention at all. Then, Fielding's research redefined scalable distributed systems for the next 20 years, transcending any alternatives on pure intellectual ground. This is systems and software architecture at the scale of decades. As the concepts in the thesis became more broadly understood in terms of interface design, Representational State Transfer (REST) largely displaced SOAP- and WSDL-based interfaces. 

A RESTful service displays six qualities that place it within a bounded framework of rules: 

  * Client-server 
  * Stateless
  * Cacheable
  * Layered
  * Uniform interface
  * Code-on-demand 



These boundaries are also often referred to as REST's constraints. They produce a flexible and scalable system. Client-server architecture separates the user interface (client) from data processing (server), so that each can evolve independently. Stateless means that clients send everything the server needs to process a request, independent of context about the client or the request. Cacheable responses, by contrast, allow the client to store responses and reuse them. This reduces server load. As a layered system, REST includes load balancers, proxies, and gateways as intermediaries to negotiate between the client and server. The uniform interface constraint standardizes on ways to identify resources, manipulate, and represent them. Consistent interactions between clients and servers happen through a uniform interface. Last, code-on-demand is an optional feature where servers can send executable code to clients, like JavaScript, delivering functionality on demand rather than pre-configuring the client. Put together, the constraints define the architecture of the web.

### REST API request

The following request fetches the details of a user with the ID 123:

GET /users/123 HTTP/1.1

Host: api.example.org

Accept: application/json

### REST API response

In response to the GET request for user details, the server returns the following JSON object containing the user's information:

{

"id": 123,

"name": "John Doe",

"email": "johndoe@example.org"

}

The uniform interface of a RESTful service is based on patterns that focus on resources. These patterns include resource identification, manipulation, and representation. Resource identification is all about URIs. They point to resources, and serve as stable references, like /users/123 in the preceding example. A Uniform Resource Identifier (URI) is simply the web equivalent of a street address, like the address of a building in a city. The address remains the same even if the contents of the building change over time, just as a URI should stay fixed as the resource evolves. Resource manipulation acts upon resources to alter their state, using standard HTTP methods like GET to retrieve, POST to create, PUT to update, and DELETE to remove. Resource representation formats a resource to present to the client, typically as JSON or XML. This is a snapshot of the resource's current state. By decoupling the resource, the thing in the backend, from its representation, REST adapts to clients, so they can interact with the same resource in more than one way.

A RESTful system uses standard HTTP status codes, and hypermedia as the engine of application state. Status codes refer to REST's standardization on the well-known, 200 OK, 404 Not Found, 201 Created statuses, to convey the result of interactions with resources through HTTP methods. Hypermedia as the Engine of Application State (HATEOAS) links resource representations, to help clients move through the system's space. Links that guide dynamically, instead of hard-coding the pathways between resources, ensure that clients remain decoupled from the server, using only the server-provided hyperlinks to make state transitions.

The following figure illustrates the core components and interactions in a RESTful system, highlighting how a client changes its state (State 0, State 1, State 2, State 3), by following URLs (e.g., https://example.org/users/123) that send requests to a server which returns JSON formatted resource representations:

Figure 1.5: REST architecture

REST vs. SOAP in the early to mid-2000s happened at a time when the industry needed a simpler way to interface between systems. SOAP solved many of the problems EJB could not, but a total reliance on XML made it look awkwardly heavy. Mobile applications demanded lighter, stateless APIs, which REST provided. As Roy Fielding's thesis became known, REST's focus on resources clicked with API designers. An adherence to the core architecture of the web inevitably led to REST's dominance in API design, displacing SOAP in most modern applications. As a result, RESTful APIs remain, to this day, the de facto standard for web services.

There are two vantage points from which to view REST: fundamentally, as the architecture of the web, and through a historic lens as a successor to CGI, RMI, EJB, and SOAP for interface design patterns. CGI, one of the earliest web protocols from the 1990s, allowed servers to execute scripts in response to web requests but performed poorly at scale, as each request required a new process. RMI and EJB gave way to distributed computing through methods executed across machines. These technologies represented a powerful interplay between client and server but forced a tight coupling and required big infrastructure. SOAP built on XML-based messaging standards as a platform-neutral interface for remote procedure calls, but verbose messaging, rigid structure, and a proliferation of XML schemas made it a mess for developers, and poorly suited for lightweight web and mobile applications.

REST demonstrates a much simpler approach to web interface design. Through the standard HTTP methods like GET, POST, PUT, and DELETE, REST operates in the same space as the web itself. And a focus on resources organizes the interface designer's attention in a way that earlier technologies could not. REST's decentralized adoption has rapidly expanded into a broad ecosystem of APIs supporting mobile apps, cloud services, as well as microservices. It is estimated that millions of REST APIs have been created since REST appeared in the early 2000s, fueling the exponential growth of web and mobile applications. As a result, REST is still positioned as the dominant architectural style for web services. 

## Conclusion

The progression of change in programmatic interface design discussed in this chapter resists characterization as a simple evolution. Instead, it happened through distinct paradigm shifts. Each shift departed radically from the past, as a response to failures in previous architectures to meet the growing demands of the web. CGI, RMI, EJB, and SOAP were not simply step-by-step, incremental improvements upon one another, but broke the rules of interface design by redefining them. Each technology introduced a new way to think about communication between distributed systems. These turns highlight the disruptive nature of change in this domain. Framing enterprise architecture in terms of a history sharpens our understanding of REST's widespread use, and reveals that in systems design breaking from the past is a constructive act, not a destructive one.

## Introduction

The expressive power of URIs (Uniform Resource Identifiers) lies in an infinitely flexible syntactic structure. URIs standardize a format to uniquely identify resources on the web. What looks like just another string of characters can distinguish a resource regardless of where it lives on a network. At first glance, URIs may seem unremarkable, flat, and numbingly uniform. But a simple string of everyday characters acquires a more profound meaning through the principle of addressability, which establishes a consistent syntax for locating resources. URIs address concepts, entities, or sets of entities, and mark the surface of a REST API (Application Programming Interface) with clear paths to them. As such, URIs become globally identifying constructs by referring to resources at any location. In doing so, they enable interoperability across a broad range of systems and platforms.

Addressability as a principle in REST API design emerged from the need to build systems that share resources between the nodes of a decentralized network. Early web systems relied too heavily on coupled architectures in which resource identification remained tied to local and platform-dependent semantics. Before URIs, the address of a piece of data was typically a set of instructions, like a database query coupled to a database schema, that only worked on a specific machine. Other examples include Windows network share paths that work exclusively on Windows machines authenticated on proprietary corporate networks. This dependence severely limited compatibility with other platforms. For the web to scale, a uniform framework for locating resources was needed. Addressability through URIs solved for this problem, and provided a foundation for the modern web.

## Naming conventions for resources

Consistent naming can transform an untidy mix of abstract resources into clear, interactive endpoints. APIs become much easier to navigate by following a few simple naming rules. Whether they are used to distinguish between collections and individual items, or to structure resource hierarchies, good names make APIs self-documenting.

To achieve this level of clarity, there are 5 key naming conventions that every RESTful API should consider. The following sections recommend nouns instead of verbs in resource names, pluralization to distinguish collections from individual items, hyphens as word separators, namespaces within resource hierarchies, and resource versioning strategies. The names given to resources make them simpler to maintain over time by applying these conventions.

### Using nouns instead of verbs

Since a noun naturally reflects a thing rather than an action, a naming scheme for API resources centered on nouns can maintain this distinction. If we think of the core entities exposed to clients as the subjects of the API, then nouns, not verbs, make more sense as resource names, in terms of the basic rules of English language grammar. For example, the two sentences "I kicked the ball" and "The ball was kicked" provide a grammatical example. The first sentence focuses on an action, kick, while the second is centered on an object, the ball. Neither sentence breaks any grammatical rules, but they do have a different grammatical structure, which means they are built different. A subject refers to an entity, while a verb explains what happens to it. Just as a sentence centers on a subject modified by a verb, an API endpoint centers on a resource modified by an HTTP method. So that naming a resource User focuses on the entity itself, while naming it CreateUser includes an action. For developers to understand resources (subjects) independently of operations (verbs), a separation between subjects and their predicates makes the API grammatically sound, so to speak.

#### Example 1

Let us name a resource CreateBook, by combining a name with an action.

GET https://example.org/CreateBook

This is bad practice because the rules that govern the grammar of an API break down. We do not know if we are GETing a book or CREATing one. If both, then the interaction with the API through such an endpoint would be like saying: I am going to the library to get a book and write it. To avoid this ambiguity, the action of creating, retrieving, updating, or deleting a book should be addressed by the appropriate HTTP method applied to the Book resource. As such, the API endpoint to create a new book should be defined as a POST request sent to a /books URI: 

POST https://example.org/books

While retrieving information about a specific book should be a GET request sent to a /books/{id} URI:

GET https://example.org/books/{id}

By anchoring the endpoint to a noun "Book", the entity is clearly identified, and the action becomes secondary, taking place through the HTTP method.

#### Example 2

Instead of naming a resource UpdateCustomer, name it Customer to represent the entity you are working with. The action of updating a customer should be handled by the PUT or PATCH HTTP methods. To update a customer's information, in whole or in part, the API could expose the following endpoints:

PUT https://example.org/customers/{id}

PATCH https://example.org/customers/{id}

Here, {id} is a unique identifier for the customer. By naming the resource "Customer", the focus remains squarely on the entity, and the HTTP methods (PUT, PATCH) specify the intended action, keeping the API design clean, grammatically correct, and focused on the resources themselves.

#### Example 3

Instead of using a verb like "DeleteProduct" to represent a resource, use "Product" to represent the resource itself. The action of deleting a product would then be executed through the DELETE HTTP method. For example, the endpoint would be:

DELETE https://example.org/products/{id}

This would delete the product with the specified {id}. This naming convention allows the API to focus on the entity (Product) while the verb (HTTP DELETE) conveys the operation performed on it.

### Pluralize resource names

To make it unmistakable that an endpoint returns multiple items, plural nouns over singulars aligns with the expectation that collections of resources lie underneath. For an endpoint like /users, access to a collection of user records is clearly implied, whereas /user may suggest a single user or an ambiguous reference to a particular user. Plural nouns help developers to trust the meaning in the structure of a URI, avoiding misinterpretation. Pluralization also separates collection-level operations, like retrieving all users:

GET https://example.org/users 

From item-specific operations, like retrieving or modifying a single user: 

GET https://example.org/users/{id} 

By sticking to this rule, RESTful APIs self-document and remain consistent in their collective resource naming strategy.

#### Example 1

The GitHub REST API emphasizes plurals in its resource names, such as /events, to expose collections of events happening around repository activity. If GitHub had instead used the singular form /event, developers might struggle to decipher if the endpoint refers to all events, the most recent one, or some arbitrary event. The GitHub API ensures clarity by using pluralized names.

Pluralized, to retrieve all the events for a repository, this is good:

GET https://api.github.com/repos/{owner}/{repo}/events

Retrieve the details for a specific event; this is also good:

GET https://api.github.com/repos/{owner}/{repo}/events/{eventId}

Singular, creates ambiguity about the scope of the resource retrieved, and this is bad:

GET https://api.github.com/repos/{owner}/{repo}/event

By following this convention, and avoiding the bad practice of using singulars, GitHub provides consistent use of plural names for collections and avoids confusion to ensure a predictable interaction with its API endpoints.

#### Example 2

PagerDuty's API provides endpoints to manage notifications for incident tracking purposes, using plural names like /incidents to represent collections of active or past problems or issues. Singular naming, such as /incident, would obscure the endpoint's ability to retrieve all incidents, the latest incident, or a specific one. By following a plural naming convention, PagerDuty avoids this obscurity.

Pluralized, retrieve all incidents for an account:

GET https://api.pagerduty.com/incidents

Retrieve details for a specific incident:

GET https://api.pagerduty.com/incidents/{id}

Singular, ambiguous, not sure if this returns a list, the newest incident, or a default item:

GET https://api.pagerduty.com/incident

The clarity provided by pluralized names gives PagerDuty's API self-documenting qualities and avoids ambiguities, allowing developers to manage incidents with greater precision.

#### Example 3

Stripe's API for e-commerce and payment processing uses pluralized names like /disputes to represent collections of disputed charges. If Stripe had adopted singular naming like /dispute, developers might expect the endpoint to return a single resource instead of a collection. By pluralizing resource names, Stripe avoids this problem.

Pluralized, retrieve all disputes for a customer: 

GET https://api.stripe.com/v1/disputes

Retrieve details for a specific dispute:

GET https://api.stripe.com/v1/disputes/{id}

Singular, confusing, developers might interpret this as fetching the latest dispute or some other arbitrary result:

GET https://api.stripe.com/v1/dispute

Through consistent pluralization, Stripe ensures its API adheres to RESTful best practices, making it intuitive for developers to interact with collections and individual resources.

### Use hyphens as separators

The surface of an API, much like the surface of a well-crafted piece of writing, has a structure that supports meaning and purpose. Hyphens sometimes serve at the character-level to shape this surface, by separating the words into compound resource names. Where backend data models expose composite entities (logical groupings of data that combine multiple attributes or related objects into a single unit), hyphens enhance their readability. Hyphens hold the name together while keeping it concise and uncluttered. Unlike camelCase or under_scores, hyphens flow more naturally into the API's surface structure in a way that matches how we parse, read, and process language. This simple convention helps to maintain a clean surface, and makes the API more readable for developers and automated systems, such as web crawlers.

#### Example 1

Consider a hypothetical example where the resource endpoint /user-profiles refers to a collection of user profile data. In this example, a hyphen separates user and profiles, clearly indicating that the resource is a collection of profile records for users. A hyphenated form is preferred over alternatives like userProfiles or user_profiles.

GET https://api.example.com/user-profiles

#### Example 2

Another API may use /order-items to represent a collection of individual items in a customer's order. By using a hyphen to separate "order" and "items", the API makes it clear that this resource deals specifically with items within an order, not the order itself: 

GET https://api.example.com/order-items

#### Example 3

A third sample API includes an endpoint for retrieving a collection of notifications for alerts, called /alert-notifications. The hyphen between alert and notifications suggests a direct relationship between the two entities, but not necessarily a hierarchical or subordinate one:

GET https://api.example.com/alert-notifications

Hyphens act as natural delimiters. They segment words within a path. This practice increases indexing accuracy for search engines, because each word before and after a hyphen becomes a token when it is crawled. In doing so, this practice helps automated systems to understand the structure of a resource. Unlike camelCase or under_scores, which can blur word boundaries, hyphenated resource names match the patterns expected by search engines or other web-based tools for crawling and semantic parsing.

### Resource namespaces

When backend data models exposed by REST APIs embody a tree-like structure, URI syntax becomes an important medium through which to express relationships between tree nodes. URIs can communicate containment and hierarchy, such as a folder containing files or a sub-category underneath a category. By embedding relationships into a tree-path structure, URIs reflect the branching, containment, or subordination in a data model. This approach follows a basic RESTful principle, uniform interface, in which the API surface reflects the logical organization of the data in the backend through a namespace.

#### Example 1

A URI structure for a book catalog that allows users to easily locate books by genre, author, and book title in a predictable, logical sequence, could look like this:

URI template:

https://libraryapi.com/books/{genre}/{author}/{book-title}

URI instance:

https://libraryapi.com/books/fiction/jk-rowling/harry-potter

In another example of resource namespaces, a segmentation of users by region and department creates a predictable tree structure that simplifies searching for user profiles.

URI template:

https://companyapi.com/users/{region}/{department}/{user-name}

A third example of resource namespaces makes it easy to navigate specific products through a brand/category/product path, regardless of where the resources reside physically (that is, where they are stored) across network regions. 

URI template:

URI instance:

https://storeapi.com/products/nike/sneakers/air-max

The following figure illustrates a REST API namespace organized to address resources across multiple network regions. Starting from the root node /products, the namespace branches into nodes /nike and /adidas, to represent different shoe brands. Under each brand, sub nodes such as /sneakers, contain specific product lines like /air-max or /samba, to define sneaker subcategories. The dashed line in the figure depicts a boundary between network regions A and B, that the namespace crosses to address resources in each of the regions.

Figure 2.3: Products model crossing networks

Giving thought to the namespaces of REST APIs is recommended because they provide predictability for developers interacting with the system. Predictable, logical URI structures reflect the data organization in the backend. They also guide the paths to follow to address a resource. In their reflection of a backend data structure, hierarchical URIs may become a mental map, thereby reducing cognitive load on API users by eliminating the need to reconstruct relationships.

As the API grows more complex, consistent naming schemes scale up through a framework that accommodates new resources easily. Thoughtful namespaces support growth in a hierarchy, to scale into evolving requirements. This foresight in API design sets the stage for future growth.

## Identifying resource versions

Resource versions allow APIs to update without breaking their clients. To ensure clients can continue to consume an API in one way or another, a version number is typically added somewhere as an identifier. Changes that break the contract between client and server when APIs either grow to incorporate new features, or address technical limitations built into older versions, inevitably push for a new version. Without a versioning strategy in place, any modification to a resource format or adjustment to endpoint behavior can disrupt clients that are coupled to previous versions.

When an API needs to take the next step in its evolution, API designers have at least four options to introduce versioning.

### URI versioning

One of the most commonly accepted methods for versioning APIs involves incorporating a version number directly into the URI, as in /v1/resource. API designers favor this approach because of its simplicity and visibility. With a version number embedded into the URI path, clients can manage the transition between versions by addressing resources through separate and distinct paths.

#### Examples

In the following examples, an e-commerce platform uses URI versioning to manage differences in its product catalog endpoint:

Request 1

https://api.ecommerce.com/v1/products

Response 1

[

{

"product_name": "Wireless Mouse",

"price": 25.99

},

{

"product_name": "Bluetooth Keyboard",

"price": 49.99

}

]

Request 2

https://api.ecommerce.com/v2/products

Response 2

[

"product_name": "Wireless Mouse",

"price": 25.99,

"description": "Ergonomic wireless mouse",

"category": "Electronics",

},

{

"price": 49.99,

"description": "Sleek Bluetooth keyboard",

"category": "Electronics",

"availability": "Out of Stock"

}

]

In v1/products, the API may only return a list of product names and prices. While in v2/products, the API returns more detailed product information, such as product descriptions, categories, or product availability details. By specifying a version in the URL, the platform supports clients using version 1 in a backward compatible way to continue operating as expected, as new clients can consume the enhanced output from version 2.

### Header versioning

An alternate approach that minimizes duplication in endpoint paths allows APIs to evolve without changing the URI structure, through a custom HTTP header such as 'X-API-Version: 1'. HTTP headers are key-value pairs sent between clients and servers to convey metadata, such as content type, authentication tokens, or, in this case, API versioning preferences. Clients can request a specific version in a header through the same endpoint. This approach conceals the version by tucking it inside a header, preventing the version number from being exposed in the URL, which often appears in server logs.

#### Example

A task management application might use header versioning for its API, allowing clients to access different versions of task data or task-related functionality. A client might make a request as follows:

If a client leaves the version header out, then an API gateway could assume the latest version is requested by default (for example, version 2). Or it could route the request to a stable version 1, if the client is using an older or default API version, based on system settings. A header allows for easy version management while ensuring that requests are still routed properly, even if the client does not explicitly specify a version.

### Media type versioning

To achieve precise control over versioning at the data format level, version information can be included in the 'Content-Type' or 'Accept' headers. This approach de-clutters the URI from version numbers and supports an inconspicuous way to run multiple versions simultaneously. For APIs that need to support multiple client types, such as web, mobile, and IoT devices all at once, media type versioning adapts API responses to client capabilities through a version-specific format in the response.

#### Example

In this example, an e-commerce platform uses media type versioning to serve different levels of product information to clients based on their capabilities. For instance, a web client may request a more detailed product catalog with specifications, reviews, and images, while a mobile client may only request basic information, such as product names and prices, to optimize performance and data usage. ## Introduction

HTTP methods function as the verbs in a dialog between clients and servers. They allow both parties to express themselves intentionally on the web. REST APIs, grounded in the principles of HTTP, state the action to take on a resource through GET, POST, PUT, and DELETE methods, which enable participants in a cross-network dialog to convey their intent to each other unequivocally. This intention comes from the need for specific outcomes. In most applications that manage data, regardless of their purpose, a set of operations called create, read, update, and delete (CRUD), sustain the lifecycle of data. REST APIs translate these essential operations into clear, intentional actions: POST creates new resources, GET reads them, PUT updates existing resources, and DELETE clears those that are no longer needed. Combined, they define how data persists within a REST system, making the methods fundamental protocol for API designers to master.

Beyond their practicality to communicate with intent, the methods also give shape to basic system security and behavior. By drawing clear boundaries around actions, they guide access control, for example, by allowing some users only to retrieve data while granting others the extended ability to modify and delete it. The differences between the actions HTTP methods express also reveal patterns in long-term system behavior, exposing the ratio between read and write operations, which can help to guide optimizations in the architecture. The methods become equally important in understanding a fundamental RESTful principle, idempotence, where GET can be repeated multiple times without changing the output. Mapped to CRUD, the methods reflect data interactions found in most, if not all, computer systems. As such, to master the methods gives API designers the power to pull the levers that govern a system's fundamental behavior and integrity. 

## Structure

In this chapter, we will cover the following HTTP methods:

  * GET
  * POST
  * PUT
  * PATCH
  * DELETE



## Objectives

This chapter characterizes the roles played by HTTP methods in a RESTful system. By the end of the chapter, readers should have a strong sense for each method's purpose in relation to the others. It proposes a conceptual framework for resource manipulation supported by the basic principles of idempotence, caching, and conflict resolution. What to do with HTTP methods is equally important as what not to do, so the chapter offers cautionary tales for the reader.

## GET

The GET method probes, but does not change anything, unlike other HTTP methods. Its stated purpose is simple: enable clients to request data without altering the server's state. Since the method is idempotent, no matter how many times a client sends a GET request, its response stays constant. Analogous to a scientific instrument, GET plays the role of an impartial observer. A scientist points an instrument at a subject to study it. The instrument can be delicately calibrated and finely tuned for this purpose, to provide a close examination of the subject. If observation passively views a thing, then the process itself should have no side effects on the thing observed. The instrument can be used once or hundreds of times to obtain readings, and it always delivers the same result: the current state of the subject, undisturbed. So too with GET. Just as a scientific instrument provides consistent readings under the same conditions, the GET method accesses data from an endpoint without interfering in the backend.

The semantics of a GET request convey the ability to articulate precisely what to retrieve and from where. The server's response transfers to clients a clear reflection of the server's state. A request begins with a line specifying a resource URI, such as /weather/US/FL/Pensacola, to identify what the client seeks. It may also include query parameters to refine the request, for example, to get a weather forecast for a specific time period. The server responds with a signal of success in a 200 OK status if the resource is available. Otherwise, the server may return a 404 Not Found. A 304 Not Modified response means the resource has not changed since the last request. Beyond its role as an impartial observer, GET may carry HTTP headers to optimize repeat interactions, and make sure the retrieval process is as efficient as it is precise. The HTTP header Cache-Control helps manage which responses are stored and validated. Cache-Control instructs a client or intermediary server for how long to hold onto a resource locally, reducing the need for repeat, round-trip requests.

In the following example, server response headers suggest that the client can cache a response body for 1 hour, or 3600 seconds. If the client sends the same GET request within that time, then it can rely on the cached JSON data instead of going back to the server. This approach minimizes latency while keeping load off the server:

Request

GET /weather/US/FL/Pensacola

Response

HTTP/1.1 200 OK 

Cache-Control: max-age=3600 

Content-Type: application/json

Response body

{

"city": "Pensacola",

"temperature": {

"value": 25,

"scale": "Celsius"

},

"weather": "Sunny"

}

A caching mechanism finds a parallel in the scientific method. Scientists collect data by observation. They use instruments to study a subject without disturbing it. Similarly, GET serves as a reliable instrument for the discovery of data on the web. Just as an instrument can retain prior observations for analysis, caching enables GET to extend its role as an observer by storing responses for future use. A cache works like a scientist's logbook. In storing past measurements, a logbook provides quick access to known data and reduces repeat observations. Caching also brings stability to duplicate responses. From this perspective, GET becomes more than a technical specification as a conceptual framework for read-only web interactions. It embodies the scientific principle of safe observation: do not disturb the subject. The rules of GET, its syntax, caching, and validation, ensure that every request to observe data is a reliable, repeatable interaction.

### Facsimile

If the scientist's probe successfully takes a reading, the result is not the original system itself, but a pure facsimile of its state in the moment. It is possible that the communication protocols and technology of the late 1980s and early 1990s had a significant impact on the development of the REST architectural style by its principal architect, Roy Fielding. Although little evidence exists directly linking fax machines to Fielding's work, conceptual parallels between fax technology and representational state transfer help us to understand a possible link. In both, we find replication. Faxes transmit replicas of original documents, just as GET requests fetch reproductions of records on the server. Neither technology exposes the original, but both deal out something that closely resembles it.

In the same way, a fax machine provides a clear copy of a document for viewing while keeping the original intact. GET allows a client to access a snapshot in time of a resource's state as it remains unchanged on the server. A fax transmission also operates independently because the machine itself does not retain the documents it transmits. Likewise, GET requests in REST are stateless, each request a self-contained interaction. When a client sends a GET request, the server retrieves the current version of the resource, without a memory of previous interactions with the client.

The facsimile system and REST operate on the same principle. Both enable non-invasive replication. While a fax machine transmits a copy of a document over a phone line as a series of pixels, a GET request obtains a data-level facsimile of the resource's state over the network. Both processes emphasize making a copy, where the value lies not in the original thing but in the ability to impart a facsimile of the current state of the thing over a medium. 

### Caching

The design of networking protocols in the late 80s and early 90s, clearly shows that the benefits of caching were well understood. Fielding, likely influenced by latest protocols at the time, was aware of the efficiencies gained by caching. This influence may have played into refinements of the GET method. Caching optimizes by reducing load on the server. It allows resources to be accessed without too many repeat trips to the server. Comparable to the File Transfer Protocol (FTP) in the early 90s, which efficiently transferred files sometimes through a cache, HTTP minimizes network traffic using a similar mechanism. In an environment characterized by high-volume traffic, as in most RESTful systems, caching yields huge performance gains by storing copies of resources locally or on intermediary servers.

The GET method's caching semantics work on both the client and server sides. Client-side caching, similar to the way FTP stores copies of files to avoid duplicate file downloads, allows a client to save resources in a local cache. A web browser can store local copies of images, CSS files, or an API response. The approach improves response times for repeat requests, speeding up performance. Server-side caching, also found in pre-internet protocols, keeps transient copies of data closer to the source. As such, it reduces load on the server. Mentioned earlier, the HTTP headers ETag, Cache-Control, and Last-Modified, indicate how and when a cached resource is updated. These headers define the rules of a control system that gates if and when data will be retrieved again from the server. They tell the client which local copies of resources are up to date and which ones are stale.

Reliable caching requires careful consideration. Caches have performance upside, but introduce some technical challenges. A common issue stems from the risk of running into versioning conflicts by overly conservative caching. If a resource changes but its cached copy fails to update frequently enough, clients may get inconsistent results. This is problematic for dynamic resources that change often or rely on user input. In a simple example, a GET request for the current temperature may give back an outdated reading if a cache does not refresh in time.

### Common misuses

The GET method can be used unwittingly. For example, when APIs allow GET for non-idempotent actions. Other misuses involve excessively long query strings. Early FTP faced this constraint, which may have influenced Fielding's design. Very long query strings that go over maximum URL lengths can sometimes lead to transmission issues such as truncation by an intermediary. FTP would set rigid rules on filenames and command lengths to ensure reliable file exchanges. Similarly, GET requests discourage long URLs. Instead, a large amount of data should be sent via POST, which handles big payloads with less risk of data loss.

Correct GET usage becomes predictable and safe for data retrieval by removing unintended side effects. Lessons learned from the early protocols, like FTP's reliable file transfers, remind us of the importance in giving clear roles to communication methods. Just as FTP commands "get" and "put" had well-known behaviors, the GET method in HTTP/REST is designed as pure function: same input, same output, no side effects. 

## POST

As a counterpart to GET's passive nature, POST plays a very active role with measurable impact in the backend. While GET observes, POST creates. It lets clients add new data. Like a scientist experimenting, more intrusively than by passive examination of a subject, POST alters the state of the backend system with a payload. In the same way a chemist combines ingredients to yield a chemical reaction, or to synthesize a new substance, POST transmutes the server's state. Because the POST method intentionally changes the backend with each request, unlike GET, caching its response could lead to an out-of-sync representation held by a client. These characteristics distinguish the POST method as non-idempotent and non-cacheable.

### Non-idempotent

Beyond basic data retrieval, POST extends the core functionality exposed by REST APIs. In forming a request, clients embed net new data, to add in the backend, usually subordinate to the last part of the URI's path. If the request succeeds, and creates a new resource, then the server returns a 201 Created response. The body of this response usually tells the client what is the new resource's ID, giving the client feedback about the state of the resource after creation:

Request

POST /users HTTP/1.1

Host: api.example.com

Content-Type: application/json

{

"name": "John Doe",

"email": "john.doe@example.com"

}

Response

HTTP/1.1 201 Created

Content-Type: application/json

Location: /users/123

{

"id": 123,

"name": "John Doe",

"email": "john.doe@example.com"

}

Strictly speaking, unless the backend checks for duplicate usernames, to send this POST request more than once yields redundant user accounts with identical details but different IDs. This variability is exactly what makes the POST method non-idempotent. The server's state is expected to change with each new POST request, because the client's intent is to add new users. The client in this case drives the creation process by specifying the new user details. The server's response gives the client clear feedback, to help subsequent actions like accessing each new user resource by ID.

The POST method is also well equipped to carry large payloads, a capacity that distinguishes it from GET. A client can upload a high-resolution image, say 2000 x 2000 pixels and 4 MB in size, using POST as a reliable carrier to deliver the image as a payload in its entirety, without risking data loss. The body of the method is effectively free-form, as it is allowed to carry complex data constructs, nested JSON objects, large files, or long multipart form submissions.

### Non-cacheable

In a hypothetical REST API for an e-commerce platform, let us say an /orders endpoint accepts POST requests. A request sent to the endpoint represents a new order for a product. It embeds order details. The customer's information, product IDs, quantities, and some payment are included. The endpoint picks up the order, updates inventory, and confirms with a 201 Created response code, along with an order ID. An implementation of the /orders endpoint, with good intentions but misguided design patterns, may cache POST responses. A goal in caching is to boost API performance. In such an implementation, an intermediary caching layer stores the server's response to each POST request and associates it with the request's URI and payload. This way, if a duplicate POST request is sent (from network retries or user insistence), then the caching layer intercepts it, and returns a cached response without forwarding to the server.

On the surface, this strategy may seem to have upside. But the approach becomes somewhat problematic. Let us say the client does intentionally send another identical request for the same product and customer. If the server never receives a second POST request, then it will not create a new order or deduct inventory. The client in this scenario receives a cached response that includes an order ID, which could easily be misinterpreted as an indication that another order has been processed. This response creates a conundrum: the client thinks there are two orders, but the server only processed one. The second order was not processed, so inventory levels remain unchanged. This could lead to overselling product or, the opposite, unfulfilled orders. To make matters worse, if the client tries to pay for both orders, then the server will not record a second payment because it did not process the second POST request. In the aggregate, the entire platform could suffer from systemic issues. Customers relying on the API may see the inconsistencies when order confirmations are returned for orders that do not exist in the system.

To avoid data integrity issues like these, API designers should make sure POST responses are never cached. They can set a Cache-Control, that is, no-store or no-cache header to prevent intermediaries from caching them. Adding a unique ID to every request can help to detect retries, and smarter retry logic can classify a cause (maybe due to a timeout), without relying on cached responses:

Request

POST /orders HTTP/1.1 

Content-Type: application/json 

{

"customer_id": "12345",

"products": [

{"id": "p001", "quantity": 1},

{"id": "p002", "quantity": 2}

],

"payment_method": "credit_card",

"total": 100.00

}

Non-ideal response

HTTP/1.1 201 Created 

Content-Type: application/json 

{

"order_id": "67890",

"status": "confirmed"

}

Ideal response (Non-Cacheable)

HTTP/1.1 201 Created 

Content-Type: application/json 

Cache-Control: no-store 

{

"order_id": "67890",

"status": "confirmed",

"message": "Your order has been successfully processed."

}

### Pitfalls

This section considers the pitfalls and misuses of the POST method in RESTful APIs.

#### Overloading

Misusing POST for actions handled more effectively by PUT turns into a common pitfall. Overloading POST with work that does not involve data creation, may trigger an action, like sending a notification. This type of operation suits PUT better to replace data or PATCH for a partial update. Using POST for operations on existing data can lead to confusion. It makes the operation difficult for clients to understand what happened. For instance, a social media API with an endpoint /user/123/notifications that allows clients to trigger a notification for a user, lets a developer use POST for this action. It does not create new data but only triggers an action. The API request might look like this:

Request

POST /user/123/notifications HTTP/1.1 

Content-Type: application/json 

{

"message": "You have a new follower!"

}

Response

HTTP/1.1 200 OK 

Content-Type: application/json 

{

"status": "Notification triggered"

}

Since POST is not idempotent, the main flaw in this approach becomes apparent when a client repeats the request, leading to unintended duplicate notifications. Instead, an alternate approach, well-grounded in RESTful principles, uses PUT to trigger this action in a more controlled, intentional manner. The idea would be to "replace" or "update" a user's notification status, where the operation is more logically tied to the user resource:

Request

PUT /user/123/notifications HTTP/1.1 

Content-Type: application/json 

{

"message": "You have a new follower!"

}

Response

HTTP/1.1 200 OK 

Content-Type: application/json 

{

"status": "Notification updated"

}

Using PUT makes it clear that we are updating or replacing the notification status for that specific user. As an idempotent action, it can be repeated safely. The use of PUT, in this case, ties the operation to modifying or replacing existing data.

#### Payload limits

While POST can handle large payloads, it is critically important to consider some limitations. Servers and clients can face problems in processing large POST bodies, caused by limited memory or network latency. While the HTTP specification does not explicitly set a maximum size for POST data, web servers and clients commonly set some practical limits. To work within them, API designers should compress large files, or split payload data into smaller chunks (batches). Sending big data wholesale may slow response times and degrade performance.

Take, as an example, an API endpoint /upload designed to handle video uploads from users. A client sends a POST request with a large video file in the body of the request:

POST /upload HTTP/1.1 

Host: example.com 

Content-Type: video/mp4 

Content-Length: 1.5 GB

[Video file data here ...]

The video file is about 1.5 GB in size. While not gigantic by modern standards, it is still large enough to cause issues depending on the server's resource allocation. The server may fail to load the entire video file into memory while processing it, which could increase processing time while swapping the file out to disk. Sending large files can take a good amount of time, especially for users with slower internet connections. This can become a cause for delay for other users trying to upload content within the size limitations defined. Most web servers usually have a default timeout setting for requests. If the file upload takes too long, and exceeds the timeout, then the request will fail. If too many clients upload large files all at once, then collectively, they could overload the server, slowing response times for everyone.

To mitigate these issues, API designers may consider some strategies:

  * Compress the video file before uploading to reduce the payload size.
  * Split the file into smaller parts to chunk it. The server then receives the file in segments within memory limits.
  * Provide a file upload progress indicator API. On the client side, upload progress bars can give users feedback in the event of a failure caused by the network.



The POST method introduces concepts essential to the framework of REST, for clients to create new resources, handle large payloads, and run non-idempotent operations. While it offers all this flexibility, it asks for careful consideration as a non-cacheable method. Its role in the framework is not to be confused with actions better left to PUT or PATCH. As we start into a discussion about the ways POST differs fundamentally from PUT, we emphasize PUT's role as a replacer.

## PUT

In RESTful systems, when we take something in an undesired state and carefully substitute it with a defined, intended state, we use PUT. Just as a scientist might take an existing substance and apply a reagent to alter its composition, the PUT method updates a resource to reflect the state specified in the request. The original resource becomes the old one, and replacement re-defines what is in it. This can happen in two ways. If the original has more field values than the new one, then an updated version will have fewer fields. In effect, PUT can seem as though it deletes field values but if the original has fewer field values than after it is updated, then PUT appears to extend the resource. In both cases, the operation replaces the original with something entirely new, which takes its place. When the PUT request carries existing fields with different values than the original, the operation simply overwrites them.

### Replace

Although the method carries a payload and changes the state of the backend, it differs from POST in that it always operates on an existing resource. Used correctly, PUT targets a specific resource or a collection. Repeating the operation has no real impact on state because the method is idempotent, yielding the same result: the resource is replaced with the exact data sent in the request. By pointing to a specific resource in a unique identifier on the URI, PUT precisely updates the backend. The payload it carries must contain a full representation of the resource's next state, which the server uses to replace the current state.

In an API for a library system, for example, where books have unique identifiers, to update a book's details, a client first GETs the current state of the book, then sends a PUT request targeting the book resource by ID in the URI:

GET request

GET /books/42 HTTP/1.1

GET response

{

"title": "Advanced Chemistry",

"author": "Dr. Jane Smith",

"genre": "Science"

}

PUT request

PUT /books/42 HTTP/1.1

Content-Type: application/json

{

"title": "Advanced Chemistry",

"author": "Dr. Jane Smith",

"publisher": "BPB",

"genre": "Science"

}

PUT response

HTTP/1.1 200 OK

Content-Type: application/json

{

"id": 42,

"title": "Advanced Chemistry",

"author": "Dr. Jane Smith",

"publisher": "BPB",

"genre": "Science"

}

In this case, a single field, "publisher", is added to the book resource, which is reasonably expected to be included immediately after in each subsequent GET request for the same book, given a full replacement of the book values in the backend. 

### Upsert

This simple example illustrates the basic ideas but also becomes a cautionary tale. To circumvent the principle that resource creation should be handled by POST, let us say an API designer decides to allow the PUT method not just to update existing books but also to create new ones if the specified ID does not exist, a trick known as upsert. Data integrity issues creep into this approach because it may blur the client's intent. In principle, PUT should be idempotent, so that simply sending a request again has no side effects. But if a client intends to create a new book, unaware that the book identifier already exists, and sends a PUT request to /books/45, then the upsert will update an existing book whose ID is 45. This results in an operation the client did not want. If a second client sends the exact same request, with the intent to update the book, the response may be 201 Created. For these reasons, PUT should focus squarely on updates. API designers are well advised to separate these two concerns, create versus update, to avoid ambiguity and to align with RESTful principles.

### Conflicts

In high traffic environments, race conditions and conflicts are inevitable. Because PUT directly overwrites with a completely new representation, requests frommultiple clients at the same time can lead to weird states.For example, when two clients fetch the same resource, modify it independently, and then send PUT requests to update it, the last request will always win and overwrite the earlier one, potentially losing important changes. In this situation, a classic race condition, the timing of requests determines the next state of a resource: last in, next out.

Consider the following sequence diagram:

Figure 3.1: Race condition

To resolve conflicts, API designers may use conditional requests with ETags. This provides assurances that updates will happen only if the resource's state matches the client's expected state.

The first client fetches the book by ID:

GET /books/42 HTTP/1.1

Response

HTTP/1.1 200 OK

Content-Type: application/json

ETag: "xyz123" // Current ETag for the book

{

"id": 42,

"title": "Advanced Chemistry",

"author": "Dr. Jane Smith",

"publisher": "BPB",

"genre": "Science"

}

The ETag xyz123 is a unique identifier for the current version of the book. Any changes to the book state will result in a new ETag value. The first client then modifies the book resource locally and sends a PUT request to update it. The client includes the ETag in the request's If-Match header to ensure the update only occurs if the book's state has not changed since it was fetched:

PUT /books/42 HTTP/1.1

Content-Type: application/json

If-Match: "xyz123"

{

"title": "Advanced Chemistry",

"author": "Dr. Jane Smith",

"publisher": "BPB",

"genre": "Science",

"language": "English"

}

As the first client prepares its update, the second client retrieves and modifies the same book, and then sends its own PUT request.

Request from second client:

PUT /books/42 HTTP/1.1

Content-Type: application/json

If-Match: "xyz123"

{

"title": "Advanced Chemistry: Modified by Second Client",

"author": "Dr. Jane Smith",

"publisher": "BPB",

"genre": "Science"

}

Response

HTTP/1.1 200 OK

ETag: "abc456" // ETag updated after second client's modification

When the server receives the first client's request, it compares the If-Match header "xyz123" to the current ETag "abc456". They do not match. The server rejects the request to prevent overwriting the second client's changes.

Response to first client's request:

HTTP/1.1 412 Precondition Failed

Content-Type: application/json

{

"error": "Precondition Failed",

"message": "This book has been modified by another user."

}

Refer to the following figure:

Figure 3.2: Conditional PUT

ETags can help to prevent race conditions. A robust implementation of the PUT method ensures that updates are applied only if the resource's state matches the client's copy. By combining If-Match and ETag headers, APIs can protect against update conflicts.

### Misuse

A common misuse of PUT occurs when its implementation does partial updates. Using it to update a resource partially can overwrite unintended parts of the resource. If used to update, say, just the email address of a user, but the body of the request includes a full user record, then this could overwrite other fields, such as the user's name or address, even though the client does not intend these fields to be modified. Such an ambiguity blurs the boundaries between the roles PUT should play. It also risks data integrity issues. Partial updates are better suited for the PATCH method, as explained in the next section.

Designers slip into another pitfall when they fail to target specific resources by their unique identifiers in the URI. PUT must point at a specific resource. The resource it modifies is identified by a unique URI. The method replaces the resource found at that location. Using PUT without a reference to a specific resource ID leaves a gap in the request. The server cannot guess which resource to update. If an endpoint like /users receives a PUT request without a user ID, then the server may swap out the wrong user data. This challenges the expected behavior of the method, a misuse that can lead to inconsistent backend states.

As a replacer, PUT comes with specific guidelines and potential pitfalls. Thinking in terms of the client's intent helps to design an implementation that always behaves as expected. How frequently HTTP methods are used in REST APIs can vary depending on the specific application, but broadly speaking, over time, patterns have emerged. Based on general usage in public APIs, GET represents 50-60% of traffic, POST is used 20-30% of the time, PUT comes in third place, at 10-15%, and DELETE is the least frequently used method, capturing only 5-10% of all requests. As such, PUT can be considered somewhat of a special case, an indication that clients are not expected to replace entire resources often, or the state of the backend remains calm and stable. The requirement to provide full payloads as replacements every time, may become a burden for clients, who are better served by PATCH for the more granular, partial updates.

## PATCH

With surgical precision, PATCH makes targeted modifications rather than complete replacements. While PUT replaces an entire specimen in a lab with a new one, PATCH focuses on adjusting specific components. Like a geneticist tweaking a single gene instead of modifying the entire genome, PATCH is all about focus. It applies small updates. By altering only the parts specified in the request payload, it leaves everything else intact. Unlike PUT, which requires a complete, full representation of the resource, PATCH zeroes in on the attributes that need attention. This economy of action makes it especially useful in environments where large resources would otherwise incur significant overhead to maintain. With this power comes a need for precise exactness: a poorly constructed PATCH request can cause unintended mutations, much like an imprecise tweak in a laboratory can have monstrous consequences.

### Targeted modification

As specified in RFC 5789, PATCH provides field-level CRUD write operations. Because of its fine granularity, the approach is efficient for situations where minor, gradual updates are required without affecting the whole resource. No need to send the full representation, the method can be fine-tuned to add a new field (create), edit an existing value (update), or remove a field completely (delete) by sending only the changes required. PATCH lowers the overhead by targeting specific fields.

#### Add Field

For example, given a book resource:

{

"id": 42,

"title": "Advanced Chemistry",

"author": "Dr. Jane Smith",

"publisher": "BPB",

"genre": "Science"

}

A single "language" field can be added by sending a PATCH request with the field to add:

Request

PATCH /books/42 HTTP/1.1 

Content-Type: application/json 

{

"language": "English"

}

After the operation succeeds, the updated resource is identical to the original, except for the added field. The implementation may simply check for the absence of the field in the current book resource to determine that the field should be added in this case:

{

"id": 42,

"title": "Advanced Chemistry",

"author": "Dr. Jane Smith",

"publisher": "BPB",

"genre": "Science",

"language": "English"

}

#### Update field

Given the same original book resource prior to adding a field to it, the following request is expected to update the current "title" field:

PATCH /books/42 HTTP/1.1 

Content-Type: application/json 

{

"title": "Advanced Chemistry: The Basics"

}

The implementation may simply check for the presence of the field in the current book resource to determine that the field should be updated in this case. The updated resource would then look like this:

{

"id": 42,

"title": "Advanced Chemistry: The Basics",

"author": "Dr. Jane Smith",

"publisher": "Pearson",

"genre": "Science"

}

#### Remove field

Given the same original book resource prior to adding a field to it, or modifying a field, the following request is intended to remove the current genre field:

PATCH /books/42 HTTP/1.1 

Content-Type: application/json 

{

"genre": null

}

The implementation may simply check for the 'null' field value to determine that the field should be removed in this case. The updated resource would then look like this:

{

"id": 42,

"title": "Advanced Chemistry",

"author": "Dr. Jane Smith",

"publisher": "BPB"

}

#### Add, update, and remove fields

Given a combination of fields to add, update, and remove in the body of the PATCH request, the following is expected to perform all three operations at once:

PATCH /books/42 HTTP/1.1 

Content-Type: application/json 

{

"language": "English",

"title": "Advanced Chemistry: The Basics",

"genre": null

}

As with the implementation of each individual add, update, and remove, a combined request to perform all should be detectable through a comparison between the JSON resource in the request and the current state of the resource on the server. These examples show how flexible and precise PATCH can be, making it a powerful method for targeted updates. 

### Efficiency

If we take into account these very simple, basic examples, a back-of-the-envelope calculation of efficiency can be made by looking at the differences in payload sizes, as given in the following table:

Operation |  Size (Bytes)  
---|---  
PUT |  122  
PATCH (add) |  28  
PATCH (update) |  48  
PATCH (remove) |  20  
  
Table 3.1: Operation payload sizes

We can safely say that on average, the PATCH method requires payloads that are ~75% smaller than the full, original resource, or 1/3rd the amount of data is required in each request to effect the same change. As resources to update at the individual field level become larger, it makes more economic sense to update them via PATCH. Applying Gzip compression to JSON data on average brings the payload size down by 2/3rds when compared to the raw character data.

#### Conflicts

Since PATCH operates on highly volatile fields, multi-client requests can lead to race-conditions and contradictory updates. To get ahead of these scenarios, API designers should include headers like ETag and If-Match, for the same reasons they should be used by PUT. The server can enforce conditional updates in every PATCH request, by sending an ETag header in the response and checking for a value in the If-Match header. If they match, then the operation proceeds. The approach ensures that client and server operate from the same basepoint.

#### Non-idempotent

The PATCH method is not idempotent, at least, not inherently. To achieve cumulative effects, it may rely on specific instructions in the payload. Idempotency, as defined in HTTP semantics, requires that several identical requests provide the same outcome as a single request. This is not assured for PATCH for several reasons.

Take, for instance, a resource representing a counter stored on the server. Initially, the resource might look like this:

{

"id": 101,

"name": "Page Views",

"count": 50

}

If a client wants to increment the "count" field by 1, a payload may contain these specific instructions:

PATCH /counters/101 HTTP/1.1

Content-Type: application/json

{

"operation": "increment",

"field": "count",

"value": 1

}

If the same PATCH request is re-sent, the server would apply the operation a second time, resulting in the "count" field being incremented again: 

{

"id": 101,

"name": "Page Views",

"count": 51

}

Sent a third time, the operation increments the counter by 1 from its previous value:

{

"id": 101,

"name": "Page Views",

"count": 52

}

Operations like this rely on the resource's current state. So, to append an item to a current list or concatenate existing strings, just as the example with the counter shows, PATCH operates on a mutable resource that changes differently with each PATCH request. If the state changes, due to concurrent requests or other operations in-flight, then applying the same PATCH request may produce different results.

#### Misuses

Some common pitfalls to avoid in using the PATCH method include:

  * Send a full representation of the resource in the payload, as if it were PUT.
  * Ignore conflict management.
  * Use the method for bulk updates.
  * Making PATCH idempotent, like PUT.



PATCH is specifically designed for partial updates. Using it for full updates negates its benefits and can confuse API clients about the intent of the request. Modifying individual fields makes the method prone to race conditions, so some level of conflict management is ideal to implement. Attempting to use PATCH for bulk updates complicates the logic even further and undermines the simplicity of REST principles.

## DELETE

DELETE plays a lead role in the final act of the lifecycle of a resource. Here, removal takes precedence over observation, creation, or modification. DELETE is definitive: it erases. In a scientific context, DELETE is like a researcher who discards an obsolete specimen from their collection to make room for a new one. The act is deliberate and final. What is removed leaves no lingering traces. DELETE is idempotent. Issuing the same request multiple times will yield the same result: absence. Once executed, the resource it targets goes poof, and its state changes irreversibly to nonexistent.

To remove a book with ID 42 from the backend, a client may send a request such as:

DELETE /books/42 HTTP/1.1

Response

HTTP/1.1 204 No Content

This example shows DELETE's pure, delightful simplicity. It targets a specific resource, /books/42, the book with ID 42. Then the server's response confirms the operation succeeds through a 204 No Content status code, without a response body, which indicates that the resource has been deleted.

The destructive nature of the DELETE method warrants careful use, as its effect is permanent. As a result, API designers must consider proper authorization, confirmation, and safeguards to prevent inadvertent or unauthorized deletions. The removal of a resource might affect system integrity. In some cases, a soft delete may be more appropriate, allowing the resource to be preserved in a deactivated, invisible state rather than completely obliterated.

## Conclusion

This chapter equips API designers with much more than prescriptive rules. It looks through a lens. Here we trade dry protocol for living analogy: the scientist, the fax machine, the surgical instrument. The chapter makes a case for web architecture derived from first principles. We understand GET as an act of non-invasive observation. We use POST to create. We apply PATCH with the precision of a surgeon. Put together, the methods applied as conceptual framework, that is, as a set of verbs that operate within a dialog between client and server, transform abstract REST constraints into intuitive design instincts. We move from rules to reason. We no longer just use HTTP, we speak it.

## Points to remember

  * GET is used for retrieving data without altering the server's state
  * POST creates new resources and modifies server state with each request
  * PUT replaces an entire resource or its state with a new version
  * PATCH applies partial updates to a resource, focusing on modifying specific attributes
  * DELETE removes a resource from the server, use with caution
  * Conflict resolution in methods like PUT and PATCH can be handled with ETags and If-Match headers to prevent race conditions

## Introduction

In giving shape to resources, API developers play in a space where engineering and design intersect. The process asks us to find a balance between technical precision and good form. We engineer for integrity in the output from an API, and design to shape data into a representation that engages clients. If we look for them, we find patterns in the output from well-crafted APIs that give resources specific traits. By following patterns, resources become consumable, composable, and adaptable. More than buzzwords, these traits form a foundation upon which to scale web services through a cohesive flow in the interaction with clients. Conversely, poor resource design contributes to unusable, disjointed, and inflexible output, creating difficulties to integrate. This chapter highlights patterns that make resource representations transformative for APIs and contrasts them with some anti-patterns to avoid.

For clients to build a model of the application's state, resource representations provide the building blocks. They form structures that reflect on the backend and guide the client in its navigation through the system's space. They combine into complex constructs. As such, individual resources are composable units. They can stand alone, but their attributes allow them to fit together. When clients need to rearrange or restructure their view, the API emits resources that adjust. As the client evolves, through new device types, changing requirements, or extended application features, resource representations embody qualities that allow them to adapt. An analysis of the patterns in API output that support this kind of adaptation helps us to distinguish good design from the anti-patterns that take us down the wrong path. The following sections guide designers through the engineering and craftsmanship of resource representation by defining a descriptive and an evaluative nomenclature that characterizes the most important patterns to recognize in API output.

## Structure

The sections in this chapter discuss what makes resources:

  * Consumable
  * Composable
  * Adaptable



## Objectives

Throughout this chapter, we use two sets of terminology: descriptive and evaluative. The first set helps define resource representation patterns without passing judgement. Descriptive terms depict exactly what we see and illustrate a subject for others to understand impartially. The second set of terms relates resources to their surroundings, puts them in a specific context, and criticizes them. Through evaluative terms, we become critics, with the aim to distinguish between traits that reflect positively on RESTful design principles and bad features that complicate integrations. By alternating between description and evaluation in analyzing resource representations, we proceed within a balanced framework to understand and improve how resources work. This approach equips us with tools to recognize patterns objectively while making value judgements about their impact critically.

To do so, we gradually improve resources by moving them through a sliding scale of design quality. Each step on the scale enhances usability, and the movement shows that small, deliberate adjustments lead to significant gains in client interaction. A step-by-step approach refines API output into highly effective resource representations.

## Consumable

To recognize a consumable resource, we start with a counterexample at the opposite end of a scale, with something totally unusable. In this approach, we discover what makes a resource consumable by gradually building it up, to shape it into a consumable form.

### Initial state: Unusable

In its initial state, the resource is a basic, albeit unusable unit, a small core. In this example, the resource has cryptic keys and meaningless values:

{

"a1": "x94",

"b2": "y32",

"c3": "z28"

}

There is no discernible structure to it, and it is anyone's guess what the data stands for, let alone what to do with it. Since the representation lacks clarity and context, it is simply unusable. It does not easily lend itself to be consumed by clients for any specific purpose. In its current form, this resource is ineffective because it fails to support purposeful client interaction, making it a clear example of a deficient representation. Furthermore, the example highlights a critical gap between data and information. While it qualifies, technically, as a data structure, it fails to inform or guide its consumer. Without clear keys or substantive values, the resource is more of a burden than an asset. Clients are left to make too many assumptions, increasing the chance of a misinterpretation. The resource's flaws run deeper than highlighting technical inefficiencies. They actively hinder usability.

So far, our scathing assessment has yielded descriptive terms, such as cryptic, meaningless, and indiscernible, as well as evaluative ones, including unusable, ineffective, and deficient. While the terms may influence our attitude towards the resource (they tell us how to feel about it), the overall critique also contributes to our analysis for improvement of the resource because it identifies specific problems to address. Rather than focus only on its faults, which narrows our perspective, a balanced critique opens the opportunity for improvement and reveals a potential to build something better, starting with something really small.

#### Improvement 1: Opaque

Here, the keys are human readable:

{

"title": "a94",

"author": "b32",

"isbn": "c28"

}

From the keys, we can reasonably infer that the data must relate to a book or some kind of publication. Because the values are codes or place holders, they leave gaps in the definition of the data. While this step improves upon the previous state, the result is still incomplete, barebones, and somewhat ambiguous. The resource lacks context.

This step provides a modest improvement over the unusable state. It introduces readable keys that give just a hint of explanation. Yet, the placeholder values leave gaps that undermine the resource's utility. The client can guess the kind of resource, likely a book, but cannot act on it. Adding readable keys is a step forward, but the lack of substantive values keeps the resource in a state that makes it difficult to be consumed for a purpose.

#### Improvement 2: Readable

At this stage, the keys and values provide relevant data about a book clearly:

{

"title": "The Great Gatsby",

"author": "F. Scott Fitzgerald",

"isbn": "9780743273565"

}

The structure is uncomplicated. In two discrete steps, we have moved the resource to a usable, functional, accessible form while retaining its original structure. Even though the ISBN number for the book can be cross-referenced with other systems, the resource still lacks extensibility or linked data that allows it to stand on its own. Without links it remains cut off from a broader web. This takes it out of context. As an isolated unit, its potential to integrate with related resources is effectively stunted. No connections means the resource remains a self-contained entity, unable to join with the larger system.

#### Improvement 3: Consumable

As a final improvement, the resource includes metadata and relates itself to other resources through links to author details and a purchase endpoint, and so it has context that connects to other parts of the system:

{

"title": "The Great Gatsby",

"author": "F. Scott Fitzgerald",

"isbn": "9780743273565", 

"genres": ["Classic", "Fiction"],

"availability": "In Stock",

"links": {

"author": "/authors/156",

"purchase": "/books/the-great-gatsby/purchase"

}

}

It is well structured and supports client use cases, like filtering by genre or checking availability. As such, the resource becomes consumable because it can be used directly by a client. The consumable state solves issues in the unusable, opaque, and readable versions by adding a little structure. It connects to other resources. Previous versions had cryptic, unreadable keys and obscure values which prevent clients from making practical use of the data. By building on metadata and links to related resources, such as the author and purchase endpoints, the resource becomes actionable and contextually rich.

Unlike earlier versions, which were isolated, the consumable state integrates the resource into a larger system, allowing clients to interact with it in relation to other parts of the application. A consumable resource embodies clarity, structure, and function. It is easy for clients to interact with and derive value from it. Consumable resources have keys and values in a human-readable form that contribute to the resource's meaning. By incorporating metadata and links to other resources, it offers contextual clues for a client to navigate. Through links, a consumable resource guides the client in navigating to other relevant parts of the application space.

By making slight, incremental improvements, using evaluative terms to identify problematic features, we build a better resource state in discrete steps through a process that moves it across a scale. A sliding scale of descriptive and evaluative terms provides a structured approach to think about the improvement of resource representations as a gradual, stepwise transformation. Descriptive terms (cryptic, opaque) frame a resource's state in terms of its usefulness to clients. A framing shapes the designer's perspective. It places the resource on a scale of utility. Based on its specific features, the resource may rank higher or lower on the scale. Whereas, evaluative terms (unusable, incomplete) criticize the resource by looking at it through a qualitative lens. Both sets of terms highlight patterns that limit or increase the resource's use. Thinking in terms of movement across an assessment scale, suggests an iterative nature to a design process, where progress is marked by measurable steps towards a desired state. ## Composable

Another pattern to look for in resources that REST APIs emit can be recognized in traits that make it composable. A composable resource representation combines easily with other data to generate more complex structures. On the opposite end of a descriptive scale, a disjointed resource lacks the structural elements needed for integration into a larger data model. It might break data into pieces that are more difficult to combine, leading to complications for clients to build purposeful interactions.

### Initial state: Disjointed

At this point, the resource only provides partial data about a person:

{

"id": "12345",

"name": "John Doe",

"age": 45,

"insurance_id": "5678-1234-A",

"health_conditions": [

"allergies",

"headaches"

]

}

In this pass, we only receive an insurance_id. This means we have to search for the full insurance policy details somewhere else, which is inefficient. While the presence of an insurance_id and health conditions strongly suggests this is a patient record, the connection is not clear from the data we have. As isolated pieces, they introduce a more complex and less efficient interaction with the API because to fully understand the record, the client must make additional requests to unknown endpoints and resolve details from obscure IDs.

Fragmentation makes it difficult to combine the data with other information. A lack of cohesion inhibits interaction with clients. To scale up, if we need to add multiple health conditions or other insurance types as arrays, the resource will need to be restructured. This would change the form of the output structurally, affecting its compatibility with clients.

#### Improvement 1: Modular

{

Each object is light. The patient's insurance details could expand further, as they remain somewhat isolated from related resources in the backend. While the additional elements add substance to the patient record, the absence of integration with contact details or insurance specifics reveals a giant gap that leaves the data feeling incomplete and segmented.

#### Improvement 2: Structured

In this version, the resource adds links, thereby connecting itself to related entities, such as the full insurance record:

{

Including a patient's mailing address helps us understand their community and location. The data is segmented. To get a fuller picture of the patient's health, we could connect this address to their contact details and other relevant health information.

#### Improvement 3: Composable

A composable version includes nested, linked data, putting forward a holistic view of the patient in relation to other, related resources. These features allow the client to navigate and integrate the information more easily. In a composable state, the resource overcomes the limitations in the previous versions.

{

To build more complex models, composable resources combine easily. To expand their view of the server, composable resources offer clients related data that integrate into a deeper application context. A patient resource, for instance, that includes hierarchical symptoms data becomes an organized, multi-level representation of symptoms, structured to show relationships and groupings between them. A hierarchy is useful in health informatics to categorize symptoms based on factors like severity or associated conditions. ## Adaptable

An adaptable resource representation adjusts with client requirements, such as mobile and web, in response to the needs of different client types. Variability enhances its interoperability with clients. On the opposite end of a descriptive scale, a static resource lacks the variability required to engage a broad range of clients running in different environments. It might fix data into rigid elements that are coupled to one client, leading to a loss of features that could benefit other clients.

### Initial state: Static

The resource includes some basic information about a product, but limits clients in this case, with a one-size-fits-all approach:

{

"product_id": "12345",

"name": "Wireless Headphones",

"price": 99.99,

"description": "High-quality wireless headphones",

"image_url": "https://example.com/images/headphones.jpg"

}

Web clients may prefer larger images or reviews integrated with product details, while mobile clients may want a smaller image size for better performance and could opt for a simplified version of the description or the option to load additional information, such as pricing, later. This representation cannot adjust to client types or use cases. Currently, the API is hardcoded to example.com, which creates a few challenges. If that domain changes, then every part of the API that uses it would require an update. Since the URLs point to production, this also makes it very difficult to test the API in a development or staging environment. To expose a domain like this could unintentionally reveal details about the service's infrastructure, which is a security risk.

#### Improvement 1: Variable

This version adds variability. It has a default image size, and allows for image size adjustments: small, medium, or large.

{

"product_id": "12345",

"name": "Wireless Headphones",

"price": 99.99,

"description": "High-quality wireless headphones",

"image": {

"size": "default",

"url": "/images/headphones.jpg"

}

}

The image element went from a flat, single-level element in the static design, to a structured mini-object with two attributes in a variable version. A variable design introduces a more dynamic, structured approach by nesting the image data in a small object. This format accommodates additional attributes, such as size, and scales up if more image properties, such as resolution or formats, are needed for other clients. While slightly more complex, a variable version improves adaptability, making the resource more flexible for clients. In this case, the full domain, example.com is replaced by a relative URL for the image. This adjustment makes the resource portable across environments and yet, the form is still somewhat restrictive. It provides a, default image, and 3 other sizes, but does not account for the different needs of our mobile and web clients.

#### Improvement 2: Dynamic

In this step, the resource has become less static and more dynamic, with different image sizes based on the client, web or mobile, and different numbers of reviews based on client preferences:

{

}

A dynamic state takes adaptability a step further by tailoring the resource to multiple client contexts: the image field provides separate links for web and mobile versions of the image, which optimizes the resource design for platform-specific rendering. Improved adaptability, a quality API designers strive for, comes from a reviews field that accommodates for web client and mobile device constraints as well. This feature offers clients a customized experience. Self-referencing links and a purchase endpoint help to navigate there, to interact with the resource and its related endpoints.

This design may be somewhat verbose in that web clients receive unnecessary data like mobile review data, or smaller image sizes than needed. The system transmits more data than required in certain cases, which could affect API performance.

#### Improvement 3: Adaptable

In a fully adaptable version, a web client can retrieve up to 10 reviews. This exposes a more comprehensive product view and uses the more extensive picture links for high-resolution displays. To minimize data load, the mobile client can only request a limited set of three reviews and retrieve a smaller image. As such, the API offers responses that bend for web clients and eliminate needless overhead for mobile devices by using links and parameters that leave the options open to request specific things:

{

An adaptable resource configures itself. It varies its format based on context, and flexes to meet future use cases that no one has yet envisioned. It makes itself compatible with different clients. Adaptable resources respond to specialized requirements. Through flexible parameters and contextual data, an adaptable resource equips the client to retrieve precisely what they need. On a scale, the movement across a range of adaptability is illustrated by the following figure:

Figure 4.3: Sliding scale, adaptable

Inflexible resource representations fix data and fail to accommodate specific clients, such as screen size or payload limitations. In contrast, adaptable resources tailor content through parameters, to respond efficiently in both contexts. The same endpoint can serve multiple clients with different requirements without breaking compatibility. 

## Conclusion

Good resource design serves your clients. It demands three traits: consumability, composability, and adaptability. Consumable resources are clear and usable. Their data is structured for action. Composable resources connect easily with other parts of the system. They let clients build complex models. Adaptable resources flex. They tailor the output to fit the client. Forget an all-or-nothing goal. Improve resources step-by-step. Add a readable key. Nest related data. Insert a link. Make small, incremental, stepwise changes to refine the API. This iterative process is your structured path. It improves usability now and scales into the future.

## Points to remember

  * Consumability in a resource makes it clear, structured, and simple for clients to use.
  * Composability integrates resources easily with other system components, enabling clients to combine data into models and insights.
  * Adaptability tailors resources to meet client needs, such as mobile versus web.
  * Anti-patterns are found in poorly designed resources. Unusable, disjointed, or static resources fail clients. Examples include cryptic keys, fragmented data, and rigid designs that lack adaptability.
  * Step-by-step refinements improve resource design as an iterative process. Small enhancements like nesting structures, adding metadata, or introducing links improve usability, scalability, and engagement.
  * Descriptive terms look at resources objectively, while evaluative terms critique their effectiveness.

## Introduction

In the previous chapters, we explored resource identification, manipulation, and representation, as foundational elements of REST architecture. We have observed that simple, repeatable elements combine to form more complex and functional structures. As the pathways through a REST API expand and clients become more diverse, an adaptive guidance mechanism is needed. Clients can use some guidance to navigate an API instead of relying on hardcoded logic or static API documentation to know where to go next. Such a mechanism does not constrain clients to follow preset routes but embeds navigational cues directly into API responses. These cues become a roadmap that informs clients of the next state transitions, based on the current resource state. Hypermedia as the Engine of Application State (HATEOAS), born in Roy Fielding’s dissertation, positions this mechanism as an architectural element that makes state transitions directed and dynamic. This chapter discusses resource state management, an element of REST architecture that embeds hypermedia links into every response and enables clients to navigate a web of interconnected resources, as a living roadmap that adapts to change.

## Structure

In this chapter, we will cover the following topics:

  * HATEOAS fundamentals
  * Designing hypermedia controls
  * Best practices and implementation



## Objectives

This chapter covers HATEOAS in modern REST APIs. It explores the integration of hypermedia controls in API responses to route clients through an application's space by making state transitions. It puts forward design guidelines for developers, students, and technical professionals to free API clients from static endpoints. The approach explored here offers advice to build self-guiding REST systems. 

## HATEOAS fundamentals

When a client retrieves a resource, it gets more than just data. It also gains insight into subsequent steps to take. Through HATEOAS, API responses are like interactive guides, or modern GPS systems that recalculate routes based on current conditions. Clients transition from one state to another without any hard dependencies by following hypermedia links. The server, aware of its own state, informs the client of every available option for interacting with the API at each step. As a design philosophy, and navigational aid, HATEOAS is all about giving clients direction. This section expands on these concepts by examining the anatomy of hypermedia responses.

### Principles and rationale

Roy Fielding explains that a REST API should be entered with no prior knowledge beyond an initial URI, like a bookmark that serves as the sole entry point. From there onward, Fielding insists, any application state transition must be driven solely by client selection of server-provided choices present in the received representations or implied by the user’s manipulation of those representations. This design philosophy addresses a critical gap in API interactions without forcing clients to rely on hard-coded lists of endpoints. HATEOAS lets the server define available actions at each step so that clients know where to go next based on the current state. In a sense, HATEOAS is like a guided museum tour. When you enter the space (API), you only need to walk through the front door. As you move through the museum, a guide, or server, takes you to the next exhibit, a state transition, based on your current location and interests. The guide adapts the tour in real time, suggesting new paths, guiding you through different rooms, and ensuring you have seen everything without needing a preset map or prior knowledge of the museum's layout.

HATEOAS solves a client-server coupling problem. It isolates the client by abstracting the system’s internal structure. Clients make decisions about what steps to take through the hypermedia links embedded in API responses. When a client retrieves a resource, it does not need to be hard-coded with all possible endpoints, as it can rely on links provided by the server to determine the next steps. As an abstraction, hypermedia keeps static endpoints separate from client code. As a systems design strategy, it is especially persuasive for technical professionals to make a case for an approach that easily adapts to new requirements. Its design rationale, then, is clear: a guidance system that evolves alongside the application provides an abstraction over static, brittle interfaces to guide the way clients interact with web services.

### Semantics and metadata

Semantics define the meaning of actions and shape how clients interact with resources in HATEOAS. When a client retrieves an order by issuing a GET /order/123 request, the response may be augmented with links like "self", "ship" and "cancel". These links help the client to understand that the order must either be shipped or canceled based on its current state and then the client can decide on its next move. Semantics define the operations a client can perform by making resource state transitions. Metadata, on the other hand, adds additional context to resources so that clients can better understand the data from a temporal perspective. Metadata in HATEOAS typically returns with the resource information that also helps to guide client interactions. It can include resource versioning information and timestamps. Metadata in a RESTful system defines for the client the lifecycle of a resource in terms of a progression of steps.

Together, semantics and metadata provide the framework that frees clients from any preconceived knowledge of API structure. For example, a consumer of an e-commerce application would retrieve an order's data with a GET /order/123 request and then process the link provided to cancel or ship the order. The client can dynamically react to the resource state and take the correct action based only on the semantics and metadata in the response. Consider a compact JSON example where these attributes guide the client:

{

"orderId": 123,

"status": "pending",

"totalAmount": 99.99,

"_links": {

"self": {

"href": "/order/123"

},

"ship": {

"href": "/order/123?status=ship",

"method": "POST"

},

"cancel": {

"href": "/order/123?status=cancel",

"method": "POST"

}

},

"metadata": {

"version": "1.2",

"expiresAt": "2035-03-04T10:00:00Z"

}

}

In this example, the resource's state determines which actions are available. The links, "self," "ship," and "cancel" provide the client with options to change the order's status (to ship or cancel it). Additionally, metadata, such as the version and expiration date, gives the client context about the resource's lifecycle, enabling informed decisions on how to use it.

### Basic traits of hypermedia controls

Well-designed hypermedia controls share some common traits as decoupled, self-describing, and contextual. Decoupled means the controls (or links) disentangle the client from the API's endpoints, leaving the server to define state transitions. Clients discover new states as a function of response-provided instruction rather than being tied to hardcoded endpoints. Self-describing responses ensure that all hypermedia links include metadata specifying their purpose, intended action, and relation to the resource, allowing the client to determine the action without prior knowledge. Contextual traits update current choices depending on the state of the resource and the client's location in the application flow.

## Designing hypermedia controls

Beyond the fundamentals, designing effective hypermedia controls involves the careful crafting of links not only to indicate available actions but to situate those actions within the surrounding application state. The link layout must be helpful for clients to recognize relations between resources with very little external information. A good way to achieve this interaction is through consistency of hypermedia control design, making it possible for clients to predictably and safely follow links exposed by the API. If the links are designed consistently, clients can parse and follow them safely without risk of ambiguity in any subsequent step. Consistency can be applied in the same way to naming relations, expected HTTP methods, and the structure of the link object. By making these things uniform, the API is simpler to integrate and more predictable. Consistency also encourages reuse in the design because the same type of resources can reuse the same kind of hypermedia structures. Through a consistent approach, developers can focus on what is unique to their business logic rather than constantly having to reinvent small navigation cues:

{

"title": "The Great Gatsby",

"author": "F. Scott Fitzgerald",

"isbn": "9780743273565",

"availability": "In Stock",

"links": {

"self": {

"href": "/books/the-great-gatsby",

"method": "GET",

"rel": "self"

},

"update": {

"href": "/books/the-great-gatsby",

"method": "PUT",

"rel": "update"

},

"delete": {

"href": "/books/the-great-gatsby",

"method": "DELETE",

"rel": "delete"

},

"purchase": {

"href": "/books/the-great-gatsby ",

"method": "POST",

"rel": "purchase"

}

}

}

This JSON example suggests effective hypermedia control design, and consistency of API interaction. Every hypermedia link contains a clearly defined, self-describing pattern so that clients can navigate the resource state with minimal external guidance. Relation names (self, update, delete, purchase) are clear and easy for clients to determine what operations can be performed. The link object structure is consistent, each specifying an href, method, and rel, which further helps ensure predictability in the interaction. The example also puts actions in the context of the application state overall. That is, a purchase link exists only because the book is available.

Hypermedia controls must be flexible, not just consistent. An API evolves, adding new relations and actions. A good design absorbs these changes. This approach shifts the burden: the server tells the client what to do next. The client no longer needs direct knowledge of the server's pathways in advance. Forget fixed URLs. Let the client follow links in the API response. Then the backend can change, and the client won't break. We gain even more flexibility with conditional logic. The server evaluates a resource's state and responds with conditional links. In this way, the API becomes a responsive interface, by adapting itself to the context of each request:

  * Confirm the order (confirm) via a POST request.

### Enhancing client navigation

Navigating a hypermedia system from the client perspective requires careful design of a state transition graph that maps out every potential client interaction. Hypermedia systems should be as intuitive to navigate as a visit to a well-designed website. An API state transition graph defines all valid client actions. This involves an exhaustive analysis of client steps and the permutations from and through every state transition. Starting with the retrieval of an initial resource, transitions flow through multi-step interactions. Each path in a state transition graph may then be encoded into the API’s responses through carefully placed hypermedia links.

Placing links in a resource directly impacts client navigation. Placement enhances the consumability of the API response. Where, how, and why links are aggregated can influence how easily a client can interpret the response. Logical organization of similar links, like placing all update-related links in a cluster, can make it more transparent. Adding visual indicators such as link titles or descriptive attributes also gives context to potential actions. While these cues are intended for programmatic integration, they also help human developers to consume the API. The design process should therefore incorporate iterative tests and feedback to improve the shape of hypermedia controls. Through an organization of links, the API itself becomes a self-documenting interface, more intuitive than external documentation. These characteristics of the design speed up client adoption.

In addition to thoughtful link organization, adopting a standard format like Hypertext Application Language (HAL) can also add to the readability and usability of hypermedia controls. HAL defines an Internet standard for embedding hypermedia links into API responses to implement HATEOAS. For example, by incorporating links in an object called "_links", HAL allows clients to discover supported actions without relying on custom, API specific URL schemes. The placement of an "_embedded" resource improves API interactions by reducing the number of requests used to GET related data. HAL facilitates API changes without impact on clients. Developers can embed and link the same resources across different API endpoints, following a common pattern. HAL encourages this kind of reuse, for clients to consume resources uniformly:

{

"_links": {

"self": {

"href": "/orders/67890"

},

"cancel": {

"href": "/orders/67890?status=cancel",

"method": "POST"

},

"update": {

"href": "/orders/67890",

"method": "PUT"

}

},

"_embedded": {

"customer": {

"customerId": "12345",

"name": "John Doe",

"_links": {

"self": {

"href": "/customers/12345"

}

}

}

}

}

In this example, the _links element adds relative URIs that point clients to operations available on the order, fetch (self), cancel, or edit (update). These links are relative paths, so they are adaptable based on the base URL of the API. The _embedded section has a customer in the order. No need for additional requests to fetch these details. The self-link aspect of the customer in the embedded resource allows clients to navigate to the customer resource easily without an external call.

### Iterative link curation

Deciding which links to include and exclude in a response plays an important part of hypermedia design. Each link must serve a clear purpose. Too many links may clutter responses and confuse the client. While leaving out links will create gaps in a state transition graph. Decisions about which links to include can be guided by the most critical use cases for the API. In practice, this means preferring those links that support the most frequent or critical state transitions. Each link should be evaluated based on how much it can enhance navigation, simplify client complexity, and support business logic. This level of link curation is required for hypermedia controls to play a reliable role as the engine for state management.

Where there are multiple routes into a given state transition, hypermedia controls may suggest alternatives. An order resource with links to both cancel and modify, for example. This multiplicity allows the client to choose the most appropriate action. The server may also customize available links based on user privileges, resource status, or other factors. By offering options dynamically, HATEOAS defines a highly customized client experience. The system becomes adaptive and responsive. Through multiple paths, the API not only introduces flexibility but also reduces the chance for errors originating from a more rigid process. This implicit design ethic is at the core of the contemporary REST, in which each state comes out of a transition.

Ultimately, the creation of hypermedia controls is an iterative process, informed by continuous feedback from real use. By observation, API authors must carefully monitor the way clients use hypermedia responses and adjust link structures to optimize them. 

## Best practices and implementation

For RESTful APIs to implement HATEOAS, some best practices can be followed by adopting a standard. Among the most common proposals is to guarantee that a full set of hypermedia links is present in each representation of a resource that define all valid state transitions. These relationships must be consistently formatted and named, so that clients can grok them irrespective of the context in which the resource appears. HATEOAS is more challenging to implement in an unstructured API design approach, if navigational cues are not based on a standard.

Standards expose teams to a community of developers, resources, documentation, and tooling. They also bring ready-made solutions to common problems. For example, HAL, JSON:API, and Siren have become established hypermedia formats. These specifications cover the nuances of hypermedia design and provide developers with a shared platform to work from. Standard formats drive consistency across APIs so clients can integrate regardless of the technology they use. In addition, these standards give guidance to incorporate metadata and conditional links that are essential in adaptive state management. They enable the integration of third-party libraries and tools assuming a standard format in hypermedia responses. Light standards make it simpler to maintain APIs over time.

Error handling is another key HATEOAS best practice that deserves careful attention. A hypermedia system must manage failure, not just success. Every response needs links for errors as well as navigation. If an update fails (a conflict, a bad request) the response must guide the client toward a fix. Provide links to documentation, logs, and recovery endpoints. Bake these recovery paths directly into your API responses. You will build far more resilient, fault-tolerant clients. They can recover easily from failure, reducing downtime. Robust error handling in hypermedia controls is an effective best practice in modern RESTful design:

{

"error": "Conflict",

"message": "The resource is locked",

"_links": {

"retry": {

"href": "/resource/123?status=retry",

"method": "POST",

"rel": "retry"

},

"help": {

"href": "/help/resource-lock",

"method": "GET",

"rel": "help"

}

}

}

To ensure the client can recover from a failure and handle the error gracefully, links can give the client its next steps. In this example, the API response for a conflict error integrates two hypermedia links: one for retrying the operation (retry) and another for retrieving help documentation (help). This enhances fault tolerance and makes the overall client experience more robust in case of failure.

### Optimizing hypermedia

Since the inclusion of hypermedia links can bloat responses, designers must consider performance optimization when implementing HATEOAS. A balance must be found in providing explicit navigational hints and maintaining performance. Selective link inclusion and payload compression can prevent potential performance degradation from response bloat. Hypermedia optimization delivers streamlined responses to clients without experiencing unnecessary delay. Developers should implement tools and metrics to monitor how hypermedia application affects system responsiveness. Through iterative improvement and localized optimization, developers can maximize the advantage of HATEOAS without compromising speed and efficiency. An efficient API strikes the right balance.

Hypermedia state management poses security risks. Links declare what a user can do. So generate them dynamically, based on authentication and authorization. Only users with the right privileges should see sensitive links. And secure these links. Any link that changes state must be protected against common web attacks, like Cross-site Request Forgery (CSRF). With strong security measures in place, hypermedia can drive client behavior securely. Security testing and auditing must be part of a development process on an ongoing basis, for the API to maintain integrity and trust, as it evolves and expands.

HATEOAS needs documentation. Even self-descriptive messages need a guide. Good documentation explains the purpose of links, the meaning of "rel" attributes, and how a client should traverse them. This gives developers crucial context. It makes the API usable. Documentation also aids troubleshooting. It provides a master source to check against inconsistent behavior. As such, pair self-descriptive messages with clear external docs. This two-way approach supports both novices and experts. It unlocks the full potential of HATEOAS. At its core, good documentation strengthens the API and accellerates integration. 

Even in the presence of self-descriptive APIs, documentation complements HATEOAS systems. Documentation should explain the purpose of hypermedia links, the significance of various "rel" attributes, and the expected client behavior in traversing links. This type of documentation is a great benefit to developers who consume the API, as it communicates clearly and adds context that makes the API more usable overall. Moreover, good documentation helps troubleshooting and debugging, as clients can refer to a master source to learn about inconsistent behavior. The combination of self-descriptive hypermedia responses and external documentation creates a solid support mechanism for API users. A two-way approach ensures that both novice and advanced developers can benefit from the highest possible potential of HATEOAS. At its core, good documentation enhances API design principles and fosters a cooperative development environment.

### Advanced considerations with conditional links

Beyond design, performance, and security best practices, adaptive refinements in hypermedia controls can also optimize HATEOAS. One such refinement is conditional links. Such dynamism enables the API to modify the set of potential actions based on the resource's state and external factors, including time and location. This approach ensures that the client is given tailored navigational experience. Conditional links minimize noise in the API response and steer the client to the most relevant actions. This degree of dynamic responsiveness makes it possible for developers to build APIs that are highly responsive to changing conditions.

{

"event": "Outdoor Concert",

"date": "2035-05-15",

"location": "New York, NY",

"_links": {

"self": { "href": "/events/12345" },

"buyTickets": {

"href": "/events/12345/tickets",

"method": "POST",

"rel": "buyTickets",

"condition": "date >= '2035-05-01' && date <= '2035-05-14' && user.geo == 'US'"

},

"viewReplay": {

"href": "/events/12345/replay",

"method": "GET",

"rel": "viewReplay",

"condition": "date < '2035-05-01' || user.geo != 'US'"

},

"joinLiveStream": {

"href": "/events/12345/stream",

"method": "GET",

"rel": "joinLiveStream",

"condition": "date == '2035-05-15' && user.geo != 'US'"

}

}

}

In this case, the external contexts of time and place control the available actions, so the API is context aware. This responsive design eliminates extraneous links, giving the user a relevant and personalized set of actions based on their context. To summarize:

  * The buyTickets link would only be displayed if the event is within a specific time period (May 1st to May 14th) and the user is from the US.
  * The viewReplay link is included if the event was in the past or the viewer is outside of the US, so that they may see a replay.
  * The joinLiveStream link will only be clickable when the event is occurring on the same date and the user is not in the US, offering live streaming to users in specific areas.



Log analysis can make completely transparent how clients interact with the API. Understanding how clients discover and use resources helps designers to improve hypermedia control structure. Insights gained from logs can highlight areas of difficulty for clients, such as poorly defined flows. Logs show details about the most common sequences of requests and allow developers to improve flows by removing unnecessary steps. If clients begin to struggle through specific sequences of actions, logs can show an unambiguous record of their activity, and design teams can use them as a basis to adapt to client needs.

Integration tests are beneficial in the analysis of the client-server environment. Engineering best practices in testing can be applied to hypermedia implementations. In this approach, sets of tests run automatically to confirm that every and any hypermedia link works, and to check whether a client can go through state transitions as expected. The tests can mimic a range of conditions, error states, and corner cases to determine how robust the API performs under a wide range of conditions. Continuous testing and verification give confidence in the navigational stability of the API even as it grows more complex. A rigorous testing regimen is not just a best practice but a necessity for sustainable API development. This level of quality assurance typically gates a system with HATEOAS to be released into a production environment.

### Fallbacks and semantic versioning

To offer secondary navigation routes when some state changes are unavailable, fallback mechanism becomes a pattern. An example of this is when a main link is disabled due to a temporary system condition. The API can instead offer a redundant path that will lead the client to an intermediary state. Redundancy serves not only for enhancing the interface but for improving the reliability of the API as well. A fallback mechanism never leaves clients high and dry. It also highlights the self-healing aspect of the API, since alternative routes may be rerouted programmatically around failure points. This pattern is most relevant in distributed systems, where network outages may cause blips.

Semantic versioning frees hypermedia APIs to evolve. It manages change without breaking clients. The scheme is simple: a three-part number, MAJOR.MINOR.PATCH. MAJOR for breaking changes. MINOR for new features. PATCH for bug fixes. Everyone understands the impact of an update at a glance. Embed this versioning in hypermedia links, and the API guides each client to the right endpoint. It matches links to client capabilities. This makes migration easier. It cuts integration risk. Clients can request the most compatible path for their state, without disruption when the backend changes. Semantic versioning is essential for long-term maintenance. It structures API evolution. It preserves the power of HATEOAS. With these patterns, resource state management becomes more than navigation. It becomes an engine that drives the entire API life cycle.

### Integrating and evolving hypermedia

Support for the evolution of hypermedia calls for a flexible strategy. Developers must design hypermedia controls with gradual evolution in mind to meet changing business needs. A long-term approach is to modularize hypermedia responses such that each module can be changed independently without affecting the entire system. Hypermedia responses can be modularized. Split them into parts: data, links, metadata, errors. Modules decouple. A change in one module, like adding a new link, does not force a change in another. A separation of concerns simplifies maintenance, encourages evolution, and allows for highly customized responses without rewriting entire payloads.

Strategic caching plays a crucial role in advanced hypermedia implementation by improving performance while maintaining the accuracy of dynamic state transitions in HATEOAS. While caching is commonly associated with GET requests, it also plays a role in HATEOAS. Caching of frequently accessed hypermedia responses enables the API to perform and scale. Caching also needs to be adequately controlled, to ensure some dynamic state transitions are correct and up to date. This demands a balance between maximizing performance and the need for real-time state. Techniques such as cache invalidation and conditional requests become solutions. Implemented correctly, caching can provide an unobtrusive experience for clients and still preserve the dynamic nature of hypermedia-based navigation. These techniques add performance engineering to HATEOAS design.

## Conclusion

Resource state management goes beyond data manipulation in REST. It invites clients into a guided navigational experience. As the essence of RESTful architecture, HATEOAS embodies the principle that clients should interact with a system through dynamically discovered transitions rather than by static, predetermined endpoints. By including hypermedia controls in responses, servers guide clients through the state transitions of an application just like the web does. This approach separates (decouples) clients from fixed URI structures, allowing the API to evolve independently. It removes the burden of prior knowledge from the client, and leaves it entirely to the server, where it belongs.

## Introduction

When the four basic constraints of REST apply to an application (client-server, stateless, cacheable, and uniform interface), they change the interaction style of the application components. In doing so, the scalability of the component interactions increases exponentially. And yet, by design, the basic constraints fail to address fundamental security issues, leaving components vulnerable to security threats. Even when implemented correctly (and with the best intentions), the constraints leave open some gaps that may expose APIs to malicious activity, unauthorized access, and data exfiltration. As security professionals, we can do better. This chapter introduces a fifth REST constraint called layering, which deploys intermediaries that can safely close security gaps. Layers in REST architecture protect APIs by enabling security features. In a layered system, intermediaries can authenticate, authorize, encrypt, and monitor activity to secure the application.

At an airport, when boarding a flight, a traveler goes through several similar security layers. First, they must show an ID to the airline to get a flight boarding pass. In the same way, a client provides credentials to authenticate with a REST API, which the server verifies in exchange for a security token. A boarding pass may give the traveler access to a gate, but from there they cannot just wander into any area of the airport. Likewise, in API authorization, a security token may grant a client specific permissions to some, but not all, API endpoints. The traveler may put a lock on their suitcase for protection, ensuring that even if handled during loading onto the aircraft or moved on a conveyor belt, the contents remain confidential. The only person who can open the suitcase and access its contents is the traveler, who has the key to the lock. API encryption protects client-server messages just as a lock on a suitcase protects a traveler's belongings. Encryption keeps prying eyes from reading messages in transit. Finally, like the airport's security cameras, API logging watches every request and response. Logging monitors API traffic by creating an audit trail and capturing traffic patterns that can be used to get ahead of security incidents before something goes wrong.

Just as airports rely on multiple layers of security to protect travelers and their belongings, APIs must also employ layered defenses to safeguard data and backends. Nearly 83% of current web traffic goes through REST APIs, and almost 40% of organizations hosting them have suffered an API security incident in the past year alone. These statistics reveal a harsh reality: while APIs form the backbone of modern web communication, they are also one of the most vulnerable attack surfaces. The same techniques that allow component interactions to scale: stateless, cacheable, and uniform, make them tragically ineffective against a determined hacker. This chapter helps readers to evaluate these weaknesses and to understand layering as a defensive strategy to secure APIs.

## Authentication

When an application sends a request to an endpoint, the API must first authenticate a user to authorize the request. This section describes two authentication schemes commonly supported by APIs and explains their differences, as a conceptual framework. A simple contrast between basic and bearer authentication brings out the essence of more advanced problems to solve. Fundamentally, authentication always asks for credentials up front, to establish and confirm a user's identity. Identification prevents anonymous access. This allows the system to associate requests with a specific user identity and to trace user actions. While it may be appropriate in some cases for anonymous requests to be accepted and processed, the lack of an identity opens the system up to easy exploits. Without an identity, malicious actors can operate undetected, making it trivial for them to flood endpoints and exhaust resources without a trace. As such, anonymous requests can be thought of as lacking accountability. For these reasons, API designers are well advised to protect their endpoints, at a minimum, with a basic authentication scheme. In more advanced use cases, bearer (token-based) authentication works as an alternative, and the two mechanisms typically cooperate by routing requests through security layers.

### Basic authentication

Since 1999, RFC 2617 has established a standard for clients and servers to negotiate the exchange of credentials for access over HTTP. The basic authentication scheme encodes credentials using Base64, to easily transmit character values through an HTTP header. Base64 converts a username and password pair into a standard ASCII string format to ensure that reserved characters like colons, which separate the username and password, will not interfere with header parsing.

On the client side, the encoding scheme simplifies an explanation. Starting with username and password values, alice and password123, the first step concatenates the two values with a (":") colon character:

alice:password123

By applying Base64 encoding to this new, combined pair, we get an obfuscated text value to send in an HTTP header:

YWxpY2U6cGFzc3dvcmQxMjM=

One of the reasons for encoding this way is to support the transmission of a password with a colon (":") character as part of the value. The standard also restricts the set of characters that are allowed within the header called Authorization, to a subset of ASCII to prevent intermediary servers like proxies and load balancers from stripping or filtering accented characters (á, é, í, ó, ú) and non-printable ASCII control characters (carriage returns and line feeds).

The client may then form and send an HTTP request that includes the basic authentication header, a colon-delimited, encoded value prefixed by the word Basic followed by a space:

> GET /example-api/books/ HTTP/1.1

> Host: example.com

> Authorization: Basic YWxpY2U6cGFzc3dvcmQxMjM=

Base64 encoding provides no security, only transport compatibility. Internal sites behind VPNs may tolerate this scheme, and legacy systems that may be ok with simplicity over risk. The encoding merely ensures servers can decode transmitted credentials consistently, based on the RFC standard.

At the receiving end, on the server, a simple implementation of Basic Authentication would then:

  1. Check for the Authorization header.
  2. Decode the Base64 payload.
  3. Split credentials and validate.
  4. Accept or reject the request with 401 or 403 HTTP response code on failure.



The following pseudocode snippet implements Basic Authentication steps as a Node.js/Express API:

import express from 'express';

import base64 from 'base-64';

const app = express();

app.get('/protected', (req, res) => {

const authHeader = req.headers.authorization;

if (!authHeader?.startsWith('Basic ')) {

res.set('WWW-Authenticate', 'Basic realm="Secure"');

return res.status(401).send('Unauthorized');

}

const base64Credentials = authHeader.split(' ')[1];

const credentials = base64.decode(base64Credentials);

const [username, password] = credentials.split(':');

// Validate against a database or LDAP (pseudo-code)

if (username !== 'alice' || password !== 'password123') 

{

return res.status(403).send('Forbidden');

}

res.send('Access granted!');

});

The server reverses the process performed by the client:

  * Extracts YWxpY2U6cGFzc3dvcmQxMjM= from the header.
  * Base64-decodes it to alice:password123.
  * Splits on ":" to validate username/password pair.



From a security standpoint, note that basic authentication sends username and password values as an encoded string in a header that can be read back easily into plaintext. HTTP sends data in plain text, while HTTPS encrypts to secure it. Without HTTPS, credentials are sent across the network in the clear and may be intercepted using packet sniffing or compromised by a man-in-the-middle (MITM) attack. Because HTTPS encrypts the entire request, it ensures the encoded credentials remain confidential in transit. But even with HTTPS, basic authentication remains vulnerable to replay attacks, an exploit in which intercepted credentials can be reused unless combined with additional protections like short-lived tokens or challenge-response mechanisms.

### Bearer

While basic authentication's simplicity supports legacy systems, its security is weak, as credentials are transmitted in easily decoded base64. This leaves it open to attack, even over HTTPS. Basic auth validates credentials against a data store on each and every request, which becomes a performance bottleneck in high-traffic environments. Bearer authentication addresses these flaws by replacing cleartext credentials with short-lived tokens. In contrast to the persistent exposure of password values over basic authentication, bearer tokens function as ephemeral keys that are invalidated after expiration. Bearer authentication eliminates the need for repeat password checks by issuing time-bound, opaque tokens, to shift the validation work from a computationally intensive password check to a stateless cryptographic verification. While basic auth may cripple your user store with long running password comparison checks, bearer tokens only need a signature verification which can be done many thousands of times per second, even on a low-end system.

#### JSON Web Tokens

Generic bearer tokens may solve basic auth's password reuse problem, but JSON Web Tokens (JWT) are the favored, smarter alternative. Unlike generic tokens, JWTs encode identity claims and expiration directly into a cryptographically signed payload. This allows for stateless validation without any loss of verifiable content. The token format is based on a standard and preserves the bearer token security model: possession grants access. JWT augments the model with structured data and cryptographic assurances, solving basic auth's two fundamental security flaws and providing a high-performance alternative to centralized verification in generic tokens.

A sample JWT may look like this:

eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMjM0NTY3ODkwIiwibmFtZSI6IkpvaG4gRG9lIiwiaWF0IjoxNTE2MjM5MDIyLCJleHAiOjE1MTYyNDkwMjJ9.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c

Where, its decoded structure has the following parts:

Header (algorithm and token type):

{

"alg": "HS256", // HMAC-SHA256 signing

"typ": "JWT"

}

Payload is:

{

"sub": "1234567890", // Subject (user ID)

"name": "John Doe", // Custom claim

"iat": 1516239022, // Issued-at timestamp

"exp": 1516249022 // Expiration timestamp

}

Signature:

Created by signing base64(header) + "." + base64(payload) with a secret key (for example, "your-256-bit-secret").

The signature component gives tamper proof qualities to the token. It cryptographically signs the payload and the header with a private key. When generating the JWT, the server hashes these elements and then encrypts the resulting hash value using a private key to generate the signature. After being issued, any modification of the token will fail a signature check, since a newly computed hash value will differ from the signature. The check ensures that, despite any modification of the payload (for example, user permission update), a signature mismatch will cause the server to reject the tampered token. A strong encryption algorithm, HMAC-SHA256 or RSA, guarantees the integrity of the signature in a JWT.

Despite the advantages over basic authentication, bearer authentication has its risks. Say if a token finds its way into the wrong hands, then someone can impersonate a user until the token expires. Possession grants access. To prevent this eventuality, developers may issue shorter lived tokens that limit the amount of time to harm with it. Refresh tokens go hand-in-hand with this approach, to allow clients to get a new access token without full re-authentication. By issuing tokens with a short expiration period, bearer authentication limits the risk of an impersonation attack. Through the combination of a token standard with practical time-limit constraints, the bearer authentication scheme has become an API security mainstay as it finds a good balance between utility and simplicity. The safe exchange of tokens over HTTPS with proper token management strengthens the security profile for modern REST APIs.

The following sequence diagram illustrates the transition from basic to bearer authentication in a typical REST API workflow:

Figure 6.1: Basic and bearer authentication flows

In the sequence, a client sends a POST request to an /authenticate endpoint, that includes a Base64-encoded user:pass value in an Authorization header. The server checks the credentials in the header against an identity store (database or LDAP). This step has a computational cost, when every request hits the data store with a lookup to validate credentials. The overhead of repeat lookups really adds up. Upon successful verification, the server returns a JWT (access_token). This token includes the user's identity, so there is no need to make duplicate data store lookups to authenticate the same client. The client sends the JWT in an Authorization: Bearer header in every request it makes (for example, GET /user). The server verifies the JWT signature (a cryptographic operation) instead of connecting to and querying the database each time. This approach lowers the computational overhead since signature verification is much cheaper than repeat database access. A credential verification model that looks up users on every request scales poorly for basic authentication, while a stateless JWT validation scheme in bearer authentication optimizes performance in high-traffic environments.

## Authorization

Authorization defines a client's actions as an authenticated user or application in a system. While authentication verifies an identity (through passwords and tokens), authorization defines an action scope: the specific resources or data a user can take action upon, and under what conditions. Similar to security at an airport, authorization allows a passenger with a boarding pass to go through a security checkpoint after they show some form of ID (passport or driver's license). A boarding pass usually prescribes a gate (an API endpoint) and a seat class (an accessible resource). With a pass in hand, they can freely access these areas. Without some basic authorization restrictions, authenticated users could easily overreach their permissions, like a passenger entering the cockpit of an airplane, resulting in serious risks to the other passengers, the pilots, and the airline. The following sections discuss authorization mechanisms commonly layered into REST APIs to prevent users from crossing into restricted areas.

### Role-based

A role-based access control (RBAC) model grants permissions to a user commensurate with their role in the system. Users get assigned to one or multiple roles such as administrator, editor, or viewer. In a simple hierarchy, a user with the editor role can create, read, update, and delete resources, while a viewer in a lower position can only read them. In this model, an administrator ranks highest and has full control over resources and other users.

Security architects and API designers favor RBAC as an authorization model because it effectively manages permissions in systems where user responsibilities correlate closely to well-defined organizational roles. The model fits well in organizations with multilevel user hierarchies. By consolidating permissions into roles as in admin, editor, or viewer, rather than granting them individually to all users, RBAC reduces the administrative overhead and the potential for an overly permissive environment. The model scales into growing organizations since new users can be granted access directly by being added into predefined roles, with no need to re-engineer permissions. RBAC also sets up an organization for audit and reporting, enabling companies to comply with regulations by being able to prove precisely which roles have access to sensitive data or systems.

Refer to the following figure:

Figure 6.2: Role-based access control model

Figure 6.2 depicts a hierarchical RBAC model within an organizational structure. The top-level Org branches into Practice A and Practice B, each being composed of facilities like Facility 1 and Facility 2. A node in the tree is like a container for the one beneath it. The level under Facility, Resource, represents a leaf-level node with no subordinates. Different roles can access the resources in the tree: Admins have Read/Write permissions that start from the root and emanate downward into the lower levels, while Editors can Read/Write, at or below the Facility level. The model grants Viewers read-only permissions on resources, but they cannot access the higher levels in the hierarchy. A structure like this illustrates basic RBAC principles, where access and permissions are granted to a role, giving Admins the highest privileges. The diagram shows permissions cascading into a multi-level organizational structure, scoped by a role.

RBAC has drawbacks. The model proves too rigid in dynamic environments where access needs change all the time. In these cases, RBAC struggles with temporary access to a resource outside of a user's current role, without creating yet another role, leading to role explosion. A proliferation of roles turns into a complex model to unravel and manage. Developers may consider combining RBAC with other models as a solution. RBAC does the heavy lifting in structured environments with well-defined job roles but can be less adaptive in situations that require context-sensitive access. In such cases, attribute-based access control (ABAC) may be a better fit.

### Attribute-based

ABAC sets up a more dynamic approach to authorization. Instead of determining access based on predefined roles, ABAC takes into account attributes, like user location, time of day, or device type, to make a decision. For example, an API might provide access to a resource only if the user is coming from a specific IP range during business hours. This level of granularity suits ABAC for complex systems with fluid access requirements. It is generally useful in cloud environments where resources and users are much more volatile. ABAC's flexibility is owed in part to its policies, which define rules that evaluate attributes, and the system enforces them in real time. ABAC solves the user authorization problem without rigid role hierarchies through fine-grained policy.

The challenge in ABAC is complexity. Policy definition and management can be cumbersome, usually in large systems where there are a high number of attributes. It also requires greater processing power because each request needs to be examined against potentially hundreds of policies. Regardless, ABAC has become widespread in modern use, common in applications with strong security requirements. For example, medical facilities use ABAC to restrict access to a patient's file to specific medical personnel at certain times. ABAC can be combined with RBAC to provide a hybrid, adaptive authorization mechanism.

#### OAuth 2.0 scopes and permissions

Like a customizable boarding pass for APIs, OAuth 2.0 scopes tailor authorization decisions by defining contextual boundaries around client access. Instead of static, role-based permissions, scopes enable developers to manage access at a more granular level. A [read:data] scope, for example, only permits reading resources, while [write:data] allows modifications.

Newer OAuth 2.0 flows put scopes directly into tokens, so APIs can enforce permissions without database lookups. JWT with scope: [update:profile], for instance, tells the server the client may update user profiles but not delete accounts. Standards like OpenID Connect normalize scope claims so they will work across identity providers. Even so, poorly defined scopes can make APIs vulnerable. An overly permissive wildcard scope like [delete:*] could grant destructive power to a malicious user.

Scopes combined with RBAC/ABAC extend OAuth 2.0 above traditional models. While RBAC defines roles like editor or viewer, scopes can limit the actions within roles, preventing an editor with [publish:articles] from also having permission to [delete:users]. In ABAC systems, scopes mix with attributes to enforce policy. For example, a policy can grant access to financial_records only if the user's is department finance and the token's scope includes [read:reports].

In practice, when a client requests an access token from a token service, the client can ask for specific scopes it wants to have associated with the token. Doing so, the client limits the risk of a compromised token by requesting only the scopes needed for a specific task. In the following request, a client requests scope X. That is, the client asks the token service to generate an access token that has scope X (giving the client authorization to call endpoints that have scope X). The client can send a POST request like this:

curl -i -X POST -H Authorization: Basic dXNlcjpwYXNz \

https://example.api.com/oauth/token?grant_type=client_credentials&scope=X

As the token service receives the request, because a scope query parameter is included, the server needs to decide if any of the endpoints associated with the client app have scope X. If they do, then an access token can be generated with scope X. Another way to look at the scope query parameter is as a kind of filter. If the token service recognizes scopes A,B,X, and the query parameter specifies scope=X,Y,Z, then only scope X will be assigned to the token.

Imagine a traveler attempting to secure lounge perks at an international departure terminal. The traveler (acting as our OAuth client) cozies up to the airline's exclusive service counter (our authorization server), credentials securely in hand (encoded via Basic Auth). "Priority boarding and Sky Lounge access, please" they proclaim (scope=A,B), flashing their loyalty card. Behind the counter the agent checks and verifies the traveler's status against the airline's benefits. While our jetsetter may feel entitled to chilled champagne and lavender scented hand towels in first class (scope=A), their current tier only grants them general boarding access (scope=X).

The approved boarding pass (our access token) clearly displays only the authorized privileges in its terms section (the token's scope claim), creating an unforgeable permission record. Later, when our traveler attempts to enter the fancy lounge (API endpoint), the personnel (API gateway) perform a quick visual check. This pass only shows general boarding, they note professionally, denying entry with a polite but firm gesture (403 Insufficient Scope). The analogy from the real world reflects the resilience of OAuth: no crafty social engineering can evade the printed restrictions, and all authorization decisions are recorded in the airline's systems (audit logs).

### Encryption

While OAuth 2.0 scopes and RBAC/ABAC models control what users can access, encryption ensures they can trust the method they use. Encryption, through protocols like Transport Layer Security/Secure Sockets Layer (TLS/SSL), ensures the confidentiality of data in transit. Combined, the two security layers form a more complete security system: authorization controls access, and encryption secures a communication channel.

#### TLS/SSL

HTTPS relies on the TLS protocol, a modern alternative to SSL, to encrypt data as it is passed between client and server. Encryption encodes sensitive information, especially passwords, credit card numbers, and API keys, into text that no one eavesdropping on the communication can read. Data passed through HTTP without TLS is sent in clear text, subject to MITM attacks. In simple terms, an attacker can intercept credentials if a user logs in to an API through an unencrypted link. TLS prevents this by creating a secure tunnel so that even if the data intercepted is read, it remains unintelligible. As such, HTTPS secures web communication, protecting both APIs and users from data breaches.

### Authorization

To prove the authenticity of the server's identity, HTTPS uses digital certificates, ensuring the client connects to a trusted server and not an imposter. To verify the server's identity before issuance, certificates are issued by well-known certificate authorities (CA). When a client connects to an API through HTTPS, the server displays its certificate, and the client validates it against a trusted list of CAs. If the certificate is valid, then the client will know for sure that the server is exactly who it claims to be. Without authentication, an attacker could pretend to be an API server and capture sensitive data from unsuspecting clients.

Some systems use mutual TLS (mTLS) in addition to server authentication, where both the client and server present certificates to verify each other's identities. In high-security environments, particularly banking transactions, both parties need to establish trust. There is a cost associated with the added security. Managing certificates can be expensive and requires careful lifecycle management, including issuance, renewal, and revocation. Teams must always ensure that certificates are stored securely and that private keys are never exposed.

#### Integrity

HTTPS ensures data integrity by using cryptographic techniques to check whether the data sent has been modified in transit. This is accomplished through Message Authentication Codes (MAC) or hash functions, which produce a unique data fingerprint. When the server sends data, it includes this fingerprint, which the client recalculates upon receipt. In simple terms, if the fingerprints do not match, then the client may conclude that the data has been modified and refuse to accept it. A built-in data integrity check guarantees that the information received is the same as what was sent by the server, if there is no evidence of tampering or corruption.

Data integrity is essential for APIs handling sensitive traffic in financial transactions or exchanging medical records. Even a slight change could have serious ramifications, among others, transferring money to an incorrect account or exposing a patient's diagnosis.

## Logging

Logs for APIs can become much more than a means to debug and troubleshoot backend request processing. Properly structured and managed, teams can turn logs into traceable forensic records of who accessed what and when. Logging can give security teams a way to track suspicious activity, and to fully reconstruct events during incident investigations. Without deliberately prepared logs, APIs operate in the dark. In the following sections, we will examine how logs can transform a passive system execution record into an actionable defense strategy.

### Deciding what to log

Effective security logging begins with a decision about what to capture in a system log. Logging events that provide helpful information about system security, along with authentication attempts and authorization decisions, builds a meaningful audit trail. To capture failed login attempts allows for the detection of brute force attacks, while logging legitimate access requests helps to establish the API's normal access patterns. Essential details like a timestamp, user IDs, source and destination IP addresses, and the action taken (HTTP method invoked) by clients, combined, become the context for security incident investigations. At the extreme, excessive logging may include irrelevant details that just add noise to the data set. A consistent, balanced logging strategy ensures that logs remain actionable from a security standpoint.

### Structure your logs

Optimizing logs for machine analysis, notably in large systems with high volumes of data, sets up the system for security monitoring. Use of a centralized logging system like ELK (Elasticsearch, Logstash, Kibana) Stack or Splunk to collect and analyze logs from multiple sources delegates management to an expert system and enables advanced features like real-time monitoring, alerting, and trend analysis. From events logged, alerts can notify teams when a specific event sequence is recognized as suspicious. By structuring logs for ingestion by a specialized tool like a SIEM, teams can turn raw, unstructured data into actionable insights, enhancing their ability to detect and respond to security incidents.

Here is a sample log entry (structured JSON):

{

"timestamp": "2034-02-20T14:23:45Z",

"level": "ERROR",

"service": "payment-api",

"event": "failed_authentication",

“user”: “admin@example.com”,

“ip”: “203.0.113.12”,

"endpoint": "/api/v1/transactions",

"status_code": 401,

"message": "Invalid API key"

}

### Logging pitfalls and best practices

Logs allow teams to keep an eye on security risks and debug issues, but only if done right. Never log passwords, API keys, or personally identifiable information (PII). If disclosed, such logs are goldmines for hackers. Use consistent log structures. If one service logs JSON and another dumps raw text, it complicates log correlation. Pick a structure early on and stick to it. Lock down log access. Treat logs as sensitive data. Encrypt them, restrict who can view them, and monitor them for unauthorized access. Moreover, define specific retention policies. Keeping logs forever is costly and adds compliance challenges. On the other hand, erasing them too quickly removes forensic evidence when you may need it most.

## Conclusion

The security measures discussed in this chapter constitute a first line of defense. Not a static problem to solve, API security requires an evolving strategy that adapts to new threats. Just as authentication mechanisms have moved beyond the insecure simplicity of basic auth to the use of safer and more scalable bearer tokens with cryptographically signed JWTs, authorization models have grown more sophisticated, with rigid RBAC role hierarchies being made more elastic through ABAC's expressiveness and OAuth's fine-grained scopes. Despite these basic security measures, REST APIs remain irresistibly vulnerable, not because the measures fail, but because the problem space continues to morph.

The RESTful constraints that give APIs their power, that is, statelessness, uniformity, and cacheability, expose qualities that make them attractive targets. Every new feature risks turning into a potential attack vector, and every convenience carries the seed of a potential exploit. This chapter covers some basic security features for API designers to layer into their architectures as absolute minimum safeguards, but a deeper security mindset emerges only by probing into the gaps between these concepts to confront threats we do not yet understand.

## Points to remember

  * REST constraints improve scalability but leave security gaps. Layering acts as a fifth REST constraint. Intermediaries in a layered system handle authentication, authorization, encryption, and monitoring to close security gaps. 
  * Authentication verifies identity, while authorization controls access. Basic and bearer authentication confirm user identity, whereas RBAC and ABAC access models define permissions.
  * Basic authentication is simple but insecure. It encodes credentials in Base64 for transport but offers no real security. HTTPS is mandatory to prevent interception, but replay attacks remain a threat.
  * Bearer tokens (like JWTs) improve security and performance. Tokens replace repeat password checks with cryptographically signed, time-bound credentials, reducing database load and exposure risks.
  * JWTs embed claims and expiration for stateless validation. Their tamper-proof structure (header, payload, signature) allows servers to verify identities without database lookups.
  * ABAC adds flexibility. Policies evaluate dynamic attributes (time, location, device) for granular control, which is ideal for cloud systems. 
  * OAuth 2.0 scopes refine authorization. Scopes limit token permissions (for example, [read:data]), working in conjunction with RBAC/ABAC to prevent over-privileged access.
  * Encryption (TLS/SSL) protects data in transit. HTTPS ensures confidentiality, server authenticity, and integrity, while mTLS adds client verification for high-security use cases.
  * Logging transforms passive data into active defense. Structured logs (timestamps, user/IP details, actions) enable monitoring and forensics, but sensitive data must be excluded to prevent exposure.

## Introduction

The web's success was never guaranteed. REST's architectural constraints arose out of necessity, conceived at a time when bandwidth was scarce, latency was high, and servers were brought to their knees by unrelenting demand for service. Decades later, the challenges endure, though their form has changed. Scalability is no longer an abstract concern: it measures a system's ability to bear the weight of its own success. Performance focuses attention on the transmission of data through compressed payloads to reduce latency. Scale requires foresight. To architect scalable systems, web traffic must find a balance in limited computational resources and an even distribution over time. These twin concerns are not theoretical. They make or break a RESTful service. To build for performance is to respect finite resources: budgets, bandwidth, CPU cycles, the patience of users. To build for scale contemplates the infinite: traffic, data, demand. Together, performance and scale determine whether a system will merely function in the short run, or stand the test of time.

## Performance

In [Chapter 4](ch04.xhtml), Resource Representation, we examined the importance of designing RESTful resource representations that serve dual needs. On one hand, it is easy for clients to interact with a resource that reflects on the application's state. On the other hand, the links in resources address state changes by guiding clients in their navigation through the application's space. In both cases, the resulting resource composition grows rich with metadata, embedded links, and derived fields that enhance their usability. REST resources must be packaged for transmission in a way that minimizes their size, because bandwidth constraints can increase latency. This section inspects a common data compression algorithm used by REST APIs to drastically reduce the amount of data to transmit. Beyond compression, to split up collections of resources into batches, or pages, ensures clients receive only necessary data, without over-fetching.

### Data compression

Few optimizations contribute more to performance in REST APIs than effective HTTP payload compression. On average, a good compression algorithm dwarfs a large payload into an equivalent that is nearly half its original size, turning bloated exchanges into swift transfers. To achieve this reduction, compression takes advantage of repetition in data. Gzip strips away repetitive sequences, and replaces them with smaller symbols, to repackage data into a much leaner form. While the result of good compression diminishes messages to a bare minimum required for network transmission, the reverse process, decompression, adds processing overhead that must be incurred on each and every request. Going over large amounts of data in this way may become computationally expensive, which requires API designers to consider the trade-offs.

#### Gzip basics

Data compression, in particular the compression of text, combines multiple techniques to produce a compact equivalent. Standard text compression starts by scanning for redundancy. In a simple example:

Original: HelloHelloWorld

Compressed: Hello(5,5)World

The compressed result (5,5) replaces the second, repetitive character sequence Hello with directives: go back 5 characters, then copy those same 5 characters to fill in for the original. Compared to the original 15 bytes (assuming 1 byte per UTF-8 character), the compressed output turns into 13 bytes (5 + 3 + 5), which is about 13% smaller. Scanning for repetitive sequences in text and replacing them with directives is achieved through an algorithm called LZ77 or Lempel Ziv, named after the inventors Abraham Lempel and Jacob Ziv in 1977.

The next stage in standard text compression focuses attention on frequent single characters that occur in the text. In the example above, the character l, gets a short binary code as a replacement. This character occurs 3 times in the LZ77 output, so for a 00 to stand in its place instead of the full 8-bit byte the character usually occupies, achieves a meaningful reduction in size. The replacement of characters with binary codes based on a frequency analysis is called Huffman coding. In this second pass at compression of compressed output, the following table summarizes the replacements.

Second pass, Huffman frequency analysis:

Symbol |  Frequency |  Assigned code |  Code length  
---|---|---|---  
l |  3 |  00 |  2 bits  
o |  2 |  01 |  2 bits  
5 |  2 |  10 |  2 bits  
( |  1 |  1100 |  4 bits  
) |  1 |  1101 |  4 bits  
, |  1 |  1110 |  4 bits  
H |  1 |  11110 |  5 bits  
e |  1 |  11111 |  5 bits  
W |  1 |  10000 |  5 bits  
r |  1 |  10001 |  5 bits  
d |  1 |  10010 |  5 bits  
  
Table 7.1: Huffman frequency analysis

Through Huffman coding, the 13 bytes in the LZ77 output, Hello(5,5)World, shrinks down to just 6.4 bytes, for a 51% reduction in size. A full breakdown of this two-stage process proceeds, as follows:

The original data, HelloHelloWorld (15 bytes) turns into a compressed LZ77 output of 13 bytes (104 bits):

  * Hello = 5 bytes
  * (5,5) = ~3 bytes (pointer)
  * World = 5 bytes



After Huffman encoding, it becomes 51 bits (6.375 bytes):

  * Frequent characters like l (3x) and o (2x) get 2-bit codes
  * Rare characters like H and W get 5-bit codes
  * Meta-symbols (, ), , get 4-bit codes



Total reduction:

  * From 104 bits to 51 bits
  * 51% size reduction (1 - (51/104))
  * Final size: ~6.4 bytes



For REST APIs, a two-stage compression approach typically reduces JSON text by about 60-80%. This efficiency demonstrates why Gzip (LZ77 + Huffman) remains the standard for API compression. JSON compresses particularly well because there is a lot of repetition in it. Starting from an uncompressed payload:

{

"users": [

{

"id": 1,

"name": "Alice",

"email": "alice@example.com",

"role": "admin"

},

{

"id": 2,

"name": "Bob",

"email": "bob@example.com",

"role": "user"

}

]

}

LZ77 stage of compression picks up on repeated strings, id, name, email, role, and replaces all of them with compact pointers to a single, original instance of the string. LZ77 operates like a teacher grading multiple papers with the same common mistake. Instead of writing "Capitalize proper nouns" in the margin multiple times throughout every paper graded, she writes it once at the top and just adds "See #1" next to subsequent errors. The pointer "#1" takes up much less space than writing out the full note repeatedly.

#### Encoding headers

In a client server exchange, through HTTP headers, a client may signal the server to compress a response. Since different compression algorithms may be supported by both, a client tells the server in an Accept-Encoding header, which ones it can decompress. Typically, the absence of an explicit compression preference through headers, disables it. 

For example:

Request

GET /api/users/123 HTTP/1.1

Host: example.com

Accept: application/json

Accept-Encoding: gzip, deflate

Response

HTTP/1.1 200 OK

Content-Type: application/json

Content-Encoding: gzip 

Content-Length: 153 

[Binary GZIP data...]

This exchange, albeit simplified, demonstrates how compression can be integrated into basic RESTful principles for API performance gains. The Accept-Encoding and Content-Encoding headers follow REST's uniform interface constraint, for clients and servers to negotiate a compression algorithm. The client does not count on the server having a memory. The server is stateless. The client tells the server everything it needs to know with each request. The use of HTTP headers also supports a layered system, where intermediaries like an API gateway or CDN, may compress the response even further, without breaking the client-server contract.

#### CPU tradeoffs

Compression improves how well an API performs across the network by reducing payload size, but it adds a computational cost. An algorithm like Gzip takes CPU cycles to scan data for redundancies (LZ77) and symbol encoding (Huffman). As such, for high-traffic APIs, Gzip may increase latency under load as servers spend more compute on compression than on request processing. The trade-off makes sense when the computational cost is incurred for large payloads. But CPU overhead is difficult to rationalize for responses under 1KB. Modern servers compensate using hardware acceleration. They may also use pre-compressed, static content. Applying compression to compressed file formats such as jpeg, zip, and mp4, wastes CPU. API designers must find a sweet spot between compression pay-offs and compute (CPU cycles), incurring the cost only when bandwidth savings can justify it.

On the client side, decompression has overhead too, especially on low power mobile devices. While modern browsers deliver built-in support for Gzip, applications processing high volumes of compressed API responses may see slower rendering or increased battery drain. Given client-side technical limitations, some APIs compress responses as an optional feature using Accept-Encoding, so that clients can decide if they should use it according to their network and CPU constraints. Also, other algorithms like Brotli offer higher compression ratios at a much lower compute cost on certain payloads. The decision to compress must factor in server capacity and client capabilities: what helps a data center may strain an IoT device. To ensure compression delivers a net benefit rather than becoming a burden, the API's performance metrics must be taken into account.

### Over-fetching

While compression reduces the size of data transmitted over the wire, it does so without consideration for the relevance of the content in a response. Sometimes the server sends unnecessary data that clients just throw away. Even compressed payloads waste processor time and bandwidth if they contain fields the client never touches. This inefficiency, known as over-fetching, occurs when API calls produce more than a client requires, forcing unnecessary movement of data. The consequences ripple across performance metrics. Over-fetching slows down requests. Generating bloated responses ties up server resources. This means the server cannot handle as many requests at once, so overall throughput drops. Take, for example, a user profile request from a client that returns 50 fields but only displays 5, a response that amounts to 90% over-fetching, which saturates networks and bogs down CPUs. Performance optimization must tailor as well as shrink API responses.

#### Field filtering

A common approach to prevent bloated responses, lets clients define fields of interest on the URL's query string. A parameter that accepts a list of fields to include in the response gives clients granular control over what they want to consume. Through such a parameter, the client can pare down an obese payload into something lean and purposeful:

GET /users/123?fields=id,name,email

This level of precision, when used to extract huge collections of JSON objects, transforms an indiscriminate data dump into a highly selective extraction. For apps that display summary views, the performance benefit of field-filtered payloads typically falls into a range of 50-90% reduction in size. Network transfer times fall sharply as API responses dump excess kilobytes. Instead of pulling entire rows needlessly, just to discard most fields when constructing responses, servers benefit on the backend from a lighter database load as queries select specific columns. They stop hauling entire database rows like overpacked suitcases. The benefits compound at scale: when aggregated across millions of API calls, mere milliseconds squeezed out of every response can add up to hours of reclaimed compute.

### Pagination

Pagination offers another path to precision, especially when dealing with open-ended requests for collections that threaten to overwhelm the backend. To prevent clients from requesting excessively large pages, the server should always enforce a maximum limit, using it as a safeguard value whenever the client's requested limit exceeds it. Where field filtering works vertically through attributes, pagination cuts horizontally across records, slicing whole datasets into digestible subsets. Used to divide search results which could add up to millions if not billions of results in today's big data ecosystems, pages act like a sliding viewport (a window) on a long, unbroken bookshelf that reveals only a small section of the shelf at a time. The client controls how this window moves using API parameters, focusing on a single relevant shelf segment at a time.

How the window moves is based on its size (limit) which defines how many items may fit into it. It also moves by a distance (offset) that sets the window's position relative to the first item. For example, a client can tell the server show me 5 books, starting after the first 10, by sending offset and limit parameter values: 

GET /books?offset=10&limit=5 

The offset counts books from the beginning of the set. As a visual analogy, take the full data set to be the upper-case letters A - Z, where each letter stands for a book on the shelf:

[A, B, C, D, E, F, G, H, I, J, K, L, M, N, O, ..., Z]

A request for /books?limit=5,offset=0 says show me the first 5 books, returning the subset:

[A, B, C, D, E]

A second request for /books?limit=5,offset=5 says skip the first 5, show me the next 5, which returns:

[F, G, H, I, J]

A third request for /books?limit=3,offset=12 says skip 12 books, show me just 3, moving the viewport to the 12th book and displaying the next 3:

[M, N, O]

Here, the parameters act as knobs that adjust the viewport, letting clients pan across segments of data to keep the system from becoming overwhelmed by open-ended queries. While the mechanics of limit and offset work well together, their implementation suggests deeper questions about how an API communicates page position in a dynamically changing data set and the finality of it. A well-designed pagination scheme avoids becoming misaligned if the letters shift. Say if someone inserts X in front of the data set, then after the second request, offset=5 lands on [E, F, G, H, I], off by one. This page shifting problem also happens if letters are removed during pagination.

If the bookshelf inventory is dynamic, books might be added or removed while you are searching through them. As an alternative to offset, a cursor solves this problem by acting like a bookmark that flags a specific book, so your window never loses its place:

Request

GET /books?limit=5&cursor=K 

Response

[L, M, N, O, P]

With a cursor, it does not matter if books are added or removed because a cursor anchors itself to a specific book in the set, regardless of its distance from the first.

Pagination goes beyond the mechanics of moving through data sets. It shapes how users experience search and the finality of it. Infinite scrolling deceptively mimics infinity. Behind an illusion of seamlessness lies a careful pagination scheme, with each batch sized just right to feel instant. Well-designed pagination does not leave clients guessing when they have hit the end of a page set. It provides clear navigation signals. Dynamic next and previous links, and a 204 No Content HTTP status can notify clients when they have reached the end. These approaches prevent wasteful checks for more data that is not there. The intentional limits imposed by pagination act as throughput valves that keep the system from bursting under a flood of unbounded requests. 

#### Sorting

While pagination splits data into more manageable chunks, and filtering trims off all those extra fields, sorting brings order to the results. By allowing clients to specify the sequence of records, an API can also prioritize the most relevant items within the viewport. Sorting can arrange the items for a client interested in the very last records, (e.g., the lowest price or the least popular), rather than paginating blindly from the beginning.

A client can retrieve the last four books by sorting in reverse-alphabetical order and taking the first page:

GET /books?sort=-title&limit=4

This command effectively flips the bookshelf, bringing the end into view first and returning the subset [Z, Y, X, W]. The sort parameter organizes the shelf before the viewport slides into place.

## Scale

API performance optimization attends to the speed and responsiveness of single API calls. As we have explored in this chapter, through compression and filtering, a single client gets exactly what it needs, quickly and efficiently. Scale ensures that APIs maintain this responsiveness even as the number of clients grows exponentially. At its core, scalability addresses a fundamental tension between the demand for services by clients and the distribution of work in the backend. Vertical scaling (upgrading server compute) meets the demand, but exhibits diminishing returns after a certain point. It only goes so far. While horizontal scaling (adding servers) trades a unified allocation of CPU cores and memory, for a replicated distribution of them, adding layers of complexity to the system. The divide between monolithic and microservices architectures expresses the two scalability approaches structurally in that monoliths consolidate work until the cohesive system buckles under sprawl, while microservices proliferate to distribute work at the cost of coordination in a vast network of independent nodes.

### Vertical

The most natural solution to a scarcity problem is to go for abundance. Vertical scaling follows this instinct by pouring more resources into a single server or cluster. When an application struggles under load, and the CPU is exhausted at 100% utilization, teams feel an impulse to upgrade, by adding CPU cores and expanding memory. The impulse is not misguided as the strength of a single pillar enables a structure to support more weight. This approach is analogous to the reinforcement of a bridge's central beam. It requires no code changes, and delivers capacity relief with a snap. For monolithic applications and traditional databases, vertical scaling remains a path of least resistance. And yet, like most simple solutions, it suffers in the long run. There is only so much hardware you can pack into a machine before costs skyrocket. The strategy works until it does not. To scale vertically is to define a hard ceiling in which growth beyond it demands a different solution. By then, the system may have become the proverbial basket of eggs, where a single failed server turns into a total outage. Vertical scaling buys time, but rarely endures over the long haul. It is abundance without resilience.

To illustrate diminishing returns on vertical scaling, let us put a real-world application to the test. Say we run a service that handles JSON Web Tokens (JWTs). The service provides REST API endpoints to issue and verify tokens. It generates JWTs from JSON claims it gets in request payloads. And it uses an RSA256 algorithm to sign the tokens it issues. The service also uses the same algorithm along with public and private keys from a keystore to verify a token signature. To invoke the service, a client can send the following requests:

Request: Issue token 

POST /jwt_provider/issue

Accept: text/plain

Content-Type: application/json 

{

"iss": "Issuer",

"sub": "Subject",

"aud": "Audience",

"exp": "2840169600",

}

Response: Issue token

U2FtcGxlIEpXVCBUb2tlbg

Request: Validate token

POST /jwt_provider/verify

Accept: text/plain

Content-Type: application/json

U2FtcGxlIEpXVCBUb2tlbg

Response: Validate token

{

"iss": "Issuer",

"sub": "Subject",

"aud": "Audience",

"exp": "2840169600",

}

Such that the input into the /issue endpoint is the output from /verify and vice-versa. The token service plays a critical role in authenticating clients, which it does continuously. Written in Java, the memory requirements for a single instance of the token service is about 3 MB per client.

As a baseline, we require the service to issue/verify 1,000 tokens (requests) per second while scaling up to 1,000 clients. Given one instance of the single-threaded worker process with limited resources, the service performs well for 10 clients, but as more start to use the service, it underperforms and then fails by exhausting the CPU.

Initial resource allocation: 2-core CPU/2 GB RAM (AWS t3.small equivalent). At baseline, the service handles 100 RPS smoothly but fails under 100 clients, hitting CPU limits and a 25% error rate:

Clients |  CPU utilization |  Memory utilization |  RPS |  Error rate  
---|---|---|---|---  
10 |  30% |  80 MB |  100 |  0%  
100 |  100% |  200 MB |  600 |  25%  
  
Table 7.2: Baseline limits at 2 cores

First vertical upgrade: 4-core CPU / 4 GB RAM (AWS c6g.large equivalent). Doubling resources quadruples throughput (1,800 RPS) but fails again at 500 clients (18% errors):

Clients |  CPU utilization |  Memory utilization |  RPS |  Error rate  
---|---|---|---|---  
200 |  90% |  300 MB |  1,800 |  0%  
500 |  100% |  500 MB |  3,200 |  18%  
  
Table 7.3: 4 cores improve but hit new limits

Second vertical upgrade: 8-core CPU / 8 GB RAM (AWS c6g.xlarge equivalent). With 8 cores, performance improves (5,500 RPS), but errors resurge at 1,000 clients, showing diminishing returns:

Clients |  CPU utilization |  Memory utilization |  RPS |  Error rate  
---|---|---|---|---  
500 |  85% |  600 MB |  4,200 |  3%  
1000 |  100% |  1.2 GB |  5,500 |  12%  
  
Table 7.4: 8 cores delay, but do not remove bottlenecks

Note that after the first vertical upgrade, the service scales up to 2x the number of users, through a 2x increase in resources. But after a second upgrade for 2x the hardware costs, the service only yields 1.3x more throughput at 4,200 RPS for the same number of clients, a classic diminishing returns pattern.

Vertical scaling offers a straightforward, immediate solution to performance bottlenecks by upgrading server resources, initially doubling capacity with a hardware boost but the gains are capped even as the costs increase 2-fold. Perhaps more critically, vertical scaling introduces a single point of failure: if the beefed-up server fails, the entire system collapses. This ceiling forces a shift in strategy rather than push a single server to the limit, a distribution of the same load across multiple smaller servers (or containers) removes the hard scalability cap. ### Horizontal

If vertical scaling fortifies a single pillar under a bridge to bear more weight, then horizontal scaling multiplies pillars to spread the load. Where vertical scale seeks to concentrate power, horizontal scale leverages the power of distribution. When traffic puts stress on an application, instead of investing more resources in a single machine to carry it all, we methodically disperse work across machine replicas. The approach here is not a monolithic upgrade but a systematic division of labor, splitting the traffic between replicas that are interchangeable and dispensable. This shifts the backend from a monolithic single-threaded process to a resilient multi-worker system. No single server has full responsibility, so no single failure spells complete disaster. The trade-off is in synchronization of traffic and persistence. To balance the traffic, a horizontally scaled system is typically fronted by a load balancer which ensures that service instances pick up their fair share of the work. To share persistent state, a common data store allows the worker processes to synchronize on the same data as they join the system.

Since we hit a vertical limit with the JWT token service, let us scale it horizontally. We start by deploying multiple instances of the service behind a load balancer. Each instance runs with the original 2-core CPU/2GB RAM (t3.small) configuration, but now the traffic is distributed.

Baseline (single instance, vertical limit). A single instance struggles under load, hitting 100% CPU and a 25% error rate at just 600 RPS:

Clients |  CPU utilization |  Memory utilization |  RPS |  Error rate  
---|---|---|---|---  
100 |  100% |  200 MB |  600 |  25%  
  
Table 7.5: Single instance performance.

Horizontal scaling (two instances, load balanced). With two instances, throughput doubles (1,200 RPS) and errors drop to 0%, while per-instance CPU falls to 80%:

Clients |  CPU (per instance) |  Memory (per instance) |  RPS |  Error rate  
---|---|---|---|---  
200 |  80% |  180 MB |  1,200 |  0%  
  
Table 7.6: Load balancing across 2 replicas

Scaling further (Four instances). At four instances, the system handles 3,000 RPS with minimal errors (5%), and per-instance resource use remains stable:

Clients |  Clients |  Memory (per instance) |  RPS |  Error rate  
---|---|---|---|---  
500 |  75% |  200 MB |  3,000 |  5%  
  
Table 7.7: Quadrupled capacity

At scale (ten instances). Ten instances support 6,000 RPS with low errors (2%), demonstrating near-linear scalability as demand grows:

Clients |  Clients |  Memory (per instance) |  RPS |  Error rate  
---|---|---|---|---  
1000 |  65-85% |  220 MB |  6,000 |  2%  
  
Table 7.8: Horizontal scaling

Scaling the JWT token service from 2 to 4, then to 10 instances teaches subtleties about the behavior of distributed systems. The first doubling from 2 to 4 instances yields near-linear throughput gains. The service handles 500 clients at 3,000 RPS with great ease, and can sustain 75% CPU utilization per node. This proves that horizontal scaling works well under moderate load. Once we hit 10 instances, the performance gains start to level off. We reached our target of 6,000 RPS, but with a catch. The load is unevenly distributed, so it causes CPU variance between 65% and 85% across instances. The most likely culprit is the load balancer. Memory overhead per instance goes up, slightly, from 200MB to 220MB due to shared keystore synchronization, a reminder that even semi-stateless services carry cache costs. Throwing more hardware at the problem actually worked here. Going from 4 to 10 instances dropped our error rate from 5% down to 2%, because the overcapacity makes the whole system more resilient. No single instance failure disrupts the service, which validates fault tolerance in the approach.

Compared to vertical scaling, operational complexity increases exponentially. In fact, the system's monitoring and deployment orchestration become burdensome. These observations underscore horizontal scaling's main trade-off: it exchanges predictable vertical limits for elastic capacity, but demands some form of automation to manage a distribution.

## Architectures

The scaling debate inevitably leads to an architectural fork in the road. Take one path, monoliths rule. Take the other, microservices prevail. Monolithic consolidation aligns well with vertical scaling, for a system to fortify a single host to bear more work. Horizontal scaling divides and spreads out the work, which meshes with microservices distribution. But a dichotomy between models runs much deeper than making a good technical decision because monoliths favor cohesion, they trade elasticity for unity. Microservices, by contrast, revel in fragmentation. They turn down unity for degrees of freedom. This tension between the two models betrays a philosophical divide in systems architecture: to centralize for control or decentralize for evolution. As we have seen with the JWT token service, scaling horizontally adds operational complexity. Load balancers, shared state, fault tolerance—monoliths can entertain these things, but microservices systematize all the moving parts and dance to the music they play. So, the question is not which architecture is better, but which trade-offs align with a system's lifespan.

### Monolithic

A monolith is a single, unified edifice where all components are tightly coupled, sharing memory, CPU, and fate. Like a vertical scale-up, it consolidates power: one codebase, one deployment. For early-stage systems, or those with predictable loads, this level of simplicity is a virtue. Changes propagate atomically. Debugging requires no distributed tracing. Developers share a common code repository. Transactions need no sagas. JWT token service, as a monolith, would handle authentication as a module within a larger application, leveraging shared libraries and a single database but as traffic and the feature set grows, the monolith's strengths become its evolutionary limits. Scaling demands vertical upgrades (costly) or risky modular splits. Its centralized design defines choke points. A crashed authentication module tanks the entire system. Monoliths scale until they cannot. Then, they must be taken apart.

#### Lifespan

For small, co-located teams developing in lockstep, monoliths host an ideal environment because everybody works in the same codebase, and coordination becomes trivial. Deployments, while risky, are straightforward as a single artifact undergoes exhaustive testing and then gets deployed as an integrated unit. When proving business value takes precedence over scaling to unknown demand, this simplicity speeds up feature development, initially but like a garage that suits an early-stage startup, as the team expands and demand for service increases, the monolith's constraints may begin to restrict its growth. Tightly coupled components force teams into extensive coordination on minor changes, which leads to longer, more costly development cycles, for fewer features. Architectural traits that once enabled agility turn into a drag, a sign that the system's structure struggles to keep up with its own success.

The decline of a monolith begins innocently. A 20% increase in traffic causes a single point of failure, or a trivial bug in one module brings down the entire system. So the architecture is fundamentally flawed. Vertical scaling hits a ceiling, and tight coupling turns small bugs into full system breakdowns. Teams brace for merge day. They need to deploy fixes, but they get all tangled up in tightly coupled modules. Progress grinds to a halt. The once efficient system resists change. It is an inflection point. When adding more engineers yields diminishing returns, and outages correlate directly with traffic expansion, then it is time to consider something different. The question now shifts from if to how. To divide the monolith without affecting the live system it serves, teams must find a way forward.

Dismantling a monolith requires surgical precision. Teams can start by analyzing service traffic patterns. Extract the least used services with well-defined boundaries and pull them out into standalone microservices. By doing so, teams minimize the risk of an outage because each extracted service speeds up a shift to broader distributed operation, as the monolith core remains stable. With each iteration, teams replace monolithic services with REST API calls to microservices while maintaining backward compatibility throughout the migration. Load balancers can distribute traffic incrementally, and performance can be tracked, with the possibility of roll backs if needed. This gradual metamorphosis yields a hybrid architecture, part monolith, part microservices, until the last vestiges of centralized control give way to pure distribution. Take a look at the following figure:

Figure 7.2: Monolithic vs. microservices architecture

### Microservices

Microservices reject consolidation. They disassemble systems into discrete standalone services that operate autonomously and scale independently. As a microservice, the JWT token service would scale up by isolating authentication behind an API gateway, and by replicating instances horizontally while offloading state to a shared keystore. Redundancy gives microservices their resilience. A failed instance is replaced, not repaired, and routed to by a gateway or load balancer. But this kind of elasticity has a cost. Network hops replace method calls. Eventual consistency supplants ACID transactions. Observability demands intricate distributed tooling. Where monoliths tremble to grow, microservices nervously hum under so many moving parts. They scale with ease but demand rigorous discipline, in design and operation. The trade-off is clear. Microservices dismiss operational simplicity for infinite elasticity, which makes sense if the organization can sustain the overhead of distributed systems.

Microservices architecture promotes sustainable growth. By splitting up monolithic applications into smaller services, development teams can work on them independently, thereby accelerating feature development as the organization grows. As an architectural style, microservices prevent systems from becoming bogged down by inter-dependencies between components. New capabilities integrate via clean REST APIs rather than risky, invasive code changes.

## Conclusion

Performance and scale do not operate independently. They engender an exchange between tactics and strategy. An efficient API call means very little to real-world adoption if it cannot accommodate thousands of executions per second on a whim. To build RESTful systems that endure under extreme and sudden pressure, API designers must squeeze out the redundancy in their responses by passing them through successive optimization layers that filter, compress, and paginate. Faster responses buy time, and please clients in the short run, but long-term endurance requires an architectural strategy that wins the future. The performance test results in this chapter show that vertical scaling can bring quick tactical gains but yield diminishing returns. Our token service improved throughput with better hardware, to a point. The cost-per-performance ratio dropped, and failures crept back in. These results reveal a key insight about scalability: it costs to push against the vertical limits. To scale up exponentially, systems must distribute the work while maintaining a level of modularity in the architecture that allows components to rearrange and evolve independently.

## Points to remember

  * Performance focuses on making individual requests fast and efficient. Scalability ensures the system performs under increased load.
  * REST APIs commonly use Gzip, which combines LZ77 and Huffman coding to shrink JSON payloads, cutting down on latency and lowering bandwidth use.
  * While compression helps the network, it consumes CPU cycles on both client and server. It is most effective for large payloads but may be counterproductive for small ones.
  * Sending unnecessary fields in API responses wastes bandwidth and processing time. Even compressed, bloated responses slow down systems.
  * Allowing clients to specify which fields to retrieve minimizes data transfer and database load, especially beneficial at scale.
  * Instead of delivering massive datasets all at once, pagination turns search results into subsets that maintain system responsiveness and prevent backend strain.
  * Adding more resources (CPU/RAM) to a server can increase capacity temporarily, but hits diminishing returns.
  * Distributing workload across multiple servers or microservices improves scalability and fault tolerance but adds operational complexity.

# Preface

As a vast network of interconnected devices on the web sets the stage for the future of human cognition, RESTful APIs operate quietly behind the scenes. Representational State Transfer (REST) defines the architectural style of the web to produce distributed, loosely coupled systems that enable and extend this network. 

Human thought (cognitive activity) has increasingly outsourced itself to a network of devices and data on the web. Our cognition is no longer what happens in the confines of our own minds, but extends into an interactive series of processes between minds and digital systems. A network of interconnected devices (servers, databases, sensors, personal mobile devices, API endpoints) now holds a significant portion of human knowledge and compute. When we think about anything these days, we no longer rely entirely on our own thoughts. We tap into services (Google, APIs, social networks, chatbots). As such, they augment our cognition. We think through and with them. The future of cognition is about the ability to extend our intelligence: to access, filter, synthesize, combine, and build on the services available through the web. Understanding how to interface becomes a critical skill. If the web is the future brain of a collective human intelligence, or becomes any part of it, then APIs are the fundamental pathways and protocols that allow different parts of this brain to interact. They are the quietly operating contracts that make a decentralized system work. Without consistent, reliable APIs that follow RESTful principles, this global brain runs the risk of becoming a segmented jumble of incompatible parts.

Designing APIs is not just a routine task you do for a project at work. It extends our collective digital future. Poorly designed APIs create friction and break the cognitive flow of this network. As an integrated web of nodes becomes infrastructure through which humanity reasons and acts, to any degree, the design of the interfaces that allow this system to function (REST APIs) matters a great deal. This is why we think of API design as a critical discipline for shaping the foundation of our augmented, extended intelligence.

[Chapter 1: Breaking Points](ch01.xhtml) \- Starting with the basic concepts in CGI, when the web server was extended to generate dynamic web pages, this chapter traces technical shifts over a 30-year period. Reveals why REST dwarfed RMI, EJB, and SOAP, as a resource-oriented architecture, its wide appeal supported by the lessons learned. If you're interested in the history of technology, then this chapter tells a story. Otherwise, skip ahead and get straight into learning design patterns in the next chapters. 

[Chapter 2: Resource Identification](ch02.xhtml) - Examines the principle of addressability through a simple string of characters in a URI. The chapter moves beyond syntax to the deeper implications of naming, treating URIs as a stable, grammatical surface upon which to build client interactions. This chapter guides you in crafting predictable namespaces. And it discusses deliberate versioning patterns. These patterns transform the API surface from a vague collection of endpoints into a coherent map of your domain. They decouple client from server for each to evolve independently.

[Chapter 3: Resource Manipulation](ch03.xhtml) \- Defines the role played by the core HTTP methods used in REST APIs. By the end of the chapter, readers should have a thorough knowledge of a method's purpose and behavior, as well as how to implement each successfully in real-world API design. Considers key concepts, including idempotence, caching, and conflict resolution. What to do is equally important as what not to do, so the chapter offers cautionary tales for the reader.

[Chapter 4: Resource Representation](ch04.xhtml) – We turn our attention to the substance client and server exchange: the resource representation. The chapter analyzes the structure of API output, moving beyond mere data delivery. [Chapter 4](ch04.xhtml) is a linguistic framing. Through a dual lens of descriptive and evaluative terminology, we distinguish between patterns that give resources their expressive power from the crippling anti-patterns that lead to inflexible integrations. This chapter suggests a framework to shape raw data into a cohesive representation, moving it along a sliding scale of quality.

[Chapter 5: Resource State Management](ch05.xhtml) - Explores HATEOAS. Argues for hypermedia controls that help clients make state transitions. The chapter equips readers with an approach to decouple clients from the server. It makes a case for API design that adapts to change. The approach explored by the chapter produces self-guiding systems that transform static API responses into hypermedia-driven state transitions.

[Chapter 6: API Security](ch06.xhtml) \- Surveys the basic principles of API security. Readers become familiar with the basics of authentication, how security tokens work to identify and track users. Examines encryption to protect data in transit. Concludes with an honest examination of API security's fundamental flaws, helping readers evaluate interoperability design strengths as potential security weaknesses.

[Chapter 7: Performance and Scale](ch07.xhtml) - Scalability and performance define release criteria for production ready REST APIs. Dissects the Gzip compression algorithm. Explains why it reduces payload size. Considers the CPU trade-offs in compression. Compares horizontal to vertical scaling. Helps to decide if more servers or upgrading the resources on existing ones is the way to go. Readers learn about microservices and monolithic architecture, to measure the impact on long term growth.

[Chapter 8: Further Reading](ch08.xhtml) - A journey through RESTful design does not end with the ideas in this book. This chapter serves as an extension through a carefully curated selection of books for further reading. It suggests related work that influences the preceding chapters and expands into specialized areas of REST that readers may take an interest in pursuing.

For practitioners in search of specific, practical advice, or historians tracing technology, the sources cited here invite you to look beyond the page to a much broader body of work.
