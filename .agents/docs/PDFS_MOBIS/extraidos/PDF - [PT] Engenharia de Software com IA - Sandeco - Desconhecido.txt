CAPÍTULO 1
A Morte do Vibe Coding
“Sandeco, você ficou louco? Que título é esse?” Eu sei. Parece radical. Mas antes de
fechar o livro, me dá uma chance de explicar. Quando eu digo que o Vibe Coding precisa
morrer, não estou dizendo que programar com IA é ruim. Pelo contrário. Estou dizendo que a
forma como a maioria das pessoas programa com IA hoje é insustentável; e que, para a gente
de fato evoluir no desenvolvimento de software com inteligência artificial, essa fase precisa
ficar para trás. No lugar dela, precisa nascer algo com fundamento. Algo que este livro vai
chamar de AI Engineering.
Tudo começou em fevereiro de 2025, quando Andrej Karpathy, ex-diretor de IA da Tesla e
um dos fundadores da OpenAI, ou seja, alguém com currículo suficiente para ser levado a
sério, publicou uma reflexão sobre seu próprio modo de programar com LLMs. Ele descreveu
uma prática que chamou de vibe coding: você larga o controle, entra em ’flow’, aceita o que a
IA sugere sem questionar, e vai construindo por impulso e intuição. O nome pegou na hora. A
prática já existia há meses. Karpathy só colocou palavras no que milhares de desenvolvedores
já faziam; e alguns até postavam no LinkedIn com muito orgulho.
A ascensão do vibe coding não é um acidente nem uma conspiração. É o resultado
direto da democratização das ferramentas de IA generativa. Quando o Claude Code, o Cursor,
o GitHub Copilot e similares chegaram ao mercado acessível, qualquer pessoa com uma
ideia e uma conexão com a internet passou a conseguir gerar código. Designers codando.
Empreendedores construindo MVPs em um final de semana. Estudantes entregando projetos
que antes levariam um semestre. A produtividade explodiu. E com ela, também explodiu
a ilusão de que velocidade sozinha é suficiente; ilusão que, convenhamos, já era antiga no
mundo do software, mas agora ganhou turbina.
O problema do vibe coding não está na IA: a IA não tem culpa de nada, ela só faz o que
mandamos. O problema está na ausência de processo. Quando você constrói software por
acumulação de prompts, sem definir o que o sistema precisa fazer, sem pensar em quem vai
usá-lo, sem planejar como vai evoluir, você não está desenvolvendo software, está construindo
às cegas.

A história do software está lotada de sistemas que ’funcionavam’ muito bem até o momento
em que pararam de funcionar de formas memoráveis. O foguete Ariane 5 explodiu 37 segundos
após o lançamento em 1996 por causa de um overflow de inteiro, código copiado do Ariane 4
sem revisar se ainda fazia sentido no novo contexto. Trezentos e setenta milhões de dólares
viraram confete no céu da Guiana Francesa. O sistema HealthCare.gov, o maior portal de
saúde dos EUA, colapsou no lançamento em 2013 por uma combinação de planejamento
insuficiente de capacidade, erros de software, funcionalidades incompletas e práticas frágeis
de validação antes da entrada em produção. Nenhum desses projetos falhou por falta de
código. Falharam por falta de processo. O vibe coding em escala industrial é a versão
moderna do mesmo problema, com menos foguete e mais startup.
Este livro parte de uma premissa simples: engenharia de software e agente inteligente
não são opostos; são parceiros. A IA é uma ferramenta extraordinária quando usada dentro
de um processo estruturado. Ela acelera a escrita de código, automatiza tarefas repetitivas,
sugere soluções e ajuda a explorar alternativas com uma paciência que nenhum colega
humano teria. Mas quem define o que construir, para quem, com quais restrições e com qual
qualidade, isso continua sendo responsabilidade do engenheiro. A IA não vai perguntar se
você pensou nos casos extremos. Você precisa perguntar por ela.
Este capítulo estabelece o ponto de partida: entender o que é o vibe coding, por que ele
é insuficiente para software que precisa durar, e o que a engenharia de software oferece
como alternativa. Nos capítulos seguintes, você vai aprender a usar agentes inteligentes com
método, não como substitutos do pensamento, mas como amplificadores dele. A morte do
vibe coding não é o fim da IA no desenvolvimento. É o começo do uso inteligente dela. E
talvez o fim de algumas noites traumáticas de produção às três da manhã.

1.1
O QUE É VIBE CODING E POR QUE ELE É INSUFICIENTE
Imagine a cena: são duas da manhã, um desenvolvedor abre o Claude Code ou o Cursor,
digita ’cria um sistema de login com autenticação por e-mail e senha’ e, em questão de
segundos, recebe um bloco de código razoavelmente funcional. Ele cola, roda, funciona. Cola
mais um pouco, roda de novo, funciona de novo. Em 48 horas, tem um ’produto’. Em 72
horas, já postou no LinkedIn: ’Construí um SaaS completo em um fim de semana com IA!
#buildinpublic #solofounder #vibecoding’. Cento e oitenta curtidas. Dezessete comentários de
’incrível demais!’. Esse é o vibe coding em ação; e, por enquanto, tudo parece ótimo.
O termo foi cunhado por Andrej Karpathy em fevereiro de 2025 para descrever essa nova
forma de programar: você esquece que existe código, entra em ’flow’, aceita tudo que a
IA sugere e vai em frente. Sem ler, sem revisar, sem entender. É como surfar numa onda
que você não sabe de onde veio, mas enquanto está em cima, a sensação é fantástica. O
problema, obviamente, aparece quando a onda quebra e você percebe que estava surfando
em cima de código que valida senha com if password == password.
O vibe coding opera sobre uma premissa implícita muito conveniente: que código que
funciona é código correto. Essa premissa é falsa. Sempre foi. Mas quando você tem uma IA
gerando centenas de linhas por minuto, a ilusão de progresso se torna tão rápida que fica
difícil perceber quando você saiu da estrada. Um sistema que ’funciona’ em ambiente local,
com um único usuário, sem dados reais, sem carga e com o banco de dados rodando no seu
notebook não é um sistema pronto para produção; é um protótipo bem vestido que vai entrar
em colapso na primeira sexta-feira à noite com dez usuários simultâneos.
O vibe coding não é culpa de um desenvolvedor específico; é um problema sistêmico
que acontece quando ferramentas de alta produtividade chegam às mãos de pessoas sem
nenhuma estrutura de processo. É mais ou menos o que acontece quando você dá um bisturi
de última geração para alguém que nunca estudou medicina. O instrumento é excelente. O
resultado, nem tanto.
O maior sintoma do vibe coding é a ausência de requisitos. Quando você não sabe
o que o sistema precisa fazer de verdade, quem vai usar, em que condições, com quais
restrições, em qual escala, qualquer código que ’pareça’ fazer algo já satisfaz. A IA vai
preencher os buracos com suposições razoáveis, você vai aceitar porque está no flow, e seis
meses depois vai descobrir que essas suposições eram tudo menos razoáveis no contexto
real do negócio.

sucessivos ao ChatGPT, um sistema de gestão financeira para pequenas empresas. Em
três semanas, o produto estava ’pronto’. Seis meses depois, descobriram que o cálculo

A outra face do problema é a manutenção. Código gerado sem estrutura é código que

esboço com a planta baixa. Usar martelo para apertar parafuso não é culpa do martelo.
O que falta no vibe coding, portanto, é exatamente o que a engenharia de software
oferece: processo. Um conjunto de práticas, técnicas e ferramentas que transformam a
criação de software numa atividade controlada, previsível e sustentável. Não porque processo
é burocracia corporativa inventada para torturar desenvolvedores; é porque software que
importa vive por anos, é mantido por equipes inteiras e carrega responsabilidades reais sobre
dados reais de pessoas reais. E para isso, vibe, infelizmente, não é suficiente.

O QUE É ENGENHARIA DE SOFTWARE
Em 1968, a OTAN reuniu um grupo de cientistas em Garmisch, na Alemanha, para
discutir um problema que já tinha nome: a ’crise do software’. Sistemas atrasando, estourando
orçamento, entregando menos do que prometiam ou falhando em produção de formas criativas.
O mais divertido é que isso foi em 1968, sem IA, sem internet, sem GitHub. Os desenvolvedores
da época conseguiam criar caos de altíssimo nível mesmo sem nenhuma ferramenta moderna
para ajudar. Talento puro. Daquela conferência nasceu o termo engenharia de software ;
e com ele, a ideia radical de que construir software é uma atividade de engenharia, não de
inspiração divina.

O corpo de conhecimento da área está organizado no SWEBOK, Software Engineering
Body of Knowledge, um documento mantido pelo IEEE que mapeia as competências essenciais
do engenheiro de software. Em versões recentes, ele organiza a disciplina em múltiplas
áreas de conhecimento, como requisitos, design, construção, testes, manutenção, gestão
de configuração, gestão de projetos, processo, modelos e métodos, qualidade e práticas
profissionais. Repare que ’escrever código’ aparece na área de construção, que é apenas
uma delas. O código é importante, mas representa uma fatia menor do que a maioria dos
iniciantes imagina quando pensa em fazer software de verdade.
Os modelos de processo são as formas concretas de organizar essas atividades no
tempo. O modelo cascata organiza tudo em fases sequenciais: requisitos, design, construção,
testes, entrega: simples, previsível e com a flexibilidade de um bloco de concreto. O modelo
espiral adiciona iterações e gestão de risco. O RAD prioriza protótipos rápidos para validação.
Os métodos ágeis, Scrum, XP, Kanban, organizam o trabalho em ciclos curtos com entrega
contínua de valor e reuniões diárias que muita gente ama odiar. Cada modelo tem seu contexto
ideal. Nenhum é bala de prata. Mas todos compartilham uma característica essencial: existe
um processo, e ele é seguido de forma consciente.
A manutenção merece atenção especial porque é sistematicamente subestimada por
quem está começando.
Estudos clássicos de manutenção de software frequentemente
estimam que a maior parte do custo do ciclo de vida ocorre após a entrega inicial, em
correções, adaptações e evoluções. Traduzindo: a maior parte do trabalho de um engenheiro
de software não é construir sistemas novos, é manter sistemas existentes que outra pessoa,
ou você mesmo, em outro estado de espírito, construiu. Quando esse sistema foi construído
sem método, a manutenção se torna cara, lenta, perigosa e emocionalmente desgastante.
Quando foi construído com processo, ela é apenas trabalho normal.
O que a engenharia de software oferece, no fundo, é previsibilidade; e previsibilidade
tem valor imenso em contextos onde falhar custa caro. A capacidade de dizer com alguma
confiança: ’esse sistema faz o que precisa fazer, pode ser evoluído por uma equipe, resiste
a falhas dentro de limites conhecidos e pode ser mantido por anos sem ninguém entrar em
desespero’. Não é glamouroso. Não rende post viral no LinkedIn às duas da manhã. Mas é
exatamente o que diferencia software que dura de software que vira uma história de terror
contada em retrospectivas de sprint.

1.3
A PROPOSTA DO LIVRO: IA + PROCESSO = SOFTWARE DE VERDADE
Se você chegou até aqui, o diagnóstico está claro. O vibe coding é sedutor, rápido e
insuficiente. A engenharia de software é robusta, comprovada e, sejamos honestos, às vezes
percebida como burocrática demais para o ritmo que o mercado exige hoje. É como se alguém
te oferecesse dois carros: um Fórmula 1 sem freio e um tanque de guerra sem acelerador.
A pergunta que este livro responde é: e se não precisarmos escolher? E se for possível ter
velocidade e controle ao mesmo tempo?
A resposta é sim; e a chave está nos agente inteligentes. Um agente inteligente no
contexto de desenvolvimento de software não é apenas um autocomplete glorificado nem um
estagiário digital que nunca dorme. É um componente autônomo que percebe o ambiente,
toma decisões e executa ações para atingir objetivos. Quando esse agente opera dentro de
um processo de engenharia bem definido, ele amplifica cada etapa: levanta requisitos com
mais rapidez, gera código com padrões consistentes, escreve testes, sugere refatoração e
documenta decisões. Ele não substitui o engenheiro; ele multiplica a capacidade do engenheiro.
A diferença é importante.
Ferramentas recentes de desenvolvimento agentic, como Claude Code e Google Antigra-
vity, apontam para uma nova geração de ambientes de desenvolvimento com IA integrada de
verdade. Não são apenas editores com sugestão de código; são ambientes onde agentes
podem executar tarefas complexas, operar em worktrees isoladas, gerenciar branches, rodar
testes e interagir com o sistema de arquivos de forma estruturada. Frameworks como o BMAD
e o Speckit organizam o trabalho desses agentes em fluxos de desenvolvimento que respeitam
as etapas clássicas da engenharia, dos requisitos ao deploy, sem abrir mão da agilidade
que as ferramentas modernas permitem.
O conceito de Agent Harness é central nessa proposta. Um harness é um conjunto de
configurações, skills e hooks que define como os agentes operam num projeto específico.
É a camada que transforma um agente genérico num colaborador especializado no seu
domínio, nas suas convenções e no seu processo. Pense como a diferença entre contratar um
freelancer que nunca ouviu falar da sua empresa e ter um desenvolvedor sênior que conhece
cada detalhe do sistema. Ao longo deste livro, você vai construir harnesses para diferentes
contextos e aprender a orquestrar múltiplos agentes trabalhando em paralelo sem perder o
controle, nem o juízo.
Este livro está organizado em torno das grandes etapas da engenharia de software, com
agentes inteligentes como protagonistas de cada uma delas. Começamos aqui, no capítulo
1, com o diagnóstico e a proposta. Nos capítulos seguintes, percorremos o ciclo completo:
modelagem de requisitos com agentes, design com design patterns aplicados em Python,

O que é software afinal?
Imagine que você pediu uma pizza. Não chegou pizza, chegou farinha, molho de tomate,
queijo e pepperoni numa caixa separada. Tecnicamente, são todos os ingredientes certos.
Mas não é pizza. Agora pense no código-fonte de um sistema: são linhas de instrução,
variáveis, funções, classes. Tecnicamente, são os ingredientes certos. Mas também não é
software, ainda. Software é o conjunto completo: o código que roda, a lógica que resolve um
problema real, a interface que alguém consegue usar, os dados que persistem, as falhas que
são tratadas, a documentação que permite a qualquer pessoa entender e manter o sistema e
o processo de evolução que garante que tudo isso ainda funcione daqui a três anos. Só a
massa não faz pizza. Só o código não faz software.
Essa distinção parece simples, mas tem consequências enormes.

capítulo anterior, o vibe coding falha não por falta de código, mas porque produz código
sem o restante do conjunto. Sem requisitos definidos, sem estrutura de manutenção, sem
testes que garantam que o comportamento esperado permanece o esperado. É a diferença
entre ter ingredientes e ter um prato. E da mesma forma que um chef profissional não mistura

2.4
COMO SE CONSTRÓI UM SOFTWARE
Existe uma pergunta que todo iniciante faz e que todo desenvolvedor experiente responde
de forma diferente: ’mas afinal, por onde eu começo?’ A pergunta parece ingênua. Não é.
Ela expõe um problema real: a construção de software não tem um ponto de partida único,
universal e incontestável. O que existe são atividades que precisam acontecer, em alguma
ordem, com algum nível de formalidade, para que o resultado final seja algo que funciona
e pode ser mantido. Entender essas atividades, o que cada uma produz e como elas se
encadeiam é o que separa quem constrói software de quem empilha código.
A primeira atividade é o levantamento de requisitos. Antes de escrever uma linha sequer,
é preciso entender o problema que o software vai resolver. Não o problema que o cliente
descreveu na primeira reunião, que raramente é o problema real. O problema real. Isso exige
conversas, perguntas incômodas, prototipação rápida de ideias e, sobretudo, a disposição de
ouvir ’não é bem isso’ e voltar ao início sem drama. Um requisito mal levantado não gera um
bug: gera um sistema inteiro construído na direção errada. E sistemas na direção errada não
se corrigem com refatoração. Se corrigem com descarte e reconstrução, que é a versão cara
e humilhante do mesmo problema.
Com os requisitos minimamente estabelecidos, entra a fase de projeto, ou design de
software. Aqui se decide a arquitetura: como os componentes vão se organizar, quais
responsabilidades cada módulo vai ter, como os dados vão fluir, quais integrações externas
existem e onde estão os pontos de maior risco técnico. Design não é criar um diagrama bonito
para mostrar ao cliente. É tomar decisões estruturais que vão determinar o custo de qualquer
mudança futura. Uma decisão de arquitetura ruim tomada nessa fase não custa nada agora,
mas pode custar semanas de trabalho quando o sistema precisar crescer ou mudar de direção.
A implementação é a fase que todo mundo imagina quando pensa em ’desenvolver
software’: escrever código. E é aqui que o papel dos agentes inteligentes fica mais visível na
prática atual. Um agente pode gerar estruturas de classes, implementar endpoints, sugerir
padrões, escrever testes unitários e até revisar código em busca de inconsistências. Mas
perceba: tudo isso só funciona bem quando as fases anteriores foram feitas. O agente
implementa o que foi projetado. Se o projeto estava errado, o agente vai implementar o erro
com eficiência impecável. Velocidade na direção errada não é produtividade, é aceleração do
problema.

2.5
OS MODELOS DE PROCESSO
Se a seção anterior descreveu as atividades da construção de software, esta responde à
pergunta seguinte: em que ordem essas atividades acontecem, quem decide essa ordem e
como o time sabe se está progredindo ou apenas se movimentando? A resposta é o modelo
de processo. Um modelo de processo é, essencialmente, uma receita estruturada para
organizar as atividades de desenvolvimento ao longo do tempo. E, como toda receita, existem
variações, cada uma com seus contextos ideais, suas vantagens e seus pontos cegos.
Cascata
O modelo mais antigo e mais conhecido é o cascata, formalizado por Winston Royce
em 1970. A ideia é simples e sedutora: as fases acontecem em sequência linear, uma após
a outra, como uma cascata de água. Levantamento de requisitos, depois design, depois
implementação, depois testes, depois deploy, depois manutenção. Cada fase só começa
quando a anterior termina e produz sua documentação formal. O modelo tem uma lógica

Espiral
O modelo espiral, proposto por Barry Boehm em 1988, nasceu exatamente dessa limitação.
Em vez de fases lineares, o desenvolvimento acontece em ciclos progressivos, cada um com
quatro quadrantes: identificação de objetivos, análise de riscos, desenvolvimento e avaliação.
A cada volta da espiral, o sistema ganha mais profundidade e os riscos são reduzidos de
forma explícita. É um modelo sofisticado, especialmente útil para projetos grandes e de alto
risco, onde ignorar incertezas técnicas pode custar fortunas. A desvantagem é a complexidade
de gestão: executar a espiral corretamente exige experiência, disciplina e uma organização
disposta a investir tempo em análise de riscos antes de escrever código. Em equipes pequenas
ou com prazo apertado, essa sofisticação pode ser um peso maior do que o problema que
resolve.
Prototipação
O modelo de prototipação parte de uma premissa diferente: se o cliente não sabe ao
certo o que quer, construa algo rápido para ele ver e reagir. Um protótipo não é o produto final;
é uma ferramenta de elicitação de requisitos. O time constrói uma versão simplificada do
sistema, o cliente usa, critica, pede mudanças, e o protótipo é refinado até que os requisitos
fiquem claros o suficiente para o desenvolvimento real começar. O risco clássico desse modelo
é o protótipo que nunca vira produto: o cliente aprova o protótipo, o time o coloca em produção
sem o processo de construção formal, e o que era para ser um rascunho vira a fundação do
sistema. Isso tem nome na área: vibe coding de 1985.

projetos onde o escopo é relativamente claro e a velocidade é crítica, o RAD entrega resultados

cada componente reutilizado ja passou por ciclos de teste em projetos anteriores. Na pratica,

em todo projeto que aparecer, independentemente de adequação, como quem tem um martelo

critério.

2.6
A REVOLUÇÃO ÁGIL
Em fevereiro de 2001, dezessete desenvolvedores se reuniram em Snowbird, Utah, num
resort de ski que provavelmente não esperava entrar para a história da tecnologia. O resultado
daquele encontro foi o Manifesto Ágil, um documento de quatro valores e doze princípios que
redefiniu como boa parte do mundo pensa sobre desenvolvimento de software. A proposta
central era direta: valorizar indivíduos e interações mais do que processos e ferramentas,
software funcionando mais do que documentação abrangente, colaboração com o cliente
mais do que negociação de contratos e responder a mudanças mais do que seguir um plano.
Era uma reação explícita ao peso burocrático dos modelos clássicos, especialmente ao
cascata aplicado de forma rígida em contextos onde a realidade mudava mais rápido do que
o cronograma conseguia absorver.
É fundamental entender o que o Manifesto Ágil disse e o que ele não disse. Ele não
disse que processo é ruim. Disse que processos excessivamente pesados, que impedem
a adaptação, são ruins. Não disse que documentação é dispensável. Disse que software
funcionando é mais valioso do que pilhas de documentação que ninguém lê. Não disse que
planejamento é perda de tempo. Disse que a capacidade de responder a mudanças é mais
valiosa do que seguir um plano cegamente. A distinção importa porque o mercado leu o
manifesto de forma seletiva durante anos, ficando com a parte conveniente, ’menos processo’,
e ignorando a parte difícil, ’mais disciplina técnica, mais colaboração, mais responsabilidade
com o produto’.
Scrum
O Scrum foi o framework que mais popularizou a agilidade na prática. A estrutura é
simples: o trabalho é dividido em ciclos curtos chamados sprint, tipicamente de duas a quatro
semanas. Cada sprint começa com um planejamento, onde o time seleciona itens do backlog
de produto que serão desenvolvidos. Ao longo do sprint, reuniões diárias breves mantêm
o time sincronizado. Ao final, uma revisão mostra o que foi construído e uma retrospectiva
identifica o que pode melhorar. O ciclo se repete. O cliente vê resultado funcionando a cada
poucas semanas, não após meses de desenvolvimento silencioso. E o time aprende com cada
iteração, ajustando o rumo antes que os erros se acumulem.

2.7
OS MITOS QUE NOS PERSEGUEM
Todo campo do conhecimento humano tem seus mitos. A medicina teve a teoria dos
humores. A física teve o éter luminífero. A engenharia de software tem uma coleção própria,
igualmente resistente às evidências e igualmente capaz de sobreviver durante décadas porque
é conveniente para quem a repete. A diferença é que os mitos do software não custam
apenas reputação acadêmica: custam projetos cancelados, sistemas em produção com
comportamento imprevisível e equipes esgotadas repetindo os mesmos erros com vocabulário
diferente.
O primeiro e mais antigo é a Lei de Brooks aplicada ao contrário. Fred Brooks publicou
em 1975, em ’O Mítico Homem-Mês’, a observação de que adicionar pessoas a um projeto
atrasado o atrasa ainda mais. A lógica é simples: novos membros precisam de tempo
para entender o contexto, os membros existentes precisam de tempo para ensiná-los, e a
comunicação fica exponencialmente mais complexa conforme o time cresce. Cinquenta anos
depois, a resposta mais comum quando um projeto atrasa ainda é ’vamos contratar mais dois
desenvolvedores’. Brooks está rodando no túmulo. O mito sobrevive porque admitir que o
atraso veio de requisitos mal levantados ou de processo ausente é mais difícil do que abrir
uma vaga no LinkedIn.
O segundo mito é que o cliente sabe o que quer. Não sabe. E isso não é crítica ao
cliente: é uma característica do problema. O cliente sabe qual dor quer resolver. Raramente
sabe como um sistema de software deve resolver essa dor. A elicitação de requisitos existe
exatamente para navegar essa distância, usando técnicas como entrevistas estruturadas,
prototipação, análise de fluxo de trabalho e validação incremental. Quando o time assume que
o cliente sabe o que quer e pula essa etapa, o resultado é um sistema que resolve o problema
que o cliente descreveu na primeira reunião, que raramente é o problema real que ele enfrenta
todo dia. A figura do balanço de árvore, onde cada envolvido imagina algo diferente e o
resultado não serve para ninguém, não é piada antiga do setor. É o diagrama de falha mais
comum em projetos sem levantamento de requisitos adequado.

3.1
POR QUE GIT É OBRIGATÓRIO QUANDO SE USA IA
Pense no que acontece quando você pede a um agente inteligente que implemente uma
nova funcionalidade num sistema existente. O agente lê o código, entende o contexto dentro
da janela de contexto disponível, propõe uma solução e escreve. Na maioria das vezes, o
resultado é impressionante. Mas ’na maioria das vezes’ deixa espaço para o resto. O agente
pode ter interpretado o requisito de forma ligeiramente diferente do que você imaginava. Pode
ter feito uma escolha de implementação que funciona agora, mas cria débito técnico. Pode
ter modificado uma função que era usada em cinco lugares diferentes sem perceber que três
deles tinham comportamentos especiais. Sem Git, descobrir qualquer uma dessas coisas
significa gastar tempo comparando o que você lembra que era com o que agora está na tela.
O Git resolve isso de uma forma elegante e simples: antes de pedir ao agente que faça
qualquer mudança significativa, você faz um commit do estado atual. Esse commit é um
instantâneo completo de tudo que estava funcionando naquele momento. Se a mudança
do agente trouxer problema, você reverte para esse instantâneo com um único comando e
o código volta exatamente ao estado anterior, sem perda, sem reconstrução manual, sem
depender da memória de ninguém. O agente pode tentar de novo com uma instrução diferente,
ou você pode seguir outro caminho. O ponto de retorno está garantido.
Existe um segundo motivo pelo qual o Git é obrigatório no contexto de IA: o agente
inteligente não tem acesso ao histórico do projeto, só ao estado atual dos arquivos. Isso
significa que toda a inteligência sobre ’por que esse código está assim’ está nos commits,
nas mensagens que os acompanham e no histórico de mudanças. Um time que usa Git
corretamente constrói, ao longo do tempo, uma documentação viva do raciocínio por trás das
decisões. Quando um agente precisar entender por que uma função tem aquele comporta-

git init

git clone https://github.com/usuario/repositorio.git

git add nome_do_arquivo.py
Adicionar todos os arquivos modificados de uma vez:
git add .

3.4
BRANCHES: TRABALHANDO COM IA SEM MEDO DE ERRAR
Uma branch é, em termos simples, uma linha de desenvolvimento paralela e isolada. Ela
parte de um ponto do histórico do projeto e segue em frente sem afetar as outras linhas até
que você decida integrá-la ou descartá-la. Para quem usa agente inteligentes, as branches
são talvez o conceito mais importante do Git, porque transformam qualquer experimento num
risco controlado.
O fluxo básico é criar uma branch antes de iniciar qualquer mudança significativa, especi-
almente quando você vai instruir o agente a fazer algo de maior impacto:
git checkout -b feat/nova-funcionalidade
Esse comando cria a branch e já muda para ela. Tudo que for feito a partir daqui, seja
por você ou pelo agente, ficará isolado nessa branch. Para ver quais branches existem e em
qual você está:
git branch
Para trocar para uma branch existente:
git checkout main
