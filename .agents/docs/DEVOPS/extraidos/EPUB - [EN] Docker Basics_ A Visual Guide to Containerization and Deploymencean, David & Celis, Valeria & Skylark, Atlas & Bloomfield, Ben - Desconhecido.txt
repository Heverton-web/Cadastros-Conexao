## Capítulo 1: Introdução ao Docker

O Docker transformou o desenvolvimento de software ao possibilitar aplicações leves, portáteis e escaláveis. Este capítulo apresenta os fundamentos do Docker, contêineres e virtualização. Você aprenderá como o Docker simplifica a implantação de aplicações, as principais diferenças entre contêineres e máquinas virtuais e por que a conteinerização é essencial para o desenvolvimento moderno.


## Capítulo 2: Primeiros passos com o Docker

Este capítulo orienta você na configuração do Docker em sua máquina e apresenta os comandos essenciais da CLI do Docker. Você aprenderá como instalar o Docker Desktop e o Docker Engine, interagir com contêineres e entender o fluxo de trabalho do Docker. Exercícios práticos ajudarão você a ganhar confiança no gerenciamento de contêineres e imagens a partir da linha de comando.


## Capítulo 3: Imagens e contêineres Docker

As imagens Docker funcionam como modelos para contêineres. Este capítulo explora como as imagens são criadas, armazenadas e gerenciadas. Você aprenderá a diferença entre imagens Docker e contêineres, como trabalhar com registros do Docker Hub e entenderá o ciclo de vida do contêiner — da criação à remoção. Estudos de caso práticos ajudarão você a aplicar esses conceitos em cenários reais.


## Capítulo 4: Redes e armazenamento Docker

Redes e armazenamento são componentes essenciais para aplicações conteinerizadas escaláveis e confiáveis. Este capítulo explica os modelos de rede do Docker, incluindo redes bridge, host e overlay. Você também explorará os volumes do Docker para armazenamento persistente, garantindo que os dados críticos permaneçam intactos quando os contêineres forem reiniciados. Ao final deste capítulo, você saberá como criar microsserviços em rede e armazenar dados de aplicações de forma eficiente.


## Capítulo 5: Criando e Compartilhando Imagens Docker

Imagens Docker personalizadas oferecem flexibilidade e eficiência. Este capítulo ensina como escrever Dockerfiles, otimizar builds e criar imagens reutilizáveis. Você aprenderá as melhores práticas para reduzir o tamanho da imagem, gerenciar dependências e enviar imagens para o Docker Hub para facilitar a distribuição. Isso é crucial para a colaboração e a implantação em ambientes de desenvolvimento e produção.


## Capítulo 6: Prática de Docker Compose e Implantação

Neste capítulo final, você explorará o Docker Compose, uma ferramenta para gerenciar aplicações com múltiplos contêineres com facilidade. Você aprenderá a escrever e executar arquivos Docker Compose, gerenciar dependências de serviços e implantar aplicações em ambientes de produção. O capítulo também aborda estratégias avançadas de implantação, incluindo o uso de proxies reversos, variáveis de ambiente e serviços em nuvem como a AWS. Por fim, você obterá informações sobre Kubernetes e Docker Swarm, preparando-o para uma orquestração de contêineres mais avançada.

Introdução aos Ambientes Computacionais

Um ambiente computacional fornece a base para a execução de programas. Para iniciantes, é essencial pensar nesse ambiente como um ecossistema cuidadosamente equilibrado. Mesmo uma pequena alteração, como a atualização de um único componente de software, pode perturbar toda a configuração e fazer com que os programas falhem.

O que é um ambiente computacional?

Um ambiente computacional refere-se ao conjunto de configurações de hardware e software que trabalham em conjunto para executar aplicações. Aqui está uma análise mais detalhada de seus principais componentes:

1\. Sistemas Operacionais (SO): Esta é a camada fundamental que

Gerencia recursos de hardware e software. Exemplos incluem macOS, Windows e Linux.

2\. Linguagens de Programação e suas Versões:

Os aplicativos dependem de versões específicas de linguagens como Python, Java ou JavaScript. Incompatibilidades de versão podem causar problemas de compatibilidade.

3\. Frameworks e Bibliotecas: São ferramentas pré-escritas que

Facilitam a programação. Por exemplo, Django é um framework para construir aplicações web e NumPy é uma biblioteca para computação numérica em Python.

Imagine que você está desenvolvendo um aplicativo web em um computador com macOS Sequoia, usando Python 3.12 e Django 5.1. Juntos, esses componentes formam seu ambiente computacional. Se alguma parte estiver faltando ou for incompatível, seu programa não funcionará corretamente. Dependências e seu papel em

Desenvolvimento de Software

Dependências são bibliotecas, ferramentas ou frameworks externos que seu programa precisa para funcionar. Pense nelas como blocos de construção que evitam que você reinvente a roda. Por exemplo:

• Um programa em Python pode usar a biblioteca requests para buscar dados.

Dados da web.

As dependências, por sua vez, muitas vezes dependem de outras dependências. Por exemplo, o React pode exigir versões específicas do Node.js ou de outras bibliotecas. Isso cria uma cadeia de componentes interconectados, todos os quais devem estar alinhados para que o programa seja executado com sucesso.

• Eficiência: Os desenvolvedores podem se concentrar em sua lógica exclusiva.

• Escalabilidade: Projetos de grande porte se beneficiam da estabilidade e

No entanto, o gerenciamento de dependências exige cautela. Se você instalar, sem saber, uma versão incompatível de uma dependência, isso pode levar a erros difíceis de depurar.

O impacto das atualizações de versão

As atualizações de versão são uma faca de dois gumes. Por um lado, trazem novos recursos e correções de bugs. Por outro, podem quebrar programas que dependem de versões mais antigas. Por exemplo:

• Um programa desenvolvido usando Django 3.0 pode falhar se o sistema

foi atualizado para o Django 4.0.

• Funcionalidades que funcionavam na versão anterior podem ser

obsoleto ou substituído na nova versão.

Iniciantes frequentemente se deparam com esse problema ao revisitar projetos antigos ou trabalhar em projetos iniciados por outra pessoa. Isso ressalta a importância do controle de versões em um projeto.

Conflitos de dependência comuns

Conflitos de dependência ocorrem quando diferentes componentes do seu ambiente computacional são incompatíveis. Esses problemas podem surgir em vários estágios de desenvolvimento e implantação.

Conflito no mesmo computador

Muitos desenvolvedores, especialmente os iniciantes, gerenciam vários projetos em uma única máquina. Aqui está um cenário típico:

•Você cria um projeto usando o Django 4.0.

• Posteriormente, você inicia um novo projeto que requer o Django 5.0. • Seu sistema não permite a utilização de ambas as versões do Django.

coexistir.

Essa limitação pode causar bastante frustração. Por exemplo, atualizar o Django para o novo projeto pode quebrar o projeto antigo, forçando você a abandoná-lo ou a gastar tempo extra corrigindo problemas de compatibilidade. Ferramentas como ambientes virtuais ou contêineres podem resolver esse problema isolando as dependências de cada projeto. (Essas soluções serão abordadas no próximo tópico.)

Analogia com o mundo real: Imagine uma cozinha onde você está tentando preparar duas receitas diferentes ao mesmo tempo. Uma receita pede arroz branco, enquanto a outra precisa de arroz integral. Se você só tiver um tipo de arroz disponível, não conseguirá preparar as duas receitas corretamente.

Conflito entre computadores de desenvolvedores

Ao trabalhar em equipe, os conflitos de dependência tornam-se mais complexos. O computador de cada desenvolvedor possui uma configuração única, influenciada por suas configurações pessoais e pelo software instalado. Por exemplo:

•O desenvolvedor A tem o Python 3.12 instalado, enquanto o desenvolvedor B

•O desenvolvedor A utiliza a versão 1.26 do NumPy, mas o desenvolvedor B

Se o desenvolvedor A escrever um código que funcione em sua máquina, mas que dependa da versão 1.26 do NumPy, a configuração do desenvolvedor B poderá gerar um erro devido à incompatibilidade de versões. Essa situação é comum em projetos colaborativos e destaca a necessidade de ambientes padronizados. Conflito nas fases de teste e produção.

Imagine que você criou um programa que funciona perfeitamente em sua máquina local. Você o implanta em um servidor para testes ou produção, apenas para descobrir que ele não funciona. Esse é um problema comum causado por diferenças entre os ambientes local e do servidor.

Por exemplo:

3.12, que estão ausentes no Python 3.8.

A depuração desses problemas consome muito tempo, especialmente em aplicações grandes com muitas dependências. Os desenvolvedores precisam fazer engenharia reversa do problema, verificando a compatibilidade em todas as camadas do ambiente.

Compreender ambientes computacionais e gerenciar conflitos de dependência são habilidades essenciais para desenvolvedores. Seja lidando com incompatibilidades de versões, problemas de colaboração em equipe ou erros de implantação, reconhecer esses desafios é o primeiro passo para resolvê-los.

O que é virtualização?

A virtualização cria versões virtuais de recursos computacionais, como sistemas operacionais, servidores ou dispositivos de armazenamento. Ela permite que vários aplicativos sejam executados independentemente, mesmo que compartilhem o mesmo hardware físico. Por exemplo, você pode executar um sistema operacional Windows em um computador Mac usando um software de virtualização.

Introdução ao Docker

O Docker é uma plataforma projetada para simplificar a conteinerização — um método de executar aplicativos em ambientes isolados. Ao criar contêineres leves e autossuficientes, o Docker garante que o software se comporte da mesma maneira, independentemente de onde seja executado, desde o laptop de um desenvolvedor até um servidor de produção.

A Ascensão da Conteinerização

A conteinerização tornou-se essencial para o desenvolvimento de software moderno. Ao contrário dos métodos de implantação tradicionais que exigem a configuração repetida de ambientes, os contêineres fornecem ambientes pré-configurados, portáteis e consistentes em diferentes sistemas. Isso elimina o problema do "funciona na minha máquina" e melhora a eficiência da implantação.

A ascensão do Docker está intimamente ligada à crescente adoção de arquiteturas de microsserviços, ambientes de nuvem e à necessidade de fluxos de trabalho DevOps simplificados. Com a conteinerização, os aplicativos podem ser desenvolvidos, testados e implantados de forma previsível, tornando-os mais escaláveis e gerenciáveis.

Como funciona o Docker Engine: Cliente-Servidor

Modelo

O Docker Engine opera por meio de uma arquitetura cliente-servidor, onde os usuários emitem comandos a partir de um cliente e o servidor (gerenciado pelo daemon dockerd) os processa e executa. Esse design garante o gerenciamento contínuo de contêineres em diferentes ambientes, seja em uma máquina local ou em infraestrutura remota.

Quando um usuário executa um comando Docker, o cliente envia solicitações ao servidor, que as executa usando o daemon. Esse modelo permite o gerenciamento remoto, possibilitando que os usuários controlem contêineres em máquinas diferentes, automatizem implantações e integrem o Docker com outras ferramentas.

1\. Dockerd Daemon – O Coração de

Docker Engine

O daemon dockerd é o processo em segundo plano que alimenta o Docker Engine, responsável por executar tarefas relacionadas a contêineres. Ele escuta continuamente solicitações de API e orquestra operações essenciais, incluindo:

•Gerenciar o ciclo de vida do contêiner (criação, execução,

parando e excluindo contêineres).

•Gerenciamento de imagens, redes e armazenamento, garantindo

Comunicação fluida entre contêineres.

• Monitoramento dos recursos do sistema para otimizar o contêiner

desempenho e prevenção do uso excessivo.

Sem o daemon dockerd, o Docker Engine não pode funcionar, pois ele serve como o componente central que executa todas as operações do Docker.

2\. Interfaces de API – Automatizando e

Estendendo o Docker

O Docker Engine expõe APIs RESTful, permitindo que desenvolvedores e administradores de sistemas integrem o Docker com aplicativos externos e automatizem fluxos de trabalho. Essas APIs fornecem:

• Gerenciamento remoto de contêineres sem acesso direto à linha de comando. • Automação de tarefas repetitivas, como escalonamento de contêineres.

dinamicamente com base na demanda.

•Integração perfeita com ferramentas DevOps e pipelines CI/CD,

e plataformas de orquestração de contêineres como o Kubernetes.

Ao aproveitar as interfaces de API do Docker, as empresas podem criar ambientes conteinerizados automatizados e escaláveis, reduzindo o esforço manual e melhorando a eficiência da implantação.

Instalando o Docker Desktop no macOS

Instalar o Docker Desktop no macOS é um processo simples. Siga estes passos para uma configuração perfeita: 1. Baixe o

Instalador: Acesse a página de download do Docker Desktop e baixe o arquivo .dmg específico para macOS.

2\. Execute o instalador: Abra o arquivo .dmg baixado e arraste e solte o ícone do Docker na pasta Aplicativos. Clique duas vezes no ícone do Docker na pasta Aplicativos para iniciá-lo.

3\. Conceder permissões: Durante o processo de instalação, você poderá ser solicitado a conceder permissões de acesso à rede e aos arquivos. Essas permissões são essenciais para o funcionamento correto do Docker. Aprove todas as solicitações conforme as instruções.

4\. Selecione as configurações recomendadas: Quando solicitado, escolha a opção “Usar configurações recomendadas” para garantir a configuração ideal do seu sistema.

5\. Inicie sessão no Docker Hub: Se ainda não tiver uma conta no Docker Hub, crie uma durante este passo. Utilize as suas credenciais para iniciar sessão quando o Docker Desktop o solicitar.

6\. Abra o Docker Desktop: Após fazer login, abra o Docker Desktop na pasta Aplicativos ou no Dock do macOS. O ícone do Docker na barra de menus indicará que ele está em execução.

7\. Verificar a instalação do Docker: Após a instalação do Docker Desktop, a CLI do Docker também estará configurada. Abra o terminal e execute o seguinte comando para confirmar se o Docker está instalado e funcionando corretamente.

docker --versão

Ao executar o comando no terminal, será exibida a versão do Docker instalada, juntamente com outros detalhes. Abaixo, segue um exemplo da saída.

Instalando o Docker Desktop no Windows

Siga estes passos para instalar o Docker Desktop no Windows: 1. Ative o WSL 2:

O Docker depende do WSL (Subsistema Windows para Linux) para executar contêineres Linux no Windows. O WSL é um ambiente Linux leve que se integra perfeitamente ao Windows. O WSL 2, a versão mais recente, inclui um kernel Linux real, permitindo que o Docker execute contêineres de forma eficiente sem a necessidade de uma máquina virtual tradicional. Para habilitar o WSL 2, abra o PowerShell como administrador e execute o comando.

wsl --instalar

Reinicie o computador se solicitado.

Fluxo de trabalho

O Docker é uma ferramenta poderosa que ajuda os desenvolvedores a empacotar, compartilhar e executar aplicativos de forma consistente. Para iniciantes, é uma ótima maneira de aprender como os aplicativos são implantados e executados sem se preocupar com a configuração do ambiente. Ao entender o ciclo de vida do fluxo de trabalho do Docker — imagens, contêineres, rede, armazenamento e gerenciamento de múltiplos contêineres — você pode simplificar seu processo de desenvolvimento. Vamos analisar isso passo a passo de uma maneira fácil de entender.

Introdução ao fluxo de trabalho do Docker

O Docker ajuda os desenvolvedores ao fornecer uma maneira consistente de criar e executar aplicativos, independentemente de onde eles sejam implantados. Aqui estão as principais ideias para entender:

• Imagens Docker e Registro de Imagens: Pense em imagens como

O projeto para sua aplicação. Esses projetos são armazenados em uma biblioteca chamada registro, como o Docker Hub, onde você pode encontrar imagens pré-construídas.

• Contêineres Docker: Contêineres são como casas de verdade construídas

Utilizando o modelo (imagem). São leves e portáteis, executando sua aplicação em um ambiente isolado.

•Rede e armazenamento Docker: os contêineres podem

Os contêineres se comunicam entre si por meio de redes. O armazenamento é usado para salvar dados, evitando que desapareçam quando os contêineres forem encerrados.

• Docker Compose: Se sua aplicação for composta por múltiplas instâncias

O Docker Compose facilita o gerenciamento de todas essas partes (por exemplo, um aplicativo web e um banco de dados).

Ciclo de vida da imagem Docker

As imagens Docker são o ponto de partida do fluxo de trabalho Docker. Elas são criadas, usadas e, às vezes, removidas ou compartilhadas. Abordaremos esses detalhes com mais profundidade nos Capítulos 3 e 5.

Usando imagens Docker a partir de imagens

Registro (Docker Hub)

O Docker Hub é uma plataforma onde desenvolvedores podem encontrar imagens prontas para uso. Por exemplo, se você precisa de um ambiente Python, pode baixar uma imagem Python do Docker Hub em vez de criá-la do zero. Isso economiza tempo e garante confiabilidade, já que essas imagens são mantidas pela comunidade ou por fontes confiáveis.

Verificação e remoção de imagens

As imagens do Docker podem se acumular com o tempo, especialmente se você estiver experimentando ou atualizando com frequência. Imagens não utilizadas ocupam espaço em disco, por isso é importante removê-las regularmente. A limpeza garante que seu sistema permaneça rápido e organizado. O Docker facilita a visualização das imagens que você possui e a exclusão daquelas que não são mais necessárias.

Criando imagens Docker personalizadas

Às vezes, você precisará criar sua própria imagem personalizada para sua aplicação. Isso envolve escrever um Dockerfile, que é como uma receita que diz ao Docker como construir a imagem. Você pode especificar itens como o sistema operacional, dependências de software e arquivos de configuração.

Compartilhando imagens do Docker

Depois de criar uma imagem, você pode compartilhá-la com outras pessoas enviando-a para o Docker Hub ou para um registro privado. Isso é ótimo para colaboração em equipe, pois outros podem baixar a imagem e executá-la sem precisar recriá-la.

Ciclo de vida do contêiner Docker

Um contêiner é a versão em execução da sua imagem. É onde sua aplicação reside e funciona. Compreender o ciclo de vida do contêiner — criação, operação e exclusão — é fundamental para usar o Docker de forma eficaz. Os detalhes sobre esse ciclo de vida serão explicados no Capítulo 3.


Criando e executando contêineres

Criar um contêiner é como iniciar uma nova instância do seu aplicativo. Você pode personalizar a forma como ele é executado, especificando detalhes como a porta a ser usada ou quaisquer configurações especiais que seu aplicativo precise. Os contêineres são rápidos e eficientes, o que os torna ideais para testar e implantar aplicativos.

Pausar, parar e reiniciar

Contêineres

Às vezes, é necessário pausar ou parar um contêiner sem excluí-lo. Por exemplo, você pode pausar um contêiner durante a depuração ou pará-lo temporariamente durante a manutenção. Reiniciar um contêiner parado o traz de volta à ativa rapidamente, sem a necessidade de recriá-lo. Dimensionamento e gerenciamento de contêineres

Se sua aplicação precisar lidar com mais tráfego, você pode escalá-la executando vários contêineres simultaneamente. Ferramentas como Docker Compose ou Kubernetes facilitam o gerenciamento de um grande número de contêineres, garantindo que eles funcionem em conjunto de forma integrada.

Excluindo contêineres

Após concluir o uso de um contêiner, é recomendável excluí-lo para liberar recursos. A limpeza regular ajuda a manter o sistema organizado e garante seu bom desempenho.

Rede e armazenamento Docker

Redes e armazenamento são partes essenciais da execução de contêineres. Eles permitem que os contêineres se comuniquem e garantem que dados importantes não sejam perdidos. Abordaremos esses tópicos com mais detalhes no Capítulo 4.

Gerenciando a rede Docker

As redes Docker permitem que os contêineres se comuniquem entre si ou se conectem ao mundo externo. Por exemplo, se o contêiner do seu aplicativo web precisar enviar dados para um contêiner de banco de dados, eles podem se comunicar por meio de uma rede Docker. Você pode usar redes padrão, como a rede bridge, ou criar redes personalizadas para ter mais controle.

Gerenciando o armazenamento e os dados do Docker

Persistência

Por padrão, os dados dentro de um contêiner desaparecem quando o contêiner é interrompido. Para evitar a perda de dados importantes, você pode usar volumes ou montagens vinculadas. Essas opções de armazenamento permitem que os dados persistam mesmo se o contêiner for excluído ou reiniciado, tornando-as cruciais para aplicativos que dependem de bancos de dados ou arquivos enviados pelo usuário.

Gerenciamento de múltiplos contêineres com

Docker-Compose

Os aplicativos geralmente são compostos por várias partes. Por exemplo, um aplicativo de comércio eletrônico pode precisar de um servidor web, um banco de dados e um serviço de cache. Gerenciá-los manualmente pode ser complexo, mas o Docker Compose simplifica o processo. Abordaremos isso com mais detalhes no Capítulo 6.

Importância da gestão de múltiplos

Contêineres

Quando as aplicações são divididas em vários contêineres, cada contêiner se concentra em uma tarefa específica. Por exemplo, um contêiner pode lidar com o tráfego da web enquanto outro armazena dados em um banco de dados. Gerenciar esses contêineres individualmente pode ser tedioso, mas usar uma ferramenta como o Docker Compose mantém tudo organizado.

Utilizando Docker-Compose

O Docker Compose permite definir todas as partes da sua aplicação em um único arquivo de configuração. Esse arquivo inclui detalhes sobre cada contêiner, como a imagem que ele usa, as portas que expõe e como se conecta a outros contêineres. Com um único comando, você pode iniciar ou parar toda a pilha da aplicação, economizando tempo e reduzindo a complexidade.

Ao aprender o ciclo de vida do fluxo de trabalho do Docker, os iniciantes podem ganhar confiança na criação e execução eficiente de aplicações. As ferramentas e os conceitos do Docker podem parecer complexos à primeira vista, mas dividi-los em etapas gerenciáveis os torna acessíveis. Dominar imagens, contêineres, redes e configurações com múltiplos contêineres o preparará para o sucesso no desenvolvimento de software moderno.

Entre imagens Docker e

Embora exploremos esses dois conceitos em detalhes neste capítulo, aqui está uma breve visão geral de suas principais diferenças. Iniciantes costumam achar as imagens e os contêineres do Docker confusos porque, embora ambos sejam essenciais, desempenham funções diferentes. Uma imagem do Docker é um modelo estático, enquanto um contêiner do Docker é uma instância em execução desse modelo.

Imagens Docker: O Plano

Uma imagem Docker é um modelo somente leitura que inclui o sistema operacional, as dependências, as configurações e o código do aplicativo necessários para criar um contêiner. As imagens são imutáveis — se você precisar fazer alterações, deverá criar uma nova versão. Elas são portáteis e controladas por versão (por exemplo, marcadas como v1.0 ou latest), garantindo consistência entre os ambientes de desenvolvimento, teste e produção.

Contêineres Docker: A Execução

Exemplo

Um contêiner Docker é uma instância executável de uma imagem onde as aplicações são executadas. Cada contêiner opera em um ambiente isolado com seu próprio sistema de arquivos, processos e configurações de rede, o que evita conflitos entre aplicações. Os contêineres são modificáveis — podem gerar arquivos temporários ou armazenar dados de tempo de execução. No entanto, sem armazenamento persistente, quaisquer dados criados são perdidos quando o contêiner é interrompido ou removido. Eles são leves e efêmeros, projetados para serem criados e descartados rapidamente, conforme necessário.

O que são imagens Docker?

Uma imagem Docker é como um projeto ou modelo para a criação de contêineres Docker. É um arquivo somente leitura que contém todas as instruções, bibliotecas, dependências e arquivos necessários para executar um aplicativo dentro de um contêiner.

Imagine como uma receita de bolo. A receita (imagem Docker) inclui todos os ingredientes e instruções necessários para assar o bolo (container Docker). Você pode usar a mesma receita para assar vários bolos, e cada bolo (container) será independente dos outros.

Etiquetas

Uma tag no Docker é um rótulo ou identificador usado para diferenciar e versionar imagens do Docker. Ela faz parte do nome completo da imagem e ajuda a especificar qual versão da imagem você deseja baixar, construir ou executar.

Estrutura de um nome de imagem Docker

A estrutura geral do nome de uma imagem Docker é:

<registro>/<repositório>:<tag>

• Registro: O local onde a imagem está armazenada (por exemplo,

Docker Hub, Amazon ECR).

• Repositório: O nome da imagem (por exemplo, nginx ou my-

aplicativo).

• Etiqueta: Um rótulo opcional para identificar uma versão específica do

imagem (ex.: mais recente, 1.0, v2.3.4).

Se nenhuma tag for especificada, o Docker usará a tag mais recente por padrão.

Por que usar tags?

As etiquetas são usadas para:

1\. Imagens de versão: Etiquetas diferentes representam versões diferentes.

da mesma imagem (por exemplo, nginx:1.21, nginx:1.22).

2\. Garantir a consistência: Ao especificar uma tag, você garante que

estão utilizando a versão exata de uma imagem em desenvolvimento ou produção.

3\. Facilitar atualizações: Etiquetas como "mais recente" podem ser usadas para

Sempre utilize a versão mais recente de uma imagem como referência, enquanto as tags versionadas (v1.0, v2.0) proporcionam estabilidade.

O comando `docker rmi` permite excluir manualmente uma imagem específica do seu sistema local. Isso é útil quando você deseja remover uma única imagem sem afetar as outras.

1\. Criado: Um contêiner é criado a partir de uma imagem Docker usando

2\. Em execução: O contêiner inicia a execução de seus processos quando

Iniciado com `docker start` ou `docker run`. Nesse estado, o contêiner consome recursos do sistema, como CPU e memória.

Pausado: Opcionalmente, os processos de um contêiner podem ser

3\. Congelado temporariamente usando o comando `docker pause`. Isso interrompe a CPU.

Construindo Microsserviços Escaláveis

Com Docker: Noções básicas de rede e armazenamento

A ascensão dos microsserviços e por que eles

Matéria

Aplicações modernas exigem velocidade, escalabilidade e flexibilidade. Essa necessidade levou ao surgimento dos microsserviços, onde as aplicações são construídas como uma coleção de pequenos serviços independentes. Ao contrário das aplicações monolíticas tradicionais, onde todos os componentes são fortemente interligados, os microsserviços permitem que as equipes desenvolvam, implementem e atualizem diferentes partes de uma aplicação separadamente.

Grandes empresas como Netflix, Amazon e Spotify adotaram microsserviços para melhorar a agilidade e a confiabilidade. Com o crescimento contínuo da computação em nuvem e da conteinerização, mais empresas estão migrando para microsserviços para lidar com cargas de trabalho dinâmicas e de grande escala de forma eficiente.

Por que o Docker é essencial para

Microsserviços

O Docker torna os microsserviços viáveis ao empacotar cada serviço em um contêiner leve e isolado. Essa abordagem garante que os aplicativos sejam executados da mesma maneira em qualquer ambiente, reduzindo problemas de compatibilidade e simplificando a implantação. Os contêineres iniciam rapidamente e usam menos recursos do que as máquinas virtuais tradicionais, o que ajuda a escalar os microsserviços de forma eficiente.

Para gerenciar múltiplos contêineres, ferramentas de orquestração como Kubernetes e Docker Swarm automatizam a implantação, o escalonamento e o gerenciamento de redes. Essas ferramentas garantem uma operação tranquila mesmo com o crescimento das aplicações. Mantendo os microsserviços conectados e

Armazenamento de dados

Para que os microsserviços funcionem corretamente, eles precisam se comunicar entre si e acessar dados de forma confiável. A rede Docker permite que os contêineres se conectem — seja na mesma máquina ou em vários servidores — para que serviços como aplicativos web e bancos de dados possam trocar informações sem problemas.

Como os contêineres são temporários por natureza, o armazenamento persistente é essencial para salvar dados críticos, como informações de usuários ou logs. O que é uma rede Docker?

O que é uma rede?

Uma rede é um sistema de comunicação entre computadores ou dispositivos. No Docker, uma rede é a forma como os contêineres podem se conectar e se comunicar entre si ou com o mundo externo. Pense nisso como as ruas entre os prédios de uma cidade. Se você quiser que diferentes contêineres (prédios) se comuniquem, eles precisam estar na mesma rede (sistema viário).

Rede Docker

Por padrão, os contêineres Docker são conectados à rede bridge (uma ponte Ethernet virtual) se nenhuma outra rede for especificada. Para casos de uso simples, especialmente quando você está executando um único contêiner ou contêineres que não precisam se comunicar entre si, não é necessário criar uma rede Docker personalizada.

O que é uma porta Docker?

Uma porta é como uma entrada ou ponto de acesso a um computador ou serviço. Todo serviço em execução dentro de um contêiner possui uma porta na qual fica à escuta de dados recebidos (como um site, um banco de dados etc.). As portas permitem que sistemas externos (como um navegador web ou outro contêiner) se conectem a um serviço específico dentro do contêiner.

Porta Docker

Ao executar contêineres, o mapeamento de portas (usando a flag -p) é necessário quando você deseja expor um serviço em execução dentro do contêiner para a máquina host (ou sistemas externos).

Quando você pode não precisar definir uma porta: Comunicação interna: Se o contêiner estiver executando um

Para serviços que não precisam ser acessados de fora do contêiner (por exemplo, uma tarefa em segundo plano ou um serviço usado por outros contêineres), você não precisa expor nenhuma porta. O Docker manterá o serviço interno ao contêiner.

docker run -d meu-serviço-em-segundo-plano

O serviço dentro do contêiner será executado, mas não estará acessível através do host.

Quando você deve definir uma porta: Acesso externo: Se você estiver executando um servidor web (como Nginx ou Apache) ou qualquer serviço que precise ser acessado de fora do contêiner (por exemplo, por meio de um navegador web).

Neste caso, a porta 80 do contêiner (a porta do servidor web) é exposta à porta 8080 na máquina host.

Rede vs. Porta

No Docker, redes e portas têm funções diferentes, mas podem parecer semelhantes para iniciantes. Compreender as diferenças entre elas é fundamental para configurar contêineres de forma eficaz.

1\. Criando uma rede Docker

O comando `docker network create` permite criar uma rede personalizada. Por padrão, os contêineres usam a rede bridge, mas criar sua própria rede oferece maior flexibilidade e isolamento.

criar rede docker

Sintaxe do comando:

docker network create <nome_da_rede>

Exemplo:

docker network criar minha_rede

Isso cria uma nova rede chamada my_network. Em seguida, você pode conectar contêineres a essa rede para garantir que eles se comuniquem dentro do mesmo ambiente isolado.

2\. Listando as redes Docker

Para visualizar todas as redes Docker disponíveis, use o comando `docker network ls`. Isso é útil para ver todas as redes existentes no seu host.

docker rede ls

Sintaxe do comando:

docker rede ls

Isso exibirá uma lista de todas as redes, incluindo as redes padrão como bridge, host e none, juntamente com quaisquer redes personalizadas que você tenha criado.

3\. Inspecionando uma rede Docker

O comando `docker network inspect` fornece informações detalhadas sobre uma rede específica, incluindo os contêineres conectados a ela, endereços IP e configuração.

docker network inspect <nome_da_rede>

Exemplo:

docker network inspect minha_rede

Este comando exibirá uma saída JSON detalhada sobre a minha_rede, incluindo informações sobre contêineres conectados, alocações de endereços IP e configurações de rede.

4\. Conectando um contêiner a uma rede

O comando `docker network connect` permite conectar um contêiner em execução a uma rede existente, possibilitando a comunicação com outros contêineres nessa rede.

Sintaxe do comando docker network connect:

docker network connect <nome_da_rede>

<nome_ou_id_do_container>

Exemplo:

docker network conectar minha_rede meu_container

Isso conecta o meu_container à minha_rede, permitindo a comunicação entre contêineres nessa rede.

5\. Desconectando um contêiner de uma rede Se você precisar remover um contêiner de uma rede, use o comando `docker network disconnect`.

Sintaxe do comando docker network disconnect:

desconectar rede docker <nome_da_rede>

<nome_ou_id_do_container>

Exemplo:

docker network desconectar minha_rede meu_container

Isso desconecta o contêiner `my_container` da rede `my_network`, isolando-o efetivamente de outros contêineres nessa rede.

6\. Eliminação de redes não utilizadas

Com o tempo, redes Docker não utilizadas podem se acumular. Para limpar redes não utilizadas, você pode usar o comando `docker network prune`, que remove todas as redes que não estão sendo usadas por nenhum contêiner no momento.

poda de rede docker

Por que a persistência de dados no Docker?

Assuntos

Os contêineres Docker são projetados como ambientes de computação descartáveis, o que significa que são temporários e podem ser interrompidos, reiniciados ou removidos sem afetar o sistema subjacente. Essa característica torna os contêineres leves e eficientes para aplicações sem estado, mas cria desafios para aplicações que exigem dados persistentes. Sem a configuração adequada, quaisquer dados dentro do contêiner são perdidos assim que o contêiner é interrompido ou removido.

O que é persistência de dados no Docker?

A persistência de dados no Docker refere-se à capacidade de reter dados mesmo quando os contêineres são interrompidos, reiniciados ou removidos. O Docker oferece mecanismos como volumes e montagens de diretórios que permitem que os dados persistam independentemente do ciclo de vida do contêiner. Essas opções de armazenamento permitem que os dados sobrevivam a reinicializações, migrações e até mesmo à remoção do contêiner, tornando o Docker adequado para aplicações com estado.

Ao utilizar essas técnicas, o Docker garante que dados importantes, como registros de banco de dados, arquivos de configuração ou logs, possam ser preservados com segurança ao longo dos ciclos de vida do contêiner.

Comparando o armazenamento de dados do Docker

Abordagens

Ao gerenciar dados em contêineres Docker, é essencial escolher a abordagem correta para persistência. O Docker oferece diversos métodos para garantir que os dados permaneçam intactos ao longo do ciclo de vida do contêiner. Cada abordagem tem suas vantagens, e algumas são mais adequadas para casos de uso específicos. Nesta seção, compararemos volumes e montagens de diretórios (bind mounts), que fornecem armazenamento persistente, e abordaremos as montagens tmpfs, adequadas para dados temporários.

1\. Volumes do Docker

Os volumes são o método mais comum e recomendado para gerenciar dados persistentes no Docker. Eles são totalmente gerenciados pelo Docker, armazenados fora do sistema de arquivos do contêiner e podem ser compartilhados entre contêineres. Os volumes são portáteis e fáceis de fazer backup, o que os torna ideais para ambientes de produção.

Prós:

•Gerenciados pelo Docker, o que facilita a criação e o gerenciamento,

e faça backup.

• Armazenados fora do sistema de arquivos do contêiner, garantindo que os dados sejam

conservado mesmo se o recipiente for removido.

• Pode ser compartilhado entre vários contêineres, simplificando os dados.

Gerenciamento em aplicações com múltiplos contêineres.

• Compatível com as ferramentas Docker para backup e restauração.

2\. Montagens de ligação

Os bind mounts permitem montar um diretório ou arquivo específico do sistema de arquivos do host no contêiner. Isso oferece mais flexibilidade, especialmente em ambientes de desenvolvimento onde você pode querer editar arquivos localmente e ver as alterações imediatamente no contêiner.

Prós:

• Oferece flexibilidade para montar qualquer diretório ou arquivo específico de

o host.

•Ideal para desenvolvimento local, pois as alterações no hospedeiro são

refletido imediatamente no recipiente.

• Permite locais de armazenamento personalizados, fora dos padrões do Docker.

diretórios.

Contras:

•Pode levar a potenciais problemas de permissão de arquivos ou

3\. Montagens Tmpfs

O sistema de arquivos tmpfs armazena dados na memória do sistema em vez de em disco, oferecendo velocidades de acesso rápidas. No entanto, os dados não são persistentes, o que significa que são perdidos quando o contêiner é interrompido ou reiniciado. O tmpfs é normalmente usado para dados temporários que não precisam ser salvos a longo prazo.

• Montagens Tmpfs: Perfeitas para armazenamento temporário de arquivos rápidos,

Dados efêmeros que não precisam persistir após a interrupção do contêiner, como dados de cache ou de sessão.

Cada uma dessas abordagens tem sua utilidade, dependendo dos requisitos da sua aplicação. Ao entender as diferenças entre elas, você poderá selecionar o método mais adequado para seus contêineres Docker.


Gerenciando volumes do Docker com

CLI

O gerenciamento de volumes do Docker é essencial para lidar com dados persistentes em ambientes conteinerizados. O Docker oferece diversos comandos para criar, inspecionar, montar e gerenciar volumes, facilitando a garantia de que dados importantes sejam preservados mesmo após reinicializações e remoções de contêineres. Nesta seção, abordaremos alguns comandos importantes do Docker que ajudam a gerenciar volumes de forma eficaz.

1\. Criando um volume do Docker

O comando `docker volume create` permite criar um novo volume. Volumes são o método preferido para persistir dados no Docker, pois são gerenciados pelo Docker e armazenados fora do sistema de arquivos do contêiner.

criar volume docker

Sintaxe do comando:

docker volume create <nome_do_volume>

Exemplo:

docker volume criar meu_volume

Isso cria um novo volume chamado my_volume. Você pode então montar esse volume em contêineres para garantir que os dados persistam mesmo se o contêiner for removido ou reiniciado.

2\. Listando os volumes do Docker

Para visualizar todos os volumes Docker disponíveis, use o comando `docker volume ls`. Isso é útil para ver todos os volumes disponíveis no seu host.

docker volume ls

Sintaxe do comando:

docker volume ls

Exemplo de saída:

NOME DO VOLUME DO DRIVER

meu_volume local

outro_volume local

Isso exibirá uma lista de todos os volumes, incluindo os volumes padrão e personalizados que você criou. A saída inclui o driver do volume (por exemplo, local) e o nome do volume. Neste exemplo, existem dois volumes: my_volume e another_volume.

3\. Inspecionando um volume do Docker

O comando `docker volume inspect` fornece informações detalhadas sobre um volume específico, como seu ponto de montagem e configuração. Isso é útil para entender onde os dados estão armazenados no host e quaisquer outros metadados associados ao volume.

inspecionar volume docker

4\. Montagem de um volume em um recipiente

Para persistir os dados entre reinicializações do contêiner, você precisa montar um volume no contêiner. O comando `docker run` permite montar um volume usando a opção `-v` ou `--mount`.

docker run -v (ou --mount)

Sintaxe do comando:

docker run -v <nome_do_volume>:<caminho_do_container>

<nome_da_imagem>

Exemplo:

docker run -v meu_volume:appdata minha_imagem

Este comando monta o volume my_volume no diretório appdata dentro do contêiner, garantindo que os dados gravados neste diretório sejam persistidos no volume.

5\. Removendo um Volume do Docker Se um volume não for mais necessário, você pode removê-lo usando o comando `docker volume rm`. Isso é útil para limpar volumes não utilizados e liberar recursos do sistema.

docker volume rm

Sintaxe do comando:

docker volume rm <nome_do_volume>

Exemplo:

docker volume rm meu_volume

Isso remove o volume my_volume do seu sistema. Observe que você só pode remover volumes que não estejam sendo usados por nenhum contêiner no momento.

6\. Eliminação de volumes não utilizados

Com o tempo, volumes não utilizados podem se acumular. Para limpar volumes não utilizados, você pode usar o comando `docker volume prune`, que remove todos os volumes que não estão sendo usados por nenhum contêiner no momento.

poda de volume docker

poda de volume docker

Isso solicitará que você confirme a remoção de volumes não utilizados, ajudando a liberar espaço em disco.

Configurando Bind Mounts e

Montagens Tmpfs

Nesta seção, abordaremos como configurar dois tipos de montagens de armazenamento no Docker: montagens Bind, para vincular diretórios do host a contêineres, e montagens Tmpfs, para armazenar dados na memória.

Configurando um Bind Mount

Uma montagem de diretório (bind mount) vincula um diretório ou arquivo específico da máquina host a um contêiner. Isso é útil quando você deseja trabalhar com arquivos do host e refletir as alterações imediatamente dentro do contêiner.

Sintaxe do comando:

docker run -v <diretório_do_host>:

<container_directory> <image_name>

Exemplo:

docker run -v pathon/host:appdata minha_imagem

Isso monta o diretório do host pathon/host em appdata dentro do contêiner. As alterações feitas no diretório do host serão refletidas no contêiner.

Observações importantes:

• Certifique-se de que o contêiner tenha direitos de acesso ao diretório do host. • Os bind mounts estão vinculados a caminhos do host, portanto, movê-los ou removê-los pode causar problemas.

O diretório do host irá interromper a montagem.

Configurando uma montagem Tmpfs

Uma montagem tmpfs armazena dados na memória (RAM) em vez de armazenamento persistente. É ideal para dados temporários que não precisam ser retidos após a parada do contêiner.

Sintaxe do comando:

docker run --tmpfs <diretório_do_container>:<tamanho>

<nome_da_imagem>

Exemplo:

docker run --tmpfs /tmp:rw,size=200m minha_imagem

Isso cria uma montagem tmpfs de 200 MB em /tmp dentro do contêiner. Os dados serão perdidos quando o contêiner for interrompido.

Observações importantes:

• As montagens Tmpfs não são persistentes; os dados são perdidos quando o sistema é conectado.

Antes de criar suas próprias imagens Docker, é importante entender os diferentes tipos de imagens disponíveis. No Docker, uma imagem é um ambiente empacotado que contém tudo o que um aplicativo precisa para ser executado, incluindo seu sistema operacional, bibliotecas, dependências e configurações.

O que são imagens oficiais do Docker?

As imagens oficiais do Docker são imagens pré-construídas e disponíveis publicamente, mantidas pelo Docker e sua comunidade de código aberto. Essas imagens seguem as melhores práticas, recebem atualizações de segurança e são verificadas quanto à confiabilidade. Exemplos comuns incluem nginx, mysql e node, que fornecem ambientes prontos para uso para executar servidores web, bancos de dados e aplicativos.

Ao longo dos capítulos anteriores, trabalhamos com imagens oficiais para obter e executar contêineres sem esforço. Como essas imagens são revisadas e mantidas por colaboradores confiáveis, elas servem como uma base sólida para muitos projetos.

O que são imagens personalizadas?

Embora as imagens oficiais sejam um ótimo ponto de partida, elas podem não incluir tudo o que sua aplicação exige. Imagens personalizadas são imagens Docker que você cria para atender a necessidades específicas. Elas permitem:

• Adicionar o software e as dependências necessárias • Configurar as definições da aplicação

• Otimize o desempenho para o seu caso de uso.

• Garantir a consistência em diferentes ambientes

Até agora, usamos apenas imagens pré-existentes do Docker Hub, mas não abordamos como criar e personalizar imagens do zero. Este capítulo apresentará esse processo e mostrará como compartilhar suas próprias imagens por meio de um repositório.

Quando usar cada um?

• Use as imagens oficiais do Docker quando precisar de uma solução estável,

Uma base segura e bem testada para sua aplicação. Essas imagens são ideais para casos de uso padrão onde a confiabilidade é fundamental. Já abordamos como obter e usar imagens oficiais em capítulos anteriores.

• Use imagens personalizadas quando seu projeto exigir imagens adicionais.

Software, configurações específicas ou otimizações que vão além do que uma imagem oficial oferece. Criar imagens personalizadas permite que você tenha controle total sobre o ambiente do aplicativo.

Ao entender quando usar cada tipo, você pode escolher a melhor abordagem para o seu projeto. Seja trabalhando com uma imagem oficial pré-construída ou criando uma personalizada do zero, cada decisão o aproxima do domínio da conteinerização com Docker.

Introdução aos Dockerfiles

O que é um Dockerfile?

Um Dockerfile é um simples arquivo de texto que contém uma lista de instruções para criar uma imagem Docker. Os Dockerfiles permitem que os desenvolvedores otimizem o processo de conteinerização, fornecendo uma maneira clara e automatizada de definir ambientes de aplicação. Eles garantem consistência entre os ambientes de desenvolvimento, teste e produção. Pense nele como uma "receita" que diz ao Docker, passo a passo, como preparar um ambiente onde sua aplicação possa ser executada.

Benefícios de usar Dockerfiles

Os Dockerfiles simplificam o gerenciamento de contêineres ao automatizar a criação de imagens Docker. Eles proporcionam consistência, garantindo o mesmo ambiente em desenvolvimento, teste e produção. A automação reduz a configuração manual, economizando tempo e prevenindo erros. O controle de versão permite o rastreamento de alterações de ambiente dentro dos repositórios. Os Dockerfiles são leves, pois são simples arquivos de texto que definem como construir uma imagem sem adicionar sobrecarga. Eles também ajudam a criar imagens otimizadas e leves, minimizando componentes desnecessários. Com a portabilidade, as equipes podem compartilhar e implantar aplicativos sem problemas.

Sintaxe fundamental do Dockerfile

A sintaxe de um Dockerfile é simples, mas entender as nuances é fundamental para criar imagens otimizadas. Aqui estão as principais regras e práticas:

1\. As instruções devem estar em letras maiúsculas.

Todas as instruções do Dockerfile, como FROM, RUN e CMD, são escritas em maiúsculas. Isso ajuda a distinguir palavras-chave reservadas e garante consistência.

Exemplo:

DE ubuntu:20.04

EXECUTE apt-get update

2\. Os argumentos seguem as instruções.

Cada instrução é seguida por argumentos específicos da sua função:

• FROM define a imagem base. • COPY especifica os caminhos de origem e destino.

Exemplo:

COPIE ./app usrsrc/app 3. A ordem importa

A ordem das instruções afeta tanto a funcionalidade quanto o desempenho. Comece com FROM, adicione instruções de configuração como WORKDIR e COPY e termine com CMD ou ENTRYPOINT.

Exemplo:

DO nó:14

DIRETÓRIO DE TRABALHO /aplicativo

CÓPIA . .

EXECUTE npm install

CMD ["node", "app.js"]

4\. Cada instrução cria uma camada.

O Docker constrói imagens camada por camada. Para minimizar o tamanho da imagem, agrupe comandos relacionados em uma única instrução.

Exemplo:

# Ineficiente

EXECUTE apt-get update

EXECUTE apt-get install -y curl

# Eficiente

EXECUTE apt-get update && apt-get install -y curl

5\. Os comentários melhoram a legibilidade.

Use # para adicionar comentários para fins de documentação e clareza.

Exemplo:

# Defina a imagem base

A PARTIR de python:3.9

# Instalar dependências

EXECUTE pip install flask

6\. Os caminhos diferenciam maiúsculas de minúsculas

Certifique-se de que os nomes de diretórios e arquivos correspondam exatamente à capitalização (maiúsculas e minúsculas).

Exemplo:

COPY MyApp /myapp # Funciona apenas se a pasta for

chamado 'MyApp'

7\. Use .dockerignore

Exclua arquivos desnecessários do contexto de compilação com um arquivo .dockerignore.

Exemplo:

node_modules

*.registro

.env

Orquestração de múltiplos contêineres

A orquestração de múltiplos contêineres é o processo de gerenciar diversos contêineres interconectados como um único sistema coeso. No desenvolvimento de aplicações modernas, diferentes componentes — como bancos de dados, servidores web, camadas de cache e APIs — frequentemente são executados em contêineres separados. A orquestração garante que esses contêineres funcionem juntos sem problemas, automatizando tarefas como iniciá-los na ordem correta, gerenciar dependências e se recuperar de falhas.

Essa coordenação é crucial porque simplifica a implantação de aplicações complexas, reduz a probabilidade de erros e facilita a escalabilidade. À medida que as organizações adotam cada vez mais arquiteturas de microsserviços e soluções nativas da nuvem, a demanda por orquestração eficiente de contêineres cresce rapidamente. As tendências do setor mostram que as empresas estão migrando para sistemas distribuídos para aumentar a agilidade e melhorar a utilização de recursos. Nesse contexto, a orquestração de múltiplos contêineres não é apenas uma conveniência, mas uma necessidade para gerenciar a complexidade das aplicações modernas.

O Docker Compose se destaca como uma solução fundamental para orquestrar múltiplos contêineres em menor escala. Ele permite que desenvolvedores definam e executem aplicações com múltiplos contêineres usando um único arquivo de configuração YAML, simplificando o processo de desenvolvimento e implantação. Para cenários mais avançados, onde as aplicações precisam operar em múltiplas máquinas ou requerem recursos aprimorados, ferramentas como Docker Swarm e Kubernetes oferecem soluções robustas com funcionalidades como escalonamento automático, balanceamento de carga e tolerância a falhas.

Ao automatizar a gestão de cada contêiner, a orquestração não só melhora a eficiência, como também fornece uma base sólida para a construção de aplicações confiáveis e escaláveis no cenário dinâmico da indústria atual.

O Docker Compose foi introduzido pela Docker, Inc. para lidar com a crescente complexidade de aplicações com múltiplos contêineres. Ele permite que desenvolvedores definam serviços, redes e volumes em um formato YAML simples, centralizando a configuração da aplicação. Com o Compose, executar e escalar múltiplos contêineres torna-se mais fácil, possibilitando implantações mais rápidas e um desenvolvimento mais eficiente.

Sintaxe fundamental do YAML

Segue abaixo uma análise da sintaxe YAML e suas regras essenciais:

1\. Pares chave-valor

O YAML representa dados usando pares chave-valor, onde a chave é seguida por dois pontos (:) e o valor. As chaves são sempre strings e os valores podem ser de qualquer tipo de dado válido, incluindo strings, números ou outras estruturas aninhadas.

chave: valor

Exemplo:

nome: Docker Compose

versão: "3.8"

• O nome é a chave e o Docker Compose é o valor. • Os dois pontos (:) separam a chave do valor.

2\. Recuo para Aninhamento

A indentação é usada para definir estruturas ou hierarquias aninhadas. O YAML exige o uso consistente de 2 espaços para cada nível de indentação, e o uso de tabulações não é permitido.

Serviços:

web:

construir: .

portas:

\- "8080:80"

•A chave de serviços contém um dicionário aninhado onde web

é outra chave com sua própria estrutura aninhada.

• O recuo define a relação entre o elemento pai e o elemento pai.

(serviços) e chaves filhas (web).

3\. Listas

As listas YAML são representadas por um hífen (-) seguido de um espaço para cada item. As listas podem existir independentemente ou como parte de uma estrutura aninhada.

Serviços:

\- web

\- db

Ou em uma estrutura aninhada:

portas:

\- "8080:80"

\- "443:443"

• O hífen (-) indica itens individuais em uma lista. • Para listas aninhadas, o recuo posiciona a lista dentro de um espaço delimitado.

chave específica.

4\. Escalares (Cadeias de caracteres, Números, Booleanos) Escalares

são os tipos de dados básicos em YAML, incluindo

Cadeias de caracteres, números e valores booleanos.

Cadeias de caracteres: Cadeias de caracteres podem ser escritas com ou sem aspas. Use aspas se a cadeia de caracteres contiver caracteres especiais ou símbolos YAML reservados (por exemplo, : ou #).

string_key: "Olá Mundo"

Números: Os números são escritos sem aspas.

chave_número: 42

Valores booleanos: Os valores booleanos são escritos como verdadeiro ou falso (sem distinção entre maiúsculas e minúsculas).

é_ativo: verdadeiro

5\. Comentários

O YAML suporta comentários que começam com o símbolo #. Os comentários são ignorados pelo analisador YAML.

versão: "3.8"

O YAML oferece duas maneiras de lidar com strings de várias linhas: Bloco Literal ( | ): Preserva as quebras de linha exatamente como aparecem.

Descrição: |

Esta é uma string com várias linhas.

Mantém as quebras de linha.

Descrição: >

Será condensado em um único parágrafo.

• Use o caractere de bloco literal (|) quando a formatação ou as quebras de linha forem importantes.

(por exemplo, trechos de código).

• Use o bloco dobrado (>) para texto simples que pode ser lido como um

parágrafo único.

O YAML representa valores nulos explicitamente usando `null` ou implicitamente omitindo o valor.

chave_sem_valor:

• `key_with_null`: Especifica explicitamente um valor nulo. • `key_without_value`: Atribui implicitamente um valor nulo.

Deixando o valor em branco.

• Recuo inconsistente: Sempre use espaços (e não tabulações) para

recuo.

• Dois pontos em strings: Use aspas para strings que contenham dois pontos:

2\. docker-compose up

Inicia os contêineres para os serviços definidos no arquivo docker-compose.yml, criando-os se necessário.

Sintaxe do comando:

docker-compose up [opções]

Opções comuns:

Exemplo:

Inicia todos os serviços em segundo plano e os mantém em execução. Gerenciamento de serviços e contêineres.

Exibe o status dos contêineres em execução no projeto.

Sintaxe do comando:

Exemplo:

Explicação:

Lista todos os contêineres, incluindo os que estão parados. Mostra os nomes, IDs, portas e status dos contêineres.

Explicação:

Inicia o serviço web caso ele tenha sido interrompido anteriormente.

Os comandos acima executam um contêiner único para o serviço web definido em um arquivo docker-compose.yml e executam o comando `python manage.py makemigrations` ou `python manage.py migrate` dentro dele.

Analisando o comando:

• docker-compose run: Inicia um novo contêiner para um serviço definido

no arquivo docker-compose.yml sem afetar os contêineres em execução.

•--rm: Remove automaticamente o contêiner assim que ele sair, garantindo que ele

• web: O nome do serviço (normalmente definido em docker-

compose.yml ).

• python manage.py migrate: O comando executado dentro de

o contêiner, comumente usado em aplicações Django para aplicar

migrações de banco de dados.

O que é a opção --rm?

A opção `--rm` garante que o contêiner temporário criado pelo comando `docker-compose run` seja excluído automaticamente após a conclusão da execução. Isso evita o acúmulo desnecessário de contêineres parados.

Principais benefícios de --rm:

• Mantém o sistema limpo, evitando o acúmulo de recipientes obstruídos. • Reduz a limpeza manual, pois o recipiente é automaticamente removido.

removido.

• Útil para comandos únicos em que você não precisa guardar o

contêiner após a execução.

Diferenças entre docker-compose run e docker exec

Quando usar qual?

• Use `docker-compose run` quando precisar criar um ambiente separado.

Contêiner temporário para executar um comando.

• Use o comando `docker exec` quando quiser executar um comando dentro de um diretório.

Contêiner já em execução.

Para migrações de banco de dados como no comando fornecido, o comando `docker-compose run --rm` é preferível, pois garante que uma nova instância do serviço seja executada com as configurações de ambiente mais recentes, sem interferir no aplicativo em execução.

Etapa final: Limpeza

Agora que você concluiu a configuração do ambiente de desenvolvimento local, é importante limpá-lo, interrompendo e removendo os contêineres, volumes e imagens criados durante o processo de desenvolvimento. Isso ajuda a manter o sistema organizado e evita o consumo desnecessário de recursos. Para fazer isso, execute o seguinte comando:

docker-compose down --volumes --rmi all

Este comando irá interromper os contêineres, remover os volumes associados (que armazenam dados de banco de dados e outros arquivos persistentes) e excluir as imagens criadas durante o desenvolvimento. É uma boa prática realizar uma limpeza após cada sessão, especialmente quando se trabalha em um ambiente conteinerizado.

Quais são os principais riscos do uso do Docker?

Uma grande preocupação são os contêineres executados com privilégios de root. Se um invasor obtiver acesso a um contêiner em execução como root, ele poderá escalar seus privilégios e obter acesso ao sistema host. Os contêineres também podem compartilhar recursos, como redes ou volumes, o que pode criar vulnerabilidades se não forem devidamente isolados. Imagens Docker desatualizadas ou não confiáveis também representam uma ameaça; imagens comprometidas ou inseguras podem introduzir malware ou vulnerabilidades em seu ambiente.


Como minimizar os riscos

Para minimizar esses riscos, restrinja os privilégios dos contêineres e evite executá-los como root sempre que possível. Atribua usuários e grupos específicos aos contêineres e utilize ferramentas como Docker Secrets ou variáveis de ambiente para um gerenciamento seguro de senhas. Analise regularmente as imagens do Docker em busca de vulnerabilidades e utilize apenas fontes confiáveis, como repositórios oficiais. Além disso, proteja sua rede de contêineres isolando-os em redes privadas, aplicando regras de firewall rigorosas e garantindo que a comunicação ocorra apenas entre os serviços necessários.

Explorando o Kubernetes para Docker

Usuários

O Kubernetes é amplamente reconhecido como o padrão da indústria para o gerenciamento de aplicações conteinerizadas em grande escala. Ele oferece recursos avançados de orquestração, incluindo escalonamento automático, autorrecuperação e sofisticadas capacidades de rede. Enquanto o Docker Swarm é adequado para ambientes menores, o Kubernetes foi projetado para lidar com as demandas de sistemas grandes e complexos.

Quais são as principais diferenças entre

Docker e Kubernetes?

Embora Docker e Kubernetes sejam frequentemente usados em conjunto, eles têm funções diferentes. O Docker se concentra na conteinerização: empacotar aplicativos e suas dependências em contêineres isolados. O Kubernetes, por outro lado, é um poderoso orquestrador que gerencia contêineres em vários nós. O Kubernetes se destaca em áreas como agendamento de recursos, balanceamento de carga e escalonamento automático, que são essenciais para o gerenciamento de aplicações dinâmicas e de grande escala.

O Kubernetes suporta vários ambientes de execução de contêineres, incluindo o Docker, mas seus recursos avançados de orquestração — como recuperação automatizada, atualizações contínuas e gerenciamento granular de recursos — o diferenciam do Docker Swarm. Para aplicações complexas com múltiplos serviços, o Kubernetes oferece a flexibilidade e o controle necessários para garantir alta disponibilidade, escalabilidade e utilização eficiente de recursos.

Por que o Kubernetes é ótimo para grandes empresas?

Manuseio de contêineres em balança

O Kubernetes se destaca no gerenciamento de aplicações que exigem alta disponibilidade e tolerância a falhas. Ele permite o escalonamento automático com base na demanda de tráfego em tempo real e possui ferramentas robustas para o gerenciamento de ambientes complexos com múltiplos serviços. Em aplicações de grande escala, como uma plataforma global de e-commerce ou uma arquitetura nativa da nuvem, o Kubernetes pode lidar com eficiência com centenas ou milhares de contêineres em diversas regiões.

Em contrapartida, o Docker Swarm, embora mais simples, carece de parte do controle preciso oferecido pelo Kubernetes, especialmente para grandes implantações. Como resultado, o Kubernetes tornou-se a solução preferida para empresas e organizações com cargas de trabalho conteinerizadas complexas que precisam de escalabilidade dinâmica.

Docker e WebAssembly

(WASM)

WebAssembly (WASM) é uma tecnologia emergente que permite a execução de código de alto desempenho em navegadores web. Ela permite que desenvolvedores executem aplicações computacionalmente exigentes diretamente no navegador, com velocidades quase nativas, sem depender do JavaScript tradicional. O Docker, conhecido por seus recursos de conteinerização, pode fornecer um ambiente de execução isolado e consistente para a implantação de aplicações WASM.

Por que Docker e WASM podem funcionar bem juntos?

Junto

Embora o Docker e o WASM sejam poderosos individualmente, eles podem funcionar bem juntos, aproveitando a conteinerização do Docker para implantar aplicações WASM em diferentes plataformas. O alto desempenho do WASM, combinado com a portabilidade do Docker, significa que você pode executar aplicações com uso intensivo de computação em diversos ambientes sem se preocupar com diferenças de configuração. Os contêineres Docker empacotam a aplicação e suas dependências, garantindo que os módulos WASM sejam executados de forma consistente e eficiente em diferentes sistemas.

É importante notar que o Docker não executa código WASM diretamente dentro dos contêineres. Em vez disso, o Docker serve como ambiente para um runtime WASM (como o Wasmer ou o Wasmtime), que lida com a execução do código WASM. Essa distinção é crucial para entender como essas tecnologias podem funcionar juntas.

Possíveis casos de uso para Docker e

WASM

A integração do Docker com o WASM ainda está em fase inicial, mas o potencial para melhorar o desempenho das aplicações já é evidente. Por exemplo, os desenvolvedores podem usar o WASM para executar tarefas computacionalmente intensivas, como processamento de imagens, enquanto o Docker garante que o ambiente de execução do WASM permaneça consistente em diversas plataformas. Embora essa integração ainda não seja amplamente difundida, a combinação da velocidade do WASM com a portabilidade do Docker representa uma promessa significativa para aplicações que exigem muitos recursos.
