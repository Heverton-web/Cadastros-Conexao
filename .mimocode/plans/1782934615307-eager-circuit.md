# Plano: Skill `documentar-modulo`

## Objetivo

Criar a skill `.agents/skills/documentar-modulo/SKILL.md` que, ao ser acionada com `/doc <nome_modulo>`, analisa o código-fonte do módulo e gera uma documentação completa em Markdown em `docs-projeto/modulos/`.

---

## 1. Estrutura da Skill

**Arquivo:** `.agents/skills/documentar-modulo/SKILL.md`

**Trigger:** `/doc <nome_modulo>`

**Destino:** `docs-projeto/modulos/<modulo>.md`

**Formato:** Seguir o padrão do projeto (YAML frontmatter + seções numeradas)

---

## 2. Seções da Documentação Gerada

A skill vai extrair automaticamente do código-fonte:

### 2.1 Cabeçalho
- Nome do módulo (`module.ts` → `nome`)
- Descrição (`module.ts` → `descricao`)
- Ícone (do Lucide)
- Key do módulo

### 2.2 Core do Módulo
- **O que o usuário realiza** — inferido da descrição + rotas + permissões + componentes
- Exemplo: "O consultor planeja rotas de visita, executa as visitas e registra despesas no campo"

### 2.3 Estrutura do Módulo (Árvore)
- Listagem em estilo árvore de `src/features/<modulo>/`
- Inclui: module.ts, permissions.ts, index.ts, types.ts, components/, hooks/, services/, lib/, pages/, utils/
- Contagem de arquivos por diretório

### 2.4 Rotas
- Tabela com: Caminho da Rota | Componente | Descrição | Quem Acessa
- Extrair de `module.ts` → `routes` + arquivo de rota em `src/routes/`
- Mapear qual permissãoControla acesso (via nav items ou padrão)

### 2.5 Permissões
- Tabela com: Chave | Label | Descrição | Grupo
- Extrair de `permissions.ts`

### 2.6 Defaults por Papel
- Tabela com: Permissão | Cadastro | Consultor | Tecnologia | Suporte
- Extrair de `module.ts` → `registerPermissionDefaults()`

### 2.7 Navegação (Nav Items)
- Itens registrados na sidebar
- Quais permissões controlam visibilidade
- Ordem de exibição

### 2.8 Eventos / Webhooks
- Tabela com: Chave | Label | Descrição | Tipo
- Extrair de `module.ts` → `events`

### 2.9 Flags de Funcionalidade
- hasDesignConfig, hasCredentialScopes, hasLaboratorio, hasFormulario, hasCustomActions, hasApiConnectors
- Routes de config/design associadas

### 2.10 Dependências entre Módulos
- Se o módulo importa de outros features (análise de imports)
- Tabelas Supabase referenciadas nos services

---

## 3. Algoritmo da Skill

```
PASSO 1: Validar que o módulo existe
  - Verificar se src/features/<modulo>/ existe
  - Se não existe → sugerir módulos disponíveis

PASSO 2: Ler module.ts
  - Extrair: nome, descricao, icon, routes, permissions, ambientes,
    abas, events, hasDesignConfig, designRoute, etc.

PASSO 3: Ler permissions.ts
  - Extrair array de permissões com key, label, description, group

PASSO 4: Mapear diretório
  - Listar estrutura completa src/features/<modulo>/
  - Contar arquivos por diretório

PASSO 5: Analisar rotas
  - Para cada rota em module.ts → routes:
    - Buscar arquivo correspondente em src/routes/
    - Ler o componente importado
    - Determinar permissão de acesso (nav items)

PASSO 6: Extrair nav items
  - Ler chamadas registerNavItem() em module.ts
  - Extrair: id, label, to, permissionCheck, order

PASSO 7: Extrair permission defaults
  - Ler registerPermissionDefaults() em module.ts
  - Montar tabela por papel

PASSO 8: Analisar imports
  - Verificar imports de outros módulos
  - Verificar imports de services/ (tabelas Supabase)

PASSO 9: Gerar documento Markdown
  - Montar todas as seções
  - Salvar em docs-projeto/modulos/<modulo>.md

PASSO 10: Retornar resumo
  - Caminho do arquivo gerado
  - Estatísticas (nº de rotas, permissões, componentes)
```
