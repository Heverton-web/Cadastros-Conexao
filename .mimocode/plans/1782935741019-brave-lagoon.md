# Análise das Anotações "->" — Cadastros

## Anotação 1 (linha 65): `gerenciar_credenciais` → DELETAR

**Contexto:** Permissão definida em `cadastros/permissions.ts:81`. Controla acesso à página `/credenciais` via `credenciais.tsx:52` e visibilidade do nav item via `useNavItems.ts:50`.

**Veredicto:** Concordo. A página de credenciais é funcionalidade transversal, não exclusiva do módulo cadastros. Manter essa permissão dentro de cadastros cria acoplamento indevido.

**Plano:** Migrar para módulo `credenciais` (ou `empresas`). Não deletar direto — precisa de migração antes.

---

## Anotação 2 (linha 66): `gerenciar_credenciais_admin` → DELETAR

**Contexto:** Definida em `cadastros/permissions.ts:87`. Controla botões de criar/editar/deletar na página de credenciais (`credenciais.tsx:53`).

**Veredicto:** Mesma lógica da anterior. Migrar junto.

**Plano:** Migrar para módulo `credenciais`.

---

## Anotação 3 (linha 68): `gerenciar_config` → DELETAR

**Contexto:** Definida em `cadastros/permissions.ts:99`. Controla nav item "Config" apontando para `/global/acoes` via `useNavItems.ts:53`. Todos os 4 ambientes têm defaults `false` — só Super Admin tem acesso.

**Veredicto:** Concordo. Essa permissão controla acesso a uma rota global (`/global/acoes`), não a funcionalidade interna do módulo cadastros.

**Plano:** Migrar para módulo `empresas` (já que `/empresa/acoes` é a rota de config).

---

## Anotação 4 (linha 74): Defaults por Papel → "não precisamos"

**Contexto:** `registerPermissionDefaults` em `module.ts:166-243` define valores iniciais por ambiente.

**Análise:** Os defaults funcionam como fallback na criação de novas credenciais. Sem eles, credenciais novas teriam permissões `undefined`. Porém, na prática o Super Admin define tudo manualmente. A tabela na doc é volumosa e redundante com o código.

**Veredicto:** Concordo parcialmente. Manter `registerPermissionDefaults` no código (é necessário), mas simplificar a documentação — trocar a tabela por uma nota curta.

**Plano:** Substituir tabela por: "Defaults configurados em `module.ts → registerPermissionDefaults()`. Valores iniciais aplicados na criação de credenciais."

---

## Anotação 5 (linha 127): Feature `Credenciais` → DELETAR

**Contexto:** `hasCredentialScopes: true` em `module.ts:106`. No código, controla apenas texto explicativo na aba Credenciais do Super Admin (`global.modulos.$key.tsx:254-277`). O checkbox de escopo aparece independente do flag.

**Veredicto:** Concordo. O flag muda uma frase de "suporta escopo" para "não suporta". Funcionalidade real não depende dele. Pode ser removido.

**Plano:** Definir `hasCredentialScopes: false` ou remover da definição do módulo.

---

## Anotação 6 (linha 128): Feature `Laboratório` → "é SUPER ADMIN?"

**Contexto:** `hasLaboratorio: true` em `module.ts:107`. Adiciona aba "Laboratório" no detalhe do módulo (Super Admin). **Atualmente mostra "Em desenvolvimento"** — é placeholder puro.

**Análise:** O laboratório é uma feature do Super Admin para testar funcionalidades do módulo. Não é algo que o usuário final do módulo usa. O flag indica se o módulo terá área de testes.

**Veredicto:** O usuário tem razão — é funcionalidade admin. Pode ser removido se cadastros não terá laboratório próprio.

**Plano:** Definir `hasLaboratorio: false`.

## Anotação 7 (linha 129): Feature `Formulário` → "pode explicar?"

**Contexto:** `hasFormulario: true` em `module.ts:108`. Adiciona aba "Formulários" no Super Admin. **Placeholder** — mostra "Em desenvolvimento".

**Estado atual:** Não implementado — é um stub.

**Plano:** Manter `true` se haverá config de formulários. Senão, definir `false`.

## Anotação 10 (linha 147): Dependências → "tornam o módulo dependente?"

**Contexto:** Tabela lista `credenciais` e `empresas` como módulos relacionados.

**Análise de `credenciais`:** As permissões `gerenciar_credenciais` e `gerenciar_credenciais_admin` estão em `cadastros/permissions.ts` mas controlam acesso à página `/credenciais`. Isso é acoplamento indevido — funcionalidade de outro módulo vivendo dentro de cadastros. **Resolução:** migrar essas permissões (já coberto nas anotações 1 e 2).

**Análise de `empresas`:** `empresa_id` é padrão multi-tenant da plataforma. Toda tabela tem. **Não é dependência de módulo** — é infraestrutura.

**Veredicto:** Após migrar as permissões de credenciais, o módulo cadastros fica truly independente. O `empresa_id` não conta como dependência.

**Plano:** Migrar permissões → remover `credenciais` da tabela de dependências → manter apenas `empresas` como "padrão multi-tenant".
